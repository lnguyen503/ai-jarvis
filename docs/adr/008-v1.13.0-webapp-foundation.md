# ADR 008 — v1.13.0 Telegram Web App Foundation

**Status:** Proposed (Phase 1, v1.13.0)
**Date:** 2026-04-24
**Deciders:** Architect Agent (iteration)
**Extends:** ADR 001 (module boundaries, gateway/messaging split), ADR 005 (multi-user posture, security-sensitive design discipline), ADR 007 + 007-revisions (callback-data namespace conventions, distinguishing-catch pattern, R10 `editMessageReplyMarkup` adapter precedent). Does NOT supersede any prior ADR — this iteration introduces an entirely new external surface (HTTPS endpoint behind a cloudflared tunnel) that lives next to the existing health Express app, an HMAC verifier under `src/safety/`, a vanilla skeleton page under `public/webapp/`, and a `MessagingAdapter.sendWebAppButton` extension.

This ADR records the architectural decisions for the v1.13.0 platform-foundation iteration. Eight items ship together. **No real feature UI** is in scope — the skeleton page proves the platform end-to-end, after which v1.14.0+ chooses which feature gets the Web App first. Three downstream design decisions are explicitly deferred per the user's brief and are repeated in §15 (Boundaries) so reviewers cannot push for scope creep:

  1. Which feature gets the Web App first (`/organize` Kanban, `/plan` dashboard, etc.)
  2. Production HTTPS hosting decision (real domain vs cloudflared long-term)
  3. Web App scope cut (read-only vs read-write, which views)

Items shipping in v1.13.0:

  1. **`verifyTelegramInitData(initData, botToken, opts?)` HMAC verifier** — pure helper that validates the `WebAppInitData` HMAC per Telegram's Mini App spec, with `auth_date` freshness window and constant-time hash comparison.
  2. **Webapp Express server (sibling to health)** — new Express app on its own port (`config.webapp.port`) serving `/webapp/*` static files and `/api/webapp/*` JSON endpoints. Health Express stays untouched, 127.0.0.1-bound.
  3. **Skeleton webview page** — `public/webapp/index.html` + a tiny vanilla `app.js`; loads Telegram's `telegram-web-app.js` SDK from CDN, calls back to `/api/webapp/echo`, renders a "Hello \<first_name\>!" + verified `userId` + a "ping" button.
  4. **`MessagingAdapter.sendWebAppButton(chatId, text, label, url)`** — interface extension + Telegram impl using grammY's `InlineKeyboard().webApp(label, url)`. URL must be HTTPS; adapter throws a typed error on http: scheme.
  5. **`/webapp` slash command** — DM-only. Reads `config.webapp.publicUrl`; if empty, replies with a "setup needed" message pointing at the README; else fires `adapter.sendWebAppButton`.
  6. **Config schema extension** — `config.webapp` stanza: `publicUrl`, `port`, `staticDir`, `initDataMaxAgeSeconds`, plus rate-limit knobs (decision 3) and CSP toggle (decision 7).
  7. **README cloudflared docs** — a "Telegram Web App (v1.13.0)" section with cloudflared install link, the quick-start command, the rotating-trycloudflare-URL note, and a one-paragraph placeholder for the real-domain migration path (intentionally light — the decision is deferred to v1.14.0+).
  8. **Tests** — unit tests for the verifier (vector-driven, deterministic), integration tests for `/api/webapp/echo` (real Express server, real verifier), unit tests for the `/webapp` command + `sendWebAppButton` adapter validation.

Scope is design only — no code is in scope here. Developer agents in Phase 2 implement the decisions below without deviating; deviations require an ADR 008 addendum.

**Non-negotiables set by the iteration brief (not up for debate in this ADR):**

  - **No real feature UI.** Skeleton page only. The `Telegram.WebApp.sendData` ping button writes a single `web_app_data`-typed log entry server-side and acks; it does not drive any tool, mutate any state, or extend any existing surface.
  - **Cloudflared as the dev HTTPS terminator.** No real domain. No nginx/Caddy reverse proxy. No production hosting decision. The user runs `cloudflared tunnel --url http://localhost:<port>` against the new Express port and pastes the resulting `*.trycloudflare.com` URL into `config.webapp.publicUrl`.
  - **Vanilla HTML/JS only.** No Vite, no React, no bundler, no npm dep additions for the page. The Telegram WebApp SDK loads from `https://telegram.org/js/telegram-web-app.js` (the only CDN dependency, which is mandatory per Telegram's guide).
  - **Web App is OFF by default.** `config.webapp.publicUrl` defaults to `''`. With no URL, the Express server still boots (so health-check style "is the surface alive" can be probed in dev), but `/webapp` command short-circuits to the setup message and `sendWebAppButton` is never wired into a flow.
  - **No new npm dependencies.** `express` is already in `package.json`. `node:crypto` covers HMAC. The Telegram WebApp SDK loads in the BROWSER from telegram.org — not via npm.
  - **No changes to existing surfaces.** `/plan`, `/debate`, `/organize`, `/scheduled`, `/audit`, `/research` — untouched. No migration of `/plan` to ProgressPanel (orthogonal). No changes to allowlist middleware. No changes to the bot's command list. ARCHITECTURE.md update is a Phase 5 Docs deliverable.

**Baseline (verified against working tree at 2026-04-24):**
  - 1582/1582 tests passing; eslint clean; tsc clean.
  - pm2 running v1.12.1 (PID 50996). HEAD = `e87eb31`.
  - `package.json` already has `express ^4.19.2`. No npm install required for the Express layer.
  - `src/gateway/health.ts` exists and binds 127.0.0.1 only — verified line-by-line. Body shape `{ok, uptimeSec, version}`.
  - `MessagingAdapter` interface (in `src/messaging/adapter.ts`) currently has `sendMessage`, `editMessageText`, `sendDocument`, `sendPhoto`, `sendVoice`, `sendChatAction`, `resolveDmChatId`, `editMessageReplyMarkup`. No web-app-button method today.
  - `bot.on('callback_query:data', ...)` router branches on `plan.*`, `rec:*`, `debate.*`. No webapp-related callback routing needed in v1.13.0 (the skeleton uses `Telegram.WebApp.sendData` which lands as a `message:web_app_data` event, not a callback query — see decision 12).
  - `config.webapp` is NOT a config namespace today — entirely new in this iteration.
  - `public/` directory does NOT exist today; will be created at the project root (next to `src/`, `tests/`, `docs/`).

## Item 1 — `verifyTelegramInitData` HMAC helper

### 1. Express layout — sibling Express app on its own port

**Status:** Accepted (sibling).

**Context (rigor ask 1).** Two candidate paths for hosting the webapp routes:

  - **(a)** Extend the existing `createHealthServer` Express app — add `/webapp/*` + `/api/webapp/*` routes to it. The health Express today binds `127.0.0.1`. Cloudflared cannot reach 127.0.0.1 from outside the process (it CAN reach localhost addresses, but adding webapp routes to a 127.0.0.1-bound app means either (i) re-binding the whole app to `0.0.0.0` so cloudflared can connect, which exposes `/health` externally too, or (ii) binding cloudflared to 127.0.0.1 explicitly via `cloudflared tunnel --url http://127.0.0.1:7878` — which DOES work since cloudflared runs in the same host network namespace). Path (a)(ii) is technically valid and saves one process.
  - **(b)** New sibling Express app on a different port (e.g. default `7879`). Health stays 127.0.0.1, body unchanged. Cloudflared targets the new port. Two processes; one tunnel; one purpose per server.

**Decision.** (b), sibling Express app. Justifications:

  1. **Surface-isolation invariant.** `/health` is intentionally a private operator probe, not a public endpoint. Even though its body is benign today (`{ok, uptimeSec, version}`), conflating "private operator endpoint" and "public HTTPS endpoint reachable via cloudflared" on one Express app means every future addition to either side has to re-prove the boundary. Keeping them physically separate makes the threat model trivially auditable: anything bound to `127.0.0.1` is private; anything bound to `0.0.0.0` (or whatever cloudflared targets) is public-by-construction.
  2. **Different middleware stacks.** The webapp Express app needs CSP headers (decision 7), per-IP rate limiting (decision 3), `Authorization: tma` header parsing (decision 5). The health app needs none of that. Adding it conditionally per-route is a path to mistakes; two apps lets each one configure its own pipeline once.
  3. **Independent lifecycle.** The webapp server can be skipped at boot when `config.webapp.publicUrl === ''` (decision 14 rollback posture). Skipping a sibling app is a one-line conditional; skipping `/webapp/*` routes inside the health app would require route-level gating that's harder to reason about during an incident.
  4. **No coupling to operator-tooling assumptions.** `/health`'s 127.0.0.1 binding is a Tier 1 sandbox invariant per `docs/BUILD_SANDBOX.md` (the operator probes from the host, not from outside). Re-binding it to satisfy cloudflared would weaken that invariant for the lifetime of the deployment.

**Module shape:**

```typescript
// src/gateway/webappServer.ts (NEW)
import express from 'express';
import path from 'node:path';
import type { Server } from 'http';
import type { AppConfig } from '../config/index.js';

export interface WebappServer {
  /** Boot the Express app. Resolves once listening. */
  start(): Promise<void>;
  /** Stop the server. Resolves once closed. Idempotent. */
  stop(): Promise<void>;
}

export interface WebappServerDeps {
  config: AppConfig;
  /** Bot token for HMAC verification. NOT logged. */
  botToken: string;
}

export function createWebappServer(deps: WebappServerDeps): WebappServer;
```

**Bind address:** the webapp Express app binds `127.0.0.1` (loopback) by design — cloudflared runs in the same host and connects via loopback. This is the SAME defense as the health server (no remote process can connect directly), but the public reachability comes from the cloudflared tunnel termination, not from the Express bind. Cloudflared TLS-terminates in front; the Express app sees plain HTTP from `127.0.0.1`. Documenting explicitly so reviewers don't read "0.0.0.0" into the spec.

**Default port:** `config.webapp.port` defaults to `7879`. The health port is `7878`; staying adjacent makes operator memory simple. Range clamped to `[1024, 65535]` (zod-level). Conflicts with the health port → fail fast at config load (decision 6).

**Boot/shutdown order:** see decision 8.

**Consequences.**

  - **Positive.** Single binding per Express app; no shared-route surprises. Health endpoint's 127.0.0.1-only invariant preserved.
  - **Positive.** Skipping the webapp server on `publicUrl === ''` is a clean conditional in `src/index.ts`, no per-route gates.
  - **Positive.** Cloudflared, ngrok, or any future tunnel can be swapped without touching the Express app — the contract is "an HTTPS terminator that proxies to `127.0.0.1:<webapp.port>`."
  - **Negative.** Two Express apps in the process. Memory cost is negligible (~5 MB total per app at baseline). Two ports to remember. Operationally minor.

### 2. Package layout — `src/safety/webappAuth.ts` for the verifier

**Status:** Accepted (`src/safety/`).

**Context (rigor ask 2).** Three candidate homes for `verifyTelegramInitData`:

  - **`src/webapp/auth.ts`** — a brand-new top-level module under `src/`. Clean separation; mirrors `src/organize/`, `src/debate/`, etc. But the verifier is ~120 LOC of pure HMAC logic with no domain types, no state, no other functions to live alongside. A module of one tiny file is overweight infrastructure for what's actually a small piece of safety code.
  - **`src/gateway/webapp/auth.ts`** — under the gateway tree, alongside `webappServer.ts`. Keeps "all webapp infrastructure" colocated. But the verifier has zero gateway-layer dependencies (no grammY, no `MessagingAdapter`, no callback routing) — it's a pure crypto check. Putting it under `src/gateway/` would cross-contaminate the gateway layer's identity (gateway = "platform-specific bot wiring + Express servers + callback routing").
  - **`src/safety/webappAuth.ts`** — alongside `scrubber.ts`, `groupScrub.ts`, and the existing `src/safety/index.ts` factory. The verifier's role is "validate that this externally-supplied blob is authentic and fresh" — a security primitive, exactly the kind of thing that lives in `src/safety/`. Imports are limited to `node:crypto` and standard library.

**Decision.** `src/safety/webappAuth.ts`.

Justifications:

  1. **The verifier is a security primitive, not a webapp concern.** Its job is "given a blob and a key, return whether the blob is authentically signed and recent." That's an authentication check in the same family as the scrubber's "redact credentials" primitive — shared `src/safety/` lineage.
  2. **Import direction is clean.** `src/safety/webappAuth.ts` imports only `node:crypto`. The webapp Express server (`src/gateway/webappServer.ts`) imports from `src/safety/webappAuth.ts`. Same direction as `src/gateway/index.ts` importing `src/safety/index.ts` today. No circular risk.
  3. **`src/safety/index.ts` already aggregates the safety surface** — extending it to re-export `verifyTelegramInitData` is a one-line change and gives the rest of the codebase a single import path: `import { verifyTelegramInitData } from '../safety/index.js'`.
  4. **No new top-level module.** `src/webapp/` would be one file deep, which fights ADR 001's module-boundary discipline. Adding it just to host one verifier creates a precedent for "every external surface gets a top-level module" that we'd regret as the surface grows.

**Module location: `src/safety/webappAuth.ts`. Re-exported from `src/safety/index.ts`.**

```typescript
// src/safety/webappAuth.ts
import crypto from 'node:crypto';

export interface VerifiedInitData {
  /** Raw user object as Telegram delivered it (already a JSON-parsed record). */
  user: {
    id: number;
    first_name: string;
    last_name?: string;
    username?: string;
    language_code?: string;
    is_premium?: boolean;
    photo_url?: string;
  };
  /** Unix epoch seconds (Telegram-supplied). */
  authDate: number;
  /** Optional chat the Mini App was launched from. */
  chat?: {
    id: number;
    type: 'private' | 'group' | 'supergroup' | 'channel';
    title?: string;
    username?: string;
    photo_url?: string;
  };
  /** start_param from the launch URL, if any. */
  startParam?: string;
  /** chat_type from the launch URL ('sender' | 'private' | 'group' | 'supergroup' | 'channel'). */
  chatType?: string;
  /** chat_instance — opaque per-chat identifier. */
  chatInstance?: string;
}

export type VerifyInitDataResult =
  | { ok: true; data: VerifiedInitData }
  | { ok: false; reason: 'malformed' | 'bad-hash' | 'stale' | 'no-user' };

export interface VerifyInitDataOptions {
  /**
   * Reject when (now - auth_date) > maxAgeSeconds. Default 86400 (24h).
   * Tightening to ~3600 in dev surfaces clock-skew issues fast; production
   * stays at 24h to tolerate a user opening a saved Web App link.
   */
  maxAgeSeconds?: number;
  /**
   * Reject when (auth_date - now) > maxFutureSkewSeconds. Default 60s.
   * NTP skew is sub-second on modern systems; >60s forward implies forgery.
   */
  maxFutureSkewSeconds?: number;
  /** Override Date.now() for deterministic testing. Receives no args, returns ms epoch. */
  now?: () => number;
}

export function verifyTelegramInitData(
  initData: string,
  botToken: string,
  opts?: VerifyInitDataOptions,
): VerifyInitDataResult;
```

**Algorithm** (verbatim per https://core.telegram.org/bots/webapps#validating-data-received-via-the-mini-app):

  1. `params = new URLSearchParams(initData)`. If parse throws (the input is not a valid query string), return `{ok: false, reason: 'malformed'}`.
  2. Extract `hash = params.get('hash')`. If absent or empty, `{ok: false, reason: 'malformed'}`. Remove the `hash` key from the param set (work on a copy — do NOT mutate the input string).
  3. Build the data-check-string: take all remaining `(key, value)` pairs, sort alphabetically by key, join as `key=value\nkey=value\n...` (no trailing newline). The values are URL-decoded by `URLSearchParams.get`/`URLSearchParams.entries` already; that's the spec-correct form (Telegram signs the decoded values, not the URL-encoded ones).
  4. `secretKey = HMAC-SHA256(botToken, key='WebAppData')` — that is, HMAC with key=`'WebAppData'` (literal ASCII string) and message=the bot token. Output is 32 raw bytes.
  5. `expectedHashHex = HMAC-SHA256(dataCheckString, key=secretKey).digest('hex')` — output is 64 hex chars.
  6. Compare in constant time: `crypto.timingSafeEqual(Buffer.from(hash, 'hex'), Buffer.from(expectedHashHex, 'hex'))`. If buffers differ in length (the supplied `hash` was malformed), wrap in try/catch — `timingSafeEqual` throws on length mismatch, so a malformed hex string surfaces as `{ok: false, reason: 'malformed'}` (NOT `'bad-hash'` — a wrong-length hash is a parse failure, not a signature mismatch). If the lengths match but bytes differ, `{ok: false, reason: 'bad-hash'}`.
  7. Parse `auth_date = parseInt(params.get('auth_date') ?? '', 10)`. If `Number.isNaN(authDate)`, `{ok: false, reason: 'malformed'}`.
  8. Compute `nowSec = Math.floor((opts?.now?.() ?? Date.now()) / 1000)`. Reject:
     - `nowSec - authDate > maxAgeSeconds` → `{ok: false, reason: 'stale'}`.
     - `authDate - nowSec > maxFutureSkewSeconds` → `{ok: false, reason: 'stale'}` (decision 4 — future-skew rejection collapses into the `'stale'` reason since both indicate "auth_date does not match real-world time").
  9. Parse `user`: `JSON.parse(params.get('user') ?? '')`. Wrap in try/catch — invalid JSON → `{ok: false, reason: 'no-user'}`. Validate `user.id: number` and `user.first_name: string` exist. Missing → `{ok: false, reason: 'no-user'}`.
  10. Parse optional fields (`chat`, `start_param`, `chat_type`, `chat_instance`) opportunistically; failures are tolerated (set the field to `undefined`). These are decoration, not authentication.
  11. Return `{ok: true, data: {user, authDate, chat, startParam, chatType, chatInstance}}`.

**Constraints on the bot token at the call site:** the function takes the raw `botToken` as a string. Callers MUST NOT log it. The function itself doesn't log anything (decision 9 — verifier is pure; callers do the logging).

**File length budget:** 80–120 LOC for the function + types. Tests sit alongside (decision 22). Anti-Slop §5 (Separation of Concerns) — no other safety primitive belongs in this file.

**Consequences.**

  - **Positive.** Pure function; no I/O; trivial to test with deterministic vectors (decision 22).
  - **Positive.** `node:crypto` only — no new npm dep.
  - **Positive.** Constant-time comparison defends against timing oracles (Anti-Slop §9 — security non-negotiable).
  - **Positive.** Future migration to a different platform's WebView auth (e.g. Slack's interactive blocks) reuses the same package layout: another `src/safety/<platform>Auth.ts` next to this one.
  - **Negative.** None material.

## Item 2 — Express server (static + API routes)

### 3. Per-IP rate limiting on `/api/webapp/echo`

**Status:** Accepted — ship NOW (in-memory, ~30 LOC).

**Context (rigor ask 3).** Devil's Advocate would flag a public HTTPS endpoint with no rate limiting as a 1000 req/sec attack surface — a single attacker hammering `/api/webapp/echo` with malformed initData drives ~30µs of HMAC work per request, which is cheap per-request but amplifies into CPU pressure across thousands of requests/sec. Two paths:

  - **Defer to v1.13.x.** Single-user deployment doesn't need it day one. Wait until the surface actually attracts traffic.
  - **Ship in v1.13.0.** Cheap to write, cheaper to ship before public exposure than to retrofit after a leak or DoS incident.

**Decision.** Ship in v1.13.0. A public HTTPS endpoint is a different threat surface than a Telegram-mediated bot endpoint — Telegram's per-bot rate limit (30 messages/sec) shaped the bot's threat model; cloudflared puts ZERO native rate limit in front of `/api/webapp/echo` (it's a transparent tunnel). Adding rate limiting now is the cheapest moment.

**Shape:**

```typescript
// src/gateway/webappRateLimit.ts (NEW; ~30 LOC)
export interface RateLimiter {
  /** True if request should proceed; false if it should be rejected with 429. */
  allow(key: string): boolean;
}

export function createRateLimiter(opts: {
  windowMs: number;       // sliding window
  maxRequests: number;    // per key per window
  maxKeys: number;        // LRU cap on the bucket map
}): RateLimiter;
```

Implementation: a `Map<key, number[]>` where the value is a sorted array of recent request timestamps. On each `allow(key)` call:
  1. Drop timestamps older than `now - windowMs`.
  2. If the remaining length is `>= maxRequests`, return `false`.
  3. Else push `now`, return `true`.
  4. LRU eviction: when the map's size exceeds `maxKeys`, drop the entry with the oldest most-recent timestamp.

**Key derivation:** the request's IP. Take `req.ip` from Express (which honors `X-Forwarded-For` ONLY when `app.set('trust proxy', ...)` is configured — see below). The webapp app sets `app.set('trust proxy', 'loopback')` so it trusts the cloudflared-injected `X-Forwarded-For` only when the upstream is loopback. Rationale: cloudflared on the same host injects the original client IP via `CF-Connecting-IP` and `X-Forwarded-For`; trusting loopback specifically (NOT a wildcard) means a public bind misconfiguration can't be tricked into trusting client-supplied headers.

**Why IP, not userId:** the rate limit fires BEFORE the verifier runs. We don't yet know who the user is. Per-IP is the right granularity for the layer.

**Defaults** (config, decision 6):
  - `windowMs: 60_000` (1 minute)
  - `maxRequests: 60` (60 requests per minute per IP)
  - `maxKeys: 10_000` (10k tracked IPs is ~80 KB at 8 bytes per timestamp × ~1 timestamp average; bounded)

These accommodate normal use (a single user opens the Web App, which fires 1 request per page load + maybe a sendData call) with massive headroom, while clipping scripted abuse at ~1 req/sec sustained.

**Response on rate-limit:** `429 Too Many Requests` with body `{error: 'Rate limit exceeded', retryAfterSeconds: <integer>}` and a `Retry-After: <seconds>` header. The retry-after is calculated as the time until the oldest in-window timestamp ages out (so a client that complies sees consistent unblock timing).

**Logged at warn level** (decision 9): `{component: 'webapp.server', event: 'rate-limit', ip, count}`. Don't log every rate-limited request — debounce so a single attacker doesn't flood our logs (decision 9 details the debounce).

**Consequences.**

  - **Positive.** Bounded CPU and log surface against a hostile public endpoint.
  - **Positive.** ~30 LOC of stdlib code; zero npm deps; trivial to test (inject `now()` like the verifier does).
  - **Negative.** In-memory state; pm2 restart resets the window. Acceptable for a fuse — an attacker who can wait for restart is already throttled relative to no-fuse.
  - **Negative.** Per-IP doesn't help if the attacker rotates IPs (botnet). Mitigated by `maxKeys` cap (LRU eviction prevents memory blowup), and the attack is uneconomic for a single-user deployment.

### 4. Future `auth_date` rejection

**Status:** Accepted — reject when auth_date is more than 60s in the future.

**Context (rigor ask 4).** The base spec rejects auth_date older than `maxAgeSeconds`. A symmetric question: should we also reject auth_date more than X seconds in the FUTURE? Three options:

  - **No future check.** Trust whatever Telegram sends. But Telegram's WebApp SDK in the browser stamps `auth_date` from the server side at Mini App init — so a far-future value implies a forgery (an attacker constructing a fake initData with `auth_date = now + 3_600` to pass our staleness check tomorrow).
  - **Tight bound (60s).** NTP-synced systems agree to sub-second precision. 60s is generous (handles a clock-drift incident) while clipping any forgery scenario where the attacker wants a long-lived ticket.
  - **Looser bound (5–10 minutes).** Tolerates clock-drift incidents but defeats the defense's intent.

**Decision.** Reject when `auth_date - now > maxFutureSkewSeconds` with default `60`. Configurable (decision 6) for ops who run on systems with known clock drift, but the default is tight.

The future-skew rejection collapses into the same `'stale'` reason as past-staleness — both indicate "the timestamp does not align with real-world time as we observe it." Callers don't need to distinguish; the audit log captures the actual `auth_date` value (decision 10) so an operator investigating can see whether it was old or new.

**Consequences.**

  - **Positive.** Defends against forged-future attacks for free (one comparison).
  - **Positive.** Makes the verifier's freshness window symmetric — easier to reason about.
  - **Negative.** A misconfigured server with a clock that is genuinely 2 minutes behind would reject all real Web App traffic. Mitigated by configurability + the default 60s is wider than any healthy system's drift.

## Item 2 — `Authorization: tma <initData>` header parsing

### 5. Authorization header — both forms supported, `tma` preferred

**Status:** Accepted (both forms; log when query-fallback is used).

**Context (rigor ask 5).** Telegram's Mini Apps spec recommends the `Authorization: tma <initData>` header for verifying initData on the server. Three options:

  - **(a) Strict header-only:** reject with 400 if no `Authorization: tma <...>` header. Cleanest; future-proofs against people copy-pasting the URL into a browser and hitting the API directly with `?initData=...`.
  - **(b) Permissive both:** accept the header OR `?initData=` query string. Most forgiving; matches what older guides show.
  - **(c) Both with deprecation:** accept both initially; log a warn whenever query-fallback is used; plan to drop the query path in a later release.

**Decision.** (c). Both supported in v1.13.0. Query-string usage emits `log.warn({component: 'webapp.server', event: 'init-data-via-query', ip}, 'initData via query — header preferred')`. The skeleton page uses the header form exclusively (decision 11) so production traffic is header-only; the query path exists for ad-hoc debugging (curl, Postman) and one-off integrations.

**Parsing:**

```
Authorization: tma <initData>     → use initData
Authorization: <anything else>    → fall through to query-string
?initData=<initData>              → use initData (with the warn above)
neither                           → 401 {error: 'No initData', reason: 'missing'}
```

The `tma ` prefix check is case-insensitive on the scheme (`Tma `, `TMA ` accepted), case-sensitive on the rest (the initData payload). The space after `tma` is mandatory; `tma<initData>` (no space) is treated as `<anything else>` and falls through.

**Why not strict:** in dev, an operator wanting to manually curl the endpoint with a captured initData should not have to hand-craft an Authorization header. The query-string path with a warn log keeps that ergonomic. Tightening to header-only is a v1.14.x decision once the surface has real traffic and we can measure whether anything legitimate is using the query form.

**Consequences.**

  - **Positive.** Matches Telegram's recommended pattern in the primary path.
  - **Positive.** Permissive enough for ad-hoc debugging and curl-based smoke tests.
  - **Negative.** Two parse paths to test (decision 22). Trivial.

### 6. Cache-Control headers for `/webapp/*` static files

**Status:** Accepted — `index.html` no-cache; future assets `max-age=86400`.

**Context.** `express.static` defaults to no `Cache-Control` header (the browser uses heuristics, which leads to surprising stale-page behavior in dev). For the skeleton page the only file is `index.html` (and a sibling `app.js`); future assets (CSS, fonts, images) will land here.

**Decision.** Two-tier policy:

  - `index.html` (and any other `.html`): `Cache-Control: no-cache, no-store, must-revalidate` and `Pragma: no-cache` and `Expires: 0`. Justification: the page's logic and content shape will change rapidly across v1.13.x → v1.14.x; a stale `index.html` cached on the user's phone for hours is a live debugging hazard.
  - All other files (`*.js`, `*.css`, `*.png`, `*.svg`, `*.woff2`, etc.): `Cache-Control: public, max-age=86400` (1 day). Future-proofed: when we add real CSS/JS bundles, they'll be served with versioned filenames (`app.v1-13-0.js`) so a 1-day cache is fine.

Implementation: a tiny middleware before `express.static`:

```typescript
app.use('/webapp', (req, res, next) => {
  if (req.path === '/' || req.path.endsWith('.html')) {
    res.set('Cache-Control', 'no-cache, no-store, must-revalidate');
    res.set('Pragma', 'no-cache');
    res.set('Expires', '0');
  } else {
    res.set('Cache-Control', 'public, max-age=86400');
  }
  next();
});
app.use('/webapp', express.static(path.resolve(config.webapp.staticDir)));
```

**Consequences.**

  - **Positive.** Page changes ship instantly; no "user has to clear cache" support burden.
  - **Positive.** Asset loading is fast on repeat visits.
  - **Negative.** None material.

### 7. CSP headers on the static page

**Status:** Accepted — strict CSP from v1.13.0 with the exact allowlist.

**Context.** The skeleton page (decision 11) has inline JS — building a vanilla page without inline boot-time wiring is awkward (the alternative is a separate `app.js` loaded via `<script src="...">`, which is fine but slightly less cohesive). Three options:

  - **No CSP.** Defer to v1.14.0 when there's actual UI. Cheapest now; weakest posture for a public HTTPS surface.
  - **Permissive CSP** with `'unsafe-inline'` for scripts and styles. Allows the boot code; weakens XSS defense.
  - **Strict CSP** with no inline scripts. Requires moving boot logic out of `index.html` into a sibling `app.js`. Strongest posture; tiny refactor cost.

**Decision.** Strict CSP. The skeleton page uses `<script src="/webapp/app.js"></script>` (no inline JS). Inline styles are minimal and acceptable under `'unsafe-inline'` for `style-src` only (Telegram's themeParams flow into CSS variables on `:root`, which inline styles set up).

CSP header on every `/webapp/*` HTML response:

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self' https://telegram.org;
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https://t.me;
  connect-src 'self';
  frame-ancestors https://web.telegram.org https://telegram.org;
  base-uri 'self';
  form-action 'self';
```

Rationale per directive:
  - `default-src 'self'`: no off-origin loads except those explicitly allowed.
  - `script-src 'self' https://telegram.org`: own JS + the WebApp SDK (`https://telegram.org/js/telegram-web-app.js` — pinned to telegram.org per Telegram's official integration guide).
  - `style-src 'self' 'unsafe-inline'`: minor inline styles for theme variables. No inline scripts.
  - `img-src 'self' data: https://t.me`: theme/photos. `data:` for SVG icons inlined as data URIs (future-proof; not used in v1.13.0).
  - `connect-src 'self'`: the page only fetches `/api/webapp/echo` from itself. No third-party telemetry.
  - `frame-ancestors`: only Telegram clients can embed us in an iframe. Defends against clickjacking. (`web.telegram.org` is the desktop/web client; native apps don't iframe us, so this header is ignored on mobile but harmless.)
  - `base-uri 'self'`: defends against `<base>` tag injection.
  - `form-action 'self'`: no `<form>` posts off-site.

**Toggle:** `config.webapp.csp.enabled` (default `true`). An operator debugging a CSP issue can flip to `false` to remove the header for one session — but the tests assert it's set in the default-config test path (decision 22).

**Consequences.**

  - **Positive.** Strong XSS posture from day one. Anti-Slop §9 (CSP headers configured).
  - **Positive.** No `'unsafe-inline'` on scripts — defends against any injected script payload that finds its way into the page.
  - **Negative.** Boot logic moves from inline `<script>` into `app.js`. ~10 LOC refactor. Trivial.

### 8. Boot order in `src/index.ts`

**Status:** Accepted.

**Context.** The current `src/index.ts` boot has 11 steps (config → logger → memory → safety → transcriber → MCP → tools → agent → gateway → scheduler → reminders/trash) and a `start()` sequence. The webapp Express server has to slot in somewhere. It needs:
  - `config.webapp.publicUrl` available (post-loadConfig).
  - `config.telegram.botToken` available (already at config load time).
  - To be UP before the Telegram polling fires the first `/webapp` command (otherwise a race could send a button to a dead server).
  - To shut down BEFORE memory closes (in case a final request is in-flight; though it's stateless, ordering matters for clean log output).

**Decision.** Insert a new step 10d (after trash evictor, before "11. Start everything"):

```
10d. v1.13.0 — Webapp Express server (skipped if config.webapp.publicUrl === '').
    const webappServer = config.webapp.publicUrl
      ? createWebappServer({ config: cfg, botToken: cfg.telegram.botToken })
      : null;
```

Start order in step 11:
```
scheduler.start();
await gateway.start();          // grammY polling begins
reminders.start();
trashEvictor.start();
if (webappServer) {
  await webappServer.start();   // webapp Express up
}
```

**Webapp starts AFTER `gateway.start()`.** Rationale: the `/webapp` command requires `gateway` to be up to deliver the button; the button targets the webapp URL; the webapp must be up by the time the user TAPS the button. The user taps the button after they receive it, which is at least one Telegram round-trip after `/webapp` runs — so `webappServer.start()` finishing AFTER `gateway.start()` is fine (the polling has begun, but no user message will have round-tripped before webapp is listening). Conversely, starting webapp BEFORE gateway means the public HTTPS endpoint is up before the bot is ready, which is harmless but provides no benefit.

Shutdown order (reverse, in the existing `shutdown` function):
```
trashEvictor.stop();
reminders.stop();
await gateway.stop();           // grammY stops polling
scheduler.stop();
if (webappServer) {
  await webappServer.stop();    // webapp Express closes
}
await mcpRegistry.close();
... browser shutdown ...
memory.close();
```

Webapp shuts down AFTER gateway stops polling (no new `/webapp` commands can arrive that would create new buttons), but BEFORE memory closes (in case a final in-flight request needs the bot token — which is ALREADY in memory, but defensive ordering for future logs/audit emission).

**Skip-when-disabled:** when `config.webapp.publicUrl === ''`, the entire webapp path is skipped: `createWebappServer` is not called, no port is bound, `start()`/`stop()` are no-ops. This means turning off the Web App is a one-line config edit + restart, no code change needed. Documented in decision 14 (rollback posture).

**Consequences.**

  - **Positive.** Clean fit into the existing 11-step pattern.
  - **Positive.** Disable-by-config produces a process that is identical to v1.12.x in every observable way.
  - **Negative.** One more step to remember. Trivial.

### 9. Logging discipline + audit emission for `/api/webapp/echo`

**Status:** Accepted — pino `child` per module; audit ONLY failures.

**Context (rigor asks 9 + 10).** Three new modules need component tags. The echo endpoint is a high-volume happy-path call (every page load fires one); auditing every success would balloon `audit_log` rows. Failures are interesting (someone trying spoofed initData, stale tickets, etc.).

**Decision.**

**Component tags per module** (used as the `component` label in pino `child`):

  - `src/safety/webappAuth.ts` → `component: 'safety.webappAuth'` (matches `src/safety/scrubber.ts` → `safety.scrubber` precedent — though the verifier itself doesn't log, callers logging on the verifier's behalf use this tag).
  - `src/gateway/webappServer.ts` → `component: 'gateway.webappServer'`.
  - `src/gateway/webappRateLimit.ts` → `component: 'gateway.webappRateLimit'` (or merged into `gateway.webappServer` since it's tiny — Phase 2 dev's call).
  - `src/commands/webapp.ts` → `component: 'commands.webapp'`.

All tags are lowercase, dotted, two-segment except where the module's own structure dictates three. Matches the existing `gateway`, `gateway.health`, `gateway.progressPanel`, `commands.audit` patterns.

**What gets logged at `info`:**
  - Webapp server start (port, bind addr, staticDir).
  - Webapp server stop.
  - `/api/webapp/echo` SUCCESSFUL verification ONLY at debug level (gated behind pino's debug threshold so prod stays quiet). Includes `userId, chatId, authDateAge` (the seconds-since-auth_date — useful for tuning maxAgeSeconds).

**What gets logged at `warn`:**
  - Echo request with no initData (`reason: 'missing'`).
  - Echo request with malformed initData (`reason: 'malformed'`).
  - Echo request with bad-hash (`reason: 'bad-hash'`) — interesting, possibly an attack. Include the ip and the userAgent header (truncated to 200 chars) to aid investigation.
  - Echo request with stale initData (`reason: 'stale'`). Include `authDateAge` so the operator can see how stale.
  - Echo request with no-user (`reason: 'no-user'`).
  - Rate-limit triggered (`event: 'rate-limit', ip, count`). DEBOUNCED: log only the FIRST rate-limit event per IP per window, not every rejected request — otherwise an attacker drives our log volume. Track per-IP `lastWarnLoggedAt`; emit warn only when `now - lastWarnLoggedAt > windowMs`. Subsequent rejections in the window are silent (the response is still 429).
  - initData supplied via query (`event: 'init-data-via-query'`) per decision 5.

**What gets logged at `error`:**
  - Webapp server failed to start (port conflict, etc.). Process exits — fatal at boot.
  - Echo handler threw (unexpected — verifier shouldn't throw on caller-supplied input). Includes stack.

**No logging of:**
  - The `initData` string itself (contains the user's hash + auth_date + user object — PII per decision 10).
  - The bot token (obviously).
  - The response body shape on success (no content; just metadata).

**Pino redaction:** the standard `src/logger/redact.ts` redaction list covers `botToken`, `authorization`, etc. — already configured project-wide. No new redaction patterns needed for v1.13.0 since we never log the sensitive fields by name.

**Audit emission for `/api/webapp/echo`:**

**Decision: audit FAILED verifications ONLY.**

  - **`webapp.auth_failure`** — NEW audit category. Emitted on any `{ok: false}` from the verifier, plus on rate-limit rejections. Detail shape:
    ```typescript
    {
      reason: 'malformed' | 'bad-hash' | 'stale' | 'no-user' | 'missing' | 'rate-limit',
      ip: string,                   // anonymized? — see decision 10 below
      userAgent?: string,           // truncated to 200 chars
      authDateAge?: number,         // seconds, only for 'stale'
    }
    ```
    Actor fields: `actor_user_id = null` (we couldn't authenticate), `actor_chat_id = null`, `session_id = null`. The audit row is anonymous.

  - **No `webapp.auth_success`.** A successful verification on every page load is high-volume and unremarkable — adding 10–100 rows/day per active user (most of which carry the same userId we already know about) for no operational signal. The pino `debug`-level log is enough for ad-hoc traceability; `audit_log` stays focused on interesting events.

  - **`webapp.session_start`** — POSSIBLY emitted ONCE per "first page load in a 24h window per user" (deduped against a recently-seen userId in the in-memory rate-limiter map). DEFERRED to v1.14.0 — for v1.13.0, session-tracking is YAGNI given there's no real Web App feature to attribute usage to.

The category name `webapp.auth_failure` follows ADR 005's audit-naming convention (`organize.nudge`, `debate.complete`, etc. — `<surface>.<event>` lowercase).

**Consequences.**

  - **Positive.** Failure-only audit keeps row count flat under nominal load.
  - **Positive.** Each component tag is greppable (`component: gateway.webappServer`).
  - **Positive.** Rate-limit log debounce defends against attacker-driven log flooding (Anti-Slop §3 — error handling without flooding).
  - **Negative.** Lose the ability to query "which users opened the Web App this week" from `audit_log` directly. Mitigated by the pino debug log path; if the question becomes operationally important, v1.14.0 adds `webapp.session_start`.

### 10. PII posture in audit + logs

**Status:** Accepted — IPs hashed, userAgent truncated; auth_date in audit rows.

**Context.** The audit row from decision 9 captures `ip` and `userAgent`. For multi-user deployments these are PII per GDPR/PIPEDA-style frameworks. Single-user deployment doesn't need it for compliance, but the design should not regress posture relative to the rest of the codebase (the bot's existing audit rows never store IPs).

**Decision.**

  - **IP storage:** hash the IP before storing in audit. `crypto.createHash('sha256').update(ip + config.telegram.botToken).digest('hex').slice(0, 16)` — 16-hex-char salted hash. The salt is the bot token (already a secret, already in config) so the hash is non-reversible without compromising the bot. Different deployments produce different hashes for the same IP, so they're not cross-deployment correlatable.
  - **userAgent storage:** raw, truncated to 200 chars. UAs are not strictly PII in most frameworks but are coarsely fingerprinting; 200 chars captures the platform and browser version without storing extension lists.
  - **auth_date:** stored as the original Unix epoch seconds. Not PII.
  - **userId/chatId on FAILURES:** NOT stored. We couldn't verify, so there's no trustworthy userId/chatId anyway (the failure case is "the verifier said no" — any embedded userId in the malformed payload is not something we want to attribute).

The IP hash is short (16 hex chars = 8 bytes effective) — collision-resistant enough for "is this the same IP that hit us 5 minutes ago" investigation, not so long that it bloats the audit row. Not cryptographically secure for "find the original IP" — that's intentional.

**Consequences.**

  - **Positive.** Audit rows don't carry raw IPs into long-term storage.
  - **Positive.** Operator investigating a clustered attack pattern can group by hashed IP without needing the originals.
  - **Negative.** Can't exactly reverse-lookup a specific IP from a row. Acceptable — that's the point.

### 11. Page robustness when `Telegram.WebApp` is undefined

**Status:** Accepted — graceful degradation with a "open from /webapp" message.

**Context (rigor ask 11).** A user (or attacker) opening the cloudflared URL directly in a browser sees `window.Telegram === undefined`. The skeleton's first lines call `Telegram.WebApp.ready()`; an undefined-deref crashes the page. Need a graceful path.

**Decision.** The page's boot logic in `app.js`:

```javascript

if (typeof Telegram === 'undefined' || !Telegram.WebApp) {
  // Browser load (not Telegram) — render the explainer, exit.

    <h1>Jarvis Web App</h1>
    <p>Open this link from the <code>/webapp</code> button in Telegram.</p>
    <p>Direct browser access is not supported.</p>
  `;
  return;
}

const tg = Telegram.WebApp;
tg.ready();
tg.expand();

// Read initData (auth) and initDataUnsafe (cosmetic display)

})

    // sendData fires a web_app_data event on the bot side. Server logs and acks.

    // sendData closes the Mini App on most clients — no further UI needed.
  });

```

**Behavior contract:**

  1. Non-Telegram browser → static "open from /webapp" message; no network calls.
  2. Telegram + missing initData (somehow) → error message; no fetch.
  3. Telegram + initData + verifier success → "Hello \<first_name\>!" + verified userId display + a "ping" button.
  4. Telegram + initData + verifier failure → error message: `"Verification failed: <statusText>"`. The 401 body's `reason` is NOT shown to the user (low-value to surface; could leak which check failed to a probe). Only the HTTP status text.
  5. CSS variables set from `tg.themeParams.bg_color`, `tg.themeParams.text_color`, etc. — the page auto-themes light/dark per the user's Telegram settings.

**HTML structure** (`index.html`, ~50 LOC):


**Consequences.**

  - **Positive.** Page works in three modes (real Telegram embed, browser direct, broken initData) with appropriate UX in each.

### 12. `Telegram.WebApp.sendData` ping handling

**Status:** Accepted — log only; no side effects.

**Context (rigor ask 12).** When the user taps the "ping" button, the page calls `tg.sendData(JSON.stringify({type: 'ping', ts}))`. This dispatches a `message:web_app_data` event to the bot via Telegram's update stream — the bot receives a `Message` object whose `web_app_data` field has `data` (the JSON string) and `button_text` (the inline button label). For v1.13.0 we have no real feature to act on; the brief says: log it server-side and ack.

**Decision.**

  1. **Add a grammY handler:** `bot.on('message:web_app_data', async (ctx) => {...})`.
  2. **Behavior:**
     - Parse `ctx.message.web_app_data.data` as JSON. Wrap in try/catch — malformed JSON → log warn + reply "Could not parse Web App data."
     - Validate `data.type === 'ping'`. Other types → log warn (`'unknown web_app_data type'`), reply "Unknown Web App message type."
     - Log info: `{component: 'gateway', event: 'webapp_data_received', userId, chatId, type: 'ping', age: now - data.ts}`. The `age` field is roughly the round-trip latency — useful for ops.
     - Reply with a tiny ack: `"🏓 Pong (received \<ms\>ms ago)"`. This proves the round-trip works end-to-end.
  3. **No state mutation.** No tool dispatch. No agent invocation. No audit emission (the log is the record). No queue interaction.
  4. **Allowlist middleware applies.** The allowlist that gates `/start`, `/plan`, etc. also gates `web_app_data` events — non-allowlisted users can't push data into the bot via this surface. (grammY's middleware chain runs on every update; the allowlist middleware is registered first.)

**Why no audit row:** ping is the only `data.type` and it carries no security-relevant content. The pino info log is the historical record. When a real feature is wired in v1.14.0+ (e.g. "user dragged a card on the Kanban Web App, dispatch a tool"), THAT call will emit its own audit row per ADR 005's per-tool audit pattern — `webapp.action` or similar. v1.13.0 doesn't need it.

**Why log the round-trip age:** end-to-end latency from user-tap → server-receipt is the most operationally meaningful number for "is the platform alive and fast" — useful baseline for the v1.14.0 rollout.

**Consequences.**

  - **Positive.** End-to-end smoke test: user taps button, bot replies "Pong" — the entire stack (cloudflared → Express → CSP-strict page → Telegram client → grammY → bot) is exercised.
  - **Positive.** No accidental side effects from ping messages.
  - **Negative.** None material.

## Item 4 — `MessagingAdapter.sendWebAppButton`

### Decision (numbered for parity with item-1 entries)

The `sendWebAppButton` method extends the existing `MessagingAdapter` interface. It is the SAME pattern as `editMessageReplyMarkup` (added in v1.12.0 per ADR 007 R10): platform-neutral surface with Telegram-specific impl in `src/messaging/telegram.ts`.

**Interface extension** (additive, non-breaking):

```typescript
// src/messaging/adapter.ts
export interface MessagingAdapter {
  // ... existing methods ...

  /**
   * v1.13.0 — Send a message with a single Web App button.
   * Telegram: uses InlineKeyboard().webApp(label, url) — Telegram's Mini Apps surface.
   * Slack/WhatsApp (future): no-op or sendMessage with the URL inline; document
   * platform-specific behavior in each adapter.
   *
   * @param chatId — destination chat
   * @param text — message body shown above the button
   * @param label — button label (Telegram caps at ~64 chars; longer is rejected by the API)
   * @param url — MUST be HTTPS. Adapter throws WebAppButtonValidationError on http: scheme.
   *              Telegram's Mini Apps API rejects non-HTTPS URLs; we fail fast at the adapter.
   * @returns the platform-assigned message id
   */
  sendWebAppButton(
    chatId: number,
    text: string,
    label: string,
    url: string,
  ): Promise<{ messageId: number }>;
}
```

**Telegram impl** (`src/messaging/telegram.ts`):

```typescript
import { Bot, InlineKeyboard } from 'grammy';

export class WebAppButtonValidationError extends Error {
  constructor(message: string, public readonly code: 'NOT_HTTPS' | 'EMPTY_URL' | 'EMPTY_LABEL') {
    super(message);
    this.name = 'WebAppButtonValidationError';
  }
}

// Inside createTelegramAdapter(api):
async sendWebAppButton(chatId, text, label, url): Promise<{ messageId: number }> {
  if (!url) throw new WebAppButtonValidationError('URL is empty', 'EMPTY_URL');
  if (!label) throw new WebAppButtonValidationError('Label is empty', 'EMPTY_LABEL');
  if (!url.startsWith('https://')) {
    throw new WebAppButtonValidationError(`Web App URL must be https, got: ${url}`, 'NOT_HTTPS');
  }
  const keyboard = new InlineKeyboard().webApp(label, url);
  const result = await api.sendMessage(chatId, text, {
    reply_markup: keyboard,
  });
  return { messageId: result.message_id };
}
```

**Validation at adapter level** (NOT at command level): `/webapp` shouldn't have to re-validate; the adapter is the canonical gate for HTTPS-only. A typed error class (`WebAppButtonValidationError`) lets callers distinguish validation failures from grammY/network failures. Anti-Slop §3 (explicit error handling, no generic Error).

**No CONTRACT change to existing methods:** `sendMessage`, `editMessageText`, etc. are untouched. Existing tests continue to pass without modification (decision 22 covers the new method).

**Future platform support:** Slack has no native "open external HTTPS as a webview" button surface today — it has `actions` blocks with URL targets that open in the browser, which is materially different. WhatsApp's "interactive buttons" don't open arbitrary URLs as webviews. So `sendWebAppButton` is effectively Telegram-only in 2026. The interface still belongs in `MessagingAdapter` because the contract — "send a message with a single button that opens an HTTPS surface" — is platform-neutral; the FALLBACK on platforms without native support is `sendMessage(chatId, text + '\n\n' + url)`, documented as the Slack/WhatsApp default. v1.13.0 ships only the Telegram impl.

**Consequences.**

  - **Positive.** Pattern-match with `editMessageReplyMarkup` (R10). One more method on the canonical interface.
  - **Positive.** HTTPS validation happens once, at the lowest layer. Anti-Slop §13 (module isolation — the command doesn't reach into the platform's surface).
  - **Negative.** None material.

## Item 5 — `/webapp` slash command

### `/webapp` handler

**Module:** `src/commands/webapp.ts` (NEW). Thin handler — ~50 LOC including types and the empty-URL fallback message.

```typescript
// src/commands/webapp.ts (skeleton)
import type { Context } from 'grammy';
import type { AppConfig } from '../config/index.js';
import type { MessagingAdapter } from '../messaging/adapter.js';
import { child } from '../logger/index.js';
import { WebAppButtonValidationError } from '../messaging/telegram.js';

const log = child({ component: 'commands.webapp' });

export interface WebappCommandDeps {
  config: AppConfig;
  adapter: MessagingAdapter;
}

export async function handleWebapp(ctx: Context, deps: WebappCommandDeps): Promise<void> {
  const chatId = ctx.chat?.id;
  if (chatId === undefined) {
    log.warn({}, '/webapp invoked with no chat context — ignoring');
    return;
  }

  // DM-only — same posture as /organize, /scheduled.
  if (ctx.chat?.type !== 'private') {
    await ctx.reply('The Web App is DM-only — message me privately.').catch(() => {});
    return;
  }

  const url = deps.config.webapp.publicUrl.trim();
  if (!url) {
    await ctx.reply(
      'Web App is not configured.\n\n' +
      'Set `webapp.publicUrl` in config/config.json to a public HTTPS URL ' +
      '(e.g., the cloudflared tunnel URL — see README → Telegram Web App setup).',
      { parse_mode: 'Markdown' },
    ).catch(() => {});
    return;
  }

  try {
    await deps.adapter.sendWebAppButton(
      chatId,
      'Open Jarvis Web App',
      '🚀 Open',
      url,
    );
    log.info({ chatId, url: '<redacted>' }, '/webapp button sent');
  } catch (err) {
    if (err instanceof WebAppButtonValidationError) {
      log.error({ chatId, code: err.code, err: err.message }, '/webapp config error');
      await ctx.reply(
        `Web App configuration error: ${err.message}.\n\n` +
        'Check config/config.json — webapp.publicUrl must be an HTTPS URL.',
      ).catch(() => {});
      return;
    }
    const msg = err instanceof Error ? err.message : String(err);
    log.error({ chatId, err: msg }, '/webapp send failed');
    await ctx.reply(`Web App error: ${msg}`).catch(() => {});
  }
}
```

**Why DM-only:** the Web App is a single-user surface — group-chat invocation would put a button visible to the group, but the verifier's `chat.type === 'group'` path opens a different threat model (which group member is allowed to see what data?). v1.13.0 dodges that question entirely; v1.14.0+ revisits when there's a real feature to multi-tenant.

**Why Markdown not HTML for the setup message:** matches `/organize` setup-style replies. Backtick the config key for legibility.

**No log of the URL:** the cloudflared `*.trycloudflare.com` URL isn't a secret in the strict sense, but it IS a capability-bearing URL (anyone with it can probe `/api/webapp/echo`). Log redaction defaults treat it like a secret.

**Wiring into the gateway's command router:** `src/gateway/index.ts` already has a command registry (the existing pattern at line ~370+ where `cmdDeps`, `compactCmdDeps`, etc. are constructed and the bot's `command()` handlers fire). Add:

```typescript

  config,
  adapter: telegramAdapter,

bot.command('webapp', (ctx) => handleWebapp(ctx, webappCmdDeps));
```

This is a 3-line addition, mirroring the existing `bot.command('plan', ...)`, `bot.command('organize', ...)` etc. patterns. No changes to the allowlist middleware (it already wraps every command).

**Consequences.**

  - **Positive.** Clean fit into the existing command-handler pattern.
  - **Positive.** Empty-URL setup message is informative without being technical-language-only.

## Item 8 — Tests

### 22. Test plan

**Status:** Accepted.

**`tests/unit/webapp.auth.test.ts`** (new, ~200 LOC, 12+ test cases):
  - Deterministic vectors with a fixed bot token. The test harness builds an `initData` string by computing the expected hash with the same algorithm; if the implementation is wrong, the verifier and the harness disagree.
  - **Happy path:** valid initData with current `auth_date` → `{ok: true, data: {...}}`. Assert `user.id`, `user.first_name`, `authDate`.
  - **Malformed:** input is not a query string (e.g. `'?@#$'` after parsing); `hash` field missing; `hash` is non-hex; `hash` is wrong-length hex.
  - **Bad-hash:** valid query string + correct shape, but `hash` is a valid-shape SHA256 hex of something else. Verify constant-time path doesn't leak.
  - **Stale:** `auth_date = now - (maxAgeSeconds + 60)`. Reject.
  - **Future skew:** `auth_date = now + 120` (default skew is 60). Reject.
  - **No-user:** `user` field missing or empty JSON or missing `id`/`first_name`.
  - **Spoofed token:** initData built with a different bot token than the verifier is given. Reject as `bad-hash`.
  - **Boundary maxAge:** `auth_date = now - maxAgeSeconds` (exactly at boundary) → accepted (the rejection is `>`, not `>=`, per spec).
  - **Boundary future skew:** `auth_date = now + maxFutureSkewSeconds` (exactly at boundary) → accepted.
  - **Optional fields parse:** `chat`, `start_param`, `chat_type`, `chat_instance` parse correctly when present, default to undefined when absent.
  - **Hash with mixed-case hex:** Telegram emits lowercase; verify the verifier accepts uppercase too (defensive — the spec is silent on case).
  - **Tampered field:** original valid initData with one byte flipped in the `user` JSON — bad-hash.

  Use `vi.useFakeTimers()` + `vi.setSystemTime(...)` for the freshness-window tests.

**`tests/integration/webapp.echo.test.ts`** (new, ~150 LOC):
  - Spin up the actual Express app on a random ephemeral port via `app.listen(0)`.
  - Build a valid initData with a known bot token. Hit `GET /api/webapp/echo` with `Authorization: tma <initData>`. Assert `200` + body `{userId, username, chatId, authDate}`.
  - Hit with no Authorization, no query string → `401 {error: 'No initData', reason: 'missing'}`.
  - Hit with malformed initData → `401` with `reason: 'malformed'`.
  - Hit with bad-hash initData (built with a different bot token) → `401` with `reason: 'bad-hash'`.
  - Hit with stale initData → `401` with `reason: 'stale'`.
  - Hit with `?initData=...` (query path) → `200` and a warn log emitted (test the log via `pino` test transport).
  - Hit `/webapp/index.html` (or `/webapp/`) → `200`, body contains `<script src="https://telegram.org/js/telegram-web-app.js">`. Assert CSP headers on the response.
  - Hit `/webapp/index.html` → `Cache-Control: no-cache, no-store, must-revalidate`. Hit `/webapp/app.js` → `Cache-Control: public, max-age=86400`.
  - Rate-limit: hit `/api/webapp/echo` 61 times from the same fake IP within 60s → 61st gets 429 + `Retry-After` header.

**`tests/unit/commands.webapp.test.ts`** (new, ~80 LOC):
  - `publicUrl` empty → reply contains `'webapp.publicUrl'` and `'cloudflared'` keywords.
  - `publicUrl` HTTP (not HTTPS) → reply contains `'configuration error'`.
  - `publicUrl` HTTPS → `adapter.sendWebAppButton` called once with `(chatId, "Open Jarvis Web App", "🚀 Open", url)`.
  - Group-chat invocation → reply `"DM-only"`.
  - Adapter throws `WebAppButtonValidationError` → reply contains `'configuration error'` and the error message.
  - Adapter throws generic Error → reply contains `'Web App error:'` + the message.

**`tests/unit/messaging.adapter.test.ts`** EXTENSION (existing file, add ~30 LOC):
  - `sendWebAppButton` with `https://...` → resolves with `{messageId}`. Asserts grammY `InlineKeyboard().webApp(...)` was called.
  - `sendWebAppButton` with `http://...` → throws `WebAppButtonValidationError` with `code: 'NOT_HTTPS'`.
  - `sendWebAppButton` with empty URL → throws with `code: 'EMPTY_URL'`.
  - `sendWebAppButton` with empty label → throws with `code: 'EMPTY_LABEL'`.

**Test isolation:** the integration tests use `app.listen(0)` to bind an ephemeral port; tests that share a fixture port are flaky in CI. The Express factory must be a function (it is — `createWebappServer(deps)`), not a top-level singleton.

**Coverage target:** the verifier and the Express handlers are pure-logic surfaces; expect >95% line coverage. The skeleton page (HTML/JS) is NOT unit-tested — vanilla browser JS without a bundler is hard to harness; manual smoke testing in the CP4 pre-QA pass covers it.

**No e2e (Playwright) tests in v1.13.0.** Setting up a real Telegram client to drive the page is high-cost for low value at the skeleton stage. v1.14.0 (when there's real UI) is the right time to add Playwright + a Telegram embed simulator.

**Consequences.**

  - **Positive.** Vector-driven verifier tests will catch any algorithm regression.
  - **Positive.** Integration tests exercise the real Express stack — middleware, CSP headers, rate-limit, the verifier, and the response shape — all in one place.
  - **Negative.** No browser-side test for the page itself. Documented as a v1.14.0 deferral.

## 13. Risk register

| Risk | Severity | Mitigation |
|---|---|---|
| Cloudflared tunnel goes down (process crash, network glitch) — user opens the Web App and sees a connection error | LOW (dev-only) | The page renders a static error inside Telegram. `/webapp` command still works; user re-runs after fixing the tunnel. README documents the `cloudflared tunnel --url` quick-start. |
| Leaked initData via browser history on a non-Telegram browser | LOW | `auth_date` freshness window (default 24h) bounds the leak. Decision 11's "open from /webapp button" message means a leaked URL opened in a regular browser shows nothing useful (no auto-fetch). The verifier rejects stale initData, so a leaked URL+initData becomes useless after 24h. |
| User opens the cloudflared URL directly in a regular browser (curiosity, debugging) | LOW | Decision 11 — graceful "open from /webapp button" message; no network calls; no information disclosure beyond "this is a Jarvis Web App." |
| Rate-limit bypass via IP rotation (botnet) | LOW | Single-user deployment impact is small. `maxKeys` cap on the bucket map prevents memory blowup. The verifier itself is constant-time per-call (~30µs); even unbounded RPS only burns CPU bounded by the request inflow rate, which cloudflared's free-tier already throttles. |
| CDN fetch of `https://telegram.org/js/telegram-web-app.js` fails | LOW | Page degrades gracefully (decision 11's `Telegram === undefined` branch fires). User sees the static "open from Telegram" message instead of a half-loaded UI. Telegram CDN uptime has been 99.99%+ historically; this is a non-issue in practice. |
| HMAC verifier algorithm bug (e.g. wrong delimiter, wrong sort order) accepts spoofed initData | HIGH (mitigated to LOW) | Vector-driven tests with deterministic harness — if the algorithm is wrong, tests fail before commit. The deterministic-gates phase (tsc + eslint + tests) blocks merge on test failure. |
| Future-skew rejection (decision 4) too tight; legitimate users with clock drift get rejected | LOW | Default 60s exceeds any healthy system's drift. `initDataMaxFutureSkewSeconds` is configurable. Logged with `authDateAge` so an operator can spot it immediately. |
| `Authorization: tma` header parsing accepts query-string fallback (decision 5) — attacker crafts a phishing URL with embedded initData | LOW | Initial concern: a leaked initData + the public URL can be combined into a phished URL the attacker passes around. But the leaked initData is bound to a real user's `auth_date`, so it expires per the freshness window. The query-string warn log surfaces unusual traffic. v1.14.x can tighten to header-only based on observed traffic. |
| Express server bound to 0.0.0.0 by misconfiguration, exposing to local network | MEDIUM | The factory binds 127.0.0.1 explicitly in `createWebappServer`. No config knob to override the bind address (intentional — operators who need that can edit the source, but the default is locked down). Documented in the source comment. |
| Skeleton page's inline-style fallback CSP `style-src 'unsafe-inline'` allows injected style attacks via theme params | LOW | Theme params come from Telegram, NOT from user input. Telegram-controlled keys/values; no user-content path into the inline styles. Page sets CSS variables on `:root` (which are values, not selectors), which is the safest inline-style usage. |
| The bot token is logged accidentally during initData verification | MEDIUM (mitigated to LOW) | Verifier doesn't log. Pino redaction list already covers `botToken`, `authorization`, etc. project-wide. Phase 2 dev agent runs `grep -i 'botToken' src/safety/webappAuth.ts` — must match only the function-parameter name and a comment, never a `log.info` call. |
| `sendWebAppButton` called from a non-DM context (a future caller misuses the adapter) | LOW | The adapter doesn't enforce DM-only — it's chat-agnostic by design. The `/webapp` command (the only v1.13.0 caller) gates DM-only at the command layer. Future callers add their own gates. Documented in adapter JSDoc. |
| pm2 restart loses the rate-limiter state — an attacker times restarts to reset their bucket | LOW | The bucket is a fuse, not a ledger. Restart resets the count, but the attacker also has to wait for restart. Net: a bursty attacker on a single IP across a restart cycle gets `2 × maxRequests` per cycle, still bounded by the cloudflared edge. |
| Express default behavior leaks request URL into error stack traces shown to the client | LOW | `app.use((err, req, res, next) => { ... res.status(500).json({error: 'Internal'}); })` final error middleware — never `res.send(err.stack)`. Tested. |
| The static page payload (HTML+JS+CSS) grows past the v1.13.0 budget (~5 KB) into multi-megabyte territory by accidental commit of node_modules | MEDIUM | `staticDir: public/webapp` is a separate path from any node_modules location. `.gitignore` already excludes node_modules. Linter's no-restricted-imports catches accidental cross-imports from `src/` (the page's app.js never imports from `src/`). |
| The `web_app_data` event handler (decision 12) panics on a maliciously-crafted ping payload from a compromised page | LOW | The handler wraps JSON.parse in try/catch and validates `data.type` before any further use. Allowlist middleware blocks non-allowlisted users from reaching the handler at all. A compromised page would still need a valid Telegram user account to even reach this path. |

## 17. Open questions
