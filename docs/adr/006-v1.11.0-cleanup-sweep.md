# ADR 006 — v1.11.0 Cleanup Sweep (/organize reconcile, nag cost, trash TTL, calendar update/delete tools)

**Status:** Proposed (Phase 1, v1.11.0)
**Date:** 2026-04-24
**Deciders:** Architect Agent (iteration)
**Extends:** ADR 003 (`/organize` storage + tool layer, `CalendarApi` surface), ADR 004 (/organize reminders, `organize.nudge` audit category), ADR 005 (/scheduled + `schedule` tool, audit category naming convention). Does NOT supersede any prior ADR — all four items are additive, backwards-compatible polish on the surfaces shipped in v1.8.6 → v1.10.0.

This ADR records the architectural decisions for the v1.11.0 cleanup sweep. Four small items ship together before the Web App UI rework (deferred to v1.12.0):

  1. **`/organize reconcile` subcommand** — DM-only command that reads recent `organize.inconsistency` audit rows, checks current disk + state against them, and offers per-item fixes via grammY inline-button callbacks.
  2. **`/organize nag cost [days]` subcommand** — DM-only cost aggregation over `organize.nudge` audit rows, per model per day.
  3. **`.trash/` TTL eviction** — daily cron in a new sibling module under `src/organize/` that hard-deletes trashed items older than `config.organize.trashTtlDays` (default 30).
  4. **`calendar_update_event` + `calendar_delete_event` standalone tools** — thin wrappers over the already-present `CalendarApi.updateEvent` / `deleteEvent` methods (surfaced for /organize in v1.8.6; now promoted to agent tools).

Scope is design only — no code is in scope here. Developer agents in Phase 2 implement the decisions below without deviating; deviations require an ADR 006 addendum.

**Non-negotiables set by the iteration brief (not up for debate in this ADR):**

  - No new npm dependencies. `node-cron`, `grammY`, `better-sqlite3`, `zod`, `googleapis` are already vendored.
  - No Telegram Web App UI (v1.12.0).
  - No multi-tenant sidecar SQLite index for `/organize` items.
  - No config hot-reload.
  - No `proper-lockfile` (the v1.9.0/v1.10.0 TODO still stands).
  - No changes to reminder-loop triage logic — Item 3's cron is a sibling module, NOT embedded in `reminders.ts`.
  - No UI changes to `/scheduled`.

## Item 1 — `/organize reconcile` subcommand

### 1. Trash-evictor module location vs. folding into `reminders.ts`

**Status:** Accepted (sibling module).

**Context.** Item 3 needs a 4am cron that walks every user's `.trash/` dir. Two candidate homes:

  - (a) **Fold into `src/organize/reminders.ts`** — `initReminders` already owns a node-cron handle (ADR 004 §1); reuse the same lifecycle. Advantage: one cron registration point; minor code reuse for user enumeration (the `readdir(organize/)` + `parseInt` filter in `tickAllUsers`, reminders.ts:856).
  - (b) **New sibling module `src/organize/trashEvictor.ts`** — its own node-cron; its own init/start/stop lifecycle, mirroring `RemindersApi`. `src/index.ts` wires it in the same boot step (after reminders). No coupling to the triage loop; the evictor doesn't load `.reminder-state.json`, doesn't read per-user organize items' front-matter beyond what's under `.trash/`, doesn't hit any LLM.

**Decision.** (b), sibling module. The two loops have zero shared state: the evictor does NOT read `.reminder-state.json`, doesn't care about `nudgesToday`, doesn't use the provider handles, doesn't touch the pool's abort-signal plumbing. Folding it in would make `reminders.ts` (already ~950 LOC) a grab-bag for "anything that runs on a cron under `src/organize/`." The <~20 LOC of user-enumeration overlap (readdir + `Number.parseInt` filter) is not worth the conceptual coupling.

The evictor's public surface mirrors `RemindersApi`:

```typescript
export interface TrashEvictorApi {
  start(): void;
  stop(): void;
  /** Exposed for tests and ad-hoc trigger; runs one pass over all users. */
  evictAllUsers(): Promise<{ usersProcessed: number; evicted: number; errors: number }>;
}

export interface TrashEvictorDeps {
  config: AppConfig;
  memory: MemoryApi;          // audit-log repo only
  dataDir: string;
  abortSignal?: AbortSignal;  // optional — mirrors reminders R6 pattern
}

export function initTrashEvictor(deps: TrashEvictorDeps): TrashEvictorApi;
```

Internal node-cron schedule: `'0 4 * * *'` (4am server-local, every day). `start()` registers one cron handle; `stop()` calls `task.stop()` on the handle and sets a module-scoped `stopped` flag so an in-flight `evictAllUsers` bails gracefully (same pattern as `reminders.ts` + the abort-signal propagation it uses).

Boot order in `src/index.ts` gets one more step:

```
... 11. reminders (NEW in v1.9.0) → 12. trash evictor (NEW in v1.11.0) →
13. scheduler.start() → 14. gateway.start() → 15. reminders.start() → 16. trashEvictor.start()
```

Shutdown reverses: trashEvictor → reminders → gateway → scheduler → memory.

**Consequences.**

  - **Positive.** `reminders.ts` stays focused on the triage loop. `src/organize/trashEvictor.ts` is ~150 LOC of self-contained module state — readable, testable, replaceable.
  - **Positive.** Two cron instances in the process (reminders + evictor), both in `src/organize/`, both lightweight. `node-cron` internal cost is negligible; the v1.9.0 scheduler module already has its own cron handles per scheduled task.
  - **Negative.** Slight code duplication of the `readdir(organize/) → parseInt → filter` user-enumeration block. ~10 lines. A future refactor could extract it to `src/organize/userDirs.ts`; deferred.

### 2. Reconcile UX at scale — inline buttons, scale-safe defaults

**Status:** Accepted.

**Context.** The brief asks: "inline buttons vs. confirmation DM per item; with 50+ inconsistencies, do we paginate / auto-apply safe actions / require admin override?"

Inline-keyboard fragility across Telegram clients is real (older iOS clients, in-place message-edit timing issues). But for the deployed installation today (single user, personal assistant), inline buttons are the lowest-friction shape — the user sees "5 drifts; fix each?" and taps `✅ Fix` / `❌ Skip` per item. The alternative — text-only DM confirmation per item — requires the user to type `/organize reconcile apply <itemId>` for each one, which turns a 5-item cleanup into a 10-message ping-pong.

The scale question is legitimate. If the audit has 50+ drifts:

  - Rendering 50 inline-keyboard messages in a row is spammy.
  - Telegram's `reply_markup` payload is bounded (~4KB of callback_data across a single message's keyboard); at a cap of ~50 items one message is safe, but multiple messages deliver fine.
  - The real cost is the user's time — looking at 50 items without context is a DoS on attention.

**Decision.**

  1. **Per-item inline-keyboard UX is the default.** Two buttons per item: `✅ Fix` and `❌ Skip`. Callback data encodes `rec:<action>:<itemId>` (see decision 7 for namespace). Each item is its OWN Telegram message with its own keyboard (not one giant keyboard) — that keeps each callback_data short, keeps the conversation scrollable, and lets the user skip-by-scroll.
  2. **Hard cap at 20 items per `/organize reconcile` invocation.** If the 30-day window has more than 20 unresolved inconsistencies, the command renders the first 20 (sorted by audit-row `ts` ASC — oldest first, so long-tail drifts surface) and appends a final summary message: `N older inconsistencies exist. Run /organize reconcile again after handling these to see more.` No pagination UI — the rerun IS the pagination.
  3. **No auto-apply, not even for "safe" actions.** The two action categories (prune orphan state entry, null out orphan `calendarEventId`) look safe in isolation, but orphan state can be recovered by a user who restores a file from `.trash/` — auto-applying "prune state" would silently wipe nudge history and cooldown state they might have wanted. The user is in the loop per item.
  4. **No admin override.** `/organize reconcile` is a per-user command (like `/organize nag`) — it reads THIS user's audit rows, touches THIS user's state/calendar-id fields. There's no cross-user fanout where an admin would need to intervene. A future multi-tenant operation panel (deferred) might have an admin-broad reconcile; not this ADR.
  5. **Group-chat posture.** DM-only. Group invocation replies "Organize is DM-only — message me privately." (mirrors v1.8.6 /organize posture, ADR 003 decision 7).

The per-item message shape is plain text + inline keyboard, e.g.:

```
Inconsistency 1 of 5 — orphan-local
Item: 2026-04-20-a1b2  (file on disk; no .reminder-state entry)
Proposed: prune the stale state entry (no file change).
[✅ Fix] [❌ Skip]
```

And for the other variant:

```
Inconsistency 2 of 5 — orphan-gcal (calendar event id remembered but no file)
Item: 2026-04-18-c3d4 → eventId: abc...xyz
Proposed: null out calendarEventId on item (file remains; Calendar event preserved).
[✅ Fix] [❌ Skip]
```

**Consequences.**

  - **Positive.** At the deployed scale (1 user, <~10 drifts likely) the UX is fast and forgiving.
  - **Positive.** The 20-item cap bounds message spam and puts the user in control of the cleanup rhythm.
  - **Negative.** At 50+ inconsistencies, the user must run the command multiple times. Explicit design tradeoff — see §4 below.
  - **Negative.** Inline-keyboard callbacks depend on the gateway's new callback_query router branch (decision 7). If that branch breaks or the Telegram client drops a callback, the user sees a dead button. Mitigation: the command lists the items; the user can also re-run after the issue self-resolves.

### 3. Existing `deletedAt` absence and migration posture (for Item 3 context; checked here)

**Status:** Accepted — extend serializer; legacy trash falls back to mtime.

**Context.** Verified against `src/organize/storage.ts:632` and `src/organize/types.ts:20`:

  - `OrganizeFrontMatter` has NO `deletedAt` field today.
  - `softDeleteItem(userId, dataDir, itemId)` calls `fs.rename` from `<userId>/<itemId>.md` into `<userId>/.trash/<itemId>.md` without rewriting the file (storage.ts:653). The trashed file retains the exact front-matter it had at live-edit time.
  - ADR 003 §4 explicitly noted this: "the trashed file preserves its original front-matter so a future 'undelete' command has everything it needs."

So Item 3 can't reach for `deletedAt` today. Two options:

  - (a) **Extend the front-matter schema** — add an optional `deletedAt: string | null` field. Rewrite `softDeleteItem` to: read the file, parse front-matter, set `deletedAt: <now>`, call `serializeItem`, rename to `.trash/<id>.md`. One more write on the soft-delete hot path (negligible cost; soft-delete is user-initiated and rare).
  - (b) **Rely on file `mtime`** — use `fs.stat(trashedPath).mtime` as the age anchor. No schema change. Works for every existing trashed file today (which have no `deletedAt`) as well as newly-trashed ones.

**Decision.** Extend the serializer (a), with mtime fallback for legacy files (b).

  - **New field.** `OrganizeFrontMatter.deletedAt?: string | null` — optional, omitted from serialization when null/undefined. The serializer (`serializeItem` in storage.ts:129) emits `deletedAt: <ISO>` as a new key between `calendarEventId` and `tags` ONLY when the field is non-null.
  - **`softDeleteItem` change.** Before `fs.rename`, read + parse the file, set `fm.deletedAt = new Date().toISOString()`, serialize via `serializeItem`, write back to the ORIGINAL path atomically (temp-then-rename inside the user dir), THEN rename to `.trash/`. Atomic write to the live path keeps ADR 003's "last writer wins; no torn writes" invariant — if the rewrite fails, the rename doesn't happen and the user sees FILE_DELETE_FAILED with the item still live (safer than the current "always renames even on read failures" path — see consequence below).
  - **Parser tolerance.** The front-matter parser (`parseItemFile` at storage.ts:181) already treats unknown keys as pass-through. `deletedAt` simply becomes one more known key; missing → `null`. Malformed ISO → treated as `null` (same posture as `due`).
  - **Legacy fallback.** `evictExpiredTrash` (decision 4 below) reads `deletedAt` from parsed front-matter first; if absent or unparseable, it falls back to `fs.stat(path).mtime`. Rationale: a file trashed before v1.11.0 has no `deletedAt`, but its filesystem mtime is approximately when the rename happened (`fs.rename` on the same filesystem preserves mtime of the source file, not of the rename, so mtime is actually the last-live-edit time — typically close-to or earlier-than the true delete time; this biases the evictor toward MORE-aggressive eviction, which is the user-preferred direction for a cleanup feature). Document in code comment.
  - **Migration posture.** No one-shot migration needed. New trashes carry `deletedAt`; old trashes use mtime. After `trashTtlDays` days (default 30), all legacy trashes will have aged out regardless and the fallback path stops being exercised.
  - **This is a SILENT schema change.** No version bump on the item file; the serializer's "canonical shape" grows by one optional field. Consumers that expected the exact old field set (if any exist outside Jarvis) see an extra line; tolerant parsers don't care. Documented here for traceability.

**Consequences.**

  - **Positive.** Item 3 has a deterministic age signal for every new trash, and a safe fallback for legacy trash.
  - **Positive.** The rewrite-before-rename actually makes `softDeleteItem` slightly more correct — today a partial parse failure still renames; the new flow stops if rewrite fails.
  - **Negative.** `softDeleteItem` pays one extra read + parse + serialize + atomic-write on the path. At <1 delete/minute user rate, this is invisible (< 20ms worst case).
  - **Negative.** Front-matter serializer emits a new line when present — any tests pinning the exact serialization of a post-delete file need updating. Developer agents will catch this in Phase 3 Sub-Phase A.

### 4. Callback-query namespace `rec:<action>:<itemId>` and gateway routing

**Status:** Accepted.

**Context.** The brief claimed v1.10.0 /scheduled uses `sch:<action>:<taskId>` callback_data. **Verified against source: this is NOT the case.** `src/commands/scheduled.ts` does not emit inline keyboards; the only `callback_query:data` handler in `src/gateway/index.ts:438` routes `plan.*` (from `src/commands/plan.ts` / `src/plan/panel.ts`). So `/organize reconcile` will be the SECOND inline-keyboard surface after /plan, and the FIRST under the organize family.

**Decision.**

  - **Callback-data format:** `rec:<action>:<itemId>` where `action ∈ {fix, skip}` and `itemId` is the existing `YYYY-MM-DD-[a-z0-9]{4}` shape (ADR 003 decision 9). Total length: `rec:skip:YYYY-MM-DD-xxxx` = 24 chars, well under Telegram's 64-byte callback_data limit.
  - **Router branch.** Extend `src/gateway/index.ts:438` (the `bot.on('callback_query:data', …)` handler) with an `else if (data.startsWith('rec:')) { await handleReconcileCallback(data, ctx, organizeCmdDeps); await ctx.answerCallbackQuery({ text: toast }).catch(() => {}); }` branch after the `plan.*` branch. Follow the same toast + catch pattern — `answerCallbackQuery` always fires so the button's spinner stops.
  - **Handler location.** A new module `src/commands/reconcileHandler.ts` (or a new exported function in `src/commands/organize.ts` — either is fine; developer agent chooses based on organize.ts's current LOC). Single function: `handleReconcileCallback(data: string, ctx: Context, deps: OrganizeCommandDeps): Promise<{ toast: string }>`. It parses the callback, performs the fix (decision 5), answers with a toast ("Fixed." / "Skipped." / "Already applied." / "Error: …"), and removes the keyboard on the original message via `ctx.editMessageReplyMarkup(undefined).catch(() => {})` so the button can't be double-tapped.
  - **Dispatch gating.** The callback is accepted only when the tap source is the DM chat (`ctx.chat?.type === 'private'`). In a group (unlikely but not impossible if the user's `reconcile` message was somehow mirrored) we answer with `Organize is DM-only.` and do nothing.

**Consequences.**

  - **Positive.** Namespace is short, distinct, greppable.
  - **Positive.** The branch adds ~5 lines to the existing handler; routing is fan-out on prefix, same as plan.
  - **Negative.** The organize command surface now has state that persists across Telegram messages (the sent reconcile message with a live keyboard). If pm2 restarts between the send and the user's tap, the handler still fires and the action still works — the callback carries all the info needed. Acceptable.

### 5. Audit category — `organize.reconcile` (NEW)

**Status:** Accepted — add a new category.

**Context.** The brief asked: reuse `organize.inconsistency` or add `organize.reconcile`?

`organize.inconsistency` (v1.8.6, `src/memory/auditLog.ts:24`) is emitted BY the tools that produce a drift (organize_create, organize_delete, etc.). Its detail shape is `{kind: 'orphan-gcal' | 'orphan-local' | 'deferred-orphan-gcal', ...}` — it's a diagnostic trail.

Reconcile actions are the OPPOSITE — they resolve drifts. Reusing `organize.inconsistency` for resolutions would conflate creation and cure in one category and force every future audit reader to switch on a new `resolved: true` flag to tell them apart. That's a type-union inside a string category, which ANTI-SLOP §1 and §4 both discourage (explicit categories > stringly-typed discriminators inside a string).

**Decision.** Add `'organize.reconcile'` to the `AuditCategory` union in `src/memory/auditLog.ts`. Detail shape:

```typescript
{
  action: 'prune-state-entry' | 'null-calendar-event-id' | 'skipped',
  itemId: string,                           // YYYY-MM-DD-xxxx
  originalInconsistencyKind: 'orphan-local' | 'orphan-gcal' | 'deferred-orphan-gcal',
  originalInconsistencyTs: string,           // ISO; the audit row this reconciles
  result: 'ok' | 'no-op' | 'failed',
  reason?: string,                          // only on failed / no-op
  calendarEventId?: string | null,          // for null-calendar-event-id action
}
```

Actor fields: `actor_user_id = userId`, `actor_chat_id = userId` (DM), `session_id = null` (reconcile is not tied to an agent turn — it's an out-of-band command handler like /organize itself).

**No raw user text in detail.** Mirrors the v1.8.6 invariant (ADR 003 decision 7, reinforced in ADR 004 §11): no title, no notes, no tags — just ids.

**Consequences.**

  - **Positive.** `/audit filter organize.reconcile` surfaces every fix attempt with provenance. A future `/audit verify organize.*` sweep can ensure every emitted `organize.inconsistency` eventually has a matching `organize.reconcile` row (or was explicitly skipped).
  - **Positive.** Type-exhaustive switch over `AuditCategory` in consumers forces acknowledgment of the new case. Anti-Slop §4 compliant.
  - **Negative.** One more category string. Low cost; high clarity.

### 6. New audit-log query helper — `listByCategoryAndActorSince`

**Status:** Accepted.

**Context.** `AuditLogRepo` today exposes `insert`, `listRecent(n)`, and `listForSession(sessionId, n)` only. `/organize reconcile` needs rows filtered by `category = 'organize.inconsistency' AND actor_user_id = ? AND ts >= ?`. `/organize nag cost` (Item 2) needs a similar filter on `organize.nudge`. Neither is a `listRecent` nor a `listForSession`.

**Decision.** Add one new helper:

```typescript
/**
 * Return audit rows filtered by category AND actor, since a cutoff timestamp.
 * Newest first. No LIMIT — callers pass a narrow window (days, not months).
 */
listByCategoryAndActorSince(
  category: AuditCategory,
  actorUserId: number,
  sinceIso: string,
): AuditLogRow[];
```

Prepared statement:

```sql
SELECT id, ts, category, actor_user_id, actor_chat_id, session_id, detail_json
FROM audit_log
WHERE category = ?
  AND actor_user_id = ?
  AND ts >= ?
ORDER BY ts DESC
```

The `audit_log` table's existing indexing (if any — developer agent verifies; a `CREATE INDEX IF NOT EXISTS idx_audit_category_actor_ts ON audit_log(category, actor_user_id, ts)` is likely prudent at 1k+ rows but defer until a profiling sweep shows it matters). At today's deployed row count (single-digit thousands at most), the scan is sub-millisecond unindexed.

**Consequences.**

  - **Positive.** Both Items 1 and 2 share one helper. DRY.
  - **Positive.** Single prepared statement, typed return, no interpolation. Anti-Slop §11 safe.
  - **Negative.** One more method on `AuditLogRepo`. Tiny addition.

## Item 2 — `/organize nag cost [days]` subcommand

### 7. `TOKEN_COSTS_USD_PER_MTOK` location — inline constants in the command file

**Status:** Accepted (inline constants).

**Context.** The brief offered three candidates: inline in the command file, config entry, or a separate `src/costs.ts` module.

Evaluating each against Anti-Slop §4 (Magic Numbers — "the why matters"):

  - **Config entry.** A user shouldn't be able to change the pricing table via `config.json` — pricing is a vendor fact, not a per-deployment preference. If Anthropic or Ollama Cloud moves a price, the user editing their config would be masking a bug (their code expects one number, their config says another). WRONG seat.
  - **Separate `src/costs.ts` module.** Premature extraction. There's exactly one caller today (the nag cost command). Creating a module called "costs" that has two scalar constants is YAGNI. If a second surface (e.g. a future "this week's model spend" dashboard) needs the same constants, THEN extract — that's the right refactor trigger, not this one.
  - **Inline in the command file.** Localized, greppable, change-in-one-place when pricing shifts. The "why" goes inline as a docstring above the constants.

**Decision.** Inline constants in `src/commands/organizeNagCost.ts` (new file, extracted from `/organize` since the subcommand has nontrivial logic; the parent `/organize` handler delegates to it):

```typescript
/**
 * Token cost table for the nag triage models. In USD per million tokens,
 * input/output separately. Source of truth for /organize nag cost aggregation.
 *
 * - deepseek-v4-flash:cloud — free (bundled in Ollama Cloud subscription,
 *   no per-call token cost). Represented as 0/0 so the math works without
 *   a null-check branch.
 * - claude-haiku-4-5 — Anthropic's published price card as of the ship date
 *   of v1.11.0. Update IN THIS FILE when the card moves. Do not move these
 *   to config.json — pricing is a vendor fact, not a deployment preference.
 *   Grep for the constant name if a second surface needs these; a shared
 *   src/costs.ts is warranted only when a second caller exists.
 *
 * Inputs matter more than outputs for triage (triage is short-reply-dominant);
 * we still show both so the aggregate is faithful.
 */
const TOKEN_COSTS_USD_PER_MTOK: Record<string, { input: number; output: number }> = {
  'deepseek-v4-flash:cloud': { input: 0, output: 0 },
  'claude-haiku-4-5':        { input: 1.00, output: 5.00 },  // verify against Anthropic card at ship time
} as const;
```

Developer agent in Phase 2 verifies the Haiku 4.5 card values against Anthropic's public pricing at implementation time and updates the `// verify…` comment to a bare attribution comment.

**Unknown model fallback.** If the audit row contains a `model` string not in the map, treat it as zero-cost (with a warning log: `log.warn({model}, 'unknown model in organize.nudge audit row; charging zero')`) and render it in the output as `<model>: usage unknown — price table missing`. Future-proofs against silent price drift when someone edits `triageModel` in config to something new.

**Consequences.**

  - **Positive.** Change in ONE place when pricing moves.
  - **Positive.** Greppable via `TOKEN_COSTS_USD_PER_MTOK`.
  - **Positive.** Comment tells the next maintainer WHY it's not in config.
  - **Negative.** If v1.12.0 needs cost data in a second surface, the refactor-to-`src/costs.ts` is a two-line import shuffle. Cheap.

### 8. Aggregation query and output shape

**Status:** Accepted.

**Context.** Input: `days` integer, default 7, clamped [1, 90]. Reject values outside that range with a usage message. Aggregate across rows of `organize.nudge` within the window, grouped by `(model, local-date)`.

**Decision.**

  1. **Parse argument.** `/organize nag cost` → days=7. `/organize nag cost 30` → days=30. Any token after "cost" that is not an integer in [1,90] → reply `Usage: /organize nag cost [days]  (days: 1-90, default 7)`. Return.
  2. **Compute `sinceIso`** = `new Date(Date.now() - days * 86400 * 1000).toISOString()`.
  3. **Query** via `auditRepo.listByCategoryAndActorSince('organize.nudge', userId, sinceIso)` (decision 6).
  4. **Filter** to rows where `detail.result === 'ok'` (delivered nudges — the ones that actually consumed tokens). Suppressed / skipped rows don't cost anything; include them in a separate counter for the summary but not the cost math.
  5. **Tally** tokens per (model, local-YMD). The nudge detail shape from ADR 004 §11 already has `provider`, `model`, `fallbackUsed`, plus any `_inputTokens` / `_outputTokens` the provider returns. **Verify token counts are persisted today** — if ADR 004 §11 rows don't currently carry input/output token counts, developer agent adds them (they come off `providers.call()` return value; the triage call has access). This is a small detail-shape extension; document as a new invariant in the v1.11.0 implementation notes.
  6. **Render.**
     - If `days <= 14`: full per-day table with ASCII bars scaled to the max-day cost. Format:
       ```
       Cost of /organize nag over the last 7 days (USD):
       2026-04-18  deepseek-v4-flash:cloud  2 nudges  $0.0000  (bar)
       2026-04-18  claude-haiku-4-5         1 nudge   $0.0003  (bar)
       ...
       Total:                              15 nudges  $0.0041
       3 skipped (Ollama failures; fallback triggered)
       ```
       Bar width: `'#'.repeat(Math.round(cost / maxCostInWindow * 20))` (20-char max bar), always render at least 1 char when cost > 0 so zero-cost rows are visibly distinct.
     - If `days > 14`: summary only — totals per model, overall total, skipped count. No ASCII bars (too sparse to be useful; too long to fit a Telegram message cleanly).
  7. **Empty-state UX.** If no rows in window: `No nudges in the last N days. Either nag is off for you (run /organize nag status) or the triage loop hasn't fired yet.` (addresses decision 11 below.)
  8. **Response format.** Plain text in a monospace code block (triple backticks) so column alignment reads correctly across clients.

**Consequences.**

  - **Positive.** Zero tokens = no cost attribution confusion (Ollama zero-line renders as $0.0000). Haiku fallback rows are distinct and can be eyeballed for "how often did I fall back this week."
  - **Positive.** `days > 14` branch keeps large queries bounded in output.
  - **Negative.** The token-count extension to `organize.nudge` detail is a silent-ish shape change to existing audit rows. Audit consumers that parse `detail_json` and don't tolerate extra keys would break — but the only consumer is `/audit` (human reader). Documented; no migration.

## Item 3 — `.trash/` TTL eviction

### 9. Config knob — `config.organize.trashTtlDays`

**Status:** Accepted.

**Context.** The TTL is a single scalar. Zod enforces the bounds.

**Decision.** Extend `src/config/schema.ts` `OrganizeConfigSchema` (currently contains only `reminders`, at schema.ts:361):

```typescript
organize: z
  .object({
    reminders: OrganizeRemindersSchema.default({}),
    // v1.11.0 — .trash/ eviction TTL (days). Default 30. The sibling evictor
    // module runs a 4am cron that hard-deletes trashed items older than this.
    trashTtlDays: z.number().int().min(1).max(365).default(30),
    // v1.11.0 — cron expression for the trash evictor. Server-local time.
    // Default 4am daily. Override only if you need a different quiet window.
    trashEvictCron: z.string().default('0 4 * * *'),
  })
  .default({}),
```

The cron expression is configurable so a deployment can move the eviction time without a code change. Defaulted to 4am to avoid the reminder-tick boundaries (reminders fire at even hours 8–22 per `cronExpression: '0 8-20/2 * * *'` schema.ts:366).

**Consequences.**

  - **Positive.** Single-knob TTL. Operator-tunable.
  - **Positive.** Operator can reschedule the evictor if 4am conflicts with backups.
  - **Negative.** One more config field. Trivial.

### 10. `evictExpiredTrash` function shape and concurrency at 100 users

**Status:** Accepted — sequential scan; document wall-time envelope.

**Context.** The brief's rigor-ask: "single-threaded scan (simpler, 4am window is wide) vs. `tickConcurrency: 5` pattern from v1.10.0. For O(100) items × 100 users = ~10k readFile calls. Acceptable?"

Wall-time math at the feared scale (100 users × 100 trash items each, sequential):

  - Per item: readFile + parse + (conditional) stat + unlink. On SSD, ~1-3ms per file end-to-end. Parse is cheap (front-matter only).
  - Per user: 100 items × 3ms = 300ms.
  - 100 users × 300ms = 30 seconds.

30 seconds sequential is fine at 4am. There's no downstream load — no LLM calls, no Telegram traffic, no competing cron (reminders fires at 04 is not an hour in the default, so no conflict). Even a pessimistic 10x (30ms per item due to contention with pm2's own I/O) yields 5 minutes, still inside the 4am–8am-next-reminder gap.

The `tickConcurrency: 5` sliding-window pool from reminders.ts:874 is the right pattern when wall-time pressure matters (2-hour cron intervals with LLM calls). Trash eviction has neither constraint. Adding a pool here is over-engineering.

**Decision.** Sequential scan. Function signature in `src/organize/storage.ts`:

```typescript
/**
 * Hard-delete trashed items for a user whose age exceeds ttlDays.
 *
 * Age source priority:
 *   1. Parsed front-matter deletedAt (new in v1.11.0)
 *   2. fs.stat(path).mtime (fallback for legacy trash with no deletedAt)
 *
 * Tolerant: malformed front-matter → falls to mtime. Stat failure → skip
 * and record in errors array. Unlink failure → skip and record. Never throws.
 *
 * Idempotent: running twice back-to-back produces no second eviction for the
 * same files (first pass deleted them; second pass sees an empty trash).
 *
 * @returns {evicted: count of files unlinked, errors: per-file error tuples}
 */
export async function evictExpiredTrash(
  userId: number,
  dataDir: string,
  ttlDays: number,
  now: Date = new Date(),
): Promise<{ evicted: number; errors: Array<{ path: string; err: string }> }>;
```

Inner loop: `readdir(.trash/)` → for each `.md` file → try `readFile + parseItemFile` → read `frontMatter.deletedAt` → if absent, `fs.stat.mtime` → compute age → if age >= ttlDays, `fs.unlink`. Errors accumulate; non-fatal.

The sibling cron loop (`evictAllUsers` in `trashEvictor.ts`) iterates users sequentially and sums the per-user results. One audit emission per user with a non-zero batch (see decision 11).

**Wall-time envelope (documented for ops).** At deployed scale (1 user, <~10 trashed items), <1 second. At "feared" 100 users × 100 items each, <30 seconds. Graceful stop signal via `deps.abortSignal?.aborted` checked at user-boundary (same pattern as reminders). `stop()` during eviction → current user finishes → next user's iteration check bails.

**Consequences.**

  - **Positive.** Simplest code that meets the scale requirements.
  - **Positive.** Mirrors the v1.9.0 original sequential `tickAllUsers` pre-v1.10.0 — proven pattern at single-digit-user scale, graceful to extend later if needed.
  - **Negative.** If someone deploys at 10,000 users per process (beyond any current plan), this becomes a problem. Out-of-scope; document as a known limit in `TrashEvictorApi`'s doc-comment.

### 11. Audit emission — `organize.trash.evict` with no-op skipping

**Status:** Accepted.

**Context.** The brief specified: emit `organize.trash.evict` per non-zero batch; 0-item batches emit no row.

**Decision.** Add `'organize.trash.evict'` to `AuditCategory`. Detail shape:

```typescript
{
  userId: number,
  evicted: number,             // always > 0 per the emission rule
  errors: number,              // count only; individual errors go to log.warn
  ttlDays: number,
  elapsedMs: number,

```

Actor fields: `actor_user_id = userId`, `actor_chat_id = userId`, `session_id = null`.

  - **Negative.** No audit trail that the evictor ran for a given user if it was a no-op. If that ever matters for compliance, re-emit a "ran, nothing to do" row — not today.

### 14. Authorization model — rely on Google's 403 response, no pre-check

**Status:** Accepted.

**Context.** The brief's rigor-ask: "if the agent calls `calendar_delete_event` with an eventId it didn't create, what happens? Walk the Google Calendar scope model: user-owned calendar vs. shared calendar. Decide whether a pre-check (fetch-before-delete to verify ownership) is required, or whether relying on Google's 403 response is enough."

Walking the scope model:

  - Jarvis's OAuth scope is `https://www.googleapis.com/auth/calendar` (full read-write on the user's calendars and calendars they have permissions on). The token is Boss's; the operations all run as Boss.
  - **User-owned calendar.** Boss is owner. Every event — created by Boss, by Jarvis acting as Boss, or by an external attendee who has write access and created the event inside Boss's calendar — can be deleted by Boss's token. Google does NOT gate delete by "who originally created it"; it gates by "can the requester edit this calendar." No 403 for the owner on any event in their own calendar.
  - **Shared calendar (Boss has write permission).** Same as owned — Boss can delete any event unless the calendar has "limited" sharing rules (e.g. view-only). In that case Google returns 403 at delete time.
  - **Shared calendar (Boss has read-only permission).** Google returns 403 at delete time. Relying on the 403 is correct.
  - **Event not on any calendar Boss knows about** — Google returns 404.

Pre-check (fetch-before-delete) would:

  - Cost one extra round-trip per delete.
  - Not actually prevent anything — the API call with Boss's token is authoritative; a pre-check from the SAME token telling us "yes you can see this event" doesn't prove "you can delete it." Delete permission is a separate capability and Google doesn't expose it as a boolean in the event resource.

**Decision.** No pre-check. Rely on Google's response:

  - 404 (update or delete) → translate to `EVENT_NOT_FOUND` (update) or `already-gone success` (delete), per decision 13.
  - 403 → surface as `GOOGLE_API_ERROR` with the verbatim message (Google's error body is already informative: "The user does not have the necessary permissions for this resource"). The LLM sees the message, relays to the user, user resolves at the Google permissions level.
  - Any other error → surface as `GOOGLE_API_ERROR`.

**Consequences.**

  - **Positive.** One API call per operation. No speculative reads.
  - **Positive.** Behavior matches the user's actual token scope — no Jarvis-invented authorization layer on top of Google's.
  - **Negative.** The LLM could ask the agent to delete a shared-calendar event and be told "permission denied" at exec time rather than at plan time. Acceptable — the failure message is clear and the user can then choose to ask the calendar owner to delete.

    - `organize_update` / `organize_delete` — USE when the event was created via
