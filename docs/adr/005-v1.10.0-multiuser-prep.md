# ADR 005 — v1.10.0 Multi-User Prep (Per-User Budget, Parallel Triage, Scheduler Owner, Scheduling Tool)

**Status:** Proposed (Phase 1, v1.10.0)
**Date:** 2026-04-24
**Deciders:** Architect Agent (iteration)
**Extends:** ADR 003 (`/organize` storage + tool layer), ADR 004 (/organize reminders v1.9.0) + its CP1 revisions, ADR 001 (module boundaries and SQLite schema). Does NOT supersede any prior ADR — all four items are additive, backwards-compatible changes.

This ADR records the architectural decisions for the v1.10.0 scaleup iteration. Four items ship together:

  1. **Per-user Haiku fallback budget** — split v1.9.0's global counter into per-user + global (outer-cap) layers so one user's Ollama failures can't starve another.
  2. **Parallel per-user triage** — bounded concurrency (`tickConcurrency`, default 5) replaces the sequential loop that would blow out the 2h cron interval at 50+ users.
  3. **Scheduler `owner_user_id` plumbing** — nullable column on `scheduled_tasks` + nullable pass-through through `scheduler` → `gateway.enqueueSchedulerTurn` → `agent.turn({userId})` so scheduled tasks can call `organize_*` / `memory_*` tools. Closes the v1.8.6 CP1 C2 gap.
  4. **`schedule` agent tool + `/scheduled` slash command** — the user-facing surface that actually creates tasks with an owner. Without this, Item 3 is foundational plumbing for a feature nobody can invoke.

Scope is design only — no code is in scope here. Developer agents in Phase 2 implement the decisions below without deviating; deviations require an ADR 005 addendum.

**Non-negotiables set by the iteration brief (not up for debate in this ADR):**

  - No new npm dependencies. `node-cron`, `zod`, `better-sqlite3` are already vendored.
  - No multi-tenant SQLite index for `/organize` items — still deferred (ADR 003 §1 and ADR 004 §16.12 carryover).
  - No standalone `calendar_update_event` / `calendar_delete_event` tools — still deferred.
  - No Telegram Web App UI.
  - No rewrite of the triage LLM prompt.

## Item 1 — Per-user Haiku fallback budget

### 1. Per-user field placement + dual-layer cap

**Status:** Accepted.

**Context.** v1.9.0 stored `haikuFallbacksToday` in `data/organize/.reminder-global-state.json` with `haikuFallbackMaxPerDay` default 20. At multi-user scale this fails two ways: (a) a single noisy user that causes repeated Ollama failures burns the budget for everyone, (b) legitimate one-off transient failures across N>10 users exhaust the 20 cap before lunch. The global cap is still useful as a DDoS / runaway-cost fuse, but it must not be the only gate.

**Decision.** Two counters, two caps:

  - **Per-user counter** lives in `data/organize/<userId>/.reminder-state.json` as `haikuFallbacksTodayPerUser: number`. Resets on the existing daily-reset path alongside `nudgesToday`. Gated by the existing `config.organize.reminders.haikuFallbackMaxPerDay` (default 20 — the v1.9.0 knob is REPURPOSED from global-cap to per-user-cap).
  - **Global counter** retains its position in `data/organize/.reminder-global-state.json` as `haikuFallbacksToday: number` (unchanged field name). Gated by a NEW knob `config.organize.reminders.globalHaikuFallbackMaxPerDay` (default 500 — 25× the per-user cap, sized to absorb a noisy day across the expected user population without becoming an independent per-user scaling bottleneck).

Gate order in `triageForUser` (§18.2): per-user check first (cheap — state is already loaded and in-memory), then global check (requires the globalState read already done once per tick), then make the Haiku call, then increment BOTH counters, then persist BOTH files.

**Consequences.**

  - **Positive.** One noisy user exhausts their own per-user budget but leaves the global outer cap intact; other users continue to get Haiku fallback during the same day.
  - **Positive.** The outer global cap remains a fuse against pathological patterns (e.g. 500 users × 20 fallbacks = 10,000 Haiku calls per day, which is a runaway-cost event that the global cap of 500 catches).
  - **Positive.** Repurposing `haikuFallbackMaxPerDay` (instead of renaming it) avoids a config migration — existing single-user deployments retain their intent ("no more than 20 fallbacks per day") at the per-user layer.
  - **Negative.** Two files must be written atomically per fallback. The per-user file write was already in the hot path; the global write is already there. Order: per-user write first (`writeReminderState` — atomic rename), global write second (`writeGlobalState` — atomic rename). If the second throws, the per-user counter is ahead-of-global; on the next read the global counter is lower than the sum of per-user counters. Acceptable: the next tick's checks use whichever value was persisted. Global drift is not a correctness problem — the outer cap is a fuse, not an accounting ledger.

### 2. Migration posture for existing `.reminder-global-state.json`

**Status:** Accepted.

**Context.** Deployed installs have a `haikuFallbacksToday` field in global state that represented the global counter under v1.9.0's single-layer model. On first load after the v1.10.0 upgrade, this field's value must be interpreted correctly: it is the global counter (unchanged semantics), NOT a per-user value (which doesn't exist on disk yet — per-user files lack the new `haikuFallbacksTodayPerUser` field).

**Decision.** No migration work. The global state file's `haikuFallbacksToday` field retains its field name and semantics — it's the global counter that we now gate against `globalHaikuFallbackMaxPerDay`. Per-user `.reminder-state.json` files gain `haikuFallbacksTodayPerUser` via zod's `.default(0)` — existing files parse without breaking (the new field fills in on first load; first write persists it). One day of stale global-counter value is fine: tomorrow it resets via the existing date-based daily-reset in `loadGlobalState`.

**Consequences.**

  - **Positive.** Zero-downtime upgrade. No backfill needed. No schema version bump on either JSON file.
  - **Negative.** For the first day post-upgrade, the global counter's pre-upgrade value may cause one user's fallbacks to count against it — briefly conflating v1.9.0 "global-only" accounting with v1.10.0 "global-outer-cap" accounting. Low impact: the cap is 500, and the pre-upgrade value is almost certainly much smaller.

### 3. Audit-log extension for cost attribution

**Status:** Accepted.

**Context.** `organize.nudge` detail currently includes `nudgesToday` (per-user) and `haikuFallbacksToday` (global, from v1.9.0). Under v1.10.0's dual-layer model, the field's meaning becomes ambiguous unless renamed or split.

**Decision.** `organize.nudge` detail is extended as follows:

  - `haikuFallbacksToday` (existing field) is REPURPOSED: its value is now the PER-USER counter (`state.haikuFallbacksTodayPerUser`), post-increment when `fallbackUsed === true`. Field name retained to avoid churn in audit consumers.
  - `globalHaikuFallbacksToday` is ADDED: the post-increment global counter. Present on every nudge row (not just fallback rows) so a per-row `/audit organize.nudge` sweep can attribute cost over time.

**Consequences.**

  - **Positive.** Audit queries distinguish user-level cost (the repurposed field) from cross-user pressure (the new field).
  - **Negative.** Audit consumers that interpreted `haikuFallbacksToday` as "global" will now see it as "per-user." Low impact: the only consumer is the human reader of `/audit`; there is no tooling on top. Document in `docs/reviews/` as a v1.10.0 audit-shape note.

## Item 2 — Parallel per-user triage

### 4. Concurrency knob + default value

**Status:** Accepted.

**Context.** The v1.9.0 `tickAllUsers` loop is sequential:

```
for (const userId of userIds) {
  try { await tickOneUser(userId, deps); } catch { /* isolate */ }
}
```

At 50 users with occasional 60s Ollama cold-starts, wall-clock tick time can exceed the 2h cron interval, triggering the v1.9.1 `wallTimeWarnRatio` warn and eventually the `tickInFlight` skip on the next fire. The fix is bounded concurrency: process N users at a time.

**Decision.** `config.organize.reminders.tickConcurrency: number`, default **5**, min 1, max 20. Clamped in zod at the schema level. The 5 default:

  - At single-user scale: no behavior change — only one user to process, so effective concurrency is 1.
  - At 10 users: roughly half the wall-time of sequential, with only 5 LLM calls in flight concurrently.
  - At 50 users + 5 concurrency + 60s avg tick: ~600s (10 min) worst-case tick wall-time, well under the 2h cron interval.
  - Upper bound 20: rationale — Ollama Cloud rate-limit observed at ~30 RPM on our free tier, so 20 concurrent calls × potential retries can start 429-ing. Haiku has higher headroom, but the primary path is Ollama. Capping at 20 keeps the module from becoming the rate-limit violator.

Re-tune conditions (documented for ops):

  - If ticks routinely hit the `wallTimeWarnRatio` warn despite concurrency: raise toward 10–15.
  - If `organize.nudge` audit rows start showing `reason: 'rate-limit'` clustered within one tick: lower toward 2–3.
  - Multi-tenant commercial operation (out of scope today): reconsider entirely — probably needs a separate provider-facing rate-limiter module, not just this knob.

**Consequences.**

  - **Positive.** Wall-time bounded; one user's cold-start no longer blocks N-1 other users.
  - **Negative.** Five concurrent Haiku fallbacks cost 5× the per-second rate of a single fallback. Cost bounded by the daily caps (Item 1); instantaneous spike is a non-issue.

### 5. Pool shape — sliding-window vs slice-batches

**Status:** Accepted (sliding-window pool).

**Context.** Two candidate shapes:

  - **(a) Slice batches** — `for (let i = 0; i < users.length; i += N) await Promise.allSettled(users.slice(i, i+N).map(tick))`. Simple to write; predictable batch boundaries; trivial to reason about in tests. Downside: a batch waits for its slowest member before starting the next batch, so throughput degrades when tail latency is high.
  - **(b) Sliding-window pool** — keep exactly N promises in flight at all times; as each resolves, start the next user. Better throughput (no tail-latency dip). Slightly more code.

**Decision.** (b) sliding-window pool. Shape:

```typescript
async function runPool<T>(
  items: T[],
  concurrency: number,
  worker: (item: T) => Promise<void>,
): Promise<{ processed: number; failed: number }> {
  let idx = 0;
  let processed = 0;
  let failed = 0;
  const inFlight: Promise<void>[] = [];

  const startNext = (): Promise<void> | null => {
    if (idx >= items.length) return null;
    const item = items[idx++];
    return worker(item)
      .then(() => { processed++; })
      .catch(() => { failed++; }); // per-user error isolation
  };

  // Prime the pool
  for (let i = 0; i < Math.min(concurrency, items.length); i++) {
    const p = startNext();
    if (p) inFlight.push(p);
  }

  // Drain: wait for any to settle, then start another until exhausted
  while (inFlight.length > 0) {
    await Promise.race(inFlight.map((p, i) => p.then(() => i)));
    // remove settled; note: since Promise.race returns the first SETTLED index,
    // we need to track which promises have resolved. Implementation uses a
    // Set<Promise> with per-promise .finally(() => set.delete(p)) instead —
    // cleaner than index juggling. See §18.3 for exact shape.
    ...
  }
  return { processed, failed };
}
```

The implementation uses a `Set<Promise<void>>` with each worker's `.finally()` removing itself, then a `Promise.race` on `Array.from(set)` to wait for any to settle. This avoids the index-juggling above. Single-file, ~25 lines, fully synchronous pool bookkeeping.

**Consequences.**

  - **Positive.** Throughput-optimal. A fast user that finishes at t=1s doesn't wait for the 90s user next to it in the batch — the next queued user starts at t=1s.
  - **Positive.** Still trivial to test: mock 15 users + concurrency=3, assert via a `getCurrentlyInFlight()` hook (test-only export) that the count never exceeds 3.
  - **Negative.** Slightly more code than slice-batches. ~10 extra lines. Acceptable for the throughput gain.

### 6. Error isolation stays per-user

**Status:** Accepted.

**Context.** v1.9.0 wraps each `tickOneUser` in try/catch so one user's throw doesn't abort the batch. With parallelism, the same property must hold — the pool's worker catches and counts failures, never propagates.

**Decision.** The pool worker is:

```typescript
async (userId: number) => {
  try {
    await tickOneUser(userId, deps);
  } catch (err) {
    log.warn({ userId, err: errMsg(err) }, 'reminders tick: per-user error, continuing');
    throw err; // let pool count it as failed; doesn't affect sibling workers
  }
}
```

The pool's `.catch(() => { failed++; })` stops propagation at the pool boundary.

**Consequences.**

  - **Positive.** Identical error semantics to v1.9.0. No new failure modes introduced by parallelism.

### 7. Global-counter mutex for concurrent increment

**Status:** Accepted.

**Context.** With N users processing in parallel, the `globalState.haikuFallbacksToday += 1; await writeGlobalState(...)` sequence is a read-modify-write race. Without a guard:

  - User A reads globalState (count=5).
  - User B reads globalState (count=5).
  - User A increments to 6, writes file.
  - User B increments to 6, writes file.
  - Correct final value: 7. Observed: 6. Lost update.

At concurrency=5 with 20 per-tick fallbacks, we can lose several increments per tick, which — over a day — makes the global counter meaningfully under-count and lets the outer cap silently fail to fire. Unacceptable for a cost-control fuse.

Three candidate patterns:

  - **(a) Simple async mutex** — serialize the entire read-modify-write sequence behind a `Promise`-chained lock. Easy to understand; locks are short (~5ms in-memory + ~5ms disk write).
  - **(b) Atomic increment helper** — same as (a) but wrapped in a named helper (`incrementGlobalHaikuFallback`). Semantically identical; better encapsulation.
  - **(c) Last-write-wins with periodic reconciliation** — accept lost increments; periodically recompute from per-user counters. Noisy, complex, not recommended.

**Decision.** (b) — an atomic-increment helper `incrementGlobalHaikuFallback(dataDir): Promise<GlobalReminderState>` in `src/organize/reminderState.ts`. Internal structure:

```typescript
// Module-level mutex — one lock per process. Jarvis is single-process today;
// multi-process deployments deferred to proper-lockfile per v1.9.0 TODO.
let globalStateLock: Promise<void> = Promise.resolve();

export async function incrementGlobalHaikuFallback(
  dataDir: string,
): Promise<GlobalReminderState> {
  const next = globalStateLock.then(async () => {
    const state = await loadGlobalState(dataDir);
    state.haikuFallbacksToday += 1;
    state.totalTicksToday += 1; // v1.9.0 already increments this alongside
    await writeGlobalState(dataDir, state);
    return state;
  });
  // Set the lock to the tail of this operation; subsequent callers await it.
  globalStateLock = next.then(() => undefined, () => undefined);
  return next;
}
```

Hot path: lock acquire + loadGlobalState (one readFile + JSON.parse, ~2ms) + mutation + writeGlobalState (one writeFile + rename, ~5ms). Total ~7ms under lock. With concurrency=5 and 20 fallbacks per tick, worst-case serialized delay is ~140ms added to a ~60s tick — <0.3% overhead. Does NOT serialize the whole tick — only the narrow global-counter path. The per-user state write is outside this lock and runs concurrently.

The lock instance lives module-scoped in `reminderState.ts`, exported via a test hook `_resetGlobalStateLockForTests()` so unit tests can reset between cases.

**Consequences.**

  - **Positive.** Lost-update impossible within a single process. The final counter value after N fallbacks is exactly N.
  - **Positive.** Narrow lock. Per-user state writes, triage LLM calls, message deliveries — all run outside the mutex.
  - **Negative.** Not multi-process safe. Jarvis runs single-process today (one pm2 process, one node); the v1.9.0 TODO for `proper-lockfile` multi-process safety still stands and is now slightly more relevant. Documented in §18.2 open boundaries.

### 8. Observability — concurrency in the tick-complete log

**Status:** Accepted.

**Context.** The existing `tick complete` log line reports `usersProcessed`, `failed`, `elapsedMs`. With concurrency added, a debugger reading logs needs to know which concurrency level ran.

**Decision.** Extend the existing `log.info({ usersProcessed, failed, elapsedMs }, 'tick complete')` call to include `concurrency: tickConcurrencyValue`. No new log line; one more field on an existing row. The `wallTimeWarnRatio` warn gets the same field.

**Consequences.**

  - **Positive.** Ops can grep `tick complete` and see historical concurrency values if the config is tuned over time.

## Item 3 — Scheduler `owner_user_id` plumbing

### 9. ALTER TABLE + NULL vs data-migration

**Status:** Accepted (ALTER + NULL).

**Context.** `scheduled_tasks` is a live table with an index on `status`. Two options for adding owner_user_id:

  - **(a) ALTER TABLE ADD COLUMN nullable** — zero-downtime, fast, no data movement. Legacy rows get NULL. New code checks for null and falls back to "no owner" behavior.
  - **(b) New table + backfill + rename** — transactional, allows NOT NULL constraint. Overkill for a nullable-by-design column.

**Decision.** (a). Migration `009_owner_user_id_scheduled_tasks.sql`:

```sql
-- Migration 009: owner_user_id plumbing for scheduled tasks (v1.10.0)
-- Closes v1.8.6 CP1 C2: scheduler-originated turns can now carry a userId
-- so they can call organize_* / memory_* tools. NULL = legacy owner-less task.

ALTER TABLE scheduled_tasks ADD COLUMN owner_user_id INTEGER;

CREATE INDEX IF NOT EXISTS idx_tasks_owner_user
  ON scheduled_tasks(owner_user_id);
```

Why the index: future "list this user's scheduled tasks" queries (`/scheduled` command — Item 4) need it. SQLite scans cost ~N today (negligible at tens of tasks) but the index is cheap to create up-front and the `/scheduled` slash command is ONE of the two use cases that justifies the column existing at all.

In the brief's framing: **zero legacy rows exist today** because there is currently no task-creator path — the scheduler reads `scheduled_tasks` that the operator inserts manually (rare), but in practice the deployed instance has an empty `scheduled_tasks` table. Still: the migration is designed to be safe even if rows existed.

**Consequences.**

  - **Positive.** Zero-downtime upgrade. Migration is idempotent per the existing `schema_migrations` tracking.
  - **Positive.** Index supports Item 4's `listByOwner` query without a follow-up migration.
  - **Negative.** Nullable column means every consumer must handle `null`. Covered in decisions 10–12.

### 10. TypeScript repo changes

**Status:** Accepted.

**Context.** `ScheduledTask` interface and `ScheduledTasksRepo` live in `src/memory/scheduledTasks.ts`. Callers: `src/scheduler/index.ts` (cron registration), `src/gateway/index.ts` (nothing today — but `enqueueSchedulerTurn` is the receiving end), future `src/tools/schedule.ts` (Item 4), future `src/commands/scheduled.ts` (Item 4).

**Decision.** The repo and type gain:

```typescript
export interface ScheduledTask {
  id: number;
  description: string;
  cron_expression: string;
  command: string;
  chat_id: number;
  owner_user_id: number | null;         // NEW — NULL for legacy or owner-less
  last_run_at: string | null;
  next_run_at: string | null;
  status: 'active' | 'paused';
  created_at: string;
  updated_at: string;
}

insert(params: {
  description: string;
  cron_expression: string;
  command: string;
  chat_id: number;
  owner_user_id?: number | null;        // NEW — optional; defaults NULL
}): number;

listByOwner(ownerUserId: number): ScheduledTask[];   // NEW
setStatus(id: number, status: 'active' | 'paused'): void;  // existing; unchanged
listActive(): ScheduledTask[];          // unchanged (returns the new field)
```

The `listByOwner` helper is used by `/scheduled` (Item 4). Prepared statement:

```sql
SELECT * FROM scheduled_tasks WHERE owner_user_id = ? ORDER BY id ASC
```

Note: `listByOwner(userId)` does NOT filter by status — the command surface shows both active and paused tasks for the user. The status filter (if any) is applied in the command layer.

**Consequences.**

  - **Positive.** Backwards-compatible: `insert` without `owner_user_id` defaults NULL — existing callers (if any) don't break.
  - **Positive.** `listByOwner` is a clean, single-query path for the `/scheduled` command.

### 11. Scheduler fire path — carry owner_user_id end-to-end

**Status:** Accepted.

**Context.** Today `src/scheduler/index.ts` calls `enqueueSchedulerTurn({chatId, taskId, description, command})`. The agent turn inside `gateway.enqueueSchedulerTurn` calls `agent.turn(...)` WITHOUT `userId`, which is why `organize_*` / `memory_*` tools fail with `NO_USER_ID`.

**Decision.** The full path gets an `ownerUserId: number | null` field:

  - `src/scheduler/index.ts`: `registerTask(task)` reads `task.owner_user_id` (now on `ScheduledTask`). `enqueueSchedulerTurn({chatId, taskId, description, command, ownerUserId: task.owner_user_id})`.
  - `src/gateway/index.ts`: `enqueueSchedulerTurn` signature becomes `{chatId, taskId, description, command, ownerUserId: number | null}`. Inside the queued run function, `agent.turn({chatId, sessionId, userText: command, abortSignal, telegram: telegramAdapter, userId: ownerUserId ?? undefined})`.
  - `src/agent/index.ts`: `TurnParams.userId?: number` is already optional (v1.7.5). No change needed; the existing role resolution + session scoping handle undefined userId as the "no-role-context" case.

**Consequences.**

  - **Positive.** Scheduled tasks with an owner now CAN invoke `organize_*` / `memory_*` tools — those tools check `ctx.userId`, which flows through from `TurnParams.userId`.
  - **Positive.** Legacy tasks (owner_user_id=NULL) behave EXACTLY as v1.9.x — `userId` stays undefined, `organize_*` tools return `NO_USER_ID`. No regression.

### 12. Allowlist re-check at fire time (authorization drift)

**Status:** Accepted.

**Context.** QA rigor ask: between task creation and fire time, the owner can be removed from `config.telegram.allowedUserIds`. Without a re-check, the scheduler would fire a task for a no-longer-authorized user, potentially DMing them and (worse) running with their stale authority. The fire path must re-validate.

**Decision.** In `src/scheduler/index.ts:registerTask`, BEFORE calling `enqueueSchedulerTurn`, synchronously validate:

```typescript
// Inside the cron-fire callback, AFTER markRan but BEFORE enqueueSchedulerTurn
if (task.owner_user_id !== null) {
  const allowed = deps.config.telegram.allowedUserIds.includes(task.owner_user_id);
  if (!allowed) {
    log.warn(
      { taskId: task.id, ownerUserId: task.owner_user_id, reason: 'owner-not-allowed' },
      'Scheduled task owner no longer on allowlist, skipping fire',
    );
    deps.memory.auditLog.insert({
      category: 'admin_command',
      actor_user_id: task.owner_user_id,
      actor_chat_id: task.chat_id,
      session_id: null,
      detail: {
        tool: 'scheduler.drop_unauthorized_owner',
        taskId: task.id,
        description: task.description,
        reason: 'owner-not-in-allowedUserIds',
      },
    });
    // User-visible notice — DM them
    try {
      // Use bot.api via adapter; scheduler today has no direct adapter ref,
      // so this notice path goes via enqueueSchedulerTurn with a short-circuit
      // marker. See §18.4 for exact mechanism.
      deps.adapter?.sendMessage(
        task.chat_id,
        `Scheduled task "${task.description}" skipped: owner no longer authorized.`,
      ).catch(() => { /* non-fatal */ });
    } catch { /* non-fatal */ }
    return; // do NOT enqueue the turn
  }
}
```

The check is O(N) over `allowedUserIds.length`, which is single-digit in the deployed config — effectively free (~microseconds). The `audit_log` row is inserted synchronously (the audit repo is in-process). The DM is fire-and-forget so a send failure doesn't block the cron handler.

Why the audit category is `admin_command` with a distinguishing `detail.tool`:
  - Adding a new `AuditCategory` string value like `'scheduler.drop_unauthorized_owner'` is overkill for one code path and bloats the type.
  - `admin_command` is the pre-existing category for admin-adjacent bookkeeping rows (see `auditLog.ts:13`). Reusing it with a descriptive `detail.tool` value is the conventional pattern.

Adapter plumbing: the scheduler does not currently hold a `MessagingAdapter` reference. The `SchedulerDeps` interface is extended to accept `adapter: MessagingAdapter | null` (nullable so tests without an adapter work). `src/index.ts` passes `gateway.adapter` (getter from ADR 004 §2) into `initScheduler`.

**Consequences.**

  - **Positive.** Closes the authorization-drift window. A revoked user's tasks stop firing the next time the cron rolls, with an auditable record and a user notice.
  - **Positive.** Scheduler gains MessagingAdapter access via dependency injection — same pattern as the reminders module (ADR 004 §2).
  - **Negative.** Scheduler now depends on messaging adapter — one more edge on the dependency graph. Acceptable per §13 module boundaries — same shape as reminders, and the dependency is structurally typed (just `sendMessage`), not coupled to a concrete implementation.

### 13. Legacy task error message update

**Status:** Accepted.

**Context.** When a legacy task (owner_user_id=NULL) fires and the command triggers an `organize_*` or `memory_*` tool, the tool returns `ok:false, error.code: 'NO_USER_ID'`. v1.8.6's message reads "Cannot update memory: speaker user id is not available for this turn." — no hint about WHY, for a scheduled-task-induced failure.

**Decision.** Each tool that checks `ctx.userId` and returns `NO_USER_ID` gets its `error.message` updated to include a scheduled-task hint:

```
"Cannot [update memory / access organize items]: this task was created before v1.10.0 and has no owner. Recreate the task with the schedule tool to tie it to you, or run this command interactively."
```

Verbatim copy applies to: `update_memory`, `forget_memory`, `organize_create`, `organize_update`, `organize_complete`, `organize_list`, `organize_log_progress`, `organize_delete`. These are the tools that currently return `NO_USER_ID`.

**Consequences.**

  - **Positive.** Actionable error message — users know exactly how to fix a failed legacy task.
  - **Negative.** Minor duplication of the hint across 8 tool files. Acceptable — the string is ~50 words; a shared constant can be extracted in a follow-up.

### 14. System-prompt rule 11 amendment (scheduled-task organize gap closed)

**Status:** Accepted.

**Context.** v1.9.0's rule 11 (in `config/system-prompt.md`) reads:

> **Scheduled tasks cannot use `/organize`** — scheduler-originated turns have no user identity, so `organize_*` tools return `NO_USER_ID`. If a user schedules "8am daily: list my tasks," it will fail every fire in v1.8.6; tell the user to interact with `/organize` interactively instead.

Under v1.10.0, the statement is partially false — tasks CREATED in v1.10.0+ via the new `schedule` tool carry an owner and CAN use `organize_*` / `memory_*`. Legacy tasks still can't.

**Decision.** Rule 11's scheduled-task paragraph is replaced verbatim with:

> **Scheduled tasks and `/organize` / memory (v1.10.0):** Scheduled tasks created in v1.10.0 or later via the `schedule` tool carry an owner user id. When they fire, the agent turn runs with that user's identity, so `organize_*` and `memory_*` tools work the same as an interactive DM turn — results are scoped to the owner. **Legacy tasks created before v1.10.0 have no owner** (their `owner_user_id` column is NULL); those tasks still can't use `organize_*` / `memory_*` and will return `NO_USER_ID` with a message asking the user to recreate the task via the `schedule` tool. When a user asks "why did my 8am task fail?" and the error is `NO_USER_ID`, relay the message and offer to recreate via `schedule`.

**Consequences.**

  - **Positive.** The agent now knows when to suggest task recreation vs. when to suggest interactive interaction.

## Item 4 — `schedule` agent tool + `/scheduled` slash command

### 15. Agent tool surface

**Status:** Accepted.

**Context.** Without a user-facing way to create scheduled tasks, Item 3's plumbing is foundational for a feature nobody can invoke. The tool must:

  - Accept a description, cron expression, and command text.
  - Validate the cron expression before accepting.
  - Tie the task to the speaker's userId (read from `ctx.userId`).
  - Trigger the scheduler to pick up the new task without a process restart.
  - Be available to all users (not admin-only) — scheduling is a per-user productivity feature, parallel to `/organize` which is also non-admin.

**Decision.** New tool `src/tools/schedule.ts` with this shape (mirrors `src/tools/update_memory.ts` closely):

```typescript
const ScheduleInput = z.object({
  description: z.string().min(3).max(200).describe(
    'A short human label for the task (≤200 chars). Shown in /scheduled listings and in the DM notice when the task fires. Example: "morning goal check".',
  ),
  cron: z.string().min(1).max(100).refine(
    (expr) => cron.validate(expr),
    'Invalid cron expression. Use standard node-cron syntax: "M H D M DOW". Example: "0 8 * * *" for 8am daily.',
  ).describe(
    'Cron schedule in node-cron syntax. "0 8 * * *" = 8:00 daily, "*/15 * * * *" = every 15 min, "0 9 * * 1-5" = 9am weekdays.',
  ),
  command: z.string().min(1).max(2000).describe(
    'The prompt Jarvis runs when the task fires. It runs as if the owner typed it. Example: "list my active organize items", "check my email and summarize anything urgent". Tools marked adminOnly run only if the owner is an admin.',
  ),
});

export const scheduleTool: Tool<z.infer<typeof ScheduleInput>> = {
  name: 'schedule',
  description:
    'Create a scheduled recurring task. The task will fire at the specified cron time as if the USER typed the command. ' +
    'Use when the user explicitly says "remind me every …", "schedule X daily at Y", "every morning check …". ' +
    'Do NOT use for one-off immediate actions — those, just do them. ' +
    'The task carries the user\'s identity at fire time, so it has access to the same tools the user does interactively (organize_*, memory_*, calendar_*, etc.). ' +
    'Validation: cron expression must parse; description ≤200 chars; command ≤2000 chars.',
  parameters: ScheduleInput,
  adminOnly: false,

  async execute(input, ctx: ToolContext): Promise<ToolResult> {
    if (!ctx.userId || !Number.isFinite(ctx.userId)) {
      return {
        ok: false,
        output: 'Cannot schedule: speaker user id is not available for this turn.',
        error: { code: 'NO_USER_ID', message: 'ctx.userId missing' },
      };
    }

    const taskId = ctx.memory.scheduledTasks.insert({
      description: input.description,
      cron_expression: input.cron,
      command: input.command,
      chat_id: ctx.chatId,
      owner_user_id: ctx.userId,
    });

    // Audit
    ctx.memory.auditLog.insert({
      category: 'admin_command',
      actor_user_id: ctx.userId,
      actor_chat_id: ctx.chatId,
      session_id: ctx.sessionId,
      detail: {
        tool: 'schedule',
        taskId,
        description: input.description,
        cron: input.cron,
        commandPreview: input.command.slice(0, 100),
      },
    });

    // Reload so the new task fires without a process restart.
    // If the scheduler is not wired in (tests, old callers), gracefully no-op.
    ctx.deps?.schedulerApi?.reload();

    // Humanize next-fire if possible
    const nextFire = computeNextFire(input.cron); // null on parse failure
    const humanized = nextFire ? ` — fires next at ${nextFire.toISOString()}` : '';

    return {
      ok: true,
      output: `Scheduled: "${input.description}" — cron "${input.cron}"${humanized}.`,
      data: { taskId, description: input.description, cron: input.cron, nextFire: nextFire?.toISOString() ?? null },
    };
  },
};
```

Wiring the `schedulerApi` into `ctx`: today `ToolContext` doesn't include a scheduler handle. Rather than widening every tool's context, a narrowly-scoped `ctx.deps?: { schedulerApi?: SchedulerApi }` object is added — `deps` is optional on `ToolContext`, and `schedulerApi` is optional inside it. `src/agent/index.ts` when constructing the context reads the scheduler handle off a module-scoped late-binding reference (same pattern as `remindersApi` in `src/gateway/index.ts`).

`computeNextFire` implementation: node-cron ≥3.0 exposes `cron.validate` but not a next-fire computation. A minimal parser for the standard 5-field cron shape lives inline in `src/tools/schedule.ts` as a helper — returns `null` on anything non-trivial (e.g. cron expressions with special chars like `L`, `W`, `#`). Users see the raw cron echoed back in that case, which is acceptable for v1.10.0. Non-goal: reimplementing a full cron parser.

**Group-chat / non-DM consideration:** the tool is NOT admin-only but group chats should NOT be able to create per-user scheduled tasks — a scheduled task delivery goes to the chat where it was created (`chat_id`), so a group-chat-created task fires into that group chat. The tool's dispatch is gated by the existing group-mode tool filter (`config.groups.disabledTools` — `schedule` is ADDED to the default list). Admins can override per-chat via the existing per-group role system. DM-only by default is the v1.10.0 posture; group-task-scheduling is a future iteration.

**Consequences.**

  - **Positive.** Users can now create scheduled tasks by asking: "remind me every day at 8am to check my goals" → agent calls `schedule({description: 'morning goal check', cron: '0 8 * * *', command: 'list my active organize items and highlight anything due soon'})`.
  - **Positive.** Tasks carry owner_user_id automatically, so Item 3's plumbing works end-to-end on first use.
  - **Negative.** The inline cron-next-fire helper is a small re-implementation of functionality that node-cron doesn't export. ~20 lines; tested in isolation.

### 16. Slash command `/scheduled` (plural, not `/schedule`)

**Status:** Accepted.

**Context.** Two naming options:

  - `/schedule` — symmetric with the tool name. Risk: user confusion (same word, two different surfaces — one agent-tool, one slash-command).
  - `/scheduled` — reads as "manage my scheduled tasks." Distinct from the tool.

**Decision.** `/scheduled`. Mirror of the `/memory` / `/organize` pattern: command surface manages the corpus, tool writes new entries. Subcommands:

  - `/scheduled` (bare) — list this user's tasks via `memory.scheduledTasks.listByOwner(ctx.from.id)`. Output format:
    ```
    Your scheduled tasks:
    [123] morning goal check — "0 8 * * *" — active — last run: 2026-04-24 08:00
    [124] weekly email summary — "0 9 * * 1" — paused — last run: never
    ```
  - `/scheduled pause <id>` — set status to 'paused'. Ownership check: user can only pause their own tasks. Admins override.
  - `/scheduled resume <id>` — set status to 'active'. Same ownership check. Calls `schedulerApi.reload()` after status change (paused→active must re-register the cron job).
  - `/scheduled delete <id>` — two-step confirmation (requires `CONFIRM`, mirrors `/memory clear`). Ownership check as above. Calls `schedulerApi.reload()` after deletion.
  - `/scheduled show <id>` — full details including the `command` text (useful for debugging). Ownership check.

Ownership logic:

```typescript
const task = memory.scheduledTasks.get(taskId);  // new helper, one-row lookup
if (!task) return 'No task with that id.';

const role = resolveRole({chatId: ctx.chat.id, userId: ctx.from.id, chatType: ctx.chat.type}, config);
const isAdmin = role === 'admin';
const isOwner = task.owner_user_id === ctx.from.id;
if (!isAdmin && !isOwner) return 'You can only manage tasks you created.';
```

DM-only surface: same posture as `/organize`. Group-chat invocations reply with "DM me to manage scheduled tasks." No state mutation in groups.

Integration with scheduler reload:
  - `pause` / `resume` / `delete` call `schedulerApi.reload()` after the DB mutation. `reload()` is idempotent (it stops all jobs and re-registers from the current DB state).
  - `show` / bare `/scheduled` are read-only — no reload.

`memory.scheduledTasks.get(id)` is a new repo helper — one-row SELECT. Added to keep `/scheduled` subcommands simple without over-indexing on list + filter.

**Consequences.**

  - **Positive.** Full lifecycle surface (create via tool, list / pause / resume / delete / show via slash) with ownership + admin override.
  - **Positive.** Confirmation gate on `/scheduled delete <id>` reuses the proven two-step pattern from `/memory clear CONFIRM`.
  - **Negative.** The `get(id)` helper adds one more prepared statement to `ScheduledTasksRepo`. Tiny addition.

### 17. System-prompt rule 12 (new) — when to call `schedule`

**Status:** Accepted.

**Context.** Without instruction, the LLM will often misfire — treating "check my goals right now" as a scheduling request, or treating "remind me to check my goals every morning" as an immediate task. The system prompt needs a disambiguation rule.

**Decision.** Append a new rule 12 to `config/system-prompt.md` (verbatim for the dev agent to copy; placed AFTER existing rule 11):

> 12. **Scheduling rules (v1.10.0).** The `schedule` tool creates a recurring task that fires on a cron schedule as if the user typed the command. The `/scheduled` slash command (plural) manages existing tasks (list/pause/resume/delete/show).
>
>     **WHEN to call `schedule`:**
>     - The user gives a recurring intent: "remind me every morning to check my goals", "every Monday at 9am summarize last week's email", "daily at 8pm, check what's due tomorrow".
>     - There's a time / cadence explicitly stated: "every day", "every week", "at 8am", "on weekdays".
>     - The command is something the user wants you to do WITHOUT them being present.
>
>     **When NOT to call `schedule`:**
>     - The user says "do X now": just do it. "List my goals" → call `organize_list`, don't schedule it.
>     - One-off future time ("remind me tomorrow at 3pm") — today, `schedule` with a cron that fires once at that time works but is awkward; prefer creating a goal-type organize item with a due date and letting the reminder loop surface it.
>     - The user is unsure whether they want recurrence: ask. "Should this be recurring (every day / week) or a one-time note?"
>
>     **Cron syntax help:** `"0 8 * * *"` = 8am daily; `"0 9 * * 1"` = 9am Mondays; `"*/15 * * * *"` = every 15 min; `"0 0 1 * *"` = midnight on the 1st of each month. If the user gives a time without specifying which day (e.g. "8am"), default to daily (`"0 8 * * *"`) and confirm the cadence in your reply.
>
>     **What scheduled tasks CAN do (v1.10.0+):** tasks created via `schedule` carry the owner's identity, so `organize_*`, `memory_*`, `calendar_*`, `gmail_*`, and other user-scoped tools work the same as in an interactive DM turn.
>
>     **What scheduled tasks CANNOT do:** legacy tasks created BEFORE v1.10.0 have no owner — those fail `organize_*` / `memory_*` with `NO_USER_ID` until recreated via `schedule`.

Also: the v1.9.0 rule-11 scheduled-task paragraph is replaced per decision 14.

**Consequences.**

  - **Positive.** Clear LLM guidance on when to schedule vs. execute immediately.
  - **Positive.** Cron syntax pocket-reference in-prompt reduces the number of tool-call attempts that fail zod validation for cron shape.

## Open questions

**None.** The brief is heavily specified; all 17 decisions above follow directly from the brief's guidance. Items flagged by the brief for explicit treatment (migration strategy, concurrency default rationale, column duplication, allowlist re-check placement, mutex design) are addressed in decisions 9, 4, 12 (note on chatId vs userId distinction), 12 (fire-path placement), and 7 respectively.

**Explicit note on chatId vs owner_user_id overlap:** `sessions.telegram_chat_id` already exists on the `sessions` table. Is `scheduled_tasks.owner_user_id` duplicating it? No. A chat_id is a Telegram chat (DMs have one, groups have one). A user_id is a Telegram user (a group has many users sharing one chat_id; a DM has exactly one user whose user_id equals the chat_id by Telegram convention). A task fires into a chat (hence `chat_id`), but it runs AS a user (hence `owner_user_id`). For DM tasks the two are numerically equal; for group-created tasks (not shipped today but preserved as a future possibility) they differ. The columns are independent semantically even when they carry the same value today.
