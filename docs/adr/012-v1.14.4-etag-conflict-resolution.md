# ADR 012 — v1.14.4 `/organize` Web App: ETag + If-Match + 412 Conflict Resolution

**Status:** Proposed (Phase 1, v1.14.4)
**Date:** 2026-04-25
**Deciders:** Architect Agent (iteration)
**Extends:** ADR 011 + 011-revisions (v1.14.3 notes/progress/hierarchy/restore + `updated:` discipline), ADR 010 + 010-revisions (v1.14.2 mutation surface, R2-mtime stale-warning), ADR 009 + 009-revisions (v1.14.0 read path), ADR 008 + 008-revisions (webapp Express foundation), ADR 005 (multi-user posture, audit categories), ADR 003 + 003-revisions (`/organize` storage layout, filename-as-id, 200-active-item cap), ADR 001 (module boundaries).

This ADR records the architectural decisions for the v1.14.4 **ETag + If-Match + 412 conflict-resolution iteration** — the fifth in the webapp series (read v1.14.0 → hotfix v1.14.1 → mutate v1.14.2 → notes/progress/hierarchy v1.14.3 → ETag v1.14.4 [THIS]). ## 0. Context

**What v1.14.2 + v1.14.3 left open.**

v1.14.2 introduced three concurrent writer surfaces on `data/organize/<userId>/<itemId>.md` — chat-side `/organize update`, the chat-agent's `organize_update` tool, and the new webapp PATCH/POST/DELETE — and shipped without proper concurrency control. CP1's Devil's Advocate rated this **HIGH (BLOCKING-adjacent)** and demanded a v1.14.3 hard deadline (verified at `docs/reviews/cp1-devils-advocate-v1.14.2.md:54-90`, R2). The DA's compromise was R3: a non-blocking `X-Captured-Mtime` header that captures `mtimeMs` at GET time, the server compares on mutation, mismatch logs `webapp.stale_edit` audit + emits `staleWarning: true` in the response so the client shows a non-blocking toast — **"Saved, but item changed elsewhere; refresh to see other changes."** This shipped in v1.14.2 with the explicit caveat that **the mutation still proceeds — last-writer wins on the rename**. The R2-mtime mitigation is a SIGNAL, not a GATE.

v1.14.3 then deferred the proper fix to v1.14.4 (verified at ADR 011 §0 "non-negotiables": *"No ETag / If-Match / 412 in v1.14.3. Hard-deferred to v1.14.4 (per ADR 010 R2-mtime hard deadline)."*) and used the iteration to **stamp `updated:` on every storage write path** as the foundation for v1.14.4's ETag (verified at ADR 011 D1; storage.ts:359 `stampUpdated()` + every write path verified at storage.ts:453, 692, 743, 811, 1086).

**v1.14.4's job: close the race.** The race window is now load-bearing for two scenarios:

  1. **chat-vs-webapp progress race** (the "user opens webapp, agent appends progress, user saves" scenario, verified in ADR 011 SF-4 + race-window analysis at ADR 011 D3:251-260). Today: `staleWarning: true` toast; agent's appended line silently overwritten. v1.14.4: 412 Precondition Failed; user must Reload or explicit Save Anyway.
  2. **multi-device same-user edit race** (CP1 P1 scenario 1; verified DA review at `:60`). Today: silent last-write-wins on the same item from two devices. v1.14.4: same 412 / Reload / Save Anyway flow.

**The v1.14.3 foundation v1.14.4 builds on.**

  - `OrganizeFrontMatter.updated?: string | null` (verified at `types.ts:37-40`) — ISO timestamp, monotonically advances on every write.
  - `OrganizeItemDetail.updated: string | null` (verified at `types.ts:87-88`) — wire shape, nullable for legacy items.
  - `OrganizeListItem.updated: string | null` (verified at `types.ts:115-116`) — list projection.
  - `stampUpdated(fm, now?)` helper (verified at `storage.ts:359`) — pure, called by every write path.
  - Every write path verified to stamp:
    - `createItem` at `storage.ts:453` — initial stamp; `created === updated` at first write.
    - `updateItem` at `storage.ts:692` — every patch advances `updated`.
    - `softDeleteItem` rewrite path at `storage.ts:743` — stamp before .trash rename.
    - `appendProgressEntry` at `storage.ts:811` — agent's append path.
    - `restoreItem` at `storage.ts:1086` — strip `deletedAt`, fresh `updated`.

**Six items ship together in v1.14.4:**

  1. **`computeETag(fm, fileMtimeMs)` storage helper** (D7) — single source of truth in `src/organize/etag.ts`, returns `"<iso-or-mtime-fallback>"` (strong ETag, double-quoted).
  2. **ETag header on GET /:id, PATCH, POST /complete, DELETE responses** (D2) — sets `ETag: "<iso>"` on every successful single-item response.
  3. **If-Match parsing on PATCH, DELETE, POST /complete** (D3) — required-when-present (optional-for-backcompat); on mismatch, return 412 with `currentEtag` + `currentItem` body (D4); on absence, proceed without conflict check (no breaking change for older clients).
  4. **R2-mtime sunset** (D6) — remove `X-Captured-Mtime` from all three mutation handlers, remove `staleWarning` field from response, retire `webapp.stale_edit` audit category emission (keep schema for forward-compat).
  5. **Audit row metadata extension** (D10) — `webapp.item_mutate` detail JSON gains optional `etag: string | null` and `forced: boolean` for forensic visibility on Save Anyway overrides.
  6. **Client-side conflict UI** (D5 + D12) — inline (non-modal) conflict panel in the detail edit form with two buttons: **Reload** (refetch GET /:id, replace state, reset form) and **Save Anyway** (re-PATCH WITHOUT If-Match; logs forced-override audit row).

**Non-negotiables set by the iteration brief:**

  - **No new npm dependencies.** ETag computation is hand-rolled (~30 LOC); HTTP semantics are standard.
  - **No mutations to v1.14.0/v1.14.1/v1.14.2/v1.14.3 routing or auth shape.** The auth chain, allowlist guard, replay window, error envelope, and audit-debouncer hook are all reused by reference, not modified.
  - **No build tooling.** Vanilla JS only; no Vite, no React.
  - **No 3-way diff UI.** Conflict UI is two buttons (Reload / Save Anyway). 3-way diff filed for v1.16.0+ (D16).
  - **No `?strict=1` enforcement mode in v1.14.4.** Required-when-present (optional-for-backcompat); strict mode (428 Precondition Required if If-Match missing) is filed for v1.15.0+ multi-tenant scenarios (D16).
  - **No list-response ETag.** GET /api/webapp/items (list) does NOT carry an ETag header in v1.14.4 (D2).
  - **No HTTP caching / If-None-Match.** Cache-Control `no-store` stays on every webapp items response (verified `items.read.ts:44-48`); ETag is for conflict detection only, not caching.
  - **No `/organize trash list` companion command.** Filed for v1.14.5 (D14).

**Surprise findings (verified against actual code, flagged for CP1):**

  - **SF-1.** The v1.14.2 R2-mtime code in `items.mutate.ts` is more entangled than the brief implied. The `auditStaleEdit` helper at `:97-122` emits a `webapp.stale_edit` audit category WITH a `progressLineDelta` field that was added in v1.14.3 R2/CP1 for additional forensic detail when progress was being patched. **Decision 6 (sunset) must be explicit about this:** stop emitting `webapp.stale_edit` rows entirely; keep the row schema (the audit_log table is JSON-detail; no migration needed); flag the helper for deletion. Documented.

  - **SF-2.** The `restoreItem` chat command (v1.14.3 D9) is a write path NOT exposed to the webapp (it's invoked via `/organize restore <id>` in chat). v1.14.4 does NOT need to add If-Match to it — there's no webapp client of restoreItem. But: a future v1.14.x adding webapp restore-from-trash WILL need to think about ETag on restore. Documented in §Out-of-scope.

  - **SF-3.** The chat-agent's `organize_update` tool path (v1.13.x) is ALSO not subject to If-Match — it's a tool call, not an HTTP route. The race that v1.14.4 closes is webapp-vs-{webapp,chat-command,chat-agent}: the WEBAPP side enforces the ETag check; the chat side just writes (and advances `updated:`). This means: chat-agent edit followed by webapp PATCH with stale If-Match → 412 (correct). Webapp PATCH followed by chat-agent edit → chat-agent overwrites (chat-agent has no ETag awareness, by design — it's the user's authoritative agent and edits are explicitly user-instructed via tool calls). Documented in §Risks.

  - **SF-4.** Older items (pre-v1.14.3) have `frontMatter.updated === null`. The ETag for these falls back to file mtime (D1, D7). **The fallback ETag changes if the file is touched (e.g., backup-and-restore reset mtime).** This is acceptable for v1.14.4 — older items not yet edited will get their `updated:` stamped on the FIRST webapp edit (then the ETag stabilizes); the mtime-based fallback is best-effort for the transitional period. Two clients editing a pre-v1.14.3 item simultaneously will both see the same fallback ETag and the conflict-detection still fires correctly because the server reads the live file's CURRENT mtime, not the cached one. Documented in §Risks.

  - **SF-5.** The audit_log schema at v1.14.2 migration 011 stores detail as a JSON blob with category-keyed rows. Adding `etag` and `forced` to `WebappItemMutateDetail` is a NON-MIGRATION change — the JSON blob shape is forward-compatible. Verified at `items.shared.ts:179-184` (`WebappItemMutateDetail` interface) — adding two optional fields is a TS-level extension only. No schema migration needed.

  - **SF-6.** `items.mutate.ts` is currently 516 LOC. After v1.14.4 changes (sunset R2-mtime: -50 LOC; add If-Match parsing + 412 envelope: +60 LOC), the file lands at ~525 LOC — still over the soft 500-LOC threshold from v1.14.2 SF-6. The Anti-Slop Reviewer at v1.14.4 may push back on the size. Mitigation: extract a small `ifMatchCheck(req, currentEtag): {ok: true} | {ok: false, response412: ...}` helper into `items.shared.ts` (~25 LOC); this drops the route handler size and keeps the per-handler logic compact. D15 binds this.

  - **SF-7.** The webapp client at `public/webapp/organize/app.js` has the conflict-flow integration touching FIVE places: detail-fetch (capture ETag), edit-form-submit (send If-Match), POST /complete from the list (send If-Match for the optimistic checkbox path), DELETE handler (send If-Match), and a NEW conflict panel render + Reload/Save Anyway handlers. The list-card complete-checkbox is the trickiest because the list does NOT carry per-item ETags (D2). **Resolution:** the optimistic complete-from-list path does NOT send If-Match; it relies on absolute-write semantics (R18 from v1.14.2 — `done: true` always sets status=done regardless of current). 412 is emitted only on PATCH from the detail-form-edit path and on DELETE (which is also detail-form only). Documented in D9 + D12.

## Decisions

### Decision 1 — ETag format: strong ETag, value = `"<updated-iso-or-mtime-fallback>"`

**Status:** Accepted — strong ETag (no `W/` prefix); value is the double-quoted ISO string of `frontMatter.updated`, with file mtime ISO as fallback for legacy items.

**Context.** Three candidates:

  - **(a) Strong ETag, value = ISO timestamp string in quotes.** `ETag: "2026-04-25T14:30:00.000Z"`. Stable, exact-match required. The HTTP spec says strong ETag means "byte-identical resource representation" — but in practice for application-level conflict detection this is the right semantic anyway: any change to `updated:` means the resource changed.
  - **(b) Weak ETag, value = ISO timestamp.** `ETag: W/"2026-04-25T14:30:00.000Z"`. The `W/` prefix is the HTTP hint that two byte-different responses with the same ETag may be semantically equivalent. Used for HTTP caching where representation can vary (e.g., gzip vs not). NOT relevant to our conflict-detection case — we never have two byte-different responses with the same `updated:`. The `W/` prefix would be technically defensible but semantically misleading for our use case.
  - **(c) Hash-based ETag.** `ETag: "sha256:<hex>"` of the canonical content. More robust against clock-drift edge cases but more expensive (read entire file → hash). Overkill given `updated:` is already a stable monotonic anchor we control end-to-end.

**Decision.** (a). Justifications:

  1. **`updated:` is already the version anchor v1.14.3 built.** v1.14.3's whole point was to make `updated:` monotonically advance on every write. ETag = `"${updated}"` is the smallest possible mapping from version-anchor to wire-shape. Anti-Slop §5 (single-source-of-truth) — the server has ONE concept of "this item changed" and ETag exposes it verbatim.
  2. **Strong-vs-weak distinction is irrelevant for conflict detection.** Both flavors require exact equality on If-Match. The W/ prefix is a hint for caching middlebox behavior we don't have. Using it would be wrong-by-pretense.
  3. **Hash-based is YAGNI.** No real scenario where `updated:` is unreliable but the file content is reliably hashable. If `updated:` is wrong (clock drift, manual file edit), hashing the content has its own correctness issues (whitespace, line ending normalization). Stick with the version anchor we already trust.
  4. **Wire shape is human-readable.** A devtools-network-tab inspection shows `ETag: "2026-04-25T14:30:00.000Z"` — operator can read it, compare to file mtime, sanity-check immediately. Hash-based ETags are opaque.

**Fallback for older items missing `updated:`** (pre-v1.14.3 items not yet edited via any v1.14.3+ write path):

```
fallback = new Date(fileMtimeMs).toISOString()
```

Same ISO format, sourced from `fs.stat().mtimeMs`. Client doesn't need to know whether the value came from `updated:` or mtime; both are wrapped in the same double-quote envelope. Three implications:

  1. **Pre-v1.14.3 item, never edited:** ETag is mtime-based; first webapp PATCH → `stampUpdated` stamps `updated:` → next GET returns ETag from `updated:`. Smooth transition.
  2. **Pre-v1.14.3 item, editing simultaneously from two devices:** both clients see the same mtime-based ETag. First PATCH advances `updated:` AND mtime; second PATCH's stale If-Match (mtime-based) fails to match the new ETag (now `updated:`-based). 412 fires correctly.
  3. **Backup-and-restore that resets mtime on a pre-v1.14.3 item:** legacy ETag resets. Acceptable — the only conflict scenario this could miss is "two clients held stale ETags BEFORE the backup, and one of them survived the backup." Vanishingly rare in single-user trust boundary; no v1.14.4 mitigation required. Documented.

**Wire format (binding for Phase 2):**

  - Response: `ETag: "2026-04-25T14:30:00.000Z"` (HTTP standard, strong).
  - Request: `If-Match: "2026-04-25T14:30:00.000Z"` (HTTP standard).
  - Both use the standard double-quote envelope; no `W/` prefix.
  - Client sends VERBATIM whatever the server sent. Server compares byte-equal (after trim) to current ETag computed at mutation time.

**Why not (b).** W/ is a caching hint we don't use. Adding it would be wrong-by-pretense and could confuse future readers ("does this allow caching? no, why is it weak?").

**Why not (c).** Costs more (file read + hash) and protects against clock-drift cases that don't exist in our trust boundary (single-user, single-clock).

**Consequences.**

  - **Positive.** Smallest possible ETag mapping. Single source of truth. Forward-compatible — if v1.16.0 adds a versioning field beyond `updated:`, the ETag value changes but the wire shape doesn't.
  - **Positive.** Operator-readable in devtools.
  - **Negative.** ETag does not survive backup-and-restore on legacy items (mtime-based fallback). Acceptable; documented in SF-4 + §Risks.

### Decision 2 — Where ETags appear in responses

**Status:** Accepted — ETag header on GET /:id, PATCH success, POST /complete success, DELETE success. NOT on GET / (list).

**Context.** Three candidates for header placement:

  - **(a) Single-item endpoints only:** GET /:id, PATCH, POST /complete, DELETE. The endpoints whose responses are valid baselines for an EDIT operation.
  - **(b) Single-item AND list:** + GET /. Add `ETag: W/"<list-hash>"` to the list response for HTTP caching benefits.
  - **(c) None.** Pure 412-on-If-Match-mismatch with the server computing ETag on the fly. No header on success responses; client constructs If-Match from `updated:` it sees in the JSON body.

**Decision.** (a). Justifications:

  1. **List ETag is YAGNI.** GET / is a snapshot of MANY items; the list itself is not a baseline for editing. Editing happens through GET /:id which IS the baseline. A list-level weak ETag would benefit HTTP caching middleware — but our `Cache-Control: no-store` (verified `items.read.ts:44-48`) explicitly disables caching anyway. Adding a list ETag would be wire-noise without effect.
  2. **Reading the ETag from a response header is more robust than reading from the JSON body.** The header is a HTTP-protocol-level surface; clients can read it via `response.headers.get('ETag')` consistently across fetch APIs. Reading from `data.item.updated` and constructing `"${updated}"` in client code is fragile (legacy items have `updated: null`, which would break naive client code). Header-based ETag pushes the "and-fallback-to-mtime" logic to the server where it belongs.
  3. **Mutation responses set the new baseline.** After a successful PATCH, the response body returns the updated item AND the response ETag header carries the NEW ETag. The client updates its `currentDetailEtag` from the header, so the next mutation uses the fresh value. This is how HTTP semantics expect ETag to work and matches the W3C specs cleanly.
  4. **DELETE returns ETag of the just-deleted state, but client doesn't use it.** The DELETE response body is `{ok: true, deletedId: id}`; the ETag is technically meaningless after delete (the resource no longer exists). Decision: still emit it for protocol consistency but document that the client does not store it (after delete, the client clears all per-item state). Cost: 1 line in handler.

**Wire bindings (Phase 2):**

| Endpoint | ETag in response? | Notes |
|---|---|---|
| GET /api/webapp/items | NO | List is not an edit-baseline. Cache-Control no-store anyway. |
| GET /api/webapp/items/:id | YES | The canonical edit-baseline. ETag = `computeETag(fm, mtime)`. |
| PATCH /api/webapp/items/:id (200) | YES | New ETag from updated `frontMatter.updated`. Client refreshes baseline. |
| POST /api/webapp/items/:id/complete (200) | YES | Same as PATCH. |
| POST /api/webapp/items/:id/complete (no-op path; abandoned + done:false from v1.14.2 R14) | YES | Even on no-op, return current ETag — client still uses it as a baseline. |
| DELETE /api/webapp/items/:id (200) | YES | Emit for protocol consistency; client discards (item is gone). |
| 412 Precondition Failed | YES (`currentEtag` is in the BODY too — see D4) | Both header and body carry the new ETag; client uses body's `currentEtag` as the canonical reload target. |
| 4xx (400/401/403/404) | NO | Errors don't have a resource representation. |
| 5xx | NO | Same. |

**Why not (b).** No caching benefit (no-store), wire-noise without effect.

**Why not (c).** Pushes ETag construction into client code. Fragile for legacy items. Loses HTTP-protocol-conformance.

**Consequences.**

  - **Positive.** Standard HTTP semantics. Client just reads `response.headers.get('ETag')`.
  - **Positive.** Mutations refresh the baseline automatically.
  - **Negative.** DELETE emits a header the client ignores. ~1 line cost. Tolerated for protocol uniformity.

### Decision 3 — When If-Match is required vs optional: required-when-present (optional-for-backcompat)

**Status:** Accepted — If-Match is REQUIRED-WHEN-PRESENT and OPTIONAL-FOR-BACKCOMPAT. If absent, server proceeds without conflict check (matches v1.14.3 last-writer-wins behavior). If present and mismatched, 412.

**Context.** Three candidates:

  - **(a) Required (no backcompat).** All PATCH/DELETE/POST /complete MUST carry If-Match; absence → 428 Precondition Required. Breaks every existing client that hasn't deployed the v1.14.4 client.
  - **(b) Required-when-present (optional-for-backcompat).** If header is present, server checks; mismatch → 412. If header is absent, server proceeds without check. The v1.14.4 webapp client always sends; future API consumers can opt in.
  - **(c) Strict mode toggle.** Add a `?strict=1` query param OR config flag. Default off (option b behavior); toggle on activates option a (428 on absence).

**Decision.** (b) for v1.14.4 ship. The strict-mode toggle from (c) is FILED for v1.15.0+ multi-tenant scenarios (D16) but NOT enabled in v1.14.4.

**Justifications:**

  1. **No existing API consumers break.** The webapp is the only client today. The v1.14.4 client always sends If-Match. There's no third-party API surface. Backcompat is "self-imposed soft invariant" — the chat agent's `organize_update` tool path is not HTTP, so it doesn't intersect with the optional/required choice anyway.
  2. **Single-user trust boundary tolerates the bypass.** A malicious or buggy client COULD bypass conflict detection by omitting If-Match. Acceptable because: (a) the user trusts their own client (single-user posture), (b) the audit log records the absence — the `webapp.item_mutate` row's `etag: null` field (D10) preserves forensic visibility, and (c) future multi-tenant deployments enable strict mode via D16.
  3. **Strict mode for v1.15.0+.** When the project moves toward multi-user / API-token surfaces (per ADR 005's posture), `?strict=1` (or a config flag `enforceETagOnMutations: true`) becomes the right default. v1.14.4 specifies the future contract (428 Precondition Required if If-Match missing) so v1.15.0+ doesn't have to redesign.
  4. **CP1 will likely push back on (b) vs (a).** Expected DA pushback: "required-when-present is a soft invariant; a buggy v1.14.5 client could regress and lose data silently." Mitigation: D10's audit row captures `etag: null` on every mutation, so forensic detection of "client stopped sending If-Match" is one query away. Documented in §CP1 surface.

**Per-route binding (Phase 2):**

| Route | If-Match required? | If-Match present + mismatch → | If-Match absent → |
|---|---|---|---|
| PATCH /api/webapp/items/:id | required-when-present | 412 | proceed without check; audit `etag: null` |
| POST /api/webapp/items/:id/complete | required-when-present | 412 | proceed without check; audit `etag: null` |
| DELETE /api/webapp/items/:id | required-when-present | 412 | proceed without check; audit `etag: null` |
| GET /api/webapp/items/:id | N/A — read-only | N/A | N/A |
| GET /api/webapp/items | N/A — read-only | N/A | N/A |

**The strict-mode-future-contract (filed but NOT implemented in v1.14.4):**

  - When activated (via `?strict=1` query param OR config flag): If-Match absent → 428 Precondition Required with body `{ok: false, code: 'PRECONDITION_REQUIRED', error: 'If-Match header required for mutations'}`.
  - The audit row's `etag: null` value would never appear in strict mode (rejected before update).
  - v1.14.4 Phase 2 implementation MUST be structured so adding strict mode is a 5-LOC change (a single `if (strictMode && !ifMatch) return 428` early-return in the helper from D15).

**Why not (a).** Breaking change. No deployment story — clients in the wild that haven't upgraded would hard-fail. Also forecloses the "graceful upgrade" path for future external API consumers.

**Why not (c) in v1.14.4.** The strict-mode toggle adds wire surface (query param OR config flag) and audit category that v1.14.4 doesn't need. v1.15.0+ multi-tenant work has the right context to make the strict-mode decision.

**Consequences.**

  - **Positive.** No breaking change. v1.14.4 client always sends; older clients (none today) keep working.
  - **Positive.** Future strict mode is one early-return away.
  - **Negative.** Soft invariant. Audit row captures the absence, but a buggy client could silently regress. Documented in §Risks; D10 mitigates via forensic visibility.

### Decision 4 — 412 response body shape: `{ok: false, code: 'PRECONDITION_FAILED', error, currentEtag, currentItem}`

**Status:** Accepted — 412 carries `currentEtag` (string) AND `currentItem` (full `OrganizeItemDetail`) in the body, in addition to the response ETag header.

**Context.** Three candidates:

  - **(a) Minimal: `{ok: false, code: 'PRECONDITION_FAILED', error}` only.** Client must refetch GET /:id to get the new state. One extra round-trip on conflict.
  - **(b) Plus currentEtag: `{ok: false, code, error, currentEtag}`.** Client knows the new ETag immediately but must refetch to get the new item state.
  - **(c) Plus currentEtag AND currentItem: `{ok: false, code, error, currentEtag, currentItem}`.** Client gets everything in one round-trip; can render the new state in the conflict UI directly.

**Decision.** (c). Justifications:

  1. **Forward-compatible with 3-way diff UI.** v1.16.0+ may add a 3-way diff (your-changes vs server-changes vs original-base). The 3-way diff needs the SERVER STATE in hand. Embedding `currentItem` in the 412 response means the client doesn't have to re-fetch when displaying the diff. v1.14.4 ships the 2-button UX (Reload / Save Anyway) which doesn't render the diff, but the response shape supports the v1.16.0 upgrade without API-breakage.
  2. **One round-trip on conflict.** A 412 without `currentItem` requires the client to issue a follow-up GET /:id before it can show "here's what's on the server now." That's two round-trips for what's already a degraded experience (the user just hit a conflict). Cost of including `currentItem`: the server already READ the current item to compute the current ETag; adding it to the response body is `~5 LOC`. Server-side cost is zero; client-side simplicity gain is real.
  3. **Header AND body carry currentEtag.** The header is the HTTP-protocol-canonical place; the body's `currentEtag` is for clients that don't read headers programmatically. Redundancy is fine — both are the same value computed once. Client uses body's `currentEtag` as the canonical "next baseline" because the body is what the client renders.
  4. **Bandwidth cost is bounded.** `currentItem` includes notes (≤10KB) + progress (≤20KB) + small fields. Worst-case 412 response is ~31KB — same as the largest legitimate PATCH body. No operational concern at single-user scale; future multi-tenant paths can revisit if 412 becomes hot.

**Wire shape (binding for Phase 2):**

```typescript
// HTTP/1.1 412 Precondition Failed
// ETag: "<new-iso-or-mtime>"      ← header carries currentEtag too
// Cache-Control: no-store, no-cache, must-revalidate
// Content-Type: application/json
{
  "ok": false,
  "code": "PRECONDITION_FAILED",
  "error": "Item changed since you opened it. Reload to see the latest, or use Save Anyway to overwrite.",
  "currentEtag": "\"2026-04-25T14:35:12.345Z\"",
  "currentItem": { /* full OrganizeItemDetail — same shape as GET /:id */ }
}
```

**Error message wording (binding):**

  > "Item changed since you opened it. Reload to see the latest, or use Save Anyway to overwrite."

This wording is for the API-level field. The client UI's user-facing wording is in D5 (separate concern; UI text may differ). Phase 2 client-side prose may be slightly different ("This item was changed elsewhere…") but the API field is fixed.

**Code value: `PRECONDITION_FAILED`** — matches HTTP semantics. Aligns with the existing error envelope discipline (`AUTH_FAILED`, `NOT_ALLOWED`, `BAD_REQUEST`, `NOT_FOUND`, `INTERNAL_ERROR`, `AMBIGUOUS_TOGGLE`, etc., per items.shared.ts and items.mutate.ts).

**Why not (a).** Forces a follow-up round-trip on every conflict. Friction on an already-degraded path.

**Why not (b).** Same as (a) but with one less round-trip. (c) costs almost nothing more and unlocks future 3-way-diff without API change.

**Consequences.**

  - **Positive.** One round-trip on conflict. Forward-compatible.
  - **Positive.** Client can render the conflict UI immediately with the new state in hand.
  - **Negative.** 412 response body is large (~31KB worst case). Tolerated; 412 is rare-by-design.

### Decision 5 — Save Anyway escape hatch: allowed; logged; no second confirm

**Status:** Accepted — on a 412 conflict, the client offers two buttons: **Reload** (refetch + reset form) and **Save Anyway** (re-PATCH WITHOUT If-Match). Save Anyway is allowed without a second confirmation. The forced-override path emits a `webapp.item_mutate` audit row with `forced: true` (D10).

**Context.** Three independent sub-decisions:

  - **(5a) Should Save Anyway be allowed at all?** YES. Single-user trust boundary; explicit user override is appropriate UX. Forbidding it ("Reload only") would be paternalistic — the user IS the system's authority.
  - **(5b) Should Save Anyway log an audit row distinct from a normal mutation?** YES. The `webapp.item_mutate` row gains optional `forced: boolean` (D10). When `forced: true`, the row records that the user EXPLICITLY overrode a detected conflict. Forensic visibility: the user can later query "show me items where I overrode someone else's edit" (which in their single-user world means "show me items where I overrode my OWN earlier edit from another device or where I overrode the chat-agent's append").
  - **(5c) Should Save Anyway require a second confirmation?** NO. The conflict UI itself IS the confirmation surface — the user already saw the panel, read "this item was changed elsewhere," and explicitly chose Save Anyway. Adding a second "are you sure?" modal would be confirmation-fatigue.

**Decision.** (5a) YES, (5b) YES, (5c) NO. Justifications:

  1. **(5a) Single-user trust boundary.** The user is the only authority. Forbidding Save Anyway would prevent legitimate workflows like "I want to overwrite the chat-agent's auto-progress entry with my own description." Available, explicit, audited is the right posture.
  2. **(5b) Forensics matter.** v1.14.4's whole point is closing the silent-loss window from v1.14.2. The audit log MUST distinguish "user knowingly overrode" from "user was blissfully unaware." `forced: true` makes the override visible in logs. Single-user audit query: `SELECT detail FROM audit_log WHERE category = 'webapp.item_mutate' AND json_extract(detail_json, '$.forced') = 1`.
  3. **(5c) UI panel already explicit.** The conflict panel will say (binding wording, D5):
     > **"This item was changed elsewhere since you opened it.**
     > *[Diff summary or 'Server has updates' placeholder]*
     > **[Reload]** to discard your edits and see the latest.
     > **[Save Anyway]** to overwrite the changes."
     >
     > Two buttons, side by side, clear consequences. Adding a third "are you sure?" modal duplicates the same affordance and creates click-through fatigue.
  4. **Implementation: Save Anyway = re-PATCH WITHOUT If-Match.** The button handler simply omits the `If-Match` header on retry. Server's required-when-present (D3) means absence proceeds without check → mutation succeeds. The `forced` flag is set client-side AS A REQUEST HEADER (`X-Force-Override: 1`) which the server uses to populate `webapp.item_mutate.detail.forced = true`. **CRITICAL nuance:** the client sends BOTH the missing If-Match AND the explicit X-Force-Override header to disambiguate "buggy client forgot If-Match" from "user explicitly chose Save Anyway." The server treats them differently:
     - **No If-Match, no X-Force-Override:** legacy / buggy client. Audit `etag: null, forced: false`. Proceed.
     - **No If-Match, X-Force-Override: 1:** Save Anyway path. Audit `etag: null, forced: true`. Proceed.
     - **If-Match present, X-Force-Override absent:** standard happy path. Audit `etag: <value>, forced: false`. Proceed if matches; 412 if not.
     - **If-Match present, X-Force-Override present:** treat as Save Anyway (forced wins; If-Match is informational). Audit `etag: <value>, forced: true`. Proceed without conflict check.

**UX flow (binding for Phase 2):**

```
1. User edits title, hits Save.
2. Client: PATCH /:id with If-Match: "<old-etag>" → 412.
3. Client renders inline conflict panel inside the edit form.
4. User reads "This item was changed elsewhere".
5a. User clicks Reload:
    → Client: GET /:id → replaces currentDetailItem + currentDetailEtag.
    → Form fields reset to fresh values from server.
    → Conflict panel hides.
    → User must re-apply their edits and Save again.
5b. User clicks Save Anyway:
    → Client: PATCH /:id WITHOUT If-Match, WITH X-Force-Override: 1, body = the original patch.
    → Server: audit forced:true; mutation succeeds.
    → Client: replaces currentDetailItem + currentDetailEtag with response.
    → Form exits edit mode (mutation succeeded).
    → Toast: "Saved. Note: another change was overridden."
```

**Why allow Save Anyway.** Forbidding it would force the user into a "Reload, re-type your edits" loop that doesn't add safety in the single-user trust boundary; it just adds friction. The forced audit row makes the override visible in retrospect.

**Why no second confirm.** The conflict panel IS the confirm surface. Two buttons with explicit consequences is enough.

**Consequences.**

  - **Positive.** User is in control. Audit-trail visible. No paternalism.
  - **Positive.** Standard 2-button conflict UX maps cleanly to v1.16.0's 3-way diff (the diff render replaces the "[Diff summary]" placeholder).
  - **Negative.** A user who routinely Save-Anyway's loses the value of the conflict detection. Mitigation: audit visibility + the toast nudge ("Note: another change was overridden") on every forced save.

### Decision 6 — R2 X-Captured-Mtime + staleWarning sunset: REMOVE in v1.14.4

**Status:** Accepted — v1.14.4 REMOVES the v1.14.2 R2-mtime code path entirely. Wire deprecation, audit-emission removal. Schema retained (forward compat); helpers deleted.

**Context.** Three candidates:

  - **(a) Remove.** Rip out `X-Captured-Mtime` request-header parsing, the `auditStaleEdit` helper, the `webapp.stale_edit` audit emission, and the `staleWarning: true` response field. v1.14.4 If-Match is the SOLE conflict-detection mechanism.
  - **(b) Deprecate.** Keep the code; mark as deprecated; ignore the header. Future-removal in v1.14.5 or v1.16.0.
  - **(c) Co-exist.** Keep both; X-Captured-Mtime continues to set `staleWarning` on the response if present (defense-in-depth alongside If-Match).

**Decision.** (a). Justifications:

  1. **If-Match strictly subsumes X-Captured-Mtime.** Both detect the same condition: "the file's been modified since the client got its baseline." If-Match is BETTER (uses `updated:` not mtime; survives backup-and-restore on edited items; HTTP-standard). X-Captured-Mtime exists only because v1.14.2 didn't have `updated:` yet.
  2. **Two parallel mechanisms are confusing.** A user with v1.14.4 client + v1.14.4 server hits a conflict. Does the toast say "stale_warning: true (saved anyway)" OR does it 412? Co-existence (c) requires deciding which signal wins, which is design overhead with no benefit (the If-Match path is strictly more correct).
  3. **Backcompat path (D3) preserves the v1.14.2 behavior for clients that don't send If-Match:** they proceed without conflict check, exactly as v1.14.2 staleWarning's NON-BLOCKING behavior did. Sunsetting X-Captured-Mtime doesn't break the older-client experience meaningfully — old clients stop getting staleWarning toasts, but they were non-blocking anyway, and no v1.14.4-aware clients exist today besides the one we ship.
  4. **Code reduction is real.** `auditStaleEdit` helper (~25 LOC at `items.mutate.ts:97-122`); X-Captured-Mtime parsing in PATCH handler (~25 LOC at `:204-230`); same pattern in DELETE (~17 LOC at `:296-311`); same pattern in POST /complete (~17 LOC at `:435-450`); the `progressLineDelta` computation (~6 LOC at `:217-223`); the response-body `staleWarning` flag (~3 LOC each x 3 routes). **Sunset timeline (binding for Phase 2):**

  - **v1.14.4:**

    - REMOVE `staleWarning` field from all mutation success responses.
    - STOP emitting `webapp.stale_edit` audit rows. The category remains in the JSON-blob schema (no migration; the row shape is forward-compatible) but no NEW rows of this category will be inserted by v1.14.4+.
    - REMOVE `WebappItemMutateDetail` `progressLineDelta` field (was only used by `auditStaleEdit`).
    - Client (`app.js`):
      - REMOVE `let capturedMtime = null;` state at `:187`.
      - REMOVE `'X-Captured-Mtime'` header sends at `:528, :860, :981`.
      - REMOVE `data.staleWarning` toast handling at `:886`.
      - REMOVE `capturedMtime = data.item?.mtimeMs ?? null;` at `:1081`.
    - The `mtimeMs` field on `OrganizeItemDetail` (verified at `types.ts:84-86`) — keep on the wire for one more version (forward compat for any operator tooling that reads it). Retire in v1.14.5 if QA finds no consumer; document in TODO.

**Schema retention (audit_log table):**

### Decision 7 — ETag computation: storage-layer helper at `src/organize/etag.ts`

**Status:** Accepted — `computeETag(fm: OrganizeFrontMatter, fileMtimeMs: number): string` lives in NEW `src/organize/etag.ts` (~30 LOC). Imported by both `items.read.ts` and `items.mutate.ts`.

**Context.** Three candidates:

  - **(a) Storage layer (`src/organize/etag.ts`).** Single source of truth. ETag is an organize-domain concept (it's about the version of an organize item). Belongs with the rest of the organize feature.
  - **(b) Webapp layer (each handler computes its own).** Duplicated logic in `items.read.ts` and `items.mutate.ts`. Fragile.
  - **(c) Inline in `storage.ts`.** Single source of truth, but bloats `storage.ts` (already 1129 LOC; Anti-Slop §13 module-isolation pressure).

**Decision.** (a). Justifications:

  1. **Anti-Slop §5 (separation of concerns) + §13 (module isolation).** ETag computation is one concern. It deserves its own small module. ~30 LOC is exactly the right size.
  2. **Domain co-location.** ETag is about organize items. The `src/organize/` directory is where organize-domain concerns live. Putting ETag in `src/webapp/` would be wrong-shaped; putting it in `src/organize/storage.ts` would bloat storage.ts. A new file is the right answer.
  3. **Shareable.** Future organize-feature surfaces (CLI, other API surfaces, MCP tool wrappers) might want to compute ETag-equivalents for their own concurrency stories. Lifting `computeETag` out of `storage.ts` makes it shareable.
  4. **Test isolation.** A standalone module is trivial to unit test (`computeETag.test.ts`, ~6 cases — see D11). Putting it in `storage.ts` couples its tests to storage concerns.

**Module shape (binding for Phase 2):**

```typescript
// src/organize/etag.ts (NEW, ~30 LOC)

import type { OrganizeFrontMatter } from './types.js';

/**
 * Compute the strong HTTP ETag for an organize item (v1.14.4 ADR 012 D1 + D7).
 *
 * Returns a double-quoted ISO timestamp (HTTP-standard strong ETag format).
 * Uses `frontMatter.updated` when present (post-v1.14.3 items); falls back to
 * the file mtime as ISO for legacy items not yet edited via a v1.14.3+ write
 * path. Both cases produce the same wire shape.
 *
 * Caller MUST provide BOTH the front-matter AND the current file mtime — the
 * fallback path requires the latter, and computing it inline here would add
 * an FS-stat to a function that should be pure.
 *
 * @param fm           Item front-matter (may have `updated` null/missing).
 * @param fileMtimeMs  Current file mtime in ms (caller's fs.stat result).
 * @returns Strong ETag value, e.g. `"2026-04-25T14:30:00.000Z"`.
 */
export function computeETag(fm: OrganizeFrontMatter, fileMtimeMs: number): string {
  const anchor = fm.updated ?? new Date(fileMtimeMs).toISOString();
  return `"${anchor}"`;
}

/**
 * Compare two ETags for byte-exact equality (after trimming optional whitespace).
 *
 * Used by the If-Match check. Both ETags MUST be in the same wire format
 * (double-quoted ISO). Mismatch in format (e.g., one weak, one strong) returns
 * false — v1.14.4 only emits strong ETags so this is a non-concern, but
 * defense-in-depth.
 *
 * @param a  ETag value (from request header or response).
 * @param b  ETag value to compare.
 * @returns  True if byte-equal after trim; false otherwise.
 */
export function etagsMatch(a: string | undefined, b: string): boolean {
  if (a === undefined) return false;
  return a.trim() === b.trim();
}
```

**Cross-imports (binding):**

  - `items.read.ts`: `import { computeETag } from '../organize/etag.js';` — used in GET /:id handler.
  - `items.mutate.ts`: `import { computeETag, etagsMatch } from '../organize/etag.js';` — used in PATCH, DELETE, POST /complete handlers.
  - `storage.ts`: NO direct dependency. ETag is a wire concern, not a storage concern; storage doesn't need to know about it.

**Why not (b).** Duplicated logic across two webapp files. Drift risk. Anti-Slop §5 fail.

**Why not (c).** Bloats storage.ts (already 1129 LOC). Wrong module-isolation axis.

**Consequences.**

  - **Positive.** Single source of truth. ~30 LOC. Easy to unit-test.
  - **Positive.** Future surfaces can reuse.
  - **Negative.** A new file. Tolerated; the file is small and focused.

### Decision 8 — Concurrent PATCH semantics: serialization at storage `updateItem` (rename-as-publish boundary)

**Status:** Accepted — concurrent PATCHes serialize at the storage layer's read-then-write boundary. The atomic rename in `writeAtomically` (per-call random tmp suffix from v1.14.2 R8) is the publish point. ETag advance is monotonic across the rename ordering.

**Context.** Two clients hold `If-Match: "T1"`. Both PATCH concurrently. Express handles requests in a single Node process; async I/O interleaves but JavaScript execution is single-threaded for handler code. The race timeline:

```
t=0   Client A reads detail at "T1".
t=0   Client B reads detail at "T1".
t=10  Client A: PATCH with If-Match "T1".
t=11  Client B: PATCH with If-Match "T1".
t=12  Server (handler A): start. Read live file → fm.updated = "T1". Compute currentEtag = "T1". A's If-Match matches → proceed.
t=13  Server (handler A): build storagePatch. Call updateItem(...).
t=14  Server (handler A): inside updateItem, read live → parse → patch → stampUpdated → fm.updated = "T2" → writeAtomically (writes A's tmp; renames to live). Returns.
t=15  Server (handler A): respond 200, ETag "T2".
t=16  Server (handler B): start. Read live file → fm.updated = "T2" (NOW, after A's rename). Compute currentEtag = "T2". B's If-Match "T1" does NOT match "T2" → 412 with currentEtag "T2".
```

The serialization point that matters: between A's successful `writeAtomically` rename (t=14) and B's read at t=16. The Node event loop guarantees:

  - A's handler runs to completion (or to its next `await`) before B's handler resumes.
  - A's `writeAtomically` (rename + return) is `await`ed; B's read-current-item happens AFTER A's rename if the timing has B starting AFTER A's first await yields.
  - If B starts WHILE A is mid-`updateItem` (e.g., between A's read and A's write), B's read of the live file sees the OLD content (`updated: "T1"`); B's currentEtag = "T1" matches B's If-Match "T1" → B proceeds; B's `updateItem` then runs against `updated: "T1"` it just observed → stamps "T3" → writes → renames. **Both clients succeed; B's content lands; A's content was overwritten.** This is the bug to consider.

**Decision.** Accept the existing storage serialization boundary; do NOT add a new lock. The race window above (B starting mid-A) is closed by re-checking the ETag INSIDE the handler at the LATEST SAFE POINT — specifically, after the read-current-item-for-projection step but before the storagePatch construction. **The ETag check happens TWICE in the v1.14.4 PATCH handler:**

  1. **First check (early reject):** read current ETag from on-disk `fm.updated` + `mtime`; compare to `If-Match`. If mismatch → 412 immediately. This is the optimization for the common case (if you're already stale, fail fast without the full update overhead).
  2. **Second check (just-before-write):** the storage layer's `updateItem` itself reads the file → patches → writes. If between the handler's first read and `updateItem`'s read, ANOTHER writer raced in, `updateItem` reads the newer file. v1.14.4 augments `updateItem` with an OPTIONAL `expectedEtag` parameter that gets passed THROUGH from the route handler; if `expectedEtag` is set and doesn't match the value `updateItem` reads, `updateItem` throws `ETAG_MISMATCH` → handler converts to 412 with the now-current ETag.

  The augmentation to `updateItem` is small (~10 LOC; new optional parameter; new error code). Critically, this closes the time-of-check-vs-time-of-use (TOCTOU) window between the handler's first ETag check and the storage rename.

**Storage signature change (binding for Phase 2):**

```typescript
// src/organize/storage.ts — updateItem signature gains optional 4th argument

/**
 * v1.14.4 — optional expectedEtag for TOCTOU-safe If-Match enforcement.
 * When present, updateItem re-reads the file and verifies that the ETag
 * computed from (fm, mtime) matches `expectedEtag` BEFORE writing. Mismatch
 * throws an Error with `code = 'ETAG_MISMATCH'` carrying the actual current
 * ETag in `.actualEtag`.
 *
 * When absent, behavior is unchanged from v1.14.3.
 */
export async function updateItem(
  userId: number,
  dataDir: string,
  itemId: string,
  patch: UpdateItemPatch,
  options?: { expectedEtag?: string }
): Promise<OrganizeItem> {
  // ...existing body...
  // After read-and-parse, BEFORE applying the patch:
  if (options?.expectedEtag !== undefined) {
    const currentEtag = computeETag(parsedFm, fileMtimeMs);
    if (!etagsMatch(options.expectedEtag, currentEtag)) {
      const err = new Error(`ETag mismatch: expected ${options.expectedEtag}, got ${currentEtag}`) as Error & { code: string; actualEtag: string };
      err.code = 'ETAG_MISMATCH';
      err.actualEtag = currentEtag;
      throw err;
    }
  }
  // ...existing body continues...
}
```

`updateItem` imports `computeETag` and `etagsMatch` from `etag.ts`. The signature change is OPTIONAL parameter — fully backwards-compatible with the chat-side call sites (`organize_update.ts`, `organize_complete.ts`) which don't pass `options`.

**For DELETE:** same pattern via `softDeleteItem(userId, dataDir, id, options?: { expectedEtag?: string })`. Storage-layer ETag check happens before the rewriteContent step (which is the FIRST write); if the check fails, throw `ETAG_MISMATCH` and skip the rewrite + rename.

**For POST /complete:** uses `updateItem` internally (verified at `items.mutate.ts:491`); the new `expectedEtag` option flows through.

**Concurrent PATCH semantics — final binding:**

  - Two clients with `If-Match: "T1"` racing → exactly ONE wins (the one whose `writeAtomically` rename runs first); the other gets 412 with currentEtag = "T2".
  - The double-check at handler-level + storage-level eliminates the TOCTOU window.
  - Single-process single-Node-event-loop guarantees the ordering. No multi-process / cluster concern in v1.14.4 (the webapp is single-instance; documented in §Out-of-scope).

**Why this approach.** The handler-level check is fast (early reject for clients who are clearly stale). The storage-level check is correct (TOCTOU-safe; closes the race). Combining them keeps the common case fast AND the rare race-condition correct.

**Why not a lock.** Process-level lock (e.g., `async-mutex` per-itemId) would block legitimate sequential edits unnecessarily and adds runtime dependency. The serialization model + double-check is correct without a lock.

**Consequences.**

  - **Positive.** Race-correct without locks.
  - **Positive.** Storage augmentation is opt-in; chat-side call sites unaffected.
  - **Negative.** ETag check happens in two places (handler + storage). Slight code duplication risk; mitigated by both checks calling the same `etagsMatch` helper.

### Decision 9 — POST /complete with If-Match: YES, consistent with PATCH/DELETE

**Status:** Accepted — POST /complete requires-when-present If-Match (D3 applied). Same 412 + currentEtag + currentItem on mismatch (D4). Same audit row metadata (D10). Same Save Anyway path (D5).

**Context.** v1.14.2 R18 made POST /complete absolute-write — `{done: true}` always sets status='done' regardless of current state. Question: does the absolute-write semantic make If-Match redundant?

  - **(a) Yes, always require If-Match (consistent).** Treat POST /complete like any other mutation. Even though absolute-write is "race-safe" in the data-corruption sense, the client may benefit from "you didn't see this status change" detection.
  - **(b) No, POST /complete is intentionally idempotent.** Skip If-Match. Client doesn't need to know current state; server takes the explicit input.

**Decision.** (a). Justifications:

  1. **Consistency outweighs idempotence-savings.** Three mutation routes; three identical conflict-detection patterns. Special-casing POST /complete to NOT need If-Match adds a "but except for /complete" footnote to every piece of documentation. Anti-Slop §5 (consistency) wins.
  2. **The "you didn't see this change" case is real.** Scenario: User has the webapp open in two tabs. Tab 1 marks an item complete (✅ → ⭕ flip). Tab 2 is showing the same item as active (its baseline ETag is from before Tab 1's completion). User taps the now-stale circle in Tab 2 → POST /complete `{done: true}`. **Without If-Match:** mutation succeeds (status was 'done' in Tab 1's path, gets set to 'done' again — no-op-ish). User in Tab 2 sees nothing weird. This is fine for the "set done" case. **But:** if Tab 2 taps `{done: false}` (uncomplete), the mutation succeeds and reverts the completion — Tab 1's user sees their just-completed item flip back to active without explanation. This IS a user-visible race. If-Match closes it: Tab 2's stale ETag → 412 → conflict UI explains.
  3. **The list-card complete-checkbox is the special case (SF-7).** The list does NOT carry per-item ETags (D2). The optimistic-flip path from the list MUST NOT send If-Match (it doesn't have one). This is allowed by D3's required-when-present semantic — absent If-Match → server proceeds. Audit row records `etag: null, forced: false` (the legacy/list-flow case). Acceptable.
  4. **The detail-form-edit path DOES send If-Match.** When the user opens detail and taps the complete checkbox in the detail panel, the detail's ETag IS available — that path SENDS If-Match. The two paths have different audit signatures (`etag: null` for list-flow vs `etag: "<value>"` for detail-flow), making the source distinguishable in audit queries.

**Per-source binding (Phase 2):**

| Source of POST /complete | If-Match sent? | Rationale |
|---|---|---|
| List card optimistic checkbox (`app.js` checkbox click handler at ~`:520`) | NO | List doesn't have per-item ETags (D2). Audit `etag: null, forced: false`. |
| Detail panel checkbox / Save button after edit form changes status | YES (the detail's ETag) | Standard conflict-detection. Audit `etag: "<value>"`. |
| (Future) MCP tool / webapp-from-script | OPTIONAL (consumer-decided) | Backcompat path; consumers may opt in. |

**Status:** Accepted — client `app.js` retires `capturedMtime` (sunset per D6) and adopts `currentDetailEtag`. Conflict UI is INLINE in the edit form (NOT modal/native dialog — CSP + browser-tools rule per ADR 009 decision 6).

### Decision 13 — Header naming: standard `ETag` and `If-Match`; `X-Force-Override` as the only X- custom header

**Status:** Accepted — use HTTP-standard `ETag` and `If-Match`; no `X-ETag` or other custom variants. Save Anyway uses ONE custom header `X-Force-Override: 1` for forensic disambiguation.

**Context.** Header naming choices:

  - **(a) Pure HTTP standard.** `ETag` (response), `If-Match` (request). `If-None-Match` deliberately NOT used (we're not doing caching).
  - **(b) Custom X- prefix.** `X-ETag`, `X-If-Match`. Avoid HTTP semantics confusion. Verbose.
  - **(c) Hybrid.** Standard `ETag` + `If-Match` for the conflict-detection flow; `X-Force-Override: 1` to mark the Save Anyway path (D5). Clean separation: standard headers for the standard semantics; ONE custom header for the application-specific override signal.

**Decision.** (c). Justifications:

  1. **Use standards where they apply.** `ETag` and `If-Match` are HTTP/1.1 RFC-defined. Browsers, devtools, fetch APIs, server middleware all understand them. No reinvention.
  2. **One X- header is acceptable.** The Save-Anyway forensic disambiguation needs SOME wire signal (we can't infer "user explicitly chose to override" from "client just omitted If-Match"). `X-Force-Override: 1` is the smallest custom-header surface that does the job. The X- prefix is technically deprecated by RFC 6648 but still widely used; acceptable.
  3. **Alternative considered: a query parameter.** `PATCH /:id?force=1`. Pro: no custom header. Con: query parameters are part of the URL and SHOULD be idempotent / cacheable; mutations in query params are an anti-pattern (Anti-Slop §5/§9). Header is the right place.
  4. **Alternative considered: a body field.** `PATCH /:id` with body `{title: '...', _force: true}`. Pro: no custom header, no URL pollution. Con: pollutes the validated PATCH body shape; `validatePatchBody` would have to strip `_force` before validation; one more thing to special-case. Header is cleaner.

**Wire bindings (binding for Phase 2):**

  - **Response:** `ETag: "<iso>"` (per D1).
  - **Request (standard):** `If-Match: "<iso>"` (per D1).
  - **Request (force):** `X-Force-Override: 1` (per D5; only when client wants to force-override past a 412).
  - **Request (legacy):** No `If-Match`, no `X-Force-Override` — server treats as backcompat path (D3).

**Why no other custom headers.** Every additional custom header adds a wire-surface for QA to verify and for an attacker to probe. Minimize.

**Consequences.**

  - **Positive.** Standard-conformant. Devtools-readable.
  - **Positive.** One X- header. Tolerated; documented.

```typescript
// src/webapp/items.shared.ts — NEW helper

import type { OrganizeFrontMatter } from '../organize/types.js';
import { computeETag, etagsMatch } from '../organize/etag.js';

/**
 * Result of an If-Match precondition check.
 * - { ok: true, force: false } → standard happy path; proceed with mutation.
 * - { ok: true, force: true }  → Save Anyway path; proceed without conflict check.
 * - { ok: false, currentEtag, currentItem } → 412 conflict; caller renders 412 envelope.
 * - { ok: false, code: 'PRECONDITION_REQUIRED' } → reserved for v1.15.0 strict mode (D3 + D16); not emitted in v1.14.4.
 */
export type IfMatchCheckResult =
  | { ok: true; force: false; ifMatch: string | null }
  | { ok: true; force: true; ifMatch: string | null }
  | { ok: false; code: 'PRECONDITION_FAILED'; currentEtag: string };

/**
 * Compare the request's If-Match header to the current item's ETag.
 * v1.14.4 D3 — required-when-present (optional-for-backcompat).
 *
 * @param req            Express request (reads If-Match + X-Force-Override).
 * @param currentFm      Current front-matter from the live file.
 * @param currentMtimeMs File mtime (for legacy-item ETag fallback).
 * @returns IfMatchCheckResult — caller branches on .ok and .force.
 */
export function ifMatchCheck(
  req: Request,
  currentFm: OrganizeFrontMatter,
  currentMtimeMs: number,
): IfMatchCheckResult {
  const ifMatch = req.header('If-Match') ?? null;
  const force = req.header('X-Force-Override') === '1';
  const currentEtag = computeETag(currentFm, currentMtimeMs);

  // Force overrides everything (Save Anyway path, even if If-Match present)
  if (force) return { ok: true, force: true, ifMatch };

  // No If-Match → backcompat path; proceed without check
  if (ifMatch === null) return { ok: true, force: false, ifMatch: null };

  // If-Match present → must match
  if (etagsMatch(ifMatch, currentEtag)) {
    return { ok: true, force: false, ifMatch };
  }

  return { ok: false, code: 'PRECONDITION_FAILED', currentEtag };
}
```

## 5. CP1 surface — what the Architect expects DA + Anti-Slop to push back on

The Architect anticipates the following CP1 pushbacks. Each has a prepared response.

  1. **DA: "Required-when-present is a soft invariant; a buggy v1.14.5 client could regress and lose data silently."** (D3)
     - *Response:* D10's audit row captures `etag: null` on every backcompat-path mutation. Forensic detection is one query away. Strict mode (D3 future contract) is filed for v1.15.0+. The trade-off is conscious; backcompat for the (single) existing client is the deciding factor.
  2. **DA: "Save Anyway has no second confirm — fat-finger users will routinely overwrite."** (D5)
     - *Response:* The conflict panel itself IS the confirm surface; two buttons with explicit consequences. Adding a second modal is confirmation-fatigue. Audit visibility (`forced: true`) lets the user retrospectively review their overrides; the toast nudge ("Note: another change was overridden") on every forced save reinforces awareness.
  3. **DA: "412 with `currentItem` in the body is up to ~31KB. On a slow connection the conflict UX feels slow."** (D4)
     - *Response:* Bandwidth cost is bounded by the largest legitimate PATCH body. 412 is rare-by-design. The alternative (412 minimal + follow-up GET) is two round-trips on an already-degraded path. Single round-trip wins.
  4. **DA: "POST /complete from list-flow doesn't send If-Match — that's a hole."** (D9)
     - *Response:* The list does NOT carry per-item ETags (D2). Adding them is wire-noise (the list is not an edit-baseline). The hole is mitigated by R18's absolute-write semantic: the data-corruption case is closed; only the user-visible-revert case from another tab is uncloseable, and the audit row distinguishes list-flow (`etag: null`) from detail-flow. Acceptable.
  5. **DA: "Legacy items use mtime-based ETag, which resets on backup-restore. A user who restores from backup loses conflict detection on the first edit."** (SF-4 + D1)
     - *Response:* Acceptable for the transitional period. Pre-v1.14.3 items get `updated:` stamped on first webapp edit; ETag stabilizes thereafter. The window where mtime-based ETag is used IS narrow: only items the user created pre-v1.14.3 AND has not edited since AND has restored from backup. Vanishingly rare in single-user trust boundary.
  6. **DA: "TOCTOU: handler-level check could pass while storage rename runs concurrently. Storage-level check (via `options.expectedEtag`) is correct but introduces a new error code in `updateItem` — chat-side call sites might not handle it."** (D8)
     - *Response:* `options.expectedEtag` is OPTIONAL. Chat-side call sites (`organize_update.ts`, `organize_complete.ts`) DON'T pass options → never get `ETAG_MISMATCH`. Backwards-compatible with v1.14.3 chat-side behavior. Test T24 verifies the storage-layer behavior in isolation.
  7. **DA: "X-Force-Override is an X- custom header (RFC 6648 deprecates the X- prefix). Should be a body field or a query param."** (D5 + D13)
     - *Response:* RFC 6648 deprecates X- but it's still widely used; the alternatives (body field pollutes validated PATCH shape; query param mixes mutation with cacheable URL semantics) are worse. Tolerated.
  8. **Anti-Slop §13: "items.mutate.ts is at 516 LOC; v1.14.4 changes drop it to 496 — but that's still close to the soft threshold."** (D15)
     - *Response:* 496 is below the 500 soft threshold from v1.14.2 SF-6. The `ifMatchCheck` helper extraction (D15) is the explicit mitigation. Anti-Slop §13 satisfied.
  9. **Anti-Slop §5: "ETag check happens TWICE (handler + storage). Code duplication."** (D8)
     - *Response:* Both checks call the same `etagsMatch` helper from `etag.ts`. Different points in the call stack do the same comparison; the duplication is intentional (TOCTOU mitigation), not accidental. Documented.
  10. **DA: "Sunsetting `webapp.stale_edit` audit emission means historical analysis of v1.14.2/v1.14.3 stale events stops; a v1.14.4-onward audit query won't see the new conflict events under that category."** (D6)
      - *Response:* v1.14.4 conflicts go to `webapp.item_mutate` with `etag` populated AND to no-emission for the soft-conflict case (since 412 IS the conflict signal — there's no "soft conflict that succeeded" any more). Operator audit query for "did a conflict happen" changes from `category = 'webapp.stale_edit'` (v1.14.2/v1.14.3) to `<some 412 record>` (v1.14.4+). v1.14.4 does NOT add a `webapp.conflict_412` audit category — the 412 returns to client-only and no audit row is inserted on 412. **CP1 may push back on this:** "should 412 emit an audit row?" Decision: NO, because (a) it's a client-detectable error (the server already returned 412), (b) audit is for SUCCESSFUL state changes, and (c) the forced-override CASE (when the user follows up Save Anyway) DOES emit an audit row with `forced: true` + the original `etag`, which captures the conflict-and-override. Documented; if CP1 disagrees, the prepared revision is to add a `webapp.conflict_412` audit category as a pure-counter (no IP, no detail) — ~10 LOC.

## 7. Risks

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | A buggy v1.14.5+ client regresses to not sending If-Match; data loss returns silently. | MEDIUM | D10's audit row records `etag: null` per mutation; forensic query detects regression. v1.15.0+ strict mode (D3 future contract) closes for good. |
| R2 | User routinely uses Save Anyway; forced-override audit accumulates without user noticing. | LOW-MEDIUM | Toast nudge ("Note: another change was overridden") on every forced save. Audit visibility lets the user retrospectively review. |
| R3 | Pre-v1.14.3 items use mtime-based ETag; backup-restore on such an item resets ETag mid-edit-cycle. | LOW | First webapp edit stamps `updated:`; window is narrow and limited to the transitional period. |
| R4 | Chat-agent's autonomous `organize_update` overwrites webapp's just-saved edit (SF-3); webapp had no chance to send If-Match because chat-agent path has no HTTP layer. | MEDIUM | The user trusts the agent (it's their explicit instructions). If the user wants the agent to NOT overwrite their webapp edits, they tell the agent to read first. v1.14.4 doesn't claim to close agent-path races; documented. |
| R5 | Two-tab same-user race on POST /complete from list-flow: Tab 1 marks done, Tab 2 (stale) marks not-done; Tab 1's user sees the revert (D9 SF-7). | LOW | List-flow can't carry per-item ETag (D2). Detail-flow path closes the race. List-flow uncloseable; tolerated. |
| R6 | Concurrent PATCH from two clients with matching If-Match → both pass handler-level check; one wins via storage-level rename; the other gets ETAG_MISMATCH from storage layer (D8). The losing client gets a 412 that LOOKS LIKE the storage threw — slightly different error path. | LOW | Test T23 verifies the TOCTOU path returns 412 with the actual newer ETag; user UX is identical to handler-level 412. |
| R7 | If-Match header is large (operator pasted a 8KB string). HTTP server may reject before route handler runs. | LOW | Express defaults handle ~8KB total header; our `If-Match` is ~32 chars. Adversarial probing in Sub-Phase B verifies. |
| R8 | `WebappItemMutateDetail` schema extension: a future iteration adds another field; old detail-blob queries break. | LOW | JSON-blob schema; new fields are optional; old queries still work. Standard forward-compat. |
