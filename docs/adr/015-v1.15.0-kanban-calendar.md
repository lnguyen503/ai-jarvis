## D1 — Mobile DnD strategy

**Decision: B (tap-pick-tap-drop) primary; native HTML5 DnD as desktop coexisting fallback.**

The user picks a card (tap a "Move" button on the card OR long-tap the card body — D1.a) and the card enters "selected for move" state: a 2px primary-colored border + cursor change + the card stays in place visually. The user then taps any goal-column header (or the calendar's target day) to drop. A single tap-anywhere-else (card body of an unselected card, an empty area outside the column gutters, or the same card again) cancels.

Native HTML5 DnD (`dragstart`, `dragover`, `drop`) is wired in parallel. On desktop, mouse-drag works directly. On mobile, the long-press → native DnD path is NOT used (D1.b explains why). On mobile, the user takes the tap-pick-tap-drop path; on desktop, both work.

**D1.a — Pick-up trigger.** A "⋮⋮" handle icon on the right edge of each card is the explicit pick-up affordance. Tap = enter "selected for move" state. The whole card is also draggable on desktop via native DnD (the handle is for mobile UX clarity, not a desktop drag-by-handle constraint). This keeps DnD discoverable on both platforms without timing-based interactions.

**D1.b — Why not C (long-press to enter native DnD).** Long-press is a fragile cross-platform input: iOS Safari pre-15 had native DnD coverage gaps; Telegram WebApp on iOS uses WKWebView which inherits Safari's quirks; Android WebView's native DnD support varies by Chrome version. The 200-300ms long-press detection also competes with the OS's own long-press (iOS context menu; Android selection menu). Tap-pick-tap-drop sidesteps all of this: regular click handlers are universal.

**D1.c — Accessibility.** Tap-pick-tap-drop is keyboard-friendly with minimal additional work: a follow-up TODO files keyboard navigation (Space to pick up; Tab to navigate; Enter to drop) for v1.15.x+. Native HTML5 DnD is NOT keyboard-accessible by default; the tap pattern is structurally better here.

**Options weighed.**

  - **A (HTML5 DnD only).** Mobile users are SOL. Telegram WebApp is mobile-first. Skip.
  - **B (tap-pick-tap-drop).** PICK. Universal; no timing dependencies; accessible; coexists with native DnD on desktop.
  - **C (long-press → native DnD).** Cross-platform fragility. Skip.

**Visual state binding.**

  - Card in "selected for move" state: `.card-pickup-selected` class; 2px solid var(--tg-theme-button-color) outline; `cursor: grabbing`.
  - Drop target hover: `.column-drop-target` class on the column; faint `var(--tg-theme-button-color)` background tint.
  - On drop: card visually moves IMMEDIATELY (D12 optimistic UX); column-drop-target class clears.

## D2 — Calendar date math

**Decision: A — roll our own minimal date helpers (~80 LOC). Pure functions; UTC-only internally; format for display in user's locale.**

New module `public/webapp/organize/dates.js` — ES module, importable by both `calendar-view.js` and tests. Pure functions; no DOM; no side effects.

**Helper signatures (binding for D11):**

```javascript
// public/webapp/organize/dates.js — v1.15.0

/** Parse 'YYYY-MM-DD' into a UTC Date (midnight UTC). Returns null on malformed. */
export function parseISO(s);

/** Format a Date into 'YYYY-MM-DD' (UTC components). */
export function formatISO(d);

/** First day of the month containing d (UTC). */
export function firstOfMonth(d);

/** Last day of the month containing d (UTC). */
export function lastOfMonth(d);

/** Add n calendar days to d (UTC). Negative ok. */
export function addDays(d, n);

/** Add n calendar months to d (UTC; date-clamps to end-of-month if overflow). */
export function addMonths(d, n);

/** Returns the first day of the week containing d, given firstDayOfWeek (0=Sun, 1=Mon). */
export function weekStart(d, firstDayOfWeek);

/** Returns true if a and b are the same calendar day in UTC. */
export function isSameDay(a, b);

/** Returns the month-grid for d: 6 weeks × 7 days = 42 cells. Each cell is {date: Date, inMonth: boolean}. */
export function monthGrid(d, firstDayOfWeek);

/** Format a month label for display ("April 2026") in user's locale via Intl.DateTimeFormat. */
export function formatMonthLabel(d, locale);

/** Format a day-of-week label ("Mon") for the column header. */
export function formatDowLabel(dayIdx, locale, firstDayOfWeek);
```

**Why pure functions.** Testability + no Date-mutation foot-guns. Every helper takes a Date and returns a new Date or primitive; never mutates the input.

**Why UTC-only internally.** D3 binding — `due` is a calendar date, not a timezone-bearing instant. Treating it as UTC midnight gives stable arithmetic; display formatting uses the user's locale via Intl.

**Why no library.** Anti-Slop §3 boundary (no new deps). The 10 helper functions cover everything calendar-view.js needs; date-fns would be ~30KB minified for ~80 LOC of equivalent logic. Skip.

**Options weighed.**

  - **A (roll our own).** PICK. ~80 LOC; testable; no deps.
  - **B (Intl/Date built-ins, scattered).** Bug magnet. Skip.
  - **C (date-fns).** Boundary violation. Skip.

## D3 — Calendar timezone strategy

**Decision: A — calendar-date semantics (no timezone). `due: 'YYYY-MM-DD'` always renders on that calendar date regardless of user's timezone.**

The `due` field has been timezone-naive since v1.8.6 (stored as `YYYY-MM-DD` in front-matter; serialized as a plain string). Calendar-view.js continues this contract: `parseISO('2026-04-25')` yields a UTC midnight Date; `monthGrid()` / `weekStart()` / `addDays()` operate on UTC; the user sees "April 25" regardless of whether they are in UTC, PST, or JST.

**Display formatting (locale-aware).** Month label, day-of-week label, and "today" highlighting use the user's locale via `Intl.DateTimeFormat` for human-readable text. The grid CELL the item lands in is determined by UTC date arithmetic, NOT by the user's timezone.

**Why this matters.** A user in Pacific creating a task at 11pm with `due: '2026-04-25'` expects the task to render on April 25 (the calendar date they typed), not April 24 (the date `2026-04-25T00:00Z` falls on in their local timezone). Calendar-date semantics matches user intent.

**Why not B (convert to user's local timezone).** Would render `2026-04-25` on April 24 for users in Pacific, April 25 for users in UTC, April 25 for users in JST (depending on the conversion direction). Confusing. The chat-side `add a task: ... by tomorrow` parser already produces a calendar date in the user's local time interpretation; round-tripping through UTC conversion would shift it. Skip.

**Today highlighting.** The "today" cell is determined by `formatISO(new Date())` — that is, UTC's current calendar date. For users in extreme timezones (UTC-12, UTC+14), "today" might briefly appear off-by-one near midnight UTC. Acceptable: a known doc'd quirk; users near midnight don't typically open the calendar to check what's due "today" relative to a date that just changed.

**Documented in:** D15 KNOWN_ISSUES entry 3 + an inline comment block in dates.js explaining the contract.

## D4 — Kanban column structure

**Decision: A — one column per active goal + one "Standalone" column for items without parentId. Done items hidden by default (filter chip toggle).**

**Column ordering.** Active goals sorted by `created` ASC (oldest goal leftmost). The "Standalone" column always renders LAST (rightmost). Rationale: "created ASC" is the simplest deterministic order; the user's mental model of project progress is "goals I created earliest are most established"; Standalone is a catch-all bucket and visually belongs at the end.

**Empty goal column.** Renders as a column header with the goal title + a placeholder text under it: "(empty — drop here to add tasks)". Drop targets remain functional even when empty. The column doesn't disappear when the user drags the last task out — visual stability beats compactness.

**Done items.** Filtered out by the existing v1.14.0 filter chip (`status: active` is the default; the chip lets the user widen to `all`). When `status: all` is active, done items render in the same goal column as their active siblings, visually distinguished by a strikethrough + faded color (matches list view).

**Abandoned goals.** Goals with `status: abandoned` do NOT get a column. Their children (if any) appear in the "Standalone" column as orphans — same rule as v1.14.3 hierarchy view (the picker excludes abandoned goals; kanban excludes abandoned goal columns; consistent).

**Goal cards INSIDE goal columns?** The goal card itself does NOT appear inside its own column (the column header IS the goal). The user can edit the goal by tapping the column header → opens the detail panel.

**Standalone-column contents.** All items where `parentId == null` AND `type ≠ 'goal'`. Goals without `parentId` render as columns (their natural place); goals are not "standalone children." Top-level goals with no children still render as columns with the empty placeholder.

**Drag-to-reparent semantics.** Drop a task/event card onto a goal column → PATCH `parentId: '<goal-id>'`. Drop onto the Standalone column → PATCH `parentId: null`. Goal cards are NOT draggable (Goals can't have parents — v1.14.3 R13 enforces; UI matches by not offering DnD on goal cards). Goal column headers are NOT drop targets for OTHER goals.

**Options weighed.**

  - **A (column-per-active-goal + Standalone).** PICK. Matches the natural project-hierarchy mental model.
  - **B (columns by status — Active/Done/Abandoned).** Status is already a filter chip; turning it into columns dilutes the "what is this work for" axis.
  - **C (columns by type — Task/Event/Goal).** Type is already represented by card icons; doesn't add information.

**Cap.** No hard cap on column count (user with 50 goals = 50 columns; horizontal scroll). Realistic deployments have <20 goals; performance implications documented in D14 risk.

## D5 — Calendar view: month/week/day subviews

**Decision: Month is the default. Week and Day are accessible via subview chips below the view switcher.**

**Month view (default).** 6-week × 7-day grid (42 cells; covers all calendar months). Cells outside the current month rendered with reduced opacity (`.cell-other-month`); cells in the current month fully opaque. "Today" cell highlighted with `var(--tg-theme-button-color)` outline.

  - Each cell shows: day-number (top-left); 0-3 item titles (truncated with ellipsis); a "+N more" indicator when >3.
  - Tap on a cell → expand to a popover/sheet showing all items for that day with full titles + drag handles.
  - Tap on an item → opens the detail panel (consistent with list view).

**Week view.** 7 columns labeled with day-of-week + date; week starts Monday (default for v1.15.0 — D5.a explains why; locale-aware Mon-Sun vs Sun-Sat is v1.15.x+). Each column shows that day's items in a vertical list; no time-of-day rendered (v1.15.0 only has all-day items per D5.b).

**Day view.** Single column for the selected day. Hours 00:00-23:00 rendered as time-slot rows but EMPTY in v1.15.0 (no time-of-day on items yet — see D16 TODO). All-day items render in a stacked all-day band at the top of the column.

**D5.a — Week-start choice (Mon vs Sun).** Default `firstDayOfWeek = 1` (Monday). Rationale: ISO 8601 week starts Monday; international users skew Monday-first; the user's locale parsed from Telegram Web App init data could provide a hint, but Telegram's `WebApp.platform` doesn't expose locale-week conventions reliably; honoring locale fully = v1.15.x+ work. v1.15.0 picks Mon as the universal default; locale-honoring is filed as a TODO (D16).

**D5.b — All-day items only.** Items have `due: 'YYYY-MM-DD'` (no time-of-day). Calendar Day view's hour slots render but are decorative for v1.15.0; all items go in the all-day band. When time-of-day lands (D16 TODO), Day view's hour slots become meaningful.

**Subview chips.** Month / Week / Day chips below the View Switcher. Chip state persists in `sessionStorage` key `organize-calendar-subview-v1` (whitelist `{month, week, day}`; default `month`; injection probe defaults to `month` on invalid value — same posture as D7).

**Calendar-internal navigation.** Prev/Next month (or week/day) buttons + a "Today" button reset to the current month/week/day. Prev/Next state lives in `calendar-view.js` module-local state; not persisted across sessions (intentional — opening the calendar lands you on "current month"; explicit choice each session).

## D6 — Calendar drag-to-reschedule

**Decision: drag within calendar only; PATCH `due: <new-iso-date>` with If-Match (v1.14.4 ETag preserved).**

**Drop on a different calendar day.** Triggers `PATCH /api/webapp/items/:id { due: '<new-iso>' }` with `If-Match: <captured-etag>` from the item's render-time ETag (same pattern as list-view edit-form's per-item ETag retention).

**Drop on the same day = no-op.** Don't fire PATCH. Avoids spurious ETag bumps + audit rows.

**Drop on a cell outside the current month** (only relevant in Month view's grayed-out trailing cells from prev/next month). Treat as a normal drop; the item's `due` updates to that date; the calendar may re-render to show the affected month if cross-month navigation is implicit (v1.15.0 keeps the month static; the item simply moves to a cell that's currently grayed out, indicating it'll be in next month).

**412 conflict path.** Same handling as list view: render the conflict panel + keep the card visually in its source position (rollback via D12 optimistic-with-rollback).

**Cross-view DnD (drag from list to calendar, etc.) — OUT OF SCOPE for v1.15.0.** Cross-view DnD requires cross-module event coordination (the dragged element's source view doesn't know about the target view's drop handlers). v1.15.0 keeps DnD scoped to within the current view: kanban DnD = within kanban; calendar DnD = within calendar. Documented in D16 TODO.

**Drag from calendar to list/kanban — OUT OF SCOPE for v1.15.0.** Same reason.

## D7 — View switcher state persistence

**Decision: `sessionStorage` key `organize-view-state-v1`; whitelist `{list, kanban, calendar}`; default `list`; injection-probe defaults to `list` on invalid value.**

**State lifecycle.**

  - On boot: read `sessionStorage[organize-view-state-v1]`; if value matches whitelist, render that view; else default to `list` AND overwrite the bad value with `'list'` (defense-in-depth — clears injection attempts).
  - On view-switch: write the new value immediately.
  - On filter chip change: state preserved (filter chips are global; same as v1.14.0 pattern).
  - On detail navigation (open detail) → back: returns to the originating view (D8 explains).

**Why sessionStorage (not localStorage).** Per-tab state. Closing the webapp tab clears it; reopening lands on default `list` view. localStorage would persist across tabs and reopens, which feels stale for a navigation-mode preference — users typically want to pick a view per session.

**Why include the whitelist (defensive).** Same pattern as v1.14.0 filters (`organize-filter-state-v1`) which whitelists chip values. A bad value in sessionStorage (manually edited; injected via DevTools probe) would otherwise crash the renderer. Whitelist + default + overwrite is the standard Anti-Slop §6 input validation discipline applied to client-stored state.

**Injection-probe coverage.** QA Sub-Phase B will set `sessionStorage[organize-view-state-v1] = 'kanban\"; alert(1)'` and reload; expected behavior is the value is rejected (whitelist mismatch), default `list` renders, the bad value is overwritten with `'list'`. No script execution path because the value is read with `JSON.parse` (no eval); the whitelist comparison is a strict-equal check on a string set.

## D8 — Cross-view state preservation

**Decision: filter chips persist; multi-select is List-only (exits on view switch); detail panel is a top-level overlay returning to the originating view.**

**Filter chips (type, status, tag).** Persist via the existing `organize-filter-state-v1` (v1.14.0 pattern). Apply to all three views — kanban + calendar consume the same filtered item set. Switching from List with `status:active` filter to Kanban shows the same active items grouped by goal; switching back to List preserves both the filter AND the selection state (per the existing v1.14.6 D9 mode-exit rules — view switch is NOT in the mode-exit transition set; ALSO see below for multi-select carve-out).

**Multi-select state (v1.14.6 in-memory).** List-only. View switch (List → Kanban or List → Calendar) EXITS multi-select mode and clears `selectedIds` — same pattern as v1.14.6 D9 filter-change-exits-select. Rationale: multi-select tools (Delete / Re-parent / Complete bulk) are list-affordance-shaped; kanban has its own DnD-driven re-parent; calendar has its own DnD-driven reschedule; running both in parallel = UX confusion. v1.15.x+ may extend multi-select to kanban (TODO D16), but v1.15.0 keeps it list-scoped.

**Detail panel.** Top-level overlay — a single `<div id="detail-panel">` that overlays the current view. Opening detail does NOT change the underlying view's state; closing detail returns to the originating view (same selection, same scroll position, same filters). Implementation: `_currentViewMode` module variable tracks list/kanban/calendar; detail panel's "back" button calls `setView(_currentViewMode)`; the underlying view's state is preserved (kanban scroll position, calendar month, etc.) via per-view module state that is NOT torn down on view switch.

**Per-view module state.** Each view module (list-view via app.js itself; kanban-view.js; calendar-view.js) keeps its OWN render state (kanban column scroll positions; calendar current month). On view switch, the inactive view's state lives in module-local variables; on view switch back, the rendered DOM is rebuilt from those variables. NOT in sessionStorage — module state is fine for "this session" UX; sessionStorage is reserved for the view-switcher mode itself + filter chips.

**Edge case: detail panel open + view switch attempt.** Disabled; the view switcher buttons are visually disabled while detail is open. User must close detail first. This matches v1.14.6 R9 mutual-exclusion pattern (only one mode active at a time).

## D9 — Configurable BroadcastChannel name via `/api/webapp/config`

**Decision: NEW endpoint `GET /api/webapp/config`; auth-protected; read-only metadata; client caches on boot; falls back to hardcoded `'organize-mutations-jarvis'` if endpoint 404 or fetch fails.**

**Endpoint shape.**

```
GET /api/webapp/config
Authorization: tma <initData>

200 OK
Cache-Control: no-store
Content-Type: application/json

{
  "ok": true,
  "botUsername": "ai_jarvis_bot",
  "broadcastChannelName": "organize-mutations-ai_jarvis_bot"
}
```

**Channel name shape.** `'organize-mutations-' + botUsername`. The botUsername comes from `bot.api.getMe()` cached in the gateway at boot (currently logged at `src/gateway/index.ts:1463` but NOT retained; Phase 2 adds a `botUsername` property alongside `botUserId` at `:349` and exposes it via the `WebappServerDeps` chain).

**Auth.** Same `authenticateRequest()` chain as items routes. 401 on missing/invalid initData; 403 on allowlist miss; 200 on success. Privacy posture: botUsername is public information (visible in any Telegram chat with the bot); leaking it to an unauthenticated probe is not a CRITICAL exposure, but the auth chain matches existing webapp patterns and keeps the surface minimal.

**No audit row.** `/api/webapp/config` is a read-only metadata fetch; matches the audit posture of `GET /api/webapp/items` (no audit on read; only on mutate). Documented in D11.

**Client integration.**

  - On boot (in `app.js` initialization), fetch `/api/webapp/config` BEFORE setting up BroadcastChannel.
  - On 200: `ORGANIZE_MUTATIONS_CHANNEL = response.broadcastChannelName`. Set up BC with the dynamic name.
  - On 404 (older server without the endpoint): fall back to the hardcoded `'organize-mutations-jarvis'`. Log a `console.info` once for forensic visibility.
  - On network error / 401 / 403 / 500: same fallback; same `console.info`.

**Why fetch on boot, not lazily.** BroadcastChannel must be set up BEFORE any user mutation fires — otherwise the first PATCH/DELETE goes unbroadcast. Boot-time fetch is the only sequence point. The fetch is cheap (no DB; just returns a constant string per-bot); ~5-10ms latency.

**Cache lifetime.** Module-local variable; persists for the lifetime of the page (no refetch). The botUsername doesn't change during a session — refetching is wasted bandwidth.

**Why expose this NOW (not v1.18.0+ when Avengers ships).** The work is small (~60 LOC server + ~20 LOC client + 3 endpoint tests). The hardcoded `-jarvis` suffix is technical debt; eliminating it now means v1.18.0+ doesn't need to touch v1.15.0 code (Avengers becomes a config rollout, not a code change). The endpoint also creates a precedent for future webapp-facing config (feature flags, version banners, etc.) — useful structural anchor.

**Where in the route mount tree.** New file `src/webapp/config.ts`; new `mountWebappConfigRoutes(app, deps)` function; called from `itemsRoute.ts` (existing aggregator) — matches the read/mutate/complete/create pattern. `deps` extends `ItemsRouteDeps` with a `botUsername: string` field. The gateway's `WebappServerDeps` (in `src/gateway/index.ts`) gains a `botUsername` field populated at `start()` from `bot.api.getMe()`.

**Edge case: gateway hasn't called `start()` yet.** Webapp server is started AFTER the gateway's `start()` per `src/gateway/index.ts:1477`; by the time the webapp accepts requests, botUsername IS cached. If `getMe()` fails (rare; network issue at boot), the gateway logs a warn (existing behavior at `:1465`); the webapp's `botUsername` falls back to `'jarvis'` (the default; matches the v1.14.5 hardcode). Phase 2 binding: `WebappServerDeps.botUsername` defaults to `'jarvis'` if the gateway's getMe failed.

## D10 — `_internals.ts` extraction model

**Decision: A — NEW `src/organize/_internals.ts`. Exports `writeAtomically`, `serializeItem`. storage.ts + trash.ts both import.**

## D12 — Optimistic vs pessimistic DnD UX

**Decision: A — optimistic. Move card immediately on drop; rollback on 412 / 4xx / 5xx / network error.**

**Drop sequence (binding for Phase 2).**

  1. **On drop:** capture source state — `sourceColumn` (kanban) or `sourceCellDate` (calendar); `sourceIndex` (position within source list); `itemEtag` (from currentEtagMap).
  2. **Visually move:** detach the card DOM node from source; insert into target column/cell; update `renderedItems[itemIdx].parentId` (kanban) or `.due` (calendar) in memory.
  3. **Fire PATCH:** with `If-Match: <itemEtag>` (D6 pattern; v1.14.4 ETag/If-Match contract honored).
  4. **On 200:** capture the new ETag from response header; update currentEtagMap; broadcast on BroadcastChannel; success toast (subtle — no toast for routine reparent/reschedule; only for errors).
  5. **On 412 (conflict):** ROLLBACK — re-attach the card to sourceColumn/sourceCellDate at sourceIndex; revert renderedItems entry; show the v1.14.4 conflict banner with the 412 envelope's `currentItem` so the user can choose Reload or Save Anyway. (Save Anyway from kanban/calendar = re-fire PATCH without If-Match; same as list view's Save Anyway path.)
  6. **On 4xx / 5xx / network error:** ROLLBACK — same as 412 minus the conflict banner; show a generic error toast ("Couldn't move this item — try again.").

**Animation.** On rollback, animate the card back to its source position (CSS transition, ~200ms) so the user sees the rejection rather than a jarring snap. Optimistic-UX foot-gun is the silent-nothing-happens case; the animation closes that.

**Source-state retention scope.** Per-drag in-flight only. If multiple drags fire concurrently (rare; tap-pick-tap-drop is single-card by construction; HTML5 DnD on desktop is single-card per `dataTransfer`), each has its own captured source. `MAX_BULK_INFLIGHT = 10` from v1.14.6 carries over; DnD fetches share the same concurrency limiter (drops queued past 10 inflight wait their turn).

**Why optimistic (not pessimistic).**

  - **Match modern UX.** Trello, Asana, Linear all do optimistic DnD. Users expect the card to move on drop.
  - **Tolerate brief flickers.** On a slow network, the card moves immediately; if 412 fires, it animates back. Users tolerate this; it's better than a 500ms-2s pause where nothing happens.
  - **Conflict UX is already the v1.14.4 banner.** The infrastructure for "show the user a conflict and let them resolve" already exists; DnD reuses it.

**Pessimistic alternative considered + rejected.** Would gray-out the card on drop, show a spinner, wait for 200, then commit. Adds visual chrome (spinner state); slow on poor networks. Skip.

## D13 — DnD source-spoofing security

**Decision: NO additional client-side guard needed. Cross-origin DnD is browser-blocked; same-origin cross-tab is server-side-auth-blocked.**

**Threat model.** Could a user drag a card from another user's webapp tab into mine, triggering a PATCH on someone else's item?

**Cross-origin (different bot's webapp at a different origin).** Browsers block dataTransfer reads across origins. The destination tab's drop handler receives an empty/sanitized `event.dataTransfer.types`. Native browser sandbox.

**Same-origin cross-tab (same bot's webapp open in two tabs, two users).** This requires both users to share a browser profile (rare; intentional sharing = different threat model). Even then, the dataTransfer DOES carry data across same-origin tabs; the dropped item's id IS readable. BUT the destination tab's PATCH would carry the destination tab's `Authorization: tma <initData>` header — that initData is bound to the destination tab's user. The server's PATCH handler scopes by userId from initData; the dragged item's id (which lives in the source tab's user's data dir) is NOT in the destination tab's user's data dir → server returns `404 ITEM_NOT_FOUND`.

**Verification (Phase 2 binding test).** QA Sub-Phase B integration test: User A's PATCH-by-id on an item that exists only in User B's data dir → 404. Already covered by ADR 013 §D14 #3 (cross-user PATCH parentId test). Reuse / extend to the full DnD path.

**No client-side guard.** Adding a "the dragged item must belong to the current user's webapp tab" check would be redundant with server-side auth + scoping AND incorrect in principle (clients shouldn't be the security boundary). Documented in D15 KNOWN_ISSUES + ADR Risks.

## D14 — Calendar at scale

**Decision: A — client-side filter for v1.15.0; defer server-side `?month=YYYY-MM` to v1.15.x with a documented trigger.**

**Behavior at typical scale.** Server returns all items in the user's data dir on `GET /api/webapp/items` (already the v1.14.0 contract; no pagination yet). Calendar-view.js filters in-memory: `renderedItems.filter(i => i.due && monthGrid(currentMonth).some(cell => isSameDay(cell.date, parseISO(i.due))))`.

**Memory + render budget at 5000 items.**

  - Items array: 5000 × ~500 bytes = ~2.5MB JSON. Already loaded for List view; no marginal cost for calendar.
  - Filter to current month: ~30-50 visible items (typical).
  - DOM: 42 cells × (header + 0-3 item titles) = ~50-150 DOM nodes for the visible month grid. <100KB DOM weight.
  - Filter loop: O(N) over 5000 items × O(42) cells = ~210K isSameDay comparisons. ~5-10ms on a typical mobile JS engine. Acceptable.

**Why defer server-side filter.** Adding `?month=YYYY-MM` query param surface adds: (a) a server-side date-range filter implementation (~50 LOC); (b) a separate fetch on every month-navigation click (extra latency vs in-memory filter); (c) cache invalidation logic (when user adds an item to current-month-but-via-PATCH-on-another-month, the cached month set goes stale). For v1.15.0 typical scale, the server-side filter is premature optimization.

**v1.15.x trigger.** If users with >2000 items report calendar slowness OR P95 calendar-render time exceeds 500ms in real-world telemetry, ship the server-side filter. KNOWN_ISSUES TODO entry (D15 + D16).

**Week and Day views.** Smaller filter window; same O(N) loop; even faster. Same defer rationale.

**Month-navigation prefetch.** Out of scope for v1.15.0. The current month's items are filtered from the already-loaded `renderedItems`; navigating to next/prev month re-runs the filter on the same array — no fetch. Future server-side filter would need to consider prefetching adjacent months.
