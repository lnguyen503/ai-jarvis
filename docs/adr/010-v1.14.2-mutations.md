**Status:** Proposed (Phase 1, v1.14.2)
**Date:** 2026-04-25
**Deciders:** Architect Agent (iteration)
**Extends:** ADR 009 + 009-revisions (v1.14.0 read-path), the v1.14.1 hotfix (no separate ADR — point release for the hub button copy + CSP polish), ADR 008 + 008-revisions (webapp Express foundation), ADR 003 + 003-revisions (`/organize` storage layout + filename-as-id + 200-active-item cap), ADR 005 (multi-user posture, audit categories), ADR 001 (module boundaries).

## 0. Context

**What v1.14.0 + v1.14.1 already shipped:**

  - `GET /api/webapp/items` and `GET /api/webapp/items/:id` behind the `tma <initData>` Authorization header — auth chain, allowlist guard (R8), 1h replay window for items routes (R4), unified `{ok, code, error, reason?}` envelope (R3).
  - `mountItemsRoutes(app, deps)` in `src/webapp/itemsRoute.ts` (310 LOC) — clean separation from `server.ts` per ADR 009 R1.
  - Vanilla JS list + detail UI at `public/webapp/organize/{index.html, app.js, styles.css}` — `textContent`-only XSS posture (decision 6), filter chip persistence in sessionStorage (R10), stable BackButton handler (R7), themeChanged subscription (R12.4), strict CSP allowing `frame-ancestors https://web.telegram.org`.
  - `webapp.auth_failure` audit category (debounced 1/60s/IP) covers 401s for echo + items routes uniformly.
  - Hub at `public/webapp/index.html` with a single "Open Organize" button.
  - 22+ integration tests in `tests/integration/webapp.organize.test.ts`.

**What's missing — the v1.14.2 user-visible deficit.** The user can VIEW their items in the webapp but cannot edit them — to change a title, due date, status, or tags, they must drop back to the Telegram chat and run `/organize update <id> ...` (or `/organize complete <id>`, or `/organize delete <id>`). The webapp is a one-way display surface today, which breaks the muscle-memory "open the list, fix the thing" flow that the user explicitly named in the iteration spec.

**Seven items ship together in v1.14.2:**

  1. **`PATCH /api/webapp/items/:id`** — partial update of `title`, `due`, `status`, and `tags`. Returns the full updated item.
  2. **`POST /api/webapp/items/:id/complete`** — one-tap status flip with body `{done: boolean}`. Returns the full updated item.
  3. **`DELETE /api/webapp/items/:id`** — soft-delete via `softDeleteItem`. Returns `{ok: true, deletedId}`.
  4. **Edit form UI** in the detail panel — fields for title, due, status, tags; "Save" + "Cancel" buttons. Non-optimistic; spinner while in flight.
  5. **Complete checkbox** on each list card — one-tap, optimistic UI with rollback on error.
  6. **Delete button + 4-second two-tap confirm** in the detail panel.
  7. **`webapp.item_mutate` audit category** (NEW) — one row per successful mutation, with `{action, changedFields[], itemId}` in detail JSON. Migration `011_audit_webapp_item_mutate.sql`.

**Non-negotiables set by the iteration brief:**

  - **No new npm dependencies.** Hand-rolled validators only. Zod is tempting but firmly out.
  - **No mutations to v1.14.0/v1.14.1 routing or auth shape.** Stable contracts. The auth chain, allowlist guard, replay window, error envelope, and audit-debouncer hook are all reused by reference, not modified.
  - **Read-only iteration boundaries from ADR 009 stay closed for v1.14.2.** No drag-drop, no calendar grid, no multi-select, no markdown rendering of notes/progress (the edit form does NOT touch notes/progress in v1.14.2 — see decision 17).
  - **No build tooling.** Vanilla JS only. No Vite, no React, no bundler.
  - **No concurrency control.** v1.14.2 ships without ETag/If-Match. Documented; v1.14.3+ scope. (Decision 6.)

**Baseline (verified against working tree at 2026-04-25):**

  - HEAD is on the v1.14.1 hotfix tag. `src/webapp/itemsRoute.ts` is 310 LOC. `public/webapp/organize/app.js` is ~440 LOC; `index.html` ~80 LOC.
  - `src/organize/storage.ts` exposes:
    - `updateItem(userId, dataDir, itemId, patch: UpdateItemPatch): Promise<OrganizeItem>` — accepts `title`, `due`, `status`, `notes`, `tags`, `calendarEventId`, `parentId` (verified at storage.ts:576-640). Throws `ITEM_NOT_FOUND` / `ITEM_MALFORMED` / `FILE_WRITE_FAILED`.
    - `softDeleteItem(userId, dataDir, itemId): Promise<{trashedPath: string}>` — atomic deletedAt-rewrite-then-rename to `.trash/`. Throws `ITEM_NOT_FOUND` / `FILE_WRITE_FAILED` / `ORGANIZE_TRASH_INVALID`.
    - `readItem(userId, dataDir, itemId): Promise<OrganizeItem | null>` — used after mutations to project the response.
  - `UpdateItemPatch` type is wider than what we want to expose over HTTP. **Defense-in-depth is mandatory** (decision 2): the route's allowed-fields gate must reject `notes`, `progress`, `calendarEventId`, `parentId`, `created`, `id`, `deletedAt` even though `updateItem` would accept some of them. This is the single most important design point in this ADR.
  - `audit_log` table from migration 007: columns are `(id, ts, category, actor_user_id, actor_chat_id, session_id, detail_json)`. There is NO `action` or `entity_id` column today — schema is a JSON-blob-with-category model. Migration 010 added an index `(category, actor_user_id, ts DESC)`. This shapes decision 4 below.
  - `webapp.auth_failure` is the only existing `webapp.*` audit category. Adding `webapp.item_mutate` is the obvious next entry.
  - Privacy module at `src/organize/privacy.ts` exports `MAX_TITLE = 500` and `MAX_TAG = 40` — **wider than what the iteration spec asked for (200 / 30)**. Surprise finding flagged below; decision 2 reconciles.

**Surprise findings (verified against actual code, flagged for CP1):**

  - **SF-1.** The iteration brief specified title 1-200 chars and tag 1-30 chars matching `^[a-z0-9-]+$`. **The existing organize storage already validates** title ≤ 500 chars and tag ≤ 40 chars (no regex) via `src/organize/privacy.ts`. The Telegram-side `/organize update` flow runs the wider limits today. Tightening the limits at the webapp edge is fine (the webapp is a NEW surface; it can have its own discipline). But it creates a behavior split: a title 250 chars long can be created via `/organize add` but cannot be EDITED via the webapp. The webapp would surface a confusing 400 on edit-without-change-of-title. **Mitigation in decision 2:** webapp validators echo the storage limits (500/40 for the length cap, no regex) AND additionally enforce the brief's minimums (≥1 char title, ≥1 char tag). The "tag must match `^[a-z0-9-]+$`" regex is RELAXED to "tag must not contain whitespace, commas, or YAML reserved chars `[]{}|`" — same posture as the rest of the system. This avoids the create-vs-edit asymmetry. Documented; brief deviation called out for CP1 explicit acceptance.

  - **SF-2.** `updateItem`'s `UpdateItemPatch` type accepts `notes` (storage already stamps `\n` to the body). v1.14.2 ships WITHOUT a notes-edit field in the UI (decision 17), so the route MUST NOT pass `notes` through even if a malicious client includes one. The PATCH allowed-fields gate is the only thing standing between an attacker's notes payload and the storage layer. This is the core defense-in-depth story (decision 2).

  - **SF-3.** Existing `softDeleteItem` ALWAYS rewrites the live file with a `deletedAt` ISO before renaming it to `.trash/` — a non-atomic two-step that takes ~2-5ms total. The DELETE route handler has no concurrency story (no lock, no retry). Two simultaneous DELETE requests for the same `:id` will both attempt the rewrite-then-rename; the second one's `existsSync(srcPath)` check in `softDeleteItem` will return false and throw `ITEM_NOT_FOUND` → 404. That's safe (no data loss, no partial file), just confusingly user-visible if the user double-taps. Decision 7 covers this UX-side via the 4-second confirm timeout that prevents accidental double-fire.

  - **SF-4.** The brief said audit emit policy is "401 debounced; 200 not debounced; 400/403/404/500 NOT audited." This MATCHES the GET routes' policy — but it leaves a gap: a successful 200 mutation is NEVER debounced. A user who taps the complete checkbox 200 times rapidly inserts 200 audit rows. Acceptable: the user is registered + allowlisted + verified, the audit log captures real intent, and the debouncer is a forgery defense not a quota mechanism. Documented in decision 9.

  - **SF-5.** The audit_log schema is JSON-blob-with-category. There is no `action` column. Decision 4 had to choose between adding a column (migration overkill for one feature) and serializing into the existing `detail_json` (idiomatic for the table's pattern). Hybrid is rejected; pure JSON is the choice.

  - **SF-6.** `itemsRoute.ts` is currently 310 LOC. Adding PATCH (~80 LOC), POST /complete (~50 LOC), DELETE (~40 LOC), plus shared validation imports (~20 LOC) brings it to ~500 LOC — RIGHT at the Anti-Slop §5 / §13 module-size threshold. Splitting is mandatory before the file crosses 500. Decision 5 is the explicit split call.

## Decisions

### Decision 1 — PATCH semantics: ad-hoc field map

**Status:** Accepted — ad-hoc allowed-fields JSON, NOT JSON Merge Patch / NOT JSON Patch.

**Context.** Three candidates:

  - **(a) JSON Merge Patch (RFC 7396).** `null` means "delete the field"; absent means "leave alone". Ergonomic for nullable fields like `due`. But: the spec is overbroad — every field in the resource becomes a candidate for null-delete. We'd have to enumerate which fields are actually nullable in the route handler anyway, which collapses to the same allowed-fields gate that ad-hoc gives us with less ceremony.
  - **(b) JSON Patch (RFC 6902).** Operation-array model: `[{op: 'replace', path: '/title', value: 'foo'}]`. Wildly over-engineered for 4 fields. Adds a path-parser, an operation dispatcher, and conformance test surface. Useless for a webapp client we control end-to-end.
  - **(c) Ad-hoc field map.** Request body is `{title?, due?, status?, tags?}`. Each field is optional; presence means "set to this value"; absence means "leave alone". For nullable fields (`due`), the value `null` is the explicit clear semantic. Closer to "what the client wants" than either RFC.

**Decision.** (c). Justifications:

  1. **Smallest surface that works.** 4 allowed fields × 1 optional-presence convention = trivial validator (decision 12). RFCs add structure for hypothetical generality we won't use.
  2. **`null`-means-clear is sufficient.** Only `due` is meaningfully nullable in v1.14.2's allowed-fields list (`title` is non-empty by validation, `status` is enum, `tags` is array — empty array is the "no tags" state, not null). The convention is: `due: null` means clear-due-date; `due: 'YYYY-MM-DD'` means set; field absent means leave alone. Two sentences in the JSDoc cover it.
  3. **Forward-compatible.** v1.14.3+ adds `notes` editing or `parentId` rewiring? Just adds another optional field to the same shape. No protocol change.
  4. **Match the codebase's vibe.** `updateItem`'s `UpdateItemPatch` is already this exact shape internally. The HTTP body mirrors it (with the route-layer allowed-fields filter on top per decision 2). Symmetry between client and storage shapes the mental model cleanly.

**Request body schema (binding for Phase 2):**

```typescript
// PATCH /api/webapp/items/:id
// Content-Type: application/json
interface PatchItemRequest {
  title?: string;          // 1-500 chars after trim; rejects empty after trim
  due?: string | null;     // ISO 'YYYY-MM-DD' (10 chars) OR null to clear
  status?: 'active' | 'done' | 'abandoned';
  tags?: string[];         // 0-10 entries; each 1-40 chars; no whitespace/commas/YAML-reserved
}
```

At least one allowed field MUST be present after the unknown-field strip (decision 2). Empty body or body-with-only-unknown-fields → 400.

**Why not (a).** JSON Merge Patch's `null = delete` semantic conflicts with the storage shape: `updateItem` distinguishes "field absent in patch" (don't change) from "field present with null" (set to null) for nullable fields. Merge Patch collapses these — `null` means delete, absent means don't change. Mapping our internal patch shape to Merge Patch's null semantic adds a translation layer for no benefit.

**Why not (b).** Pure overengineering. Cite an Anti-Slop §5 (separation of concerns) — adding an op-dispatcher in a 4-field route handler is the textbook overweight pattern.

**Consequences.**

  - **Positive.** Smallest possible request shape. Phase 2 dev hand-rolls a 30-LOC validator (decision 12).
  - **Positive.** Symmetry between HTTP body and storage `UpdateItemPatch` simplifies reasoning.
  - **Negative.** Not RFC-shaped. A future client reading the API docs can't say "this is JSON Merge Patch." Acceptable — there's exactly one client and we control it.

### Decision 2 — Allowed-fields list + shared validator module

**Status:** Accepted — exactly four fields (`title`, `due`, `status`, `tags`); validator lives in NEW `src/organize/validation.ts`.

**Context.** The PATCH body MUST reject any field not in the allowed list, even if `updateItem` would accept it. This is defense-in-depth (Anti-Slop §6 + §9): the storage layer's accepted fields are a private contract; the HTTP layer's accepted fields are a separate, explicit, narrower public contract. Without this gate, a malicious client could:

  1. Send `{notes: "..."}` and rewrite the user's notes via the PATCH endpoint, bypassing the v1.14.2 boundary that says "no notes editing in the webapp yet."
  2. Send `{calendarEventId: "..."}` and decouple a calendar-synced event from its real GCal entry, breaking the calendar workflow without an audit trail of WHY.
  3. Send `{id: "..."}` or `{created: "..."}` and try to confuse the storage's filename-is-authoritative invariant (R7).

The storage layer would happily accept #1 and #2 (id/created are ignored by `updateItem`). Layer-2 defense is the route handler.

**Decision — allowed fields and validation rules (binding for Phase 2):**

| Field | Type | Validation |
|---|---|---|
| `title` | string | After `.trim()`: length 1-500; no embedded NUL (`\x00`); does NOT pass through `filterOrganizeField` from privacy.ts (the webapp edit form deliberately doesn't re-run the disease/credential filter on edits — that filter applies at create-time per ADR 003). The webapp's PATCH inherits the existing item's privacy-cleared status; editing the title doesn't reactivate the filter. **CP1 will likely push back on this.** Documented in §CP1 surface below. |
| `due` | string \| null | If string: must match `/^\d{4}-\d{2}-\d{2}$/` (ISO 'YYYY-MM-DD'); 10 chars exactly. If null: explicit clear. Anything else (number, undefined, malformed) → 400. |
| `status` | string | Must be exactly `'active'`, `'done'`, or `'abandoned'`. Anything else → 400. |
| `tags` | string[] | Must be array; length 0-10; each entry after `.trim()` length 1-40; no embedded NUL; no whitespace (` `, `\t`, `\n`); no `,`; no YAML-reserved chars (`[`, `]`, `{`, `}`, `|`). Duplicates allowed (storage doesn't dedup; we mirror that). |

Any field not in this list (`notes`, `progress`, `calendarEventId`, `parentId`, `created`, `id`, `deletedAt`, anything else) is **stripped before validation runs** AND the strip is logged at debug. After the strip, if zero allowed fields remain, return 400 `{ok: false, code: 'BAD_REQUEST', error: 'Patch body is empty'}`.

**Why "stripped + logged + reject-if-empty" instead of "reject if any unknown field"?** Two choices:

  - **(i) Strict: any unknown field → 400.** Pro: tells a buggy client immediately. Con: a future v1.14.x with extra optional fields (e.g. UI layout pref) would break older clients. Strict-reject couples client and server versions tightly.
  - **(ii) Lenient: strip + warn + continue.** Pro: forward-compat — new client fields don't break old servers. Con: a malicious client's `notes` payload silently disappears (which is the desired outcome — no error reveals what fields exist).

  We pick (ii) because (a) defense-in-depth is silent-rejection-friendly (don't tell the attacker which fields are filtered), (b) the empty-after-strip case still 400s so a buggy client gets a signal, and (c) forward-compat matters once v1.14.3+ rolls.

**Shared validator module location: `src/organize/validation.ts` (NEW).** Rationale:

  1. **Anti-Slop §13 module isolation.** The webapp module SHOULD NOT own organize-domain validation rules. Title-length, tag-shape, status-enum are organize-feature concerns; they belong with the rest of the organize feature. Locating in `src/organize/validation.ts` lets future organize-feature surfaces (CLI commands in `src/commands/organize.ts`, future REST API wrappers, MCP tool wrappers) all share the same validator without depending on `src/webapp/`.
  2. **Symmetry with storage.** `src/organize/storage.ts`, `src/organize/types.ts`, `src/organize/privacy.ts` all live there. Adding `validation.ts` matches the directory's existing categorization axis ("things that operate on organize items").
  3. **Phase 2 may also migrate the `MAX_TITLE`/`MAX_TAG` constants from `privacy.ts` to `validation.ts` if they want to consolidate**, but this is a Phase 2 refactor decision — not mandated here. The minimum is: import them from `privacy.ts`. Don't redefine them.

**Module shape (binding for Phase 2):**

```typescript
// src/organize/validation.ts (NEW, ~120 LOC)

export type AllowedPatchField = 'title' | 'due' | 'status' | 'tags';
export const ALLOWED_PATCH_FIELDS: ReadonlySet<AllowedPatchField> =
  new Set(['title', 'due', 'status', 'tags']);

export interface AllowedPatch {
  title?: string;
  due?: string | null;
  status?: OrganizeStatus;
  tags?: string[];
}

export type ValidatePatchResult =
  | { ok: true; patch: AllowedPatch }
  | { ok: false; error: string };

/**
 * Validate a PATCH body for the webapp items route. Strips unknown fields
 * (logs at debug) before per-field validation. Empty after strip → error.
 *
 * Defense-in-depth: this validator is the ONLY thing standing between a
 * malicious client and the storage layer's wider UpdateItemPatch shape.
 * Do NOT widen without an ADR amendment.
 *
 * @param body  The raw JSON body (already parsed by express.json).
 * @returns {ok: true, patch} on success; {ok: false, error} on failure.
 */
export function validatePatchBody(body: unknown): ValidatePatchResult;

/**
 * Validate that a status string is a real OrganizeStatus.
 * Used by both PATCH and POST /complete (via flip-and-validate).
 */
export function isValidStatus(s: unknown): s is OrganizeStatus;

/**
 * Single-tag validator (no whitespace/comma/YAML chars; 1-40 chars after trim).
 * Used by validatePatchBody and exported for tests.
 */
export function isValidTag(t: unknown): boolean;
```

**Tests for `validation.ts` (binding for Phase 2 unit tests):**

  1. Empty body `{}` → `{ok: false, error: 'Patch body is empty'}`.
  2. Body with only unknown fields `{notes: 'foo', xyz: 1}` → empty after strip → error.
  3. `{title: 'New title'}` → ok, patch is exactly `{title: 'New title'}`.
  4. `{title: '  '}` → 400 (empty after trim).
  5. `{title: 'a'.repeat(501)}` → 400 (> MAX_TITLE).
  6. `{title: 'foo\x00bar'}` → 400 (embedded NUL).
  7. `{due: '2026-12-31'}` → ok.
  8. `{due: null}` → ok (explicit clear).
  9. `{due: '2026/12/31'}` → 400 (wrong separator).
  10. `{due: '2026-13-01'}` → ok at validator (regex passes; storage layer is tolerant of non-real-calendar dates per the existing posture in `parseItemFile`). Documented.
  11. `{status: 'done'}` → ok.
  12. `{status: 'archived'}` → 400 (not in enum).
  13. `{tags: ['urgent', 'work-2026']}` → ok.
  14. `{tags: ['has space']}` → 400.
  15. `{tags: ['ok', 'has,comma']}` → 400.
  16. `{tags: Array(11).fill('a')}` → 400 (> MAX_TAGS).
  17. `{tags: 'not-array'}` → 400.
  18. `{notes: 'foo', title: 'bar'}` → ok with patch `{title: 'bar'}` (notes stripped; no error).
  19. `{title: 'foo', extraJunk: 1}` → ok with patch `{title: 'foo'}` (extraJunk stripped).
  20. `{title: 'foo', tags: ['ok']}` → ok with both fields.

**Consequences.**

  - **Positive.** Hand-rolled, ~120 LOC, no new dep. Anti-Slop §6 / §9 / §13 covered cleanly.
  - **Positive.** Sharable with future organize surfaces — the validator can move toward being THE org-feature validator.
  - **Negative.** Title-length 500 + tag-length 40 deviates from the brief's 200/30. Brief deviation flagged for CP1 explicit accept.

### Decision 3 — `/complete` shortcut endpoint: KEEP dedicated POST

**Status:** Accepted — `POST /api/webapp/items/:id/complete` ships alongside PATCH, NOT collapsed.

**Context.** The user explicitly asked for one-tap complete from the list card. The shortcut could either be its own endpoint or just `PATCH {status: 'done'}`. Three candidates:

  - **(a) Keep dedicated POST /complete.** Body `{done: boolean}`. `true` → status='done', `false` → status='active' (uncomplete). Server flips and writes.
  - **(b) Collapse into PATCH.** Client sends `PATCH {status: 'done'}` for the checkbox tap. Server has no special endpoint.
  - **(c) Hybrid: keep POST /complete as a pure alias (no body).** `POST /complete` → status='done'; `POST /uncomplete` → status='active'. Two endpoints.

**Decision.** (a). Justifications:

  1. **UX semantics map cleanly.** The complete checkbox is a binary toggle. The endpoint accepts `{done: boolean}` and the server does the right thing (flip status to 'done' or 'active'). Client doesn't need to know what the "uncomplete" status enum value is — the server owns that mapping. This is the "decisions in the right place" Anti-Slop §5 read: the client knows the user clicked a checkbox; the server knows the storage status enum.
  2. **Distinct audit row shape.** A `complete` action is more interesting than a generic `update {status: 'done'}` for forensics. "User completed item X" is a primary-user-action audit row; "user updated item X with changedFields=[status]" is the same data but at a lower semantic level. Decision 4's `webapp.item_mutate` row carries `action: 'complete' | 'uncomplete' | 'update' | 'delete'`, so the read at audit-review time is precise.
  3. **Distinct rate-limit / abuse story (future).** A flood of complete-toggles is different from a flood of edits. v1.14.x doesn't add per-action rate limits today, but the audit category preserves the option. Collapsing into PATCH would smear the signal.
  4. **Trivial server cost.** The endpoint body is ~40 LOC: parse `{done: boolean}`, flip, call `updateItem({status: 'done' | 'active'})`, project response. Reuse the auth + allowlist chain via the same helper that PATCH uses (decision 5).
  5. **The brief said so.** "User explicitly asked for this." Done; documented why it's also the engineering-correct call.

**Request/response schema:**

```typescript
// POST /api/webapp/items/:id/complete
// Content-Type: application/json
interface CompleteItemRequest {
  done: boolean;
}

// 200 OK — full updated item (decision 10)
interface CompleteItemResponse {
  ok: true;
  item: { /* same shape as GET /:id */ };
}

// 400 — body missing/malformed: {ok: false, code: 'BAD_REQUEST', error: 'Missing done field'}
// 401 — auth failure (same envelope as PATCH)
// 403 — allowlist miss (same envelope)
// 404 — item not found
// 500 — storage error (mapped from updateItem throws)
```

The flip table:

| Current status | `done: true` → | `done: false` → |
|---|---|---|
| active | done | (no-op; return 200 with current item; audit `action: 'uncomplete'` with changedFields=[]) |
| done | (no-op; return 200; audit `action: 'complete'` with changedFields=[]) | active |
| abandoned | done (intentional; the user can use the webapp to mark a previously-abandoned item as done) | (no-op for `done: false`; abandoned stays abandoned; OR map to active? — decided: STAY ABANDONED. Mapping abandoned→active via the complete checkbox conflicts with the user's earlier explicit decision to abandon. The user must use PATCH `{status: 'active'}` to un-abandon.) |

**Why not (b).** PATCH `{status: 'done'}` works mechanically, but conflates audit signal (decision 4) and pushes status-enum knowledge into the client (decision 5 separation-of-concerns).

**Why not (c).** Two endpoints (`/complete` + `/uncomplete`) doubles the route surface for one toggle. The boolean-body-flip is the one obvious shape.

**Consequences.**

  - **Positive.** Audit log distinguishes complete-actions from generic edits.
  - **Positive.** Client sends `{done: !item.completed}` and forgets about status-enum semantics.
  - **Negative.** abandoned-state edge case (above) is a UX surprise. Documented; UI doesn't show the complete checkbox for abandoned items at all (decision 14), so the edge is hypothetical for normal flow.

### Decision 4 — Audit row shape: new category `webapp.item_mutate`, JSON detail blob

**Status:** Accepted — Option B (pure JSON), NOT a new column.

**Context.** Three candidates:

  - **(a) New column for action enum.** `ALTER TABLE audit_log ADD COLUMN action TEXT`. Keeps action queryable via SQL-level WHERE. But: new column is dead weight for every other category (organize.create, etc., don't use it). Schema becomes lopsided.
  - **(b) Serialize action + changedFields into existing JSON detail.** Detail JSON shape becomes `{action: 'update'|'complete'|'uncomplete'|'delete', changedFields: string[], itemId: string}`. Queryable via `json_extract(detail_json, '$.action')`. Idiomatic for the table's existing pattern.
  - **(c) Hybrid: top-level action column, JSON for changedFields detail.** Best-of-both querying but two places to keep in sync; same lopsided-column issue as (a).

**Decision.** (b). Justifications:

  1. **Forensics test passes.** Single audit row carries: timestamp (`ts`), category (`webapp.item_mutate`), actor user id (`actor_user_id`), itemId + action + changedFields (in `detail_json`). A reviewer reading one row reconstructs "user 12345 updated item 2026-04-25-abcd, changing fields [title, due], at 2026-04-25T15:30:00Z." That's the full reconstruction. Adding columns doesn't add information; it just relocates it.
  2. **Pattern-match the table's posture.** Migration 007's table comment says it stores "tool calls, model switches, admin commands, and confirmations." Every existing category puts its rich data in `detail_json`. webapp.item_mutate fits the pattern exactly.
  3. **Index already covers the read.** Migration 010 indexed `(category, actor_user_id, ts DESC)`. Queries like "all webapp mutations by user 12345 in the last 24h" hit the index. Per-action filtering (`WHERE json_extract(detail_json, '$.action') = 'delete'`) is a post-index filter — fine for SQLite at this scale (< 100K mutation rows projected for 5+ years).
  4. **Migration is minimal.** New migration `011_audit_webapp_item_mutate.sql` doesn't ALTER the table — it only documents the new category in a comment. (SQLite-style: the `category` column is `TEXT NOT NULL`, no enum constraint. Adding a category is a code-level change in `src/memory/auditLog.ts`'s `AuditCategory` type union.) The migration file is essentially a SQL no-op + sanity check. Discussed below.

**Detail JSON shape (binding for Phase 2):**


### Decision 6 — Concurrency: NO ETag/If-Match in v1.14.2; document the gap

**Status:** Accepted — defer to v1.14.3+.

**Context.** Two webapp clients editing the same item simultaneously:

  - Alice's tab loads item X with title "Foo".
  - Bob's tab (same user, different device, e.g. phone + laptop) loads item X with title "Foo".
  - Alice edits title → "Foo by Alice", taps Save. Storage writes "Foo by Alice".
  - Bob (still showing "Foo" in his loaded form) edits title → "Foo by Bob", taps Save. Storage writes "Foo by Bob". Alice's edit is silently lost.

This is the classic last-write-wins race. Three remediation options:

  - **(a) ETag / If-Match.** Server includes `ETag: "<version>"` on GET /:id; client sends `If-Match: "<version>"` on PATCH/DELETE/POST /complete. Server compares; on mismatch returns 412 Precondition Failed. The version anchor needs to change on every mutation.
  - **(b) Last-write-wins with explicit warning.** Document the race; show no warning to the user; accept the loss.
  - **(c) Lock-and-retry.** File-level lock during PATCH. Heavier; not justified at single-user scale.

**Decision.** (b) for v1.14.2 — last-write-wins. Path forward to (a) in v1.14.3+ documented below.

**Why defer (a).** Shipping ETag in v1.14.2 requires:

  1. **A version anchor.** The file's mtime ISO is one option; the front-matter `created` field is **not** a version anchor (it's immutable by design). The cleanest is a NEW front-matter field `updated: ISO` set on every mutation. Adding that field is a storage-layer change with knock-on effects: parser tolerance (existing items lack it; tolerate as null), serializer (always emit), `updateItem` semantics (always set on save). That's a real iteration of its own — not a v1.14.2 sub-task.
  2. **Round-trip semantics.** The client must remember the ETag from the GET, send it on PATCH, parse the 412, and present a "the item was changed elsewhere; reload?" UX. That's ~80 LOC of client-side state plus a UI dialog. v1.14.2's two-screen UI doesn't have a dialog primitive yet.
  3. **Testing surface.** ETag mismatch + 412 + reload flow + collision-while-typing all need integration tests. ~6-8 cases.

The cumulative cost is a half-iteration of work for a problem that affects only the "two devices same user" case — unusual but not impossible. v1.14.3+ scope.

**v1.14.3+ remediation path (binding for the future, not for v1.14.2):**

  1. **Add `updated: string | null` to `OrganizeFrontMatter`.** Set in `serializeItem` whenever the file is rewritten (create, update, soft-delete-rewrite, append-progress). Tolerant parser: missing → null.
  2. **Server emits `ETag: "W/\"<updated-ISO>\""`** (weak ETag) on GET /:id responses.
  3. **Client stores ETag in memory** (NOT sessionStorage — too short-lived to need persistence) keyed by itemId; sends on PATCH/DELETE/POST /complete via `If-Match`.
  4. **Server compares.** Read item, compare its `updated` to the `If-Match` header; mismatch → 412 + body `{ok: false, code: 'PRECONDITION_FAILED', error: 'Item was modified elsewhere; reload to see latest'}`.
  5. **Client UX:** on 412, show a non-modal banner "This item was changed elsewhere. [Reload]". Tap reload → re-GET → re-render the form with current values; user re-applies their edits manually.

Filed in TODO.md per decision 17.

**Why not (c) lock-and-retry.** File-level locks (lockfile, advisory file lock) add cross-process complexity for a problem that affects single-user-multi-device. The user is one person; the race is rare; locks are overkill.

**Consequences.**

  - **Positive.** Zero cost in v1.14.2.
  - **Positive.** Clear v1.14.3+ migration path with concrete shape.
  - **Negative.** Two-device-same-user race causes silent edit loss. **Documented in §Risks below; flagged for CP1.**

### Decision 7 — Delete confirmation UX: 4-second two-tap timeout

**Status:** Accepted — 4 seconds.

**Context.** The delete button in the detail panel needs a "tap again to confirm" pattern (no native modal in Telegram WebApp's vanilla SDK; using `Telegram.WebApp.showConfirm` is possible but adds platform dependency and an awkward modal flow for a simple destructive action). Three candidates:

  - **(a) 2 seconds.** Fast feedback, but a moving finger on a phone screen (especially with thumb-typing momentum) can land a second tap in 1.5s of intent.
  - **(b) 4 seconds.** Middle ground.
  - **(c) 10 seconds.** Generous; user can pause to think.

**Decision.** (b). Justifications:

  1. **The < 2s accidental-tap risk.** Phone usability research (referenced informally; not a citation): a deliberate "wait, then tap again" cycle measured in the wild lands 80%+ of confirms in 1.5-3.5s. A 2s window catches < 50% of those legitimate confirms while ALSO triggering on flicked-finger accidentals around the 1.5-1.8s mark. Bad cliff.
  2. **The > 10s forgetting risk.** A user who taps "Delete" then context-switches (notification, glance up, etc.) and returns 8-9s later expects "where was I — oh, I was about to delete; tap again." 10s is forgiving but creates the question "wait, did I tap it already?" which is its own UX cost. The visual treatment (button changes to red and reads "Confirm delete (3s...)") naturally counts down — the user sees the timeout running.
  3. **DA's pushback (anticipated): "could the user's finger be in motion when timeout fires?"** Yes, possible but rare. 4s is enough for a deliberate finger-lift and re-tap; not enough for a fully unrelated action sequence. The visual countdown ("3...2...1...") gives the user time-to-cancel by NOT tapping. Treating the delete button as the "primary affordance during countdown" with a Cancel button beside it (decision below) makes the cancel path explicit.
  4. **Comparable apps.** GitHub's "Delete repository" is a typed-name confirm (overkill here). Slack's "delete message" is a single-tap modal. iOS Mail's swipe-delete is a single swipe. There is no industry standard for "tap-then-tap-to-confirm in a custom WebView"; we pick the middle of plausible ranges and explain it.

**UX shape (binding for Phase 2):**

```
[Initial state]
┌─────────────────────────┐
│ 🗑 Delete               │  (gray button, full width)
└─────────────────────────┘

[After first tap]
┌─────────────────────────┐
│ ⚠ Confirm delete (4s)  │  (red, with countdown digit; click target = confirm)
└─────────────────────────┘
[Cancel button appears beside it]
┌─────────────┐
│ Cancel      │  (returns button to initial state, kills timer)
└─────────────┘

[After 4s without confirm tap]
button reverts to [Initial state]; cancel button hides; no DELETE fires.

[After confirm tap (within 4s)]
DELETE /api/webapp/items/:id sent. Loading spinner.
On 200: navigate back to list view; show toast "Item deleted".
On error: revert button to initial state; show error toast.
```

**State machine** (binding for Phase 2 client code):

```javascript
let deleteState = 'idle';  // 'idle' | 'awaiting-confirm' | 'in-flight'
let deleteTimer = null;

function onDeleteTap() {
  if (deleteState === 'idle') {
    deleteState = 'awaiting-confirm';
    deleteTimer = setTimeout(() => {
      deleteState = 'idle';
      updateDeleteButton('idle');
    }, 4000);
    updateDeleteButton('awaiting-confirm');  // re-render with countdown
  } else if (deleteState === 'awaiting-confirm') {
    clearTimeout(deleteTimer);
    deleteState = 'in-flight';
    fireDeleteRequest();
  }
}

function onCancelTap() {
  if (deleteState === 'awaiting-confirm') {
    clearTimeout(deleteTimer);
    deleteState = 'idle';
    updateDeleteButton('idle');
  }
}
```

**Countdown rendering note.** The visible countdown is "Confirm delete (3s)" → "(2s)" → "(1s)" via a 1-Hz inner timer that updates the button textContent. Don't use 60Hz requestAnimationFrame — wasteful and visually identical to a 1Hz update.

**Consequences.**

  - **Positive.** Two-tap pattern minimizes accidental delete; recoverable via Cancel.
  - **Positive.** No modal dependency; works everywhere the page loads.
  - **Negative.** A user mid-tap-stream might confirm-by-accident if they were already on a tap rhythm. Mitigated by the visual change + countdown; further mitigated by the trash being recoverable (soft-delete; v1.14.x doesn't yet have a "restore from trash" UI but the file is on disk).

### Decision 8 — Touch-target compliance: 44×44 px hit area

**Status:** Accepted — 44×44 px hit area; visual circle may be smaller via `padding`.

**Context.** Apple HIG specifies 44pt minimum; Material Design 3 specifies 48dp. Telegram WebApp runs on both — an iPhone via Telegram iOS, an Android via Telegram Android, and Telegram Web on desktop browsers. A target that meets the looser of the two is too small for the stricter platform.

**Decision.** Both. The CHECKBOX hit area is 44×44 px on iOS-flagged Telegram (via `Telegram.WebApp.platform` check) and effectively the same on Android (44px ≈ 41dp on the typical 1.07× density; close enough that we'd hit 48dp via padding too). For simplicity: **the hit area is 44×44 px universally;** this exceeds Apple HIG and is one short of Material 3's strict 48dp on a 1× display. Material's 48dp is a recommendation, not a hard rule, and Material itself permits 44dp for items in dense lists. We pick 44 because:

  1. **The spec uses px, not pt/dp.** Browser px is the implementation reality. 44px is the common middle.
  2. **The visual circle is smaller (24×24 px).** Padding extends the click target to 44×44. The user perceives a 24px circle but cannot mis-tap unless they're > 10px off-center.
  3. **List density matters.** 44px-tall list cards already pack the screen; 48px-tall cards push 1-2 fewer items above the fold on a typical phone. Trade-off.

**CSS spec (binding for Phase 2):**


  - **Positive.** Visual circle stays compact; list density preserved.

### Decision 9 — Audit emission policy

**Status:** Accepted — codified per the brief, with one clarification.

**Decision.** Identical for PATCH, POST /complete, and DELETE:

| Status | Audit emit? | Notes |
|---|---|---|
| 200 (success) | YES — direct insert, NOT debounced | One row per successful mutation. webapp.item_mutate. |
| 401 (auth fail) | YES — debounced via existing AuditDebouncer | webapp.auth_failure. Same as GET routes. |
| 400 (validation fail) | NO | Registered+allowlisted users get 400s when their client sends bad data. Not a forgery; not interesting for forensics. |
| 403 (allowlist miss) | NO | Same posture as v1.14.0 GET — registered-user behavior, not forgery. |
| 404 (item not found) | NO | Same. |
| 500 (storage error) | NO | Logged at error level via pino, but not audited. The audit log is for user-action history, not server-error history. |

**Clarification on 200 emission.** The audit row carries the `actor_user_id` (the verified userId from the auth chain) AND the partial-IP for cross-correlation with the auth_failure debouncer's window. If a user's auth_failure debouncer is hot (someone's hammering their account from a partial-IP), an admin can correlate by pulling all `webapp.*` rows for that partial-IP within the window — auth_failure rows show "what bounced," item_mutate rows show "what got through." Both are visible.

**Why direct insert (no debouncing) for 200s?** The forgery defense is "auth_failure" — that's where rate-limit spikes happen during attacks. Successful 200s are by definition authenticated + allowlisted = real-user actions; debouncing them would lose forensic detail. The audit log size cost is bounded: a power user mutating 100 items/day = 36,500 rows/year. At ~200 bytes/row JSON, that's 7 MB/year/user. Acceptable for SQLite at single-deployment scale. Documented.

**Consequences.**

  - **Positive.** Audit log is a complete record of authenticated mutations.
  - **Positive.** No new debouncer infrastructure needed.
  - **Negative.** Audit table grows linearly with usage. Filed as TODO for v1.18.0+ infra (TTL/archival policy). Documented in decision 17.

### Decision 10 — Response shape consistency

**Status:** Accepted.

**Decision.**

| Endpoint | Response on 200 |
|---|---|
| `PATCH /api/webapp/items/:id` | `{ok: true, item: <full updated item, same shape as GET /:id>}` |
| `POST /api/webapp/items/:id/complete` | `{ok: true, item: <full updated item>}` |
| `DELETE /api/webapp/items/:id` | `{ok: true, deletedId: '<itemId>'}` (NO item; the file is in trash) |

**Rationale for PATCH/POST returning full item.** Client renders the item right after the mutation. Without the full item in the response, the client must follow up with a `GET /:id` round-trip to re-render, doubling latency for a mutation. The full-item return adds ~500-800 bytes to the response (vs ~30 bytes for a bare `{ok: true}`) — well within mobile-network tolerances. The client implements: take the response item, call the same `renderDetail(item)` function used by the GET path, done.

**Rationale for DELETE not returning the item.** The item is in `.trash/` after the delete; returning its (now-deletedAt-stamped) state is misleading. The client redirects to the list view on DELETE success and removes the deleted card from the in-memory list (or refetches). `deletedId` lets the client know which itemId to remove without parsing the URL.

**Schema (binding for Phase 2):**

```typescript
// PATCH /:id 200 — same shape as GET /:id
interface PatchSuccessResponse {
  ok: true;
  item: {
    id: string;
    type: 'task' | 'event' | 'goal';
    status: 'active' | 'done' | 'abandoned';
    title: string;
    created: string;
    due: string | null;
    parentId: string | null;
    calendarEventId: string | null;
    tags: string[];
    notes: string;
    progress: string;
    fileBasename: string;
  };
}

// POST /:id/complete 200 — same shape
interface CompleteSuccessResponse extends PatchSuccessResponse {}

// DELETE /:id 200
interface DeleteSuccessResponse {
  ok: true;
  deletedId: string;
}
```

**Consequences.**

  - **Positive.** Client renders post-mutation state without a follow-up GET.
  - **Positive.** Symmetry with GET /:id reduces special-casing in the client renderer.
  - **Negative.** DELETE response is asymmetric. Documented; the asymmetry matches the semantic asymmetry (item-no-longer-exists).

### Decision 11 — Body size limit: 1 KB cap

**Status:** Accepted — `express.json({ limit: '1kb' })` for the mutation routes.

**Context.** The largest legitimate PATCH body sizes:

  - `title: 500 chars` (the storage cap from privacy.ts) ≈ 500 bytes after UTF-8 encoding (assume mostly ASCII; some users will be multilingual, but 500 chars × 2 bytes/char worst case for BMP = 1000 bytes).
  - `tags: 10 entries × 40 chars = 400 chars` ≈ 400 bytes plus JSON overhead `["...","..."]` ≈ 450 bytes.
  - `due: 'YYYY-MM-DD'` = 10 bytes.
  - `status: 'abandoned'` = 9 bytes.
  - JSON envelope `{"title":"...","tags":[...],"due":"...","status":"..."}` ≈ 50 bytes overhead.

Worst case: **≈ 1100 bytes** if the user is on the absolute storage cap on every field with worst-case UTF-8. With ASCII titles + tags (the common case): ~600 bytes.

**Decision.** **`express.json({ limit: '1kb' })`** for both PATCH and POST /complete routes. (DELETE has no body.) Justifications:

  1. **40% headroom over common-case body sizes** (~600 bytes); ~10% headroom over worst-case all-ASCII.
  2. **Real worst-case (multilingual title at 500 chars × 2 bytes)** would 413 — that's a tightening of the brief's "1 KB has 40% headroom" claim. Mitigation: a multilingual user typing a 500-char Japanese title would hit ~1500 bytes, exceeding the cap. Accept this as a documented edge: the storage cap is 500 chars, but the wire cap is 1 KB. A user editing a 500-char Japanese title via the webapp gets a 413; they can use the Telegram chat fallback. Documented in §Risks.
  3. **Blocks DoS via 10 MB JSON.** Without the limit, a malicious client could send `{"title":"<10 MB of garbage>"}`. Express's default `json` limit is 100 KB — far higher than necessary. 1 KB is correct for this surface.
  4. **`express.json({limit: '1kb'})` returns 413 Payload Too Large** automatically with no extra code.

**Where applied.** Phase 2 dev attaches the body parser ONLY to the mutation routes — not globally — so GET routes don't pay the parser cost:

```typescript
// in items.mutate.ts
const jsonParser = express.json({ limit: '1kb' });

app.patch('/api/webapp/items/:id', jsonParser, (req, res) => { /* ... */ });
app.post('/api/webapp/items/:id/complete', jsonParser, (req, res) => { /* ... */ });
app.delete('/api/webapp/items/:id', (req, res) => { /* no body parser */ });
```

**Tighter cap option considered (and rejected): 512 bytes.** Cuts headroom too thin; a user with a 250-char ASCII title + 8 tags on the worst-case length is at ~480 bytes already. 1 KB is the right size.

**Consequences.**

  - **Positive.** DoS surface bounded.
  - **Positive.** Per-route attachment keeps GET routes free of parser overhead.
  - **Negative.** Multilingual users with very long (500-char) titles can hit 413. Documented; chat fallback is available.

### Decision 12 — Validation library: hand-rolled, no Zod

**Status:** Accepted.

**Decision.** Hand-rolled validators in `src/organize/validation.ts` per decision 2. NO Zod, no Yup, no schema lib. Justifications:

  1. **No new npm deps (boundary).** The iteration brief is firm.
  2. **4 fields don't justify a schema lib.** The validator is ~120 LOC of TypeScript including JSDoc. Zod would replace ~80 LOC with `~40 LOC + 1 npm dep + a lockfile delta + bundler weight + supply-chain surface`. Not a win at this size.
  3. **Codebase precedent.** The existing organize commands (`src/commands/organize.ts`) use hand-rolled validators (verified via grep — no schema library used in `src/organize/` or `src/commands/`). Adding Zod to ONE file would be the lone outlier.
  4. **Anti-Slop §13 module isolation.** A new dep is a cross-module concern — every other organize-feature surface would have a "should I use Zod here too?" tension going forward. Hand-rolled keeps the discipline localized.

**Phase 2 dev signature** (binding):


### Decision 13 — Tap-target separation: `event.stopPropagation()` on checkbox

**Status:** Accepted — checkbox click stops propagation; documented as a regression-prone pattern.

**Context.** The list card has a click listener that navigates to the detail view. The checkbox sits inside the card. Without stopPropagation, tapping the checkbox bubbles up to the card listener — which navigates AWAY from the list, defeating the one-tap-complete goal.

**Decision.** Checkbox click handler calls `event.stopPropagation()` BEFORE doing anything else:

```javascript
checkbox.addEventListener('click', (e) => {
  e.stopPropagation();  // CRITICAL — prevents card-level navigation
  toggleComplete(item.id);
});
```

**Document as a regression-prone pattern in app.js.** Add a comment block at the top of the checkbox-rendering function:

```javascript
// REGRESSION GUARD (decision 13): the complete checkbox lives inside a list card
// that also has a click listener (navigates to detail). The checkbox click MUST
// call stopPropagation() to prevent navigating away when the user taps the
// checkbox. Anti-Slop will check that stopPropagation is the FIRST line of the
// checkbox handler.
```

**Test in Phase 2:**

  1. Render a list with one item. Programmatically click the checkbox. Assert that the detail panel did NOT become visible (i.e., the list view stayed up).
  2. Programmatically click the card (not the checkbox). Assert that the detail panel DID become visible.

This is JSDOM testable; doesn't need a real browser.

**Consequences.**

  - **Positive.** Tap-target semantics work as expected.
  - **Negative.** A future refactor that removes stopPropagation silently breaks the feature. Anti-Slop's regression-pattern check covers this.

### Decision 14 — Optimistic UI + rollback

**Status:** Accepted — checkbox optimistic; edit form NOT optimistic.

**Decision.**

**Complete checkbox: optimistic.**

  1. User taps checkbox.
  2. Client immediately flips the visual state (empty circle → filled with checkmark, or vice versa).
  3. Client fires `POST /:id/complete {done: true}` (or false).
  4. On 200: confirm — visual state is already correct, no further action.
  5. On error (4xx, 5xx, network): REVERT — flip the visual state back; show error toast "Failed to update; tap again to retry."

**Edit form: NOT optimistic.**

  1. User taps Save.
  2. Client shows "Saving..." spinner ON the Save button (button stays disabled during the call).
  3. Client fires `PATCH /:id`.
  4. On 200: hide the form, show the updated detail view with the new values from the response.
  5. On error: keep the form open; show inline error message above the form ("Failed to save. <reason>"). User can retry or cancel.

**Why the asymmetry.**

  - **Checkbox is binary.** Rollback after error is trivial: flip back. The user's mental model is "I tapped → it changed; if it changes back with a red error, I retry."
  - **Edit form is multi-field.** If the server rejects (e.g., 400 because the server's tag-regex is stricter than the client's, or a future server-side privacy filter rejects the title), rolling back means "which fields had been locally edited from the original load? which one(s) caused the rejection?" The client would need to track per-field dirty state and selectively revert. That's complex error-handling for a story the user is rarely in (validators agree client-server).
  - **Spinner is the right primitive for the form.** "Saving..." is unambiguous; the user waits 100-300ms; the form vanishes on success. No rollback complexity.

**Consequences.**

  - **Positive.** Checkbox is snappy; matches user mental model.
  - **Positive.** Edit form is correct (no rollback bugs).
  - **Negative.** Edit form has ~200ms perceived latency vs ~0ms for checkbox. Acceptable — the user expects forms to take longer than checkboxes.

### Decision 15 — XSS in edit form: DOM-property assignment only

**Status:** Accepted — `inputEl.value = item.title`, NEVER setAttribute or template-string.

**Context.** The edit form pre-populates with the current item values. The `title` field is user-authored — could contain anything including HTML/script. Three vectors:

  - **(a) `inputEl.value = item.title`.** DOM property; the browser treats the string as plain text. SAFE.
  - **(b) `inputEl.setAttribute('value', item.title)`.** HTML attribute; the browser parses the string for entity references. A `&lt;` becomes `<`. Generally safe for `<input>` but inconsistent with the rest of the codebase.
  - **(c) `formEl.innerHTML = '<input value="' + item.title + '">';`.** Template-string interpolation. UNSAFE — a title containing `"><script>alert(1)</script><input value="` injects a script element.

**Decision.** (a). Always.

**Binding pattern (Phase 2):**

```javascript
function showEditForm(item) {
  const titleInput = document.getElementById('edit-title');
  const dueInput = document.getElementById('edit-due');
  const tagsInput = document.getElementById('edit-tags');

  // CRITICAL: DOM property assignment only (decision 15). Never setAttribute,
  // never template strings. Each input's HTML structure is hardcoded in
  // index.html; only the values get set here.
  titleInput.value = item.title || '';
  dueInput.value = item.due || '';   // empty-string for null/cleared due
  tagsInput.value = (item.tags || []).join(', ');  // CSV input convention

  // Status select uses .value with the enum value
  const statusSelect = document.getElementById('edit-status');
  statusSelect.value = item.status || 'active';
}
```

**Document as regression-prone.** Add a comment block at the top of the edit form code:

```javascript
// REGRESSION GUARD (decision 15): the edit form is pre-populated with item
// values. ALL value-assignments use DOM properties (.value, .checked, etc.)
// — NEVER setAttribute('value', ...) or template-string innerHTML. This
// prevents XSS via user-authored titles/tags. Anti-Slop and QA will both
// grep for setAttribute.*value and verify it is absent.
```

**Test:** Phase 2 QA includes a test where the item title is `<img src=x onerror=alert(1)>`. The test asserts that opening the edit form does NOT execute the onerror; it asserts that `document.querySelectorAll('img[src="x"]').length === 0` after the form is shown. (The string lives only as an input value, not as DOM.)

**Consequences.**

  - **Positive.** XSS surface zero by construction.
  - **Negative.** None.

### Decision 16 — Filter-interaction "surprise vanish": don't refetch on toggle

**Status:** Accepted.

**Context.** The user is on the "Active" filter chip and taps the complete checkbox on item X. The optimistic UI flips X's status to "done". Two refresh strategies:

  - **(a) Refetch the list immediately.** With "Active" filter, item X (now "done") would be filtered out — the row VANISHES from the user's view. The user sees: tap → row disappears.
  - **(b) Don't refetch.** The row stays; checkbox stays checked. The user can see "I just completed this; it's still here." Refetch happens only on (i) filter chip change, (ii) pull-to-refresh, or (iii) navigating away and back.

**Decision.** (b). Justifications:

  1. **Predictability beats freshness for THIS action.** The user's mental model: "I tapped to complete; the action took effect; I want to see the result of my action." A row vanishing the moment they tap it makes the action feel destructive when it's not.
  2. **The list is a snapshot view.** Until the user explicitly reloads (chip tap, pull, navigate-back), the list reflects the state at last-fetch with optimistic updates layered. This is the same model as iOS Mail (a deleted message stays grayed-out in the list until the user navigates) and Reminders (a checked item stays visible-but-struck-through until sync-and-refresh).
  3. **Implementation is trivially the default.** "Don't refetch" is just "don't call fetchItems()" after the optimistic update. The default code path is the right one; no code added.
  4. **Refetch points are predictable.** Filter chip change (different query) → refetch makes sense. Pull-to-refresh (explicit user gesture) → refetch. Navigate to detail and back → refetch (treats navigation as session boundary).

**Implementation (binding for Phase 2):**

```javascript
function toggleComplete(itemId) {
  // 1. Find the item in the in-memory list
  const item = currentItems.find(i => i.id === itemId);
  if (!item) return;

  // 2. Flip optimistically
  const newStatus = item.status === 'done' ? 'active' : 'done';
  item.status = newStatus;
  rerenderListPreservingScroll();  // re-render the list WITHOUT refetching

  // 3. Fire request
  fetch(`/api/webapp/items/${itemId}/complete`, {
    method: 'POST',
    headers: { 'Authorization': `tma ${initData}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ done: newStatus === 'done' }),
  })
    .then(r => r.json())
    .then(data => {
      if (data.ok === true) {
        // Update in-memory item with full server response (status, etc.)
        Object.assign(item, projectFromDetailResponse(data.item));
        rerenderListPreservingScroll();
      } else {
        // Rollback
        item.status = newStatus === 'done' ? 'active' : 'done';
        rerenderListPreservingScroll();
        showToast(`Failed: ${data.error || data.code}`);
      }
    })
    .catch(err => {
      // Rollback on network error
      item.status = newStatus === 'done' ? 'active' : 'done';
      rerenderListPreservingScroll();
      showToast(`Network error: ${err.message}`);
    });
}
```

**Note on edit form behavior.** When the user saves an edit (PATCH succeeds), the in-memory item is replaced with the server's response item. The list re-renders with the new title/due/tags. NO refetch.

**Consequences.**

  - **Positive.** UI feels predictable; user retains visual confirmation of their action.
  - **Negative.** A user toggles a "done" item to "active" while on the "Done" filter — the item stays. Same logic; same posture. Only refetch on chip change.
