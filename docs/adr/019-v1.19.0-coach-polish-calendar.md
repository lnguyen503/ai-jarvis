# ADR 019 — v1.19.0 Coach polish + Google Calendar two-way sync + Calendar view polish

**Status:** Proposed (CP1 pending — Devil's Advocate + Anti-Slop Phase 1).
**Date:** 2026-04-25.
**Authors:** Architect Agent (opus).
**Predecessors:** ADR 018 + 018-revisions-after-cp1 (Coach Jarvis: per-item intensity dial; daily scheduled fire; coach memory keyed entries; tool dispatcher `<untrusted>` wrap; D3.a per-turn caps; D6/F1 code-level allowlist brake).
**Supersedes:** none.
**Header note for reviewers:** v1.19.0 is the **"feels like a human assistant"** revision. v1.18.0 shipped the coach machinery as a configurable surface — every item had to be opted-in with an intensity, the chat command was nested behind `/organize coach setup`, and there was no calendar surface for the items the coach was managing. v1.19.0 inverts the defaults: coach is auto-on for items with a due date or active progress; intensity infers from item shape unless the user overrides; NL overrides land in chat without the user touching frontmatter; the coach actively monitors progress and stays alive on an item until the user confirms it's done; items with due dates land on a dedicated **"Jarvis Organize"** Google Calendar with two-way sync; the webapp gets Day / Week / Month subview polish + drag-to-reschedule + a Today focus card. **NOT** a multi-coach revision (deferred to v1.20.0). **NOT** an autonomous-completion revision (Coach still cannot mark items done; D12 alive-until-confirmed-done is by design).

---

## Context

### Why v1.19.0 — quoting the user

> "It needs to be flexible but Jarvis sits in the background to monitor my list and due dates and monitor the progress and my response to my progress until I confirm it's done. Like a human assistant... We need a polish Calendar with all of these organize list whether that's in Google Calendar we need to make it look nice and let Jarvis manage it all."

> "This will be incredibly helpful to anyone so we need this working well."

The first quote sets the binding behavior: **monitor, not nag**; **confirm-done is the user's**, not the coach's; **calendar polish is the visible surface** of the coach's work; **let Jarvis manage it all** authorizes proactive coach moves on the calendar provided the item is the source of truth. The second sets the polish bar: **shared/public quality**, not just "works for Boss."

### v1.18.0 was config-heavy; v1.19.0 inverts to auto-default

| Surface | v1.18.0 default | v1.19.0 default |
|---|---|---|
| `coachIntensity` per item | `'off'` (must be set per item) | `'auto'` (inferred from item shape; user can override) |
| Coach setup | `/organize coach setup [HH:MM]` (nested under `/organize`) | `/coach on` / `/coach off` (top-level chat command; back-compat with v1.18.0 nested form) |
| Override "skip exercise this week" | Edit item frontmatter via webapp | Type "skip exercise this week" in chat; NL parser logs an override expiring in 7 days |
| Item with due date on calendar | Manual `calendar_create_event` tool call | Auto-synced to "Jarvis Organize" calendar; round-trip via `extendedProperties.private.itemId` |
| Webapp calendar subviews | Month/Week/Day exist (v1.15.0) but minimal polish | Day + Week visually polished; coach activity overlay; drag-to-reschedule with undo |
| Hub | No coach affordance | "Coach Jarvis active — daily 8am" banner with one-click disable |

### Calendar two-way sync is the natural extension

Items with due dates already conceptually belong on a calendar. v1.18.0's `calendarEventId` frontmatter field exists but is only populated when the user manually invokes the calendar tools. v1.19.0 makes that automatic and bidirectional, scoped to a **dedicated "Jarvis Organize" calendar** so the user's main calendar is never polluted and can be hidden if desired. The dedicated calendar gives us a clean delete domain (drop the calendar = drop all sync state) and a clean trust boundary (events in the Jarvis calendar are owned by Jarvis; events on the user's primary calendar are not).

### Differentiator framing (binding)

OpenClaw is a chat shell. ChatGPT-with-tasks is a list. Jarvis-with-coach in v1.18.0 was a configurable assistant. **Jarvis-with-coach in v1.19.0 is a coach that already knows what to do.** The user opens the app, the calendar is already organized, the coach already has a plan, and the only work the user does is confirm what's done. That's the "human assistant" feel.

## Decisions

### Decision 1 — `coachIntensity = 'auto'` placement: 5th value in the closed set; default for unset items

**Pick.** Add `'auto'` as the 5th value in the existing `CoachIntensity` closed set. Update the closed-set duplicate-source-of-truth invariant test to expect 5 values. Webapp pill picker grows from 4 to 5 buttons.

**Default policy (binding for back-compat):**

- Item with `coachIntensity` field absent → inferred as `'auto'` at coach selection time.
- Item with `coachIntensity: 'off'` (explicitly set) → STAYS `'off'`. The user opted out; do not silently re-engage.
- Item with `coachIntensity: 'gentle' | 'moderate' | 'persistent'` → unchanged behavior.
- Item with `coachIntensity: 'auto'` (newly written via webapp) → auto inference.

**Inference rules** (in coach prompt for v1.19.0; can promote to code in v1.19.x if telemetry shows non-determinism is annoying):

- Item has a due date in the next 7 days → `moderate` equivalent.
- Item has a due date 7–30 days out → `gentle` equivalent.
- Item has a due date > 30 days OR no due date → `gentle` equivalent (still nudge weekly so it doesn't disappear from the user's mind).
- Item is type `goal` (no due date by definition) → `gentle` equivalent.
- Item has an explicit "exercise" / "save" / "retirement" tag (regex `/(exercise|workout|gym|run)|(save|retirement|invest)/i` on title) → bump to `persistent` equivalent (these are the user's stated examples of needing the persistent push).
- Item has been ignored for ≥ 3 nudges in a row (D13 fatigue) → reduce to `gentle` for 7 days regardless of due date.

**Why prompt and not code for the inference itself:**

1. Single user, small list. The LLM has plenty of capacity to apply the rules in-prompt (mirrors D8 of ADR 018: in-prompt picking is the product).
2. **Cheap to revisit.** If users complain "auto keeps picking exercise", we ship a deterministic JS heuristic in v1.19.x. The migration is a JS function + a prompt-clause swap.
3. **The closed-set discipline still applies.** `'auto'` as a stored value is a code-level concern (`CoachIntensity` union, `COACH_INTENSITIES` array, `isCoachIntensity` guard, validator code `COACH_INTENSITY_INVALID`); that part is pure code. The behavior of `'auto'` is the prompt concern.

**Webapp UI delta.** Pill picker grows from `[off | gentle | moderate | persistent]` to `[Auto | Off | Gentle | Moderate | Persistent]`. `'Auto'` is the visually-default selection (per D16, the picker is behind an Advanced disclosure; Auto is implied by closed Advanced section).

**Static-test update.** `tests/static/coach-intensity-closed-set.test.ts` from v1.18.0 enforces the array-equality invariant between `coach/intensityTypes.ts` and `organize/validation.ts`. v1.19.0 update: assert array length is 5, contains `'auto'`. **One-line test change.**

### Decision 2 — `/coach on` / `/coach off` placement: top-level chat commands; extend `coachSubcommands.ts`

**Pick.** Add top-level `/coach on` and `/coach off` chat commands wired in `src/gateway/index.ts` (next to `/coach setup` if any) AND extend `src/commands/coachSubcommands.ts` with two new exported handlers: `handleCoachOnTopLevel`, `handleCoachOffTopLevel`. **Do not** create a new file `src/commands/coachToggle.ts` — that splits the coach command surface into two files for one feature, which violates single-source-of-truth (the user's "what coach commands exist?" question should resolve in one file).

The v1.18.0 `/organize coach setup [HH:MM]`, `/organize coach off`, `/organize coach reset` continue to work (back-compat: ADR 018 ships them; v1.19.0 must not break them). New top-level forms:

- `/coach on` — equivalent to `/organize coach setup` (default 08:00 if no time specified). User-friendly. Replies "Coach Jarvis active daily at 08:00. He'll start checking your organize list tomorrow morning."
- `/coach on 14:30` — equivalent to `/organize coach setup 14:30`.
- `/coach off` — equivalent to `/organize coach off`. Replies "Coach paused."
- `/coach status` — NEW, no v1.18.0 equivalent. Shows `[on at HH:MM | off]` + count of items with `coachIntensity != 'off'` + last fire time + next fire time. ~30 LOC. **Bundled with v1.19.0 as a polish item — the user needs to know the coach is alive.**

**Why extend, not new file:**

1. v1.18.0 already extracted `coachSubcommands.ts` per its W1 (avoid `commands/organize.ts` exceeding 500 LOC). The file is at 279 LOC; +90 for three new handlers = 369. Comfortable under 500.
2. Two files for the same product surface ("how do I control the coach?") is a navigation tax — when a user reports a bug, the dev has to grep for the command surface in two places.
3. The v1.18.0 nested forms continue to live in this file; the top-level forms live next to them. One file = one source of truth for coach command behavior.

**Why not add the top-level commands as middleware in `gateway/index.ts` directly:** the gateway already gets the `/coach …` slash; the parsing + dispatch logic belongs in `commands/`, mirroring how `/organize` parsing flows from gateway → `commands/organize.ts`.

### Decision 3 — NL override parser: standalone `src/coach/userOverrideParser.ts` module

**Pick.** New file `src/coach/userOverrideParser.ts` (~120 LOC projected). Heuristic regex-based parser with negation detection. Confidence threshold: only fire override when (a) intent verb matches AND (b) item title fuzzy-matches with score > 0.6. No new npm deps.

**Intent verbs and regex set (binding):**

| Intent | Regex (case-insensitive) | Example phrase | Effect |
|---|---|---|---|
| `back_off` | `/(skip|pause|hold off on|don'?t bug me about|back off|chill on|leave alone)/i` | "skip exercise this week" | 7-day hold (D10) |
| `push` | `/(push me on|harder on|nag me about|stay on|don'?t let me forget)/i` | "push me harder on retirement" | One-coach-run-only persistent bump (D10) |
| `defer` | `/(later|tomorrow|next week|after|once I've finished)/i` | "later on retirement" | One-day defer (D10) |
| `done_signal` | `/(I'?m done with|finished|wrapped up)/i` | "I'm done with X" | Suggest organize_complete in next coach turn (still does NOT auto-complete; D12) |

**Negation detection.** Run a pre-pass that splits the user's message on sentence boundaries; for each sentence, check if the regex `/(don'?t|do not|never|stop telling me to)/i` precedes the intent verb within 8 tokens. If yes, INVERT the intent (e.g., "don't skip exercise" → push instead of back_off). If multiple sentences resolve to different intents, log a warning and act on the FIRST resolved intent only.

**Fuzzy-match algorithm (no new deps; ~30 LOC):**

```typescript
function fuzzyMatchScore(itemTitle: string, userPhrase: string): number {
  const titleTokens = itemTitle.toLowerCase().split(/\s+/).filter(t => t.length > 2);
  const phraseTokens = userPhrase.toLowerCase().split(/\s+/).filter(t => t.length > 2);
  if (titleTokens.length === 0 || phraseTokens.length === 0) return 0;
  // Substring score: any title token appears as substring of any phrase token
  let substringHits = 0;
  for (const tt of titleTokens) {
    for (const pt of phraseTokens) {
      if (pt.includes(tt) || tt.includes(pt)) { substringHits++; break; }
    }
  }
  // Token-overlap score: |title ∩ phrase| / |title|
  const overlap = titleTokens.filter(t => phraseTokens.includes(t)).length;
  return Math.max(substringHits / titleTokens.length, overlap / titleTokens.length);
}
```

Threshold `> 0.6` requires that >60% of the item title tokens find a substring or exact match in the user phrase. Tune in v1.19.x via telemetry.

**Failure modes (binding):**

- **Multiple items match the override target.** Prefer the most-recently-mutated item (max `updated` frontmatter timestamp). Log info: `multiple_match_picking_recent`.
- **Zero items match.** Log warn: `no_match_for_override` with the user's phrase (scrubbed) and the active-item count. Skip writing an override entry. The user's regular agent turn handles the chat reply.
- **Negation is ambiguous** (e.g., "skip exercise but push me on running"). Resolve to the FIRST sentence's intent. Document; punt complex multi-sentence parsing to v1.19.x.
- **Intent verb matches but item title fuzzy score ≤ 0.6.** Skip the override write. The user's regular turn handles the chat reply.

**Where the parser is called.** Inside `src/agent/index.ts.turn()` AFTER the user message is read, BEFORE the Claude API call. If the parser returns a non-null override decision, write a `coach.<itemId>.userOverride` keyed memory entry (D10) and continue with the normal agent turn. The agent turn's reply text can mention the override ("OK, I'll back off on retirement for a week") since the chat agent has access to the parser result via `ctx.coachOverrideJustWritten`.

**Tests required:**

- T-D3-1 — "skip exercise this week" + active item titled "exercise" → back_off override written with 7-day expiry.
- T-D3-2 — "don't skip exercise, push me harder" → push override (negation flips back_off → push).
- T-D3-3 — "skip retirement" with two active items "save for retirement" and "retirement planning" → most-recently-updated wins; log shows `multiple_match_picking_recent`.
- T-D3-4 — "skip something" with no fuzzy match → no override written; warn log emitted.
- T-D3-5 — "later on retirement" → defer override with 1-day expiry.
- T-D3-6 — Sentinel-injection probe: user phrase contains `<!-- key:` → override write rejected (parser still runs, body sanitized via existing v1.17.0 R3 + F1 sentinel-injection guard).

### Decision 4 — Calendar sync trigger: post-write hook in `storage.ts updateItem`; fire-and-forget; failures audit + log only

**Pick.** Hook in `src/organize/storage.ts updateItem` AFTER the `writeAtomically` succeeds (line 717 today). Fire-and-forget via `Promise.resolve().then(() => notifyCalendarSync(...))`. Failures audit + log; do NOT block the item write.

**Wrapper mechanism.** New module-level callback registered at boot from `src/calendar/sync.ts`:

```typescript
// src/organize/storage.ts
let _calendarSyncCallback: ((userId: number, item: OrganizeItem) => void) | null = null;
export function registerCalendarSyncCallback(cb: typeof _calendarSyncCallback): void {
  _calendarSyncCallback = cb;
}
// Inside updateItem, AFTER writeAtomically:
if (_calendarSyncCallback) {
  Promise.resolve().then(() => _calendarSyncCallback!(userId, returnedItem))
    .catch(err => log.warn({ userId, itemId, err: err.message }, 'calendar sync callback rejected'));
}
```

The callback is registered ONCE at boot in `src/index.ts` step 7 (after `initMemory`, before `initGateway`):

```typescript
import { syncItemToCalendar } from './calendar/sync.js';
import { registerCalendarSyncCallback } from './organize/storage.js';
registerCalendarSyncCallback((userId, item) => syncItemToCalendar({ userId, item, memory, config, logger }));
```

**Fire-and-forget rationale (binding).**

1. **Item is the source of truth; calendar is a derived view.** Sync failures must not corrupt or delay the item write. v1.18.0 ADR 018 D4 already establishes this for coach memory writes (also fire-and-forget); same posture.
2. **Webapp PATCH must stay fast.** Webapp PATCH SLA is "~50ms typical" (per ADR 014 perf budget). Adding a Google Calendar API call (~200-500ms typical) to the synchronous write path violates that.
3. **Calendar API can be down.** Google has 99.9% SLO; ~9 hours/year of degraded service. The item must remain writable during those windows.

**Hooks at all write paths (binding inventory).** v1.14.3 stampUpdated discipline lists 5 paths in `storage.ts` that call `serializeItem`. The calendar-sync callback fires from:

| Path | Callback fires? | Rationale |
|---|---|---|
| `createItem` | YES | New item with due date should land on calendar immediately |
| `updateItem` | YES | Title / due / notes changes propagate to calendar |
| `softDeleteItem` rewriteContent | NO | The .trash rename triggers a calendar DELETE via a sibling hook (line below) |
| `appendProgressEntry` | NO | Progress entries don't change calendar-visible fields |
| `restoreItem` | YES | Restored item with due date should re-appear on calendar |
| `softDeleteItem` rename to .trash | YES (special) | Triggers a one-shot `removeItemFromCalendar(userId, itemId)` callback |

A sibling callback `_calendarRemoveCallback` is registered for the soft-delete + permanent-evict cases. Same pattern; same fire-and-forget posture.

**Audit categories added (binding):**

- `calendar.sync_success` — write to Google Calendar succeeded (record `itemId`, `calendarEventId`, `direction: 'out'`, NO title/notes content per privacy posture).
- `calendar.sync_failure` — write failed (record `itemId`, `error.code`, retry-eligible flag, NO title/notes).
- `calendar.sync_skipped` — item has no due date OR `coachIntensity === 'off'` (skip sync); record `reason`.

### Decision 5 — Calendar polling cursor: keyed memory entry `calendar.jarvis_sync_cursor`

**Pick.** Reuse the v1.18.0 sentinel parser (extended to dotted multi-segment keys) for the polling cursor. Store as a keyed memory entry with key `calendar.jarvis_sync_cursor`. Body shape:

```json
{ "lastPolledAt": "2026-04-25T12:34:56.000Z", "lastEventEtag": "syncToken-from-google" }
```

`lastEventEtag` is Google's [Sync Token](https://developers.google.com/calendar/api/guides/sync) — server-side opaque cursor that returns only events changed since the last poll. No client-side timestamp comparison needed.

**Corruption recovery (binding).** If the cursor entry is missing OR `JSON.parse` throws OR `lastEventEtag` is not a string at poll time:

1. Log warn: `cursor_missing_or_corrupt_recovering`.
2. Fall back to `events.list` with `timeMin = now - 24h` (24-hour lookback).
3. Walk every event returned, sync if our extendedProperties.private.itemId matches a live item.
4. Write a fresh cursor entry from Google's `nextSyncToken` in the response.

**Cursor never expires by default.** Only manually reset via:

- `/coach reset-calendar-cursor` chat command (NEW, ~10 LOC, fits in `coachSubcommands.ts`).
- Or implicitly via `/coach off` → wipe cursor (the user has paused the integration).

**Why not store in SQLite.** Adding a `calendar_sync_state` table would require a migration. The keyed-memory layer is right-sized for "1 row per user per state-thing"; the cursor IS such a thing. v1.17.0 R3 + R5/F3 binding: `userMemoryEntries.ts` is the SOLE WRITER for keyed entries. Calendar's writes flow through that layer.

**`calendar.jarvis_calendar_id` companion entry.** Same store, key `calendar.jarvis_calendar_id`. Body is the Google Calendar ID (string, e.g. `c_abc123@group.calendar.google.com`). Set once during D8 ensureJarvisCalendar; never expires.

### Decision 6 — Polling cadence: 5 minutes

**Pick.** 5-minute poll. Hook into `src/scheduler/index.ts` as a cron job with expression `*/5 * * * *` registered at boot for every user with a `calendar.jarvis_calendar_id` set (i.e., who has opted in via D8).

**Quota math (binding):**

- 5-min poll × 24h = 288 polls/day per user.
- Google Calendar API free quota: 1,000,000 requests/day per project.
- Even at 100 active users: 100 × 288 = 28,800 req/day. **2.88% of quota.** Trivial.
- Per-user QPS: 1 req / 300s = 0.0033 QPS. Per-user QPS limit is 100 QPS (per Google docs). **0.0033% of per-user limit.** Trivial.

**Why not faster (e.g., 1-min):** webhook push notifications are the right answer for sub-minute latency, and they're a v1.20.0 enhancement (require a public HTTPS endpoint; deferred per "no inbound network exposure" posture in ARCHITECTURE.md §1).

**Why not slower (e.g., 15-min):** the user expects calendar drags to reflect within "a few minutes" — the polish bar sets the cadence.

**Per-user enablement.** Only poll for users who have:

- A `calendar.jarvis_calendar_id` set (D8 ran).
- An active coach scheduled task (`__coach__` description present in `scheduled_tasks`).

**Configurable via `config.calendar.syncPollSeconds` (default 300, min 60, max 3600).** The 60-second floor prevents users from accidentally setting `1` and burning quota; the 3600-second ceiling keeps the polish bar.

### Decision 7 — Two-way sync conflict resolution: last-modified-wins after 1-min debounce; webapp wins ties

**Pick.** When both sides have changed the same event in the same window:

1. **Compute the "winner"** by comparing modification timestamps (item.updated vs Google event.updated).
2. **Apply 1-minute (60s) debounce.** Within 60 seconds of a webapp PATCH on the item, the WEBAPP value wins (the user is actively editing; we trust the local action).
3. **After 60 seconds**, the calendar value wins if it's newer.
4. **Tie (within ±1s of each other on opposite sides)**, the **webapp wins**. Treat the user's webapp action as more intentional than a calendar drag (the calendar is a derived surface).

**Edge cases (binding):**

| Edge | Handling |
|---|---|
| Clock skew between server and Google's servers | Accept ±60s tolerance (baked into the debounce window). If `|local.updated - google.updated| < 60s`, treat as same-time and apply the webapp-wins-tie rule. |
| Timezone confusion | All server-side timestamps stored UTC; convert at render time. Matches CLAUDE.md v1.15.0 calendar-UTC discipline (`dates.js` UTC-only). Google's API returns RFC3339 with timezone offset; we normalize to UTC before comparison. |
| Daylight saving | UTC storage neutralizes. No DST-aware code anywhere. |
| Webapp PATCH then immediate calendar drag (within 60s) | Webapp wins (debounce). Calendar drag is reverted on next poll (sync direction = `to_calendar`). User sees the drag undone in their Google Calendar within 5 min. Toast in webapp says nothing (the user already saw their PATCH succeed). |
| Calendar drag then immediate webapp PATCH (within 60s) | Webapp wins (its PATCH triggers a `to_calendar` sync that overwrites the drag). |
| Both modified simultaneously (within 60s of each other on opposite sides) | Webapp wins (tie-break rule). Audit row `calendar.sync_conflict` emitted with `winner: 'webapp', loser: 'google', deltaMs`. |
| Webapp delete + Google "uncancel" | Soft-deleted items remove from calendar. If user resurrects via Google's UI and Google re-shows the event, our next poll sees the event back; we check the `extendedProperties.private.itemId` → look up our item → it's still soft-deleted in `.trash/` → **delete from Google again**. Audit + log. (User intent: webapp delete is authoritative.) |
| Item with no due date | Never synced to calendar (D4 skip). If user adds a due date, sync on the next `updateItem` write. |

**`sync_conflict` audit category** added to `KNOWN_AUDIT_CATEGORIES`. Detail JSON: `{itemId, calendarEventId, winner, deltaMs, fields}` — NO content.

### Decision 8 — Jarvis calendar auto-create: idempotent find-then-create

**Pick.** At first sync attempt for a user, idempotently ensure the "Jarvis Organize" calendar exists in their Google account.

**Algorithm (binding):**

1. Look up keyed memory entry `calendar.jarvis_calendar_id`.
2. If present: use the stored ID. Skip to "verify alive" (one-time API check on first use per process; cache the verification).
3. If missing: call `CalendarApi.listCalendars()` (NEW method; ~30 LOC in `src/google/calendar.ts`).
4. Filter calendars by `summary === 'Jarvis Organize'`.
5. If found: store its ID via `createMemoryEntry('calendar.jarvis_calendar_id', id)`. Use it.
6. If not found: call `CalendarApi.createCalendar({ summary: 'Jarvis Organize', description: 'Items synced from Jarvis. Do not edit metadata; edit times by drag.' })` (NEW method; ~25 LOC). Store the returned ID.
7. Audit: `calendar.jarvis_created` (NEW category).

**Find-then-create order (binding).** Always list-then-create; never create-without-list. Reason: if the user (or a previous Jarvis install) already has a "Jarvis Organize" calendar and we create a second one, the user gets two calendars and we lose the merge battle. List-first is idempotent across re-installs.

**Failure handling.**

- `listCalendars` API failure → audit `calendar.jarvis_create_failed`, log warn, return `{ ok: false, code: 'GOOGLE_API_ERROR' }`. Coach setup proceeds; sync is skipped this poll cycle.
- `createCalendar` API failure → same.
- User has 100+ calendars and pagination is needed → use `pageToken` loop with cap of 10 pages (1000 calendars). If still not found at cap, fall through to create. Documented edge case; we don't expect anyone to hit this.

**Tests required:**

- T-D8-1 — First call: `listCalendars` returns 0 matches; `createCalendar` called; ID stored.
- T-D8-2 — Second call (same process or after restart): `listCalendars` not called; cached ID used.
- T-D8-3 — User manually deletes the Jarvis calendar in Google: next poll's "verify alive" detects 404; cursor + ID entries reset; D8 re-runs.
- T-D8-4 — User has a pre-existing "Jarvis Organize" calendar (e.g., from a past Jarvis install): find returns it; no duplicate created.
- T-D8-5 — `listCalendars` API error: audit emitted; sync skipped this cycle; retry on next poll.

### Decision 9 — Calendar event `extendedProperties.private.itemId` for round-trip identity

**Pick.** Every event Jarvis creates on the Jarvis Organize calendar carries `extendedProperties.private.itemId = '<itemId>'` per Google Calendar best practice for app-internal IDs. **No PII** in extendedProperties (only the itemId, which is `YYYY-MM-DD-[a-z0-9]{4}` — non-identifying).

**`CalendarApi.createEvent` extension.** Add an `extendedProperties` field to the request body:

```typescript
requestBody.extendedProperties = {
  private: { itemId: opts.itemId }, // 19-char string; safe.
};
```

**Hygiene checklist (binding for `src/calendar/sync.ts`):**

- ONLY `itemId` in `extendedProperties.private`. NO email, NO title, NO notes, NO progress.
- If a future field is added, it must pass the same "non-identifying" gate (e.g., a sync-direction enum is fine; a free-text label is not).
- `extendedProperties.shared` (visible to other Google Calendar consumers) is NEVER used.

**Reverse-sync identity.** `pollCalendarChanges` reads `event.extendedProperties.private.itemId` to find the local item. If the field is missing OR the itemId doesn't match a local item:

- Missing → event was created externally (not by Jarvis). SKIP. Document via INFO log: `external_event_skipped`.
- Mismatch → orphaned. SKIP. Document via WARN log: `orphan_event_in_jarvis_calendar`.

**Why not `event.iCalUID` for identity:** `iCalUID` is reserved for cross-calendar federation (subscribed/copied events). Using it for our own internal identity would conflict with future federation and is non-standard. `extendedProperties.private.itemId` is the documented Google-blessed pattern.

### Decision 10 — Coach NL override `expiresAt` semantics + storage shape

**Pick.** Per-intent expiry windows + JSON-body storage in coach memory.

| Intent | Expiry | Behavior during expiry |
|---|---|---|
| `back_off` | 7 days | Coach skips this item entirely during selection (D11 rule applies BEFORE auto-intensity inference) |
| `push` | One coach run only | Next coach turn bumps this item to `persistent` regardless of D1 inference; entry self-deletes on next coach turn |
| `defer` | One day | Coach skips this item for 24h after the override |
| `done_signal` | One coach run only | Next coach turn includes "I noticed you said you're done with X — want to mark it done? Reply 'done X'" line |

**Storage shape.** Keyed memory entry `coach.<itemId>.userOverride` (one per item; latest write wins; v1.18.0 sentinel parser allows up to 128-char keys, comfortable margin):

```json

```

The `fromMessage` field is captured for debugging (scrubbed via existing scrubber before write). NUL-byte ban + char cap (≤256) applied via the v1.18.0 R5/F3 validation helper.

**Coach prompt reads it (Step 0 — D11).** Active-items injection extended to include any `coach.<itemId>.userOverride` entries; coach prompt instructs:

> Before picking items, check the userOverride map injected with the active-items block. For each `back_off`-intent override with `expiresAt > now`: skip the item entirely. For each `push`-intent override: bump that item's effective intensity to persistent for THIS run only. For each `defer`-intent override with `expiresAt > now`: skip the item.

**Self-delete after one-shot use.** `push` and `done_signal` overrides are deleted after coach reads them (in the coach's own writes via a new `coach_clear_override` tool — NOT a write_file / not free dispatch; lives in `src/coach/coachTools.ts`). `back_off` and `defer` self-expire by `expiresAt`; coach skips silently when expiresAt has passed; a daily prune (in the same poll cron job at 5-min cadence) deletes expired overrides.

### Decision 11 — Active monitoring loop: in coach prompt (Step 0)

**Pick.** Add to `src/coach/coachPrompt.md` a new "Step 0 — Read recent activity" section that instructs the LLM to:

1. **Compare each candidate item's `progress` field timestamp** (from the active-items injection) to coach memory's `coach.<itemId>.lastNudge.at`; if `progress.updatedAt > lastNudge.at` → user is on it; mark as `engaged` and DEMOTE in priority (the user doesn't need a nudge if they're actively progressing).
2. **Read coach memory** `coach.<itemId>.lastNudge.userReply` and `.outcome` for the last 3 nudges per item.
   - 3 ignored (no userReply) → mark for fatigue (D13).
   - 3 with `userReply: 'pushback'` (NL override or "leave me alone") → mark for fatigue.
   - 3 with `userReply: 'engaged'` (user replied positively or made progress) → mark as `flowing`; lower-frequency check-ins.
3. **Apply NL overrides** from the injected override map (D10).
4. **Apply auto-intensity inference** (D1 rules).
5. **Then pick 1-3 items.**

**Why prompt and not code (for v1.19.0):**

1. The LLM has the context to weigh "engaged + due tomorrow" vs "ignored + due in 3 weeks" with judgment. Hard-coding the priority arithmetic in JS would be brittle.
2. Mirrors D8 of ADR 018 (item picking is in prompt; auto-intensity is in prompt). Keeps the coach behavior surface in one file (`coachPrompt.md`).
3. **Cheap to revisit.** If telemetry shows non-determinism, ship a `selectCoachItems(items, memory): Item[]` JS helper in v1.19.x.

**Code-side enabler.** `src/coach/coachMemory.ts` adds a helper `loadRecentNudgeHistory(userId, dataDir, itemId, n=3): Promise<NudgeHistoryEntry[]>` that returns the parsed last-N nudge entries. Coach prompt has access via the `organize_list` injection (which is extended to include the nudge-history summary per item). ~20 LOC.

### Decision 12 — Alive-until-confirmed-done: code-level brake holds

**Pick.** `organize_complete` STAYS in `coach.disabledTools` (v1.18.0 R6/F1 invariant from ADR 018). Coach can only SUGGEST completion in DM; user reply is a normal turn (not a coach turn) where `organize_complete` IS in the allowlist.

**Why NEVER auto-complete:**

1. **The user's exact words:** "until I confirm it's done. Like a human assistant." A human assistant doesn't unilaterally close out tasks; they confirm with the principal.
2. **Trust boundary.** Letting the coach close items removes the user's veto; if the coach is wrong (the user wasn't actually done; the user's "done" was sarcastic; the user said "done with that conversation" meaning the chat thread), the item is gone.
3. **Audit posture.** Items completed by a non-user actor are forensically harder to reconstruct.
4. **Reversibility.** Reverting an auto-completion requires `restoreItem` ceremony; reverting a "did the coach push too hard" doesn't.

**v1.18.0 invariant 2 holds.** `coach.disabledTools` defaults to: `run_command`, `schedule`, `organize_complete`, `organize_delete`, `forget_memory`, `calendar_delete_event`, `calendar_update_event`, `gmail_draft`. **No change in v1.19.0.** Adding `organize_complete` to the coach allowlist is explicitly out of scope.

**Coach prompt clause** (binding for `coachPrompt.md` Decision 12 section, EXTENDED for v1.19.0):

> 1. **NEVER mark an item done.** `organize_complete` is NOT in your tool allowlist. Even if the user signals "done with X" in their last message (you'll see this via the userOverride map's `done_signal` intent), do NOT call organize_complete. Instead, in your DM reply, write: "Sounds like you wrapped up X — want to mark it done? Reply 'done X' (or just yes)." The user's normal turn handles the closure. **This is structural, not advisory** — the dispatcher returns UNAUTHORIZED_IN_CONTEXT if you try.

### Decision 13 — Coach memory `coach.<itemId>.fatigue` policy

**Pick.** When the coach detects 3 ignored gentle nudges in a row for an item (D11 Step 0 check), write a `coach.<itemId>.fatigue` entry with `expiresAt = now + 7 days`. Coach prompt reads this and skips the item until expiry.

**Storage shape.** Keyed memory entry, body:

```json
{ "reason": "3_ignored_nudges", "writtenAt": "2026-04-25T08:00:00.000Z", "expiresAt": "2026-05-02T08:00:00.000Z" }
```

**Persistent items DON'T fatigue.** If `coachIntensity === 'persistent'` (explicit user setting OR D1 inference), skip the fatigue write. The user EXPLICITLY asked for persistent push; respecting fatigue would defeat the user's instruction.

**Audit category `coach.fatigue` added** to `KNOWN_AUDIT_CATEGORIES`. Detail: `{itemId, reason, expiresAtIso}`. NO title/notes (privacy posture).

**Self-recovery.** When the user finally engages with a fatigued item (any progress write OR any chat message about it), the next coach turn detects `progress.updatedAt > fatigue.writtenAt` (D11 Step 0) and skips the fatigue check (the user is back). The fatigue entry self-expires after 7 days; no manual cleanup needed. Daily prune (in the 5-min poll cron) deletes expired fatigue entries.

**Why 7 days.** Long enough to not annoy a user who's actively avoiding (one full week of no nudges is restful); short enough that long-term goals don't fall off the radar. Tunable via telemetry.

### Decision 19 — Drag-reschedule undo toast

**Pick.** "Moved to <date>. Undo." toast with 5-second window.

**Implementation:**

1. **On pickup:** stash original `dueDate` snapshot in a module-level variable (`_lastDragOriginalDueDate`).
2. **On drop:** optimistic UI update; PATCH with `If-Match`.
3. **On 200:** show toast `Moved to <newDate>. [Undo]` with 5-second auto-dismiss.
4. **On Undo button click:** PATCH back to `_lastDragOriginalDueDate` (no `If-Match` — accept whatever ETag is current; this is an explicit user action). On success, toast confirms "Reverted." On failure, toast says "Couldn't revert — please refresh."
5. **On 5-second timeout without Undo:** dismiss toast; clear `_lastDragOriginalDueDate`.
6. **Multiple drags within 5s:** the second drag's pickup overwrites `_lastDragOriginalDueDate`; the original-original is lost (acceptable; user is rapidly moving things).

**Edge case: user drags then Undo, then drags again.** The Undo PATCH is its own action; subsequent drag's pickup correctly captures the post-Undo `dueDate`. No state corruption.

**Toast component reuse.** `_showToast` (existing in app.js) already supports a button slot per v1.14.4 conflict-banner work. Pass `{ text, action: { label: 'Undo', onClick: () => doUndo() }, ttlMs: 5000 }`.

### Decision 20 — Drag-reschedule + Calendar sync interaction

**Pick.** PATCH lands on item, post-write hook (D4) fires sync, sync updates Calendar event. If user double-drags within 300ms client-side debounce: only the second one PATCHes.

**Client-side 300ms debounce (binding).** Calendar-day-view.js + calendar-week-view.js drop handler:

```javascript
let _dragDebounceTimer = null;
function onDrop(itemId, newDate) {
  if (_dragDebounceTimer) clearTimeout(_dragDebounceTimer);
  _dragDebounceTimer = setTimeout(() => {
    patchDueDate(itemId, newDate);
    _dragDebounceTimer = null;
  }, 300);
}
```

**Server-side: no special debounce.** The 300ms client-side window means at most ~3 PATCHes/second from a single drag-spam user; the existing per-chat queue + writeAtomically R8 random tmp suffix handles concurrency. Calendar sync's own 1-min debounce (D7) handles the out-of-order propagation.

**Sync direction during drag:** Webapp-originated PATCH → `to_calendar` sync direction. Calendar-originated drag → next poll → `from_calendar`. The 1-min debounce window (D7) means the first 60s after a webapp drag are webapp-wins; a user dragging in webapp + Google Calendar simultaneously gets the webapp value.

**Failure modes:**
- 412 mid-drag: rollback to original cell, show conflict banner (existing v1.14.4 path; calendar drag-reschedule reuses it).
- Calendar sync fails silently after PATCH: log warn; item is still correct in webapp; calendar will reconcile on next poll (D6).
- Network drop mid-drag: PATCH fails; rollback (existing kanban DnD pattern).

## R2 — Institutional memory (KNOWN_ISSUES.md additions; CLAUDE.md v1.19.0 invariants)

Plan 7 KI entries + 7 CLAUDE.md invariants:

1. **Auto-intensity inference rule placement.** Inference rules live in `coachPrompt.md`, NOT in `coach/intensityTypes.ts`. The closed-set type guard validates the value `'auto'`; it does NOT compute the effective intensity. Rationale: inference is a coach-behavior concern, not a type concern. If JS-side inference becomes necessary in v1.19.x, add a `selectEffectiveIntensity(item, memory): CoachIntensity` helper in `src/coach/index.ts` and update both the test and the prompt to reference it.

2. **NL override parser regex set + negation.** `src/coach/userOverrideParser.ts` is the SOLE owner of intent regex + fuzzy match + negation. Do not inline these regexes in the agent or coach prompt. Negation window is 8 tokens (configurable via constant `NEGATION_TOKEN_WINDOW`). Multi-sentence input resolves to the FIRST sentence's intent.

3. **Calendar sync infinite-loop defense.** Every event Jarvis writes carries `extendedProperties.private.itemId`. The reverse-sync poll skips events without this field (external) AND skips events whose itemId resolves to a soft-deleted local item (orphan). Without these two skips, a feedback loop is possible: `to_calendar` write → `from_calendar` read → `to_calendar` write…. Static test: `tests/integration/calendar.sync.no-loop.test.ts` simulates 10 poll cycles after a single webapp PATCH; assert exactly 1 calendar write occurred.

4. **Sync cursor recovery semantics.** `calendar.jarvis_sync_cursor` keyed memory entry. Corruption (missing OR JSON.parse throws OR missing field) → fall back to 24h lookback + fresh cursor write. Cursor never expires by default; manual reset via `/coach reset-calendar-cursor`. Do NOT add an "auto-reset every N days" path — that re-introduces the 24h-fallback cost on a schedule.

5. **Calendar view DnD pattern reuse from kanban.** `calendar-day-view.js` + `calendar-week-view.js` reuse `cancelPendingRollback()` discipline + `rollback-animating` class + `requestAnimationFrame` deferred reorder from `kanban-view.js`. Top-of-file comment in each new view file references this. The v1.15.0 invariant on `cancelPendingRollback()` being the FIRST call in any pickup entry point applies to the new calendar DnD entry points.

6. **Today focus card data flow.** `today-focus-card.js` reads from the SAME `/api/webapp/items` endpoint the list/kanban/calendar views use; filters client-side for items with `due === today` OR `coach.<itemId>.lastNudge.at > startOfToday`. NO new server endpoint. Card auto-refreshes on the existing `BroadcastChannel` mutation message.

7. **Coach fatigue policy.** 3-strike-then-back-off-7-days for `gentle`/`moderate`/`auto`-effective items. `persistent`-effective items NEVER fatigue (user explicitly opted in). Fatigue entries auto-expire at `expiresAt`; daily prune in the 5-min poll cron deletes expired entries. Self-recovery: any progress write OR chat message about the item resets the fatigue check on next coach turn.

## Open questions for DA + Anti-Slop reviewers

1. **D1 inference in prompt vs code** — is the LLM's non-determinism for auto-intensity acceptable for a "human assistant feel" or does it create the "why does coach keep picking exercise" complaint? Architect's call: prompt for v1.19.0, with v1.19.x escape hatch documented.

2. **D6 5-min poll vs webhook** — Google Calendar push notifications would give sub-second latency but require a public HTTPS endpoint. Architect says webhooks are v1.20.0+ (deferred per "no inbound network" posture). Anti-Slop may ask: is the 5-min polish bar tight enough for the user's "looks nice and let Jarvis manage it all" expectation?

3. **D7 webapp-wins-tie semantics** — concrete scenario: user drags an event in Google Calendar, then drags the same item in webapp 30 seconds later. The webapp PATCH wins (debounce). Is the user surprised when their Google Calendar drag silently reverts? Architect says: yes-but-acceptable; the webapp drag is the more recent intentional action; we surface NOTHING on the calendar side because Google's UI doesn't accept arbitrary toast surface. DA: probe this.

4. **D8 listCalendars on every poll vs once-per-process** — Architect picks once-per-process + cached ID. If the user manually deletes the Jarvis calendar in Google, the cached ID becomes a 404 source for ~as-long-as-the-process-lives. Detection happens on next event-write attempt; we re-run D8. Acceptable, but DA may want a periodic re-verification (e.g., every 24h).

5. **D14 one-way edge invariant under post-write hook** — the hook callback IS registered FROM `src/calendar/` and stored IN `src/organize/storage.ts`. Static-import scan would show no reverse import (correct). But behaviorally, calendar code runs DURING an organize write. Anti-Slop may probe: is this still a "one-way" edge for §5 / §13 purposes?

6. **D11 Step 0 in prompt vs code** — same flavor as Q1. Active monitoring requires the LLM to weigh "engaged + due tomorrow" vs "ignored + due in 3 weeks." Architect says: prompt for v1.19.0. Risk: token usage grows because every coach turn now reads 3 nudges × N items into context. Mitigation: we already have the per-(item, eventType) FIFO cap of 30 from v1.18.0; loadRecentNudgeHistory caps at 3 entries × ~200 chars = ~600 chars per item. At 30 active items: ~18 KB injection. Acceptable.

7. **D17 hub banner's Disable button** — the button POSTs to `/api/webapp/coach/setup` with empty body. In v1.18.0, the endpoint is "create or update"; we need an explicit "off" semantic. Architect's call: extend the endpoint to accept `{ action: 'off' }` AND keep the create/update path. Verify in Phase 2 the endpoint surface; if surface change breaks the test contract, the existing v1.18.0 tests need a passive update.

8. **Scalability — coach overhead per turn at 100 active coached items** — per-turn cost: organize_list (~50ms) + coach_read_history × 1-3 items (~10ms each) + recentNudgeHistory × 30 items (~3ms each) + LLM call (~3-5s). Bounded; trivial. No concern for single-user. For 100 concurrent multi-user: scheduler poll fires staggered (cron jitter) and coach turns share the agent queue per chatId; no contention. **Document in Scalability note in ARCHITECTURE.md.**

9. **Calendar quota for many users** — D6 quota math holds at 100 users. At 10,000 users: 2.88M req/day = 2.88x the free quota. Need quota uplift OR move to webhooks. **NOT a v1.19.0 problem** (single-user posture); document in ARCHITECTURE.md as future-scale note.

10. **`done_signal` intent — risk of false positives** — "I'm done with this conversation" → `done_signal` triggers on the chat thread, not the item. Architect says: fuzzy-match score > 0.6 against item title is the gate; "this conversation" doesn't fuzzy-match any item title. Documented test: T-D3-X (add) — "I'm done with this conversation" with active item "exercise plan" → no override written.
