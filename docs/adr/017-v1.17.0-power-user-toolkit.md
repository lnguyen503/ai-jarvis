## 1. Context

### 1.1 Where the operator surfaces are today (post-v1.16.0)

Three operator/power-user concerns live entirely in chat or in the database, with no webapp surface:

  - **Cron (`scheduled_tasks` table + `src/scheduler/index.ts`).** Created via the chat-side `schedule` tool ("schedule a task to remind me each day at 9am"). Listed via `/scheduled list`; paused/deleted via `/scheduled pause <id>` / `/scheduled delete <id>`. The user has to either (a) trust the LLM to translate "every weekday at 9" into a correct cron expression on creation, or (b) read raw cron strings in `/scheduled list` to verify what was created. For power users this is friction — they want to see and edit the cron expression directly with a "next 5 fire times" preview.
  - **User memory (`data/memories/<userId>.md` files + `src/memory/userMemory.ts`).** Edited indirectly via the `/update_memory` tool (LLM-mediated: "remember I'm a data scientist" → tool call → privacy filter → file append). Read indirectly via `/recall` and the per-turn memory-prepend in the system prompt. Power users want a direct file-level browse + edit to (a) audit what Jarvis "knows" about them, (b) prune stale entries without typing "/memory forget X" 12 times, (c) reorganize sections.
  - **Audit log (`audit_log` table + `src/memory/auditLog.ts`).** Surfaced today only via the chat-side `/audit recent`, `/audit filter <category>`, and `/audit detail <id>` commands. No filtering by time window. No pagination beyond the chat-output 4000-char limit. No way to drill into a category-specific timeline ("show me all my organize.delete events from the last week").

All three are operator-grade workflows that don't fit chat well. The webapp hub at `/webapp/index.html` already has Organize + Debate tiles (post-v1.16.0); v1.17.0 redeems three more entries from the long-standing `<!-- Future features: /schedule, /memory -->` comment in `index.html:53`.

### D2 — Cron builder visual UX (visual ↔ raw bidirectional sync + live preview)

**Picked: visual builder with bidirectional sync to raw cron string.** Six visual presets cover ~95% of human-spoken cron use cases; the seventh ("Custom") falls through to a raw cron string textbox.

**Visual presets (binding for Phase 2).**

  1. **Every X minutes** (X ∈ {5, 10, 15, 30}). Cron form: `*/X * * * *`.
  2. **Every day at HH:MM** (HH ∈ 0..23, MM ∈ {0, 15, 30, 45}). Cron form: `MM HH * * *`.
  3. **Every weekday at HH:MM** (Mon-Fri). Cron form: `MM HH * * 1-5`.
  4. **Every {Mon,Wed,Fri} at HH:MM** (or any selectable day combo). Cron form: `MM HH * * 1,3,5`.
  5. **Every {Saturday,Sunday} at HH:MM** (weekend-only is a common ask; combinable). Cron form: `MM HH * * 0,6`.
  6. **Custom** — raw cron string textbox; `presetKey === 'custom'`.

**Bidirectional sync.** Visual ↔ raw ties together via two pure functions:

```javascript
// public/webapp/cron/cron-builder.js — pure functions (binding)

/** Convert a raw cron expression into a {presetKey, params} pair if it matches one of the 5 visual presets. Returns null on no match (UI shows "Custom"). */
function cronToPreset(expr) { /* … */ }

/** Convert a presetKey + params into the raw cron expression. Throws on invalid inputs. */
function presetToCron(presetKey, params) { /* … */ }
```

When the user picks a visual preset → `presetToCron` writes the raw textbox.
When the user edits the raw textbox → `cronToPreset` re-evaluates the presets; if it matches one, that preset highlights; else "Custom" highlights.

**Live preview.** Server-side endpoint `GET /api/webapp/scheduled/preview?expr=<urlencoded>` parses + validates + computes the next 5 fire times in UTC. Client debounces input by **400ms** before fetching. Renders as a 5-line block: `Next: 2026-04-25 09:00:00 UTC`, etc. On invalid expression: 400 with `{ ok: false, code: 'INVALID_EXPR', error: '<reason>' }`. On expression with zero fire times in the next year (e.g., `0 0 31 2 *` — Feb 31): 200 with `{ ok: true, fireTimes: [], warning: 'This expression doesn't fire — check the day-of-month + month combination.' }`.

**Why one server endpoint + one client component.** The cron-parse + next-fire computation belongs server-side (single source of truth using the same `node-cron` lib that the scheduler uses; avoids client/server divergence on edge cases like leap years, DST). Debouncing belongs client-side (don't slam the server on every keystroke). 400ms is the same debounce window v1.14.6 used for char-counter updates — proven UX precedent.

### D3 — Memory browser edit scope

**Picked: A — full edit (browse + create + edit + delete).** The user gets a webapp surface that maps 1:1 to `data/memories/<userId>.md` operations. The chat-side `/update_memory` + `/recall` flows remain unchanged; both surfaces write to the same file.

**Why A.** The user explicitly asked for "browse + edit". The chat-side editing flow is LLM-mediated (the user says "remember I'm a data scientist" → Claude calls `/update_memory` → privacy filter → append). The webapp gives direct file-level edit (a power-user surface — bypass the LLM-translation step for prune/reorganize workflows). Both surfaces map to the same `userMemory.ts` storage; per-key concurrent-edit detection uses the v1.14.4 ETag pattern (D9).

**Rejected.**
  - **B (read-only).** Punt the edit problem to the LLM-mediated chat flow. But the user explicitly wants edit; rejecting it forces them to type "/memory forget X" 12 times to prune a single section.

**Surface (binding for Phase 2).**

  - `GET /api/webapp/memory` — list all memory entries for the authenticated user. Returns `{ ok: true, entries: [{key, category, body, mtime, etag}, ...] }`. Categories from `MemoryCategory` union: `'profile' | 'preferences' | 'projects' | 'people' | 'avoid'`.
  - `GET /api/webapp/memory/:key` — single entry detail. Returns `{ ok: true, entry: {key, category, body, mtime, etag} }` with `ETag` response header.
  - `PUT /api/webapp/memory/:key` — create OR update. If-Match required for updates; absent for creates (server returns 412 if `:key` already exists when no If-Match header). Body: `{ category, body }`. Privacy filter (`filterMemoryFact`) runs on `body` before write.
  - `DELETE /api/webapp/memory/:key` — delete. If-Match required.

**Note on memory model.** Today `userMemory.ts` stores ONE `.md` file per user with sections per category, not one file per key. v1.17.0 surface treats each section's bullets as logically-keyed entries; the file-level read/write contract goes through new `userMemoryEntries.ts` helpers (D8) that parse + serialize the markdown structure. The underlying file format is unchanged (backward compatible with `/update_memory` and `/recall`).

### D4 — Audit UI pagination strategy

**Picked: A — cursor-based using `(ts, id)` tuple.** Cursor is an opaque string `<ts>_<id>`. Forward-pagination only (no jump-to-page). Sort `ts DESC, id DESC`. Default page size **50**; max **200**. Filters: category (multi-select dropdown of all known categories from `AuditCategory` union including the 5 NEW per D6) + time range preset (`last_hour | last_day | last_week | last_month | custom`). For `custom`, two ISO datetime inputs. All filters compose AND.

**Why A.** Audit volume can be ~1.8M rows/year at scale (a per-message tool_call audit + per-mutation organize audit + per-debate audit + per-webapp audit). Offset-based pagination breaks under high-volume insertion (rows shift; user sees duplicates or gaps). Cursor-based stays correct under concurrent inserts. The `(ts, id)` tuple ensures stable ordering even when multiple rows share a millisecond timestamp (id is monotonically incrementing per `audit_log.id` PK).

**Rejected.**
  - **B (offset-based).** Simpler to implement (`LIMIT n OFFSET m`) but breaks under concurrent inserts. Operator scrolling through a high-traffic period sees duplicates/gaps.

**Endpoint surface.**

  - `GET /api/webapp/audit?cursor=<opaque>&limit=<n>&categories=<csv>&range=<preset>&from=<iso>&to=<iso>`. Returns `{ ok: true, rows: [...], nextCursor: <opaque> | null }`. `nextCursor === null` means end of result set.
  - `GET /api/webapp/audit/:id` — single audit row detail (pretty-formatted JSON detail block).

**Audit cursor format (binding).** `<ts>_<id>` where `ts` is the row's ISO 8601 string and `id` is the integer PK; opaquely base64-encoded over the wire so clients don't construct cursors directly. Server validates the decoded shape; rejects malformed with 400 `INVALID_CURSOR`.

### D6 — New audit categories (5 added to `AuditCategory` union)

**Picked.** Five new categories on the `AuditCategory` union in `src/memory/auditLog.ts`:

```typescript
// Additions to AuditCategory union (after 'debate.persistence_error' line 41):
| 'webapp.scheduled_view'    // v1.17.0 D6 — scheduled-task list/detail/preview read access
| 'webapp.scheduled_mutate'  // v1.17.0 D6 — scheduled-task create/update/delete
| 'webapp.memory_view'       // v1.17.0 D6 — memory list/detail read access
| 'webapp.memory_mutate'     // v1.17.0 D6 — memory create/update/delete (KEY only; no value content)
| 'webapp.audit_view'        // v1.17.0 D6 — audit list/detail read access (meta-audit)
```

**Audit detail metadata (binding for Phase 2; values NEVER stored):**

  - `webapp.scheduled_view`: `{ action: 'list' | 'detail' | 'preview', taskId?, ip }`. For `preview`, no taskId.
  - `webapp.scheduled_mutate`: `{ action: 'create' | 'update' | 'delete', taskId, ip }`. NO cron expression value, NO description, NO command — only the structural fact + IP.
  - `webapp.memory_view`: `{ action: 'list' | 'detail', memoryKey?, ip }`. Memory KEY (e.g., `'preferences'`) is NOT a value; it's structural metadata.
  - `webapp.memory_mutate`: `{ action: 'create' | 'update' | 'delete', memoryKey, ip }`. KEY only; no value content.
  - `webapp.audit_view`: `{ action: 'list' | 'detail', auditEventId?, filters?, ip }`. `filters` is a hash of the active filters (categories array + range preset), not the row contents. `auditEventId` for `detail` action.

**Why no values in audit rows.** Privacy posture inherited from ADR 010 D4 + ADR 016 D9 (debate audit). The audit log is forensic — it records THAT something happened, not WHAT the contents were. Cron expression + memory body content don't go in audit rows. (For cron, the description and command may be sensitive — "remind me my therapy is at 2pm"; for memory, the body content may include preferences the user doesn't want re-leaked into the audit_log even at structural-metadata level.)

**Why `webapp.audit_view` (meta-audit).** Reading the audit log is itself an action worth recording. If an attacker compromises a session they shouldn't be able to scroll through historic audit rows without leaving a trail. Meta-audit is the v1.14.4-style accountability anchor for this surface.

### D7 — Cron preview server-side parser

**Picked.** New helper at `src/scheduler/preview.ts` (~50 LOC). Reuses the existing `node-cron` dependency (already in `package.json` since v1.0.0; used by `src/scheduler/index.ts:1`).

**Helper contract (binding for Phase 2):**

```typescript
// src/scheduler/preview.ts (NEW; ~50 LOC)
import cron from 'node-cron';

export interface PreviewResult {
  ok: true;
  fireTimes: string[];      // ISO 8601 UTC; up to 5; empty = no fire in next year
  warning?: string;         // populated when fireTimes is empty
}

export interface PreviewError {
  ok: false;
  code: 'INVALID_EXPR';
  error: string;            // human-readable reason
}

export function previewCronFireTimes(expr: string, now?: Date): PreviewResult | PreviewError {
  // 1. Validate via cron.validate(expr) — node-cron's syntax check
  if (!cron.validate(expr)) {
    return { ok: false, code: 'INVALID_EXPR', error: `Invalid cron expression: ${expr}` };
  }

  // 2. Reject pathological forms node-cron doesn't catch:
  //    - `*/0 * * * *` (zero-step iteration → infinite loop in the iterator)
  //    - any field with negative numbers
  //    - extra or missing fields
  if (/\*\/0\b/.test(expr)) {
    return { ok: false, code: 'INVALID_EXPR', error: 'Step values cannot be zero' };
  }

  // 3. Compute next 5 fire times within next 365 days from `now` (default Date.now()).
  //    Uses cron-parser shape: iterate minute-by-minute, check matchesCronExpr, take first 5.
  //    OR use `cron-parser` lib (better fit; verify package.json — if not present,
  //    add it as a Phase 2 dep; ~30KB gzip; well-maintained; same author as node-cron).
  // 4. Return at most 5; if zero, return warning.
  // …
}
```

**Phase 2 dep note (binding):** node-cron does not expose a "next-N-fires" iterator. `cron-parser` does (`CronExpression.next()`); add it as a runtime dep if not present (verify `package.json` first; if `cron-parser` is already there, just import). Architect's preference: `cron-parser` over hand-rolled iteration to avoid leap-year / DST edge cases.

**Rejected hand-rolled iteration.** Computing fire times manually means re-implementing leap-year / month-day-bound / DST handling. `cron-parser` is the canonical Node.js library for this; reusing it is cheaper and safer than re-deriving.

**Why the 365-day cap.** Expressions like `0 0 31 2 *` (Feb 31) never fire. Without a time cap the iterator runs forever. 365 days is the longest-meaningful preview horizon (any reasonable cron fires at least yearly; if the cron user wants annual fires, the preview shows 1-5 fires across the year ahead).

### D8 — Memory storage abstraction (key validation; defense in depth)

**Picked.** Memory keys validated by the regex **`^[a-z0-9_-]{1,64}$`** at BOTH the API validator layer (`memory.shared.ts` + the route handlers) AND the storage abstraction layer (a NEW `src/memory/userMemoryEntries.ts` helper that wraps `userMemory.ts`). Defense in depth.

**Why two layers.** API-layer validation rejects bad keys before they reach storage; storage-layer validation is the last-line defense if a future API route forgets to validate. Same posture as ADR 003 item-id validation (regex enforced at the validator + at the repo layer).

**Memory key shape rationale.** Lowercase alphanumerics + underscore + hyphen, 1-64 chars. Excludes:
  - `/` and `\` — path separators (traversal defense).
  - `..` — parent-directory navigation (traversal defense).
  - `\0` (null byte) — POSIX path injection.
  - whitespace — confusing UX.
  - Unicode — limits surface area; if a user wants Unicode keys, they can encode the body content (which has no key restrictions).

**Storage abstraction (binding for Phase 2).** New file `src/memory/userMemoryEntries.ts` (~140 LOC; wraps `userMemory.ts` for entry-level CRUD):

### D9 — Memory browser If-Match concurrency

**Picked.** Per-key ETag computed at read time as `hash(file_mtime_iso || body)` (or just `hash(body)`; mtime cheaper). PATCH/PUT/DELETE on `/api/webapp/memory/:key` requires `If-Match` header. 412 on mismatch with `{ok: false, code: 'PRECONDITION_FAILED', currentEntry: {...}, currentEtag: '...'}` envelope. Reuse v1.14.4 conflict UI from organize (the v1.14.4 R1 2-button "Reload / Save Anyway" pattern).

**Why per-key not whole-file.** Whole-file ETags would conflict every time any entry changes (the file mtime advances on every write). Per-key ETag isolates conflicts to the entry the user is editing. Trade-off: synthesizing a per-key etag from `hash(body)` only catches body-level conflicts, not category/key-rename conflicts; for v1.17.0 that's acceptable (rename = delete + create; the create's existence-check produces 412 if the user races their own rename).

**Phase 2 dev note.** The conflict UI is reused from `edit-form.js` (ADR 015 R1 commit 0b extracted it, ADR 016 R7 left it there). Memory browser imports `showConflictUI` / `hideConflictUI` from `edit-form.js` — OR (cleaner) the architect proposes a small extraction of the conflict UI into a new `public/webapp/shared/conflict-ui.js` if `edit-form.js` is the wrong home for cross-webapp reuse. Architect's call: defer the extraction to v1.17.x+ if Anti-Slop CP1 doesn't flag; for v1.17.0 just have memory webapp reimplement the 2-button UI inline (~30 LOC of duplication is acceptable when the alternative is a reactive-shared-module-extraction that touches three webapps mid-iteration).

### D10 — Audit UI redaction posture (display layer)

**Picked.** **NO additional redaction at display time.** The audit_log table already has IP redaction baked in at write time (3-octet redactor per `redactIp()` in `items.shared.ts:86`; ADR 010 D4). The display layer renders stored values as-is.

**actor_user_id display:** show raw (it's the user's own actions; the audit table is per-user-scoped at the query level).

**detail_json display:** pretty-format as JSON in the detail panel (collapsible nested structures; syntax-highlight via simple class-based CSS — no syntax-highlighting lib). NO content rewriting; if a future bug leaked a value into a detail row, the audit UI would show it — that's the right behavior (operator catches the leak).

**Privacy invariant (binding for Phase 2 + carries forward).** The new audit categories per D6 emit detail rows that contain ONLY:
  - `action` (closed enum from D6)
  - structural identifiers (`taskId`, `memoryKey`, `auditEventId`)
  - `ip` (already 3-octet redacted at write)

## 7. Risks (called out per template)

  1. **Cron expression DOS.** A user (or compromised session) submits `*/0 * * * *` or a similarly pathological expression to `/api/webapp/scheduled/preview`. **Mitigation:** D7 binding rejects `*/0`; cron-parser timeout-bounded; server response budget < 1s; 400 returned with `INVALID_EXPR` code. Phase 2 test enumerates 8 pathological forms. **Severity:** HIGH if it ships unmitigated; LOW with the mitigation chain.
  2. **Memory key path traversal.** A hostile key like `../../etc/passwd` or `..\\..\\etc\\passwd` reaches the file-write path. **Mitigation:** D8 dual-layer validation (`memory.shared.ts` API-layer + `userMemoryEntries.ts` storage-layer); both enforce `^[a-z0-9_-]{1,64}$`; defense in depth. Phase 2 test enumerates 12 hostile keys at API + 4 hostile keys at storage (bypass-attempt). **Severity:** CRITICAL if a single layer is breached; LOW with both layers.
  3. **Audit volume in pagination correctness.** At 1.8M rows/year scale, naive offset-based pagination breaks under concurrent inserts. **Mitigation:** D4 cursor-based `(ts, id)` tuple; forward-only; opaque cursor base64-encoded; server validates decoded shape. Phase 2 test: scroll 200 rows during 50-row insert burst; assert no dupes/gaps. **Severity:** MEDIUM (correctness, not security); mitigation in design.
  4. **`detail-panel.js` extraction discipline.** Mechanical relocation must preserve all branches + all callsites. **Mitigation:** D1 commit -1 binding + Test D1-1 (all existing tests pass UNCHANGED); coverage measure ≥ pre-extraction; `madge` zero-cycle check; 7th iteration of this pattern (v1.14.x → v1.15.0 R1 → v1.16.0 R7) — well-rehearsed. **Severity:** MEDIUM (LOC-drift bug carryover from v1.15.0 R1 / v1.16.0 R7); LOW with discipline.
  5. **Audit privacy invariant (no field values).** A future Phase 2 dev or fix-cycle agent might add `description` or `body` to a webapp.scheduled_mutate / webapp.memory_mutate row "for forensics". **Mitigation:** D10 binding + Phase 2 deterministic scan checks audit detail rows in 5 new categories for forbidden field names (`description`, `command`, `cron_expression`, `body`, `value`). Anti-Slop §6 enforces. **Severity:** HIGH if it ships (privacy regression); LOW with the deterministic scan.
  6. **Memory inline-key sentinel collision.** D8 picks `<!-- key:my_pref -->` HTML comment as the per-bullet key marker (CP1 #11 issue). **Mitigation:** the sentinel is invisible in markdown render and can't naturally appear in conversational text. Phase 2 dev MUST verify legacy bullets without sentinels get synthetic keys (`legacy_<hash8>`) at read time. **Severity:** LOW (cosmetic worst-case if collision); medium if a future Phase 2 dev uses a different sentinel and breaks compat.
  7. **Cross-user timing side-channel.** Detail endpoints (`scheduled.detail`, `memory.detail`, `audit.detail`) must use single-query `SELECT WHERE id = ? AND user_id = ?` — not two-step. **Mitigation:** D10 + CP1 #6 probe; Phase 2 inspection of SQL; per-route timing test within ±20%. **Severity:** LOW (information leak, not data leak); LOW with the single-query enforcement.
  8. **Scheduler reload race.** Webapp creates a task; row INSERTed; scheduler doesn't see it until `reload()`. **Mitigation:** `scheduled.mutate.ts` calls `scheduler.reload()` inline after the INSERT commits; Phase 2 test asserts `getJobs(taskId).status === 'active'` within 100ms. **Severity:** MEDIUM (correctness; user expects immediate firing if cron is overdue); LOW with the inline reload.
