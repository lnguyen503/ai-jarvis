## 1. Context

### 1.1 Where the debate feature is today (post-v1.12.x)

`runDebate(params)` in `src/debate/index.ts:337` is a single in-process orchestrator:

  - Picks a 4-debater roster from `pool.ts` (4 Ollama Cloud models per topic).
  - Runs `maxRounds × exchangesPerRound × roster.length` turns. Default 2 × 2 × 4 = 16 turns.
  - Calls a Claude judge after each round; consensus stops early; otherwise a final-arbiter Claude call writes a verdict.
  - Streams progress via `ProgressPanelApi<DebateState>` into a Telegram inline panel (`src/debate/panelRender.ts`).
  - Emits a single `debate.complete` or `debate.cancel` audit row at terminal state with the full transcript in `detail_json` (`src/debate/index.ts:603-637`).
  - Sends a standalone verdict HTML message after `panel.finalize` and (when transcript > 3500 chars) attaches `.md`/`.txt`/`.docx` transcript files.

**State today is ephemeral by design.** The Telegram panel is the only live view. The audit row is the only persistence — and `detail_json` is a forensic blob, not a user-queryable history.

### 1.2 The OpenClaw differentiation gap

OpenClaw and similar competitors render `runDebate`-style multi-agent panels in their webapps live, with a DOM column per debater that streams as the debate progresses. Jarvis ships the strongest debate engine (4 debaters, judge, final-arbiter, R6 transcript truncation, scrub-by-chat-type) but the only surfaces are:

  - The Telegram inline panel (≤ 4000 chars; collapsed / expanded views; v1.12.x R6 round-1-+-last-round elision).
  - The `.md`/`.txt`/`.docx` attachments (post-hoc; no live progress).
  - The `/audit filter debate.complete` chat command (operator-grade; not user-grade).

**Users who want to see all 4 models thinking in parallel** — the most visceral demo of the multi-model reasoning idea — currently have to squint at one telegram panel that re-renders into one-line summaries. That's a regression vs OpenClaw's UX even though Jarvis's engine is stronger.

v1.16.0 closes that gap with three concurrent capabilities:

  1. **Live debate viewer in the webapp** — separate `/webapp/debate/` app that streams a running debate via SSE, with a column per debater. Reads from a new `debate_runs` + `debate_rounds` SQLite pair.
  2. **3-way diff UI for notes/progress 412 conflicts in `/webapp/organize/`** — when a longer-text-field conflict fires, show user / server / original side-by-side rather than the v1.14.4 R1 2-button "Save Anyway / Reload" UX. Short-field conflicts (title / due / status) keep the 2-button.
  3. **Minimal markdown rendering for notes/progress in `/webapp/organize/`** — read-only render of headings / bold / italic / inline + fenced code / lists / safe links. The `<textarea>` edit form stays raw.

### 1.3 Why one ADR

These three workstreams ship together because they share clients (`/webapp/...`), dev surface (HTML/JS/CSS modules + SSE + new audit row), and the same set of CP1 reviewers. Splitting into 3 ADRs would 3x the CP1 cycle for ~30% extra rigor; the architect's call is that one ADR + one CP1 + Phase 2 commit ordering keeps the cycle fast without sacrificing depth.

### 1.4 What v1.15.0 left in place

Per ADR 015 + `015-revisions-after-cp1.md` (R1 BLOCKING), v1.15.0 will land with `app.js` extracted into `list-view.js` + `edit-form.js` + `kanban-view.js` + `calendar-view.js` + `dates.js` + `hierarchy.js` + the v1.15.0 view-switcher state machine. Post-ship `app.js` projection ≈ 1,977 LOC (under the lowered 2000-LOC trigger). v1.16.0 adds `markdown.js` + `diff.js` to `/webapp/organize/`; touches `app.js` only for the markdown swap in detail view + the 3-way diff dispatch. Net `app.js` delta in v1.16.0 is small (~+40 LOC).

The Debate webapp is its own subdir (`/webapp/debate/`), zero coupling to `/webapp/organize/` source — only shared CSP + HMAC auth chain.

## 2. The 17 Decisions

### D1 — Debate webapp placement: separate app at `/webapp/debate/`

**Picked: A.** New webapp at `/webapp/debate/` with its own `index.html`, `app.js`, `styles.css`. The hub at `/webapp/index.html` gains a Debate tile alongside the existing Organize tile.

**Why.** Organize and Debate are different domain models (queryable mutable items vs append-only debate transcript); sharing a view switcher would force layout compromises (kanban columns vs debater columns are different visual primitives). Separate webapps mirror the v1.13.0 hub pattern (`<!-- Future features added here: /debate, /schedule, /memory -->` is already in `index.html:47`); v1.16.0 redeems that comment.

**Rejected.**
  - **B (4th view in `/webapp/organize/` view switcher).** Mixes concerns. The organize view switcher (List/Kanban/Calendar) all operate on the same `OrganizeItem[]`; Debate operates on `DebateRun[]` + `DebateRound[]`. Forcing them through the same switcher is a category error.
  - **C (separate webapp + cross-link from organize).** No use case in v1.16.0 (debate doesn't reference organize items). Defer cross-linking to v1.17.0+ if a use case emerges.

### D2 — Debate persistence: new SQLite tables via migration 013

**Picked: A.** Two new tables — `debate_runs` (one row per debate) and `debate_rounds` (one row per debater turn) — created by migration `013_debate_runs_and_rounds.sql`. Forwards-only.

**Schema (binding for Phase 2).**

```sql
-- D:\ai-jarvis\src\memory\migrations\013_debate_runs_and_rounds.sql
-- Migration 013: debate_runs + debate_rounds tables (v1.16.0)
--
-- New persistence layer for the live /webapp/debate/ viewer. The Telegram
-- panel + audit_log debate.complete row REMAIN — debate_runs/_rounds is
-- additive, not a replacement. Audit row stays the source-of-truth for
-- forensics; debate_runs is the user-queryable history surface.
--
-- Forwards-only (per ADR 002 migration policy). Idempotent via IF NOT EXISTS.

CREATE TABLE IF NOT EXISTS debate_runs (
  id                TEXT PRIMARY KEY,                    -- UUID v4 (lowercase, no braces)
  user_id           INTEGER NOT NULL,                    -- Telegram user id of the debate initiator
  topic             TEXT NOT NULL,                       -- scrubbed topic (matches state.topic)
  model_lineup_json TEXT NOT NULL,                       -- JSON array: [{debaterName, modelName, providerName}, ...]
  participant_count INTEGER NOT NULL,                    -- roster.length (4 today; future-proof for /debate /size)
  rounds_target     INTEGER NOT NULL,                    -- planned maxRounds (post-clamp 1..5)
  rounds_completed  INTEGER NOT NULL DEFAULT 0,          -- monotonically incremented as rounds finish
  status            TEXT NOT NULL CHECK (status IN ('running', 'complete', 'aborted')),
  verdict_json      TEXT,                                -- nullable until terminal; JSON of the verdict object
  reasoning         TEXT,                                -- nullable; final-arbiter rationale prose
  created_at        TEXT NOT NULL,                       -- ISO 8601 UTC
  updated_at        TEXT NOT NULL,                       -- ISO 8601 UTC
  abort_reason      TEXT                                 -- nullable; populated when status='aborted'
);

CREATE INDEX IF NOT EXISTS idx_debate_runs_user_created
  ON debate_runs(user_id, created_at DESC);

CREATE TABLE IF NOT EXISTS debate_rounds (
  id              TEXT PRIMARY KEY,                                    -- UUID v4
  debate_run_id   TEXT NOT NULL REFERENCES debate_runs(id) ON DELETE CASCADE,
  round_number    INTEGER NOT NULL,                                    -- 1-indexed; matches DebateState.currentRound
  debater_name    TEXT NOT NULL,                                       -- debater human label (Claude / GLM / etc.)
  model_name      TEXT NOT NULL,                                       -- Ollama model id (glm-5.1:cloud, etc.)
  content         TEXT NOT NULL,                                       -- post-scrub turn text
  ts              TEXT NOT NULL,                                       -- ISO 8601 UTC
  UNIQUE(debate_run_id, round_number, debater_name)
);

CREATE INDEX IF NOT EXISTS idx_debate_rounds_run_round
  ON debate_rounds(debate_run_id, round_number);
```

**Why SQLite tables (rejected B = JSON files).**
  - Listing user history needs ORDER BY + LIMIT — trivial in SQLite, requires a directory scan + JSON parse + sort in the JSON-files alternative.
  - Per-user scoping is a single SQL `WHERE user_id = ?` predicate; the JSON-files alternative needs careful path validation to avoid traversal.
  - The audit row already carries the canonical full transcript in `detail_json`; `debate_rounds` is a normalized copy optimized for the streaming/listing query pattern. Dual-store is fine — both write paths are append-only and the audit row is still the source-of-truth (per D2.b).
  - SQLite WAL + FK + the busy_timeout = 5000 setup (`db.ts:55`) handles concurrent writes adequately at single-user-instance scale.

**D2.b — audit row remains source of truth.** The new tables are additive. If `debate_runs` and `audit_log` ever disagree (process crash mid-debate; partial commits), `audit_log` wins forensically. The webapp lists what's in `debate_runs`; operators reading `/audit` see the full picture. No coordinating cross-table transaction in v1.16.0 — each write path is independent. (If the additivity ever feels fragile we can add a CHECK constraint binding `audit_log.detail_json` to debate_run_id in a later migration.)

### D3 — SSE auth: HMAC at connection open + userId-scoped subscription

**Picked: A.** SSE connection sends `Authorization: tma <initData>` on the initial GET (the same header pattern items endpoints use, per ADR 008 R5 + items.shared.ts:91). Server runs the same `authenticateRequest` chain, then binds the connection to the verified `userId`. All subsequent events are filtered by `(userId, debateRunId)` at the server before write to the SSE stream.

**Why.** EventSource API supports custom headers in modern browsers (and the Telegram WebApp embeds support it). HMAC at open + 1h replay window is the SAME posture as items endpoints — same `itemsInitDataMaxAgeSeconds` (3600) — so we don't introduce a new auth model. Connection is bound to one userId for its lifetime; if the user's initData expires, the next event-emission attempt fails closed (the open connection is closed; client reconnects with fresh initData and re-fetches snapshot first per D4).

**Rejected.**
  - **B (per-event re-auth).** Way too expensive — HMAC verify on every round-emission. Not justified.
  - **C (one-time SSE token from a separate authenticated POST).** Adds an endpoint and a token table. The token would either be persisted (extra schema) or in-memory (extra singleton + cleanup). The header approach is simpler and matches the existing chain.

**Implementation note.** The `EventSource` browser API does NOT natively support custom headers. The webapp uses `fetch()` + a manual stream-reader (`response.body.getReader()` + a `TextDecoder` parser) instead of `new EventSource(url)`. ~50 LOC of client-side glue. The server side is plain HTTP keepalive — no library needed.

### D4 — SSE idle timeout: 60s server-side close + 25s keepalive comment

**Picked.** Server-side: 60-second idle timeout. If no events were emitted in 60s, the server sends `\n: idle-timeout\n\n` and closes the connection. Client receives the close, re-fetches snapshot via `GET /api/webapp/debates/:id`, then re-opens the SSE stream.

**Keepalive (separate from the idle timeout).** Server emits a `: keepalive\n\n` SSE comment line every **25 seconds**. SSE comment lines reset Cloudflared / nginx idle proxy timers without delivering an event to the client (the client's stream reader sees them and discards them per the SSE spec). 25s is well below typical 60s proxy timeouts; the 60s server-side idle close ONLY fires when nothing else (round / verdict / keepalive) is emitted, which is impossible while keepalive is active. So the practical effect: keepalive holds the connection forever; the 60s idle close is a safety belt.

**Cloudflared note.** Cloudflared tunnels (the deployment posture per `BUILD_SANDBOX.md` setup) honor SSE comment lines; verified in v1.13.0 review history. No special header (no `X-Accel-Buffering: no`) needed for cloudflared, but Phase 2 should set it anyway as defense-in-depth (ngix and other proxies need it):

```
Content-Type: text/event-stream
Cache-Control: no-cache, no-transform
Connection: keep-alive
X-Accel-Buffering: no
```

**Client reconnect strategy.** On any close (idle, error, or `complete`/`aborted` event received), the client uses exponential backoff: 1s, 2s, 4s, 8s, 16s, capped at 30s. Reconnect always re-fetches the snapshot first via `GET /api/webapp/debates/:id`, then opens the SSE stream — never resumes mid-stream (debater rounds are idempotent at the `(round_number, debater_name)` UNIQUE constraint, so the snapshot covers any in-flight rounds the previous stream missed).

### D5 — SSE event bus: in-process EventEmitter singleton

**Picked: A.** New module `src/debate/eventbus.ts` (~50 LOC). Exports a singleton `DebateEventBus` with three methods:

```typescript
// src/debate/eventbus.ts
import { EventEmitter } from 'node:events';

export interface DebateBusEvent {
  kind: 'round' | 'verdict' | 'complete' | 'error';
  payload: Record<string, unknown>;
}

class DebateEventBus extends EventEmitter {
  publish(runId: string, event: DebateBusEvent): void {
    this.emit(`run:${runId}`, event);
  }
  subscribe(runId: string, handler: (event: DebateBusEvent) => void): () => void {
    const listener = (e: DebateBusEvent) => handler(e);
    this.on(`run:${runId}`, listener);
    return () => this.off(`run:${runId}`, listener);
  }
}

export const debateEventBus = new DebateEventBus();
```

**Why.** Node's built-in `EventEmitter` — zero new dependencies. `runDebate` publishes; SSE handlers subscribe. Cleanup is per-handler via the returned unsubscribe function (called on SSE connection close). Event bus carries no state — if the SSE handler attaches AFTER round 1 has fired, the snapshot from `GET /api/webapp/debates/:id` covers the missed events; subsequent events flow normally. Event keys are namespaced by `runId` so listeners only see their own run.

**Memory leak guard.** EventEmitter has a default `maxListeners = 10`. v1.16.0 sets `debateEventBus.setMaxListeners(0)` (unlimited) at module load — single-user deployment + cleanup-on-disconnect makes a fixed cap unnecessary. We keep the leak guard via the unsubscribe contract: every subscribe MUST be matched by an unsubscribe in the SSE close handler. Phase 2 dev binds this with a `try { ... } finally { unsubscribe(); }` wrapper around the SSE handler.

**Rejected.**
  - **B (poll DB every 500ms).** Lossy (round emitted between polls) + 500ms latency baseline. The poll alternative is a fallback if the event-bus pattern proves buggy; not the default.

### D6 — Gateway hook for debate persistence: `persistenceHook` callback parameter

**Picked: A.** Add an optional `persistenceHook?: DebatePersistenceHook` field to `DebateParams`. The gateway provides it; `runDebate` calls it as state transitions:

```typescript
// src/debate/index.ts — addition to DebateParams (around line 95)

export interface DebatePersistenceHook {
  onStart(state: DebateState): Promise<void> | void;
  onRound(round: { roundNumber: number; debaterName: string; modelName: string; content: string; ts: string }): Promise<void> | void;
  onVerdict(verdict: NonNullable<DebateState['verdict']>, reasoning: string | null): Promise<void> | void;
  onAbort(reason: string): Promise<void> | void;
}

export interface DebateParams {
  // ... existing fields ...
  persistenceHook?: DebatePersistenceHook;
}
```

The gateway's `/debate` command handler creates the hook. The hook calls `DebateRunsRepo.insertRun()` on `onStart`, `DebateRoundsRepo.insertRound()` on `onRound`, `DebateRunsRepo.markComplete()` on `onVerdict`, `DebateRunsRepo.markAborted()` on `onAbort`. SAME hook also publishes to `debateEventBus.publish(runId, { kind, payload })` so SSE subscribers see the event in lockstep with the DB write.

**Why.**
  - Decouples `runDebate` from the memory layer (`runDebate` already has zero direct repo imports; this preserves that).
  - Optional — chat-only invocations can pass `persistenceHook: undefined` and runDebate behaves exactly as before. v1.16.0 always passes it; later versions could omit it for ephemeral debates.
  - Hook calls are async-safe — `runDebate` `await`s each call, so a slow DB insert pauses the next debater turn (acceptable; insert is fast). Errors in the hook are caught and logged at warn (matching the `try/catch` pattern around audit emission); a failed hook does NOT abort the debate (debate continues; webapp may show stale data; audit row at terminal state is still complete).

**Rejected.**
  - **B (runDebate writes DB directly).** Couples debate to memory layer. Adds repo imports to `src/debate/index.ts`. Worse module boundary.
  - **C (post-hoc serialization).** Loses the live-streaming property. Whole point of v1.16.0 is to see rounds as they fire. Rejected.

**Hook ordering invariant.** `runDebate` calls the hook BEFORE updating `panel.updateState`, so:
  1. New round turn produced.
  2. Scrub turn text.
  3. Append to `state.transcript`.
  4. Call `persistenceHook.onRound({...})` — hook writes to DB + publishes to event bus.
  5. Call `panel.updateState(state)` — Telegram panel re-renders.

This ordering means the SSE viewer can lag behind the Telegram panel by ≤ 100ms (DB write + event-bus emit), which is fine. It also means the hook fires BEFORE the scrubbed text is visible anywhere — a bug in the hook can't leak unscrubbed content (text is already scrubbed at step 2).

### D7 — Markdown subset: roll our own ~150 LOC parser + DOM construction

**Picked: A.** New module `public/webapp/organize/markdown.js` (~150 LOC). Line-based parser; returns DOM `Node[]`, NOT HTML strings.

**Supported subset.**
  - Block: `# H1`, `## H2`, `### H3`, paragraphs (blank-line separated), `- bullet lists` (single level, no nesting), `1. numbered lists` (single level), ` ```fenced code blocks``` ` (no language hint rendering — class added but stays plain).
  - Inline: `**bold**`, `*italic*`, `` `code` ``, `[link text](url)`. (Note: `_underscore italic_` deliberately not supported — italics live in `*` only, simpler tokenizer; documented in module JSDoc.)

**Hard exclusions (per security posture).**
  - No raw HTML (treated as plain text with `<` / `>` rendered as text via `createTextNode`).
  - No images (no `![alt](url)` syntax — image embedding is an exfiltration channel; defer to v1.16.x if anyone asks).
  - No tables (defer per D17).
  - No blockquotes (defer per D17).
  - No autolinking — only explicit `[text](url)` syntax becomes a link; bare `https://...` in prose stays as text.

**Link URL safety (binding).**

```javascript
const SAFE_LINK_SCHEMES = ['http:', 'https:', 'mailto:'];

function isSafeLinkUrl(url) {
  try {
    const parsed = new URL(url, window.location.origin);
    return SAFE_LINK_SCHEMES.includes(parsed.protocol);
  } catch (_) {
    return false;
  }
}

// In the link tokenizer:
function buildLinkNode(text, url) {
  if (!isSafeLinkUrl(url)) {
    // Render as plain text, preserve original markup so users see why it didn't link.
    return document.createTextNode(`[${text}](${url})`);
  }
  const a = document.createElement('a');
  a.href = url;                              // setAttribute equivalent; URL passed validation
  a.textContent = text;                      // textContent — never innerHTML
  a.rel = 'noopener noreferrer';             // mandatory for external links
  a.target = '_blank';                       // open in new tab (Telegram WebApp default)
  return a;
}
```

**XSS posture (binding for Phase 2 + Anti-Slop §10).** EVERY text fragment passes through `document.createTextNode` or `element.textContent =`. NEVER `element.innerHTML =`. The single `setAttribute`-equivalent on `<a>` is the `href`, which has been validated. Code blocks use `<pre><code>` with `textContent` set on the inner `<code>`. Bold/italic/inline-code wrap a `textContent` node. Heading tags are constructed via `createElement('h1'|'h2'|'h3')` + `textContent`. Defense in depth: even if the parser were bypassed, no DOM property accepts arbitrary HTML.

**Tokenizer pattern.** Line-based: split content by `\n`; for each line classify as heading / list / code-fence / blank / paragraph; in the inline pass, scan paragraph + heading + list-item text for inline tokens with regex (no nested matches; greedy left-to-right). Code fences span multiple lines (track open/close state across the line loop). Tokenizer is pure (input string → output AST array of `{type, ...}`); rendering is a separate pass (AST → DOM nodes) — clean separation.

**Rejected.**
  - **B (defer; keep `<pre>` for v1.16.0).** Markdown rendering is an OpenClaw differentiation point — debate transcripts read like garbage in `<pre>`. Ship it now.

### D8 — 3-way diff: minimal LCS line-by-line ~120 LOC

**Picked: A.** New module `public/webapp/organize/diff.js` (~120 LOC). Pure function `diff(originalLines, userLines, serverLines) → { user: DiffChunk[], server: DiffChunk[], original: DiffChunk[] }`.

**Algorithm.** Standard LCS (longest common subsequence) on lines. Three pairwise diffs (`original ↔ user`, `original ↔ server`) merged into a 3-column alignment. Each cell is `{type: 'same' | 'add' | 'remove', text}`.

**UX (binding).** When the conflict response (412) fires AND the conflicted field is `notes` OR `progress` (the long-text fields):

  - Replace the v1.14.4 R1 2-button conflict UI (Save Anyway / Reload to see the latest) with the **3-way diff panel**.
  - Three columns: **Mine** (user's pending edit) | **Server** (current state from 412 envelope) | **Original** (the value the user started editing from — captured client-side at edit-mode entry).
  - Differing lines highlighted: green for `add`, red for `remove`, neutral for `same`.
  - Three buttons at the bottom:
    - **Take Mine** — re-PATCH with `X-Force-Override: 1` and the user's pending value (overwrites server). Same wire effect as today's Save Anyway.
    - **Take Theirs** — discard pending; re-fetch the item; re-render in detail view. Same wire effect as today's Reload.
    - **Save Manually-Merged** — opens an editable `<textarea>` pre-populated with the **server** value; user edits to merge; submit re-PATCHes with `X-Force-Override: 1` and the merged value.

For short fields (`title`, `due`, `status`, completion), the v1.14.4 R1 2-button conflict UI stays — diff has no value when the whole field is one short string.

**Why ship it.** OpenClaw shows raw two-column user/server only. A 3-way diff (with the original baseline) is a clear differentiation; users know whether their change overlaps with the server's change or whether they're stomping on it.

**Rejected.**
  - **B (2-column user/server, no original).** Can't tell whether to take mine — no way to know which side made the change relative to the baseline.
  - **C (defer, stay with v1.14.4 2-button).** OpenClaw differentiation is the point.

### D9 — `webapp.debate_view` audit category: read-access only, no content

**Picked.** Add `'webapp.debate_view'` to the `AuditCategory` union in `src/memory/auditLog.ts`. Three read endpoints emit one row each:

  - `GET /api/webapp/debates` → `{ action: 'list', ip, count }`
  - `GET /api/webapp/debates/:id` → `{ action: 'detail', debateRunId, ip }`
  - `GET /api/webapp/debates/:id/stream` (on connection open) → `{ action: 'stream-open', debateRunId, ip }`

**Detail JSON shape (binding).**

```typescript
export interface WebappDebateViewDetail {
  action: 'list' | 'detail' | 'stream-open';
  debateRunId?: string;        // null for action='list'
  count?: number;              // present only for action='list' (number of returned rows)
  ip?: string;                 // 3-octet redacted (per items.shared.ts redactIp)
}
```

**No content in audit rows.** No topic, no rounds, no verdict, no debater names. The audit row says "this user looked at this debate" — privacy posture mirrors `webapp.item_create` (records WHAT happened, not the values). The full transcript already lives in the `debate.complete` audit row from v1.12.x; we don't duplicate it here.

**Debouncing.** No debouncing — read endpoints fire one row per request (matches `webapp.item_mutate` posture). High-volume reads (e.g., a misbehaving client polling `/api/webapp/debates` every second) would be a per-IP rate-limit concern, not an audit-volume concern. Phase 2 dev binds `audit_log` insert costs as cheap; if rates become a problem, add a debouncer in v1.16.x (matching `webapp.auth_failure`).

**No content NOT in audit; no SSE-event-level audit emission.** The `stream-open` row is emitted ONCE on connection open. Subsequent SSE events (round, verdict, complete) do NOT each emit an audit row — the connection-open record + the existing `debate.complete` row from v1.12.x cover the forensic requirement. SSE-event-level rows would multiply the audit volume by ~16x per debate (one per turn) and add zero forensic value.

**Migration scope.** No new migration for the category itself; `audit_log.category` is `TEXT NOT NULL` so adding a string is a code-only change. Migration 013 (D2) creates the new tables; the audit category union update is a separate Phase 2 commit. (Pattern matches v1.14.6 D7 `webapp.item_create`: union string was added without a migration.)

### D10 — `items.shared.ts` split: new `debate.shared.ts` module

**Picked: A.** New file `src/webapp/debate.shared.ts` (~80 LOC). Contains:

### D11 — `GET /api/webapp/debates`: list endpoint

**Picked.** Paginated list of the authenticated user's debate history.

**Request.** `GET /api/webapp/debates?limit=N&offset=M`
  - `limit` query param: default 50; max 100 (clamp server-side).
  - `offset` query param: default 0; non-negative integer.
  - Sort: `created_at DESC` (most recent first; matches the index `idx_debate_runs_user_created`).
  - Filter: `WHERE user_id = ?` — per-user scope; no cross-user.

**Response (200 OK).**

```json
{
  "ok": true,
  "debates": [
    {
      "id": "uuid",
      "topic": "...",
      "participantCount": 4,
      "roundsTarget": 16,
      "roundsCompleted": 16,
      "status": "complete",
      "createdAt": "2026-04-25T12:34:56.789Z",
      "updatedAt": "2026-04-25T12:42:11.123Z",
      "verdictSummary": "first 200 chars of verdict_json.summary OR null if running"
    }
  ],
  "total": 12,
  "limit": 50,
  "offset": 0
}
```

`verdictSummary` is computed server-side: parse `verdict_json`, take `.summary`, slice to 200 chars. If `verdict_json` is null (status='running' or 'aborted' with no verdict), return `null`. This avoids the client parsing the full verdict JSON for the list view.

**Error envelopes (matches items.shared.ts pattern).**
  - 400 BAD_REQUEST — invalid `limit`/`offset`.
  - 401 AUTH_FAILED — no Authorization header / HMAC failed.
  - 403 NOT_ALLOWED — user not in allowlist.
  - 500 INTERNAL_ERROR — DB read failed.

**Audit emission.** `auditDebateView(memory, userId, 'list', null, debates.length, redactedIp)` — one row per call.

**Cache-Control.** `no-store, no-cache, must-revalidate` (matches items endpoints; debates can change live).

### D13 — `GET /api/webapp/debates/:id/stream`: SSE endpoint

**Picked.** SSE stream of debate events. HMAC auth at open (D3); 60s idle close + 25s keepalive (D4); event-bus subscription (D5).

**Request.** `GET /api/webapp/debates/:id/stream` with `Authorization: tma <initData>` header. (No `Accept: text/event-stream` requirement — the response Content-Type is text/event-stream regardless of Accept.)

**Initial event: `snapshot`.** Server immediately writes a `snapshot` event with the same shape as the `GET /api/webapp/debates/:id` response (D12). This means clients can open the SSE connection directly without first calling `/:id` separately — saves a round trip.

```
event: snapshot
data: {"id":"...","topic":"...","rounds":[...],...}

```

**Subsequent events.**

```
event: round
data: {"roundNumber":2,"debaterName":"GLM","modelName":"glm-5.1:cloud","content":"...","ts":"..."}

event: verdict
data: {"verdict":{"kind":"final-arbiter","decision":"...","rationale":"...","dissent":"..."},"reasoning":"..."}

event: complete
data: {"status":"complete","roundsCompleted":8,"updatedAt":"..."}

event: error
data: {"abortReason":"...","updatedAt":"..."}
```

**Heartbeat.** `: keepalive\n\n` every 25 seconds (no event name — pure SSE comment; client discards per spec).

**Server close conditions (binding).**

  1. **Terminal state event was published** (`complete` or `error`). Server emits the event, writes `: closing\n\n` (informational comment), closes the socket. Client receives the close, treats it as a normal terminal-state close, does NOT reconnect.
  2. **60-second idle.** No events emitted in 60s (impossible while keepalive runs; a safety belt). Server writes `: idle-timeout\n\n`, closes. Client reconnects with backoff (D4).
  3. **Client disconnects.** Server detects `req.on('close')`, calls the unsubscribe callback (D5), no further sends.
  4. **Debate run is already terminal at connection open.** Server emits the snapshot event (which carries the terminal state) + the relevant `complete`/`error` event + `: closing\n\n`, closes. Client sees terminal snapshot, doesn't reconnect.

**Audit emission.** `auditDebateView(memory, userId, 'stream-open', debateRunId, null, redactedIp)` on connection open. NO audit on individual events (D9 binding).

**Authorization 404 vs 401.** If `:id` doesn't exist OR doesn't belong to the verified userId, return **HTTP 404** with `text/plain` body before promoting to SSE. The SSE response headers are NOT written. (Once the SSE response is written, the client interprets HTTP errors via `event: error` instead.)

**Response headers (binding).**

```
Content-Type: text/event-stream
Cache-Control: no-cache, no-store, no-transform, must-revalidate
Connection: keep-alive
X-Accel-Buffering: no
```

## 7. Risks (called out per template)

  1. **SSE memory leak.** EventEmitter listener leak if unsubscribe is missed. **Mitigation:** unconditional unsubscribe in close handler (D5); test in `debate.eventbus.leak.test.ts`; `setMaxListeners(0)` is documented as INTENTIONAL with a unit-test backstop. **Severity:** HIGH if it ships; LOW with mitigation.
  2. **SQLite contention under SSE fan-out.** Multiple SSE clients reading + persistenceHook writing concurrently. **Mitigation:** WAL + busy_timeout=5000 already set; SSE reads are SELECT-only (no locks); writes are append-only. Phase 2 test: 4 concurrent subscribers + 1 active debate insert path; assert zero SQLITE_BUSY. **Severity:** MEDIUM; mitigation in place; verify under load.
  3. **Markdown XSS.** Parser bypass leaks raw HTML. **Mitigation:** ALL text via `createTextNode`/`textContent`; ONLY `setAttribute` is `<a href>` against an allowlist; 20 injection-probe tests (D7); plaintext fallback on parser error. **Severity:** HIGH if a bypass ships; LOW with the mitigation chain.
  4. **3-way diff DoS via large input.** LCS is O(n×m); 5000-line × 5000-line is 25M operations. **Mitigation:** MAX_DIFF_LINES = 200; truncate-with-marker for larger; document cap. **Severity:** LOW (notes/progress is bounded to 5000 chars per ADR 011 D2; lines is bounded to ~250 at worst).
  5. **persistenceHook silent failure.** Hook throws → caught → logged → debate continues but webapp shows stale data. **Mitigation:** debate.complete audit row covers forensic loss; Phase 2 dev binds hook errors to a `WARN` log with hook name + run id. **Severity:** MEDIUM; observability covers detection.
  6. **Cross-user 404 vs 403 timing leak.** D12 returns 404 for both "doesn't exist" and "exists but other user". **Mitigation:** Single-query SQL with both predicates (`WHERE id = ? AND user_id = ?`); SQLite's b-tree lookup is constant time within the index. **Severity:** LOW.
  7. **CSP regression.** SSE response on `connect-src 'self'` works in modern browsers; older Telegram WebApp embeds (Android < 9, iOS < 16) may not. **Mitigation:** Phase 2 dev tests on the lowest-supported Telegram client; if blocked, fall back to long-poll (1-second polling against `/api/webapp/debates/:id`) — degrades live UX but functional. **Severity:** MEDIUM.
  8. **`app.js` over 2000 LOC.** v1.15.0 R1 lowered the trigger; v1.16.0 puts it at 2017. **Mitigation:** if Anti-Slop CP1 flags, do the detail-panel.js extraction in Phase 2 (mechanical; ~200 LOC); otherwise defer to v1.17.0. **Severity:** LOW (mechanical fix; trigger fires the next iteration regardless).
  9. **SSE keepalive vs proxy buffering.** Cloudflared honors `: keepalive`; nginx may not without `X-Accel-Buffering: no`. **Mitigation:** header set on SSE response (D4). **Severity:** LOW (deployment is cloudflared today).
  10. **Migration 013 + concurrent restart.** A prior process at boot might run migration 013 simultaneously. **Mitigation:** SQLite migration runner already serializes (per `runMigrations` in `src/memory/migrations/index.ts`); IF NOT EXISTS makes the operation idempotent. **Severity:** LOW.
