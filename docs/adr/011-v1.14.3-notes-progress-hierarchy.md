# ADR 011 — v1.14.3 `/organize` Web App: Notes + Progress + Hierarchy + Restore + `updated:`

**Status:** Proposed (Phase 1, v1.14.3)
**Date:** 2026-04-25
**Deciders:** Architect Agent (iteration)
**Extends:** ADR 010 + 010-revisions (v1.14.2 mutation surface), ADR 009 + 009-revisions (v1.14.0 read path), ADR 008 + 008-revisions (webapp Express foundation), ADR 005 (multi-user posture, audit categories), ADR 003 + 003-revisions (`/organize` storage layout, filename-as-id, 200-active-item cap), ADR 001 (module boundaries).

This ADR records the architectural decisions for the v1.14.3 **notes + progress + hierarchy iteration** — the fourth in the webapp series (read v1.14.0 → hotfix v1.14.1 → mutate v1.14.2 → notes/progress/hierarchy v1.14.3 [THIS]). Scope is design only — no code is in scope. Phase 2 developer agents implement the decisions below verbatim; deviations require an addendum to this ADR.

## 0. Context

**What v1.14.2 already shipped:**

  - `PATCH /api/webapp/items/:id` for `{title, due, status, tags}` (four allowed fields), 1KB body cap, machine-readable validator codes (RA1's 11 codes).
  - `POST /api/webapp/items/:id/complete` with absolute-write semantics (R18) and per-status checkbox visibility (R14).
  - `DELETE /api/webapp/items/:id` with 6-second two-tap confirm + visible countdown + Cancel + the toast wording **"Deleted. Restore via Telegram chat: /organize restore <id>"** (R5 + R16-toast).
  - `webapp.item_mutate` audit category with `{action, itemId, changedFields, ip}` JSON detail (3-octet redacted ip per QA M1 fix); `webapp.stale_edit` for mtime mismatches (R2-mtime).
  - `writeAtomically` per-call random tmp suffix (R8 BLOCKING) — v1.14.3 inherits this and depends on it (every new write path also goes through it).
  - `OrganizeItemDetail` interface in `src/organize/types.ts` (W7) with `mtimeMs` for stale-edit detection.
  - `src/webapp/items.{read,mutate,shared}.ts` split, `auditItemMutate` co-located in `items.shared.ts`, `redactIp` helper, `validation.ts` with `ALLOWED_PATCH_FIELDS = {title, due, status, tags}` and 11 ValidatorErrorCode values, `ISO_DATE_RE` + `TAG_RE` named constants.
  - QA M2/M3/M4 fixes landed: client tag normalizer no longer strips chars (sends verbatim, server is sole boundary); list refresh after PATCH happens via `showList(renderedItems)`; HTML title `maxlength="500"` matches server.
  - The detail view at `public/webapp/organize/index.html` has `<pre id="detail-notes">` + `<pre id="detail-progress">` rendering existing notes/progress as read-only, but the edit form does NOT include them. The whole iteration brief is to close that gap.

**What's missing — the v1.14.3 user-visible deficit (4 things):**

  1. **Notes editing.** The user can READ notes in the webapp (v1.14.0) but cannot EDIT them — they must drop into Telegram and run `/organize update <id> --notes "..."` (which itself is currently chat-only). Webapp is still a partial-edit surface.
  2. **Progress editing.** Same gap. The agent's `organize_log_progress` tool appends timestamped lines via `appendProgressEntry`; the user has no way to fix typos or strike completed milestones from the webapp. Long-form Progress logs accumulate; webapp can render but not edit.
  3. **Hierarchy navigation.** Goals can have child Tasks/Events via `parentId` (already in the data model — `OrganizeFrontMatter.parentId: string | null` since v1.8.6). The chat-side `/organize` listing renders all items flat. The webapp also renders all items flat. There's no visual cue that a Task belongs to a Goal, no collapse/expand, no way to scan "what's under this goal?" from either surface. This is the navigation gap.
  4. **`/organize restore <id>` chat command.** The v1.14.2 delete toast says "Restore via Telegram chat: /organize restore <id>" — but **the command does not exist** (verified at `src/commands/organize.ts:115-178` — only `nag/reconcile/off/on/<id>/tag/all/tasks/events/goals/show/list` are handled). The toast is making a promise the system can't keep. v1.14.3 fulfills it.

**Bonus item that lays foundation for v1.14.4:**

  5. **`updated:` ISO front-matter field**, auto-stamped on every write path, immutable `created`. v1.14.4's planned ETag + If-Match + 412 work needs a stable, monotonic-ish "last-modified" stamp; mtime is fine for v1.14.2 stale-warning best-effort, but ETag wants something stored IN the data so backups, restores, and cross-machine clones don't reset it. Stamping `updated:` in v1.14.3 means v1.14.4's ETag is just `"W/${updated}"` rather than reading mtime. Also makes the chat-side `/organize <id>` show "last edited" cleanly.

**Five items ship together in v1.14.3:**

  1. **`updated:` front-matter field** (D1) — added to `OrganizeFrontMatter` as optional `string | null`; auto-stamped via the `stampUpdated()` helper on every write path (`createItem`, `updateItem`, `appendProgressEntry`, `softDeleteItem`'s rewriteContent, the new `restoreItem` primitive). Parser tolerates missing field; older items get stamped on first edit.
  2. **PATCH allowed-fields list expanded** to six (D2 + D3) — adds `notes` (10KB cap, `NOTES_TOO_LONG` validator code) and `progress` (20KB cap, `PROGRESS_TOO_LONG` validator code) alongside the existing four. Body size cap raised from 1KB to 32KB (D4).
  3. **Edit form gets two `<textarea>` inputs** for notes and progress (D2/D3 client side); shows existing values; same non-optimistic save path as v1.14.2; same `webapp.item_mutate` audit row with `changedFields: ['notes']` or `['progress']` (D11 — field NAMES only, never values).
  4. **Hierarchy rendering in list view** (D5/D6/D7/D8/D13) — client-side grouping module `public/webapp/organize/hierarchy.js` (D13) groups items by `parentId`; goals render as collapsible group headers with child Tasks/Events nested under them; chevron toggles persist in sessionStorage (D7); separate tap targets for goal-detail-navigation vs collapse-toggle (D8). Filter-intersection edge case: orphan-renders-top-level (D5).
  5. **`/organize restore <id>` chat command** (D9) — new branch in `src/commands/organize.ts` matching id pattern; new storage primitive `restoreItem(userId, dataDir, id)` that renames `.trash/<id>.md` back to `<id>.md`, strips `deletedAt`, stamps `updated:`, returns the restored item; new audit category `organize.restore` (one row per successful restore); 404 on miss; reply text `"Restored: <title> [<status>]"`.

  - **No `/organize trash list` companion command** (D10) — scope creep; users find ids via chat history or filename search; defer to v1.14.5 polish if useful.
  - **No backfill migration for `updated:`**. Existing items without the field stay valid; first edit stamps it. Phase 2 verifies via the parser tolerance test (D1).

## Decisions

### Decision 1 — `updated:` front-matter field discipline

**Status:** Accepted — optional `updated: string | null` added to `OrganizeFrontMatter`, stamped via a single `stampUpdated()` helper used by every write path.

**Context.** Three candidates for where the timestamp lives:

  - **(a) New optional `updated:` front-matter field, auto-stamped at every write.** Survives backups and clones. Persisted next to the data. Visible in the file when humans inspect.
  - **(b) Use file mtime exclusively.** No new field. But: rsync, git, file-restore-from-backup, and editor-save-without-content-change all reset mtime in unpredictable ways. v1.14.4's ETag wants something stable.
  - **(c) Sidecar metadata file.** `data/organize/<userId>/.meta/<id>.json` with `{updated, etag, ...}`. Adds an entire I/O surface for one field. Strictly worse than (a).

**Decision.** (a). Justifications:

  1. **Survives operator activity.** Ports cleanly across machines, backups, and `git pull`. mtime resets on `cp` / `rsync --no-times` / fresh checkout; `updated:` doesn't. v1.14.4 ETag wants this.
  2. **Single source of truth for "last edited."** Chat-side `/organize <id>` can show "last edited 2026-04-25T15:30:00Z" trivially (Phase 2 nice-to-have, not bound here).
  3. **Tolerant parser already handles missing fields.** `parseItemFile` already accepts missing optional fields; `updated` becomes another tolerated absence that gets filled on first edit. No backfill migration.
  4. **Discipline lives in ONE helper.** `stampUpdated()` is the only place that writes the field. Every write path calls it before `serializeItem`. No drift, no forgetting.

**Field definition (binding for Phase 2):**

```typescript
// src/organize/types.ts — extend OrganizeFrontMatter
export interface OrganizeFrontMatter {
  // ...existing fields unchanged...
  /** v1.14.3 — ISO timestamp of the most recent write. Stamped via stampUpdated()
   *  on every write path. Older items (pre-v1.14.3) may not have this field;
   *  parser tolerates absence and first edit stamps it. Distinct from `created`
   *  (which is immutable) and from file mtime (which resets on cp/rsync). */
  updated?: string | null;
}

// src/organize/types.ts — extend OrganizeItemDetail
export interface OrganizeItemDetail {
  // ...existing fields unchanged...
  /** Same field as front-matter `updated`; nullable for legacy items. */
  updated: string | null;
}

// src/organize/storage.ts — new helper, exported for Phase 2 unit tests
/**
 * Return a new front-matter object with `updated` set to now. Pure — does not
 * mutate the input. Use at every write path that calls serializeItem with new
 * content. Note: pure rename calls (e.g., the .trash rename at softDeleteItem
 * line 721) do NOT call this helper, because they don't change content.
 */
export function stampUpdated(fm: OrganizeFrontMatter, now: Date = new Date()): OrganizeFrontMatter {
  return { ...fm, updated: now.toISOString() };
}
```

**Write paths that MUST call `stampUpdated` (binding survey, verified by line):**

| Path | File:line | Stamps? | Why |
|---|---|---|---|
| `createItem` | `storage.ts:413-429` | YES | First write of the item; `created` and `updated` initialize to the same value at creation. |
| `updateItem` | `storage.ts:644-657` | YES | Partial patch then re-serialize; the canonical edit path. |
| `softDeleteItem` rewriteContent | `storage.ts:697-702` | YES | Rewrites the live file with `deletedAt` stamped — a content change. The .trash rename at `:721` does NOT call stampUpdated (no content change). |
| `appendProgressEntry` | `storage.ts:761-773` | YES | Appends a progress entry; this is a write. |
| `restoreItem` (NEW per D9) | `storage.ts` (new function) | YES | Strips `deletedAt`, moves from .trash to live, re-serializes. Content change. |

**Serializer change (binding):** `serializeItem` at `storage.ts:131-160` adds an `updated` line to the front-matter block, conditionally:

```typescript
// inside serializeItem, after the deletedAt-conditional block (storage.ts:148-150)
if (fm.updated != null) {
  frontMatterLines.push(`updated: ${fm.updated}`);
}
```

Emitted ONLY when present (mirrors the `deletedAt` conditional pattern from v1.11.0 R3). This means files written before v1.14.3 retain their pristine front-matter; files written from v1.14.3 onward carry `updated:`. Parser tolerates either.

**Parser change (binding):** `parseItemFile`'s front-matter loop adds one line for `updated`:

```typescript
// inside the front-matter scan loop (around storage.ts:230-260)
} else if (key === 'updated') {
  fm.updated = value || null;  // empty string → null; valid ISO → string
}
```

**Tests required (Phase 2):**

  1. **Unit test — stampUpdated purity.** `stampUpdated(fm)` returns new object; original unchanged; `updated` is a valid ISO.
  2. **Unit test — round-trip.** Create item → read back → `updated` is an ISO. Update item with `{title: 'new'}` → read back → `updated` is later than initial. Append progress → read back → `updated` advances again.
  3. **Unit test — legacy parser tolerance.** Construct a v1.14.2-format file (no `updated:` line), parse it → `fm.updated` is undefined or null (decide; recommend null for consistency with `parentId`/`deletedAt` patterns); update it once → on-disk file now has `updated:`.
  4. **Integration test — softDeleteItem stamps updated.** Soft-delete an item → read the .trash file → it has both `deletedAt` AND `updated` set (within 1s of each other; `updated >= deletedAt`).
  5. **Integration test — restoreItem stamps updated.** (Per D9 below.) Restore stamps a fresh `updated` later than the soft-delete's `updated`.

**Why not (b).** mtime resets on `cp`, fresh `git clone`, `tar -x`, IDE-touch-save-without-content-change, file-system metadata-time corruption, and the user's own backup-and-restore. v1.14.4's ETag wants persistence across all of these.

**Why not (c).** A sidecar adds an entire I/O surface (read + write + atomic-rename + delete-on-trash + restore-on-restore) for one field. Strictly worse.

**Consequences.**

  - **Positive.** Lays the foundation for v1.14.4 ETag (`W/${updated}`). Single helper, no drift.
  - **Positive.** Survives `rsync`/`git`/backup operations.
  - **Negative.** Adds a parser branch and a serializer branch. Both ~3 LOC each. Tolerated.
  - **Negative.** Existing items don't have it; older clients/tooling that grep `updated:` will miss them. Acceptable — v1.14.4 ETag uses missing-`updated:` as "first-edit-not-yet-stamped" and falls back to mtime there.

### Decision 2 — Notes editing semantics: PATCH overwrite (Option A)

**Status:** Accepted — `notes` joins the PATCH allowed-fields list with overwrite-on-present, leave-alone-on-absent semantics, identical to title/due/status/tags. Cap: 10KB (10,240 chars). Validator code: `NOTES_TOO_LONG`.

**Context.** Three candidates:

  - **(a) PATCH `{notes: "string"}` overwrites; absent leaves untouched; empty string `{notes: ""}` clears.** Same convention as title/due/status/tags. Smallest model that works.
  - **(b) Separate `POST /api/webapp/items/:id/notes` endpoint.** Dedicated route for notes editing only. Pro: more explicit. Con: more route surface; doesn't compose with simultaneous title edits.
  - **(c) Semantic field `{notes: {append: "..."}}`.** Append-only convention. Pro: matches the agent's append-side flow. Con: webapp users expect to EDIT existing notes (typo fixes, deletions), not just append.

**Decision.** (a). Justifications:

  1. **Symmetry with v1.14.2's PATCH model.** `title`, `due`, `status`, `tags` all use the same overwrite-on-present convention. `notes` is the same kind of field (a string); same convention applies. No protocol drift.
  2. **Composes with the edit form.** The v1.14.3 edit form sends one PATCH with up to six changed fields (title, due, status, tags, notes, progress). Splitting notes into a separate endpoint would force the client to issue two requests for "rename + reword notes" — and now the audit log has two rows for one logical user action.
  3. **Edit semantics match user mental model.** The user opens the form, sees existing notes in a `<textarea>`, edits, clicks Save. They expect the result to BE what's in the textarea. Append-only semantics (c) breaks this — clearing the textarea wouldn't clear notes.
  4. **The defense-in-depth posture (RA2) extends naturally.** `validation.ts` adds two more entries to `ALLOWED_PATCH_FIELDS`: `notes` and `progress`. `items.mutate.ts` adds two more `if (validated.notes !== undefined) storagePatch.notes = validated.notes;` lines. The explicit-field-copy invariant (RA2) holds verbatim.

**Validator rules (binding for Phase 2):**

| Field | Type | Validation |
|---|---|---|
| `notes` | string | After `.trim()`: length 0-`MAX_NOTES` (10240). NO embedded NUL (`\x00`). Empty string is VALID — clears notes. Code: `NOTES_TOO_LONG`. Embedded-NUL rejection reuses the title's existing pattern; new code: `NOTES_INVALID_CHARS` (or reuse `NOTES_TOO_LONG`'s sibling — Phase 2 picks; recommend `NOTES_INVALID_CHARS` for code-name consistency with `TAG_INVALID_CHARS`). |

**Why 10KB.** The brief asked for an explicit reasoning between 10KB vs 50KB. Picking 10KB:

  1. **Long-form reflective notes fit comfortably.** 10KB ≈ 1500-2000 words at average prose density. A user's "what I'm thinking about this goal" multi-paragraph note fits. A user's "every meeting note since 2024 dumped here" doesn't, and shouldn't — that's what dedicated note-taking apps are for.
  2. **Tighter cap = tighter abuse defense.** Smaller body limit = smaller request impact under abuse. The agent's `organize_log_progress` tool caps individual progress entries at 500 chars (verified at `tools/organize_log_progress.ts:19`); notes are bigger but not unbounded.
  3. **DB / storage growth predictable.** 200 items × 10KB notes × 2 (with body header overhead) = 4MB per user — fits comfortably in the existing single-user filesystem budget.
  4. **Surveys of similar product tiers.** Notion's free-plan block limit, Bear's free-tier note size, Trello's checklist text — all hover around 10-25KB. 10KB is at the conservative end; v1.16.0 markdown rendering may relax to 50KB if real users push the cap.
  5. **50KB is YAGNI for v1.14.3.** No real user has demanded > 10KB notes; if one does, the cap raises in a 1-LOC config change.

**Cap location.** Define `MAX_NOTES = 10240` in `validation.ts` next to `MAX_TITLE` / `MAX_TAG`. Privacy.ts does NOT need to mirror — privacy.ts is creation-time filter; notes editing is post-creation and goes through validation.ts. Documented.

**Storage layer change (binding for Phase 2):** None. `updateItem` at `storage.ts:612-662` already accepts `notes` in `UpdateItemPatch` (verified at `storage.ts:598-606`). The webapp's RA2 explicit-field-copy in `items.mutate.ts:200-204` adds:

```typescript
if (validated.notes !== undefined) storagePatch.notes = validated.notes;
if (validated.progress !== undefined) storagePatch.progress = validated.progress;  // per D3
```

**`UpdateItemPatch` storage type extension (binding for Phase 2):** Adds `progress?: string` (D3 below); `notes` is already present. The storage `updateItem` function gains an `if (patch.progress !== undefined) ...` branch — about 5 LOC.

### Decision 3 — Progress editing semantics: PATCH overwrite (Option A, identical to D2)

**Status:** Accepted — `progress` joins the PATCH allowed-fields list with overwrite-on-present, leave-alone-on-absent semantics, identical to D2. Cap: 20KB (20,480 chars). Validator code: `PROGRESS_TOO_LONG`. Race window with chat-agent's `appendProgressEntry` is documented as accepted v1.14.3 posture; v1.14.4 ETag closes it; v1.14.2's mtime stale-warning catches it best-effort meanwhile.

**Context.** Same three candidates as D2; same reasoning applies.

**Decision.** (a). Same justifications as D2; additionally:

  1. **Progress accumulates faster than notes.** Each `appendProgressEntry` adds `- YYYY-MM-DD: <500-char-entry>\n` ≈ 520 bytes per agent log. Over 100 entries that's ~50KB. Cap of 20KB still allows ~35 entries before the user must trim — reasonable.
  2. **Bigger cap than notes because of accumulation.** 20KB is 2× notes; matches the asymmetric-growth pattern.
  3. **Trim/compact is a v1.14.x maintenance feature**, not a v1.14.3 concern. If the user hits 20KB, the inline error tells them to manually trim. Documented.

**Why 20KB and not bigger.**

  1. **Body cap symmetry.** PATCH body cap is 32KB (D4). Notes 10 + progress 20 + headers + other fields = ~31KB max. Exactly fits. 50KB progress would need 64KB body cap, which Express's default-flow gets noisier on.
  2. **Real-user data shape.** Agent's `appendProgressEntry` adds ~520 bytes per call; 20KB ≈ 38 entries; that's ~1 month of daily progress-logging at typical use. Real-user data should rarely hit this; if it does, the user is using progress as a journal (which v1.16.0 markdown rendering will address).

**Race window analysis (SF-4 above; binding for v1.14.3):**

User opens detail view at `t=0` (mtime captured). Agent fires `appendProgressEntry` at `t=2` (mtime advances). User clicks Save with 20KB progress field at `t=5`. Webapp sends `X-Captured-Mtime: <t=0 mtime>`; server compares to current (`t=2`) mtime; mismatch → `staleWarning: true` in response + `webapp.stale_edit` audit row. **The mutation still proceeds — agent's appended entry is overwritten.** User sees the toast "Saved — but item changed elsewhere; refresh to see other changes."

This is an ACKNOWLEDGED race. Severity: MEDIUM (user-visible, recoverable via refresh-and-redo; agent's append is in audit log; user can see what got overwritten). Mitigation:

  - **v1.14.3:** mtime stale-warning toast (best-effort). User informed.
  - **v1.14.4:** Hard If-Match + 412 + ETag. Webapp's PATCH with stale ETag → 412 → "Save anyway / Reload" UX cliff. The race becomes detected-and-blocked, not detected-and-warned.
  - **No interim mitigation.** Adding a "merge progress entries" path to v1.14.3 is YAGNI — webapp users rarely edit progress while the agent is writing; the audit log preserves what was overwritten; v1.14.4 closes it properly.

Document in `docs/PRIVACY.md` (created v1.14.2) and `docs/KNOWN_ISSUES.md` (a v1.14.3 entry).

### Decision 4 — PATCH body size limit: 32KB

**Status:** Accepted — `express.json({ limit: '32kb' })` for PATCH; oversized bodies return 413 (Express default), NOT 400.

**Context.** Three candidates:

  - **(a) 32KB.** Notes 10K + progress 20K + 4 small fields + JSON overhead = ~31KB. Tightest fit.
  - **(b) 36KB.** ~10% headroom over (a). Tolerates JSON whitespace, Unicode multi-byte characters, slightly oversized base64 (no base64 in PATCH today, but defense-in-depth).
  - **(c) 64KB.** Generous. But: every byte over the minimum increases abuse surface.

**Decision.** (a) — **32KB**, with audit-relevant rationale:

  1. **Tight cap = tight abuse surface.** Each KB above the minimum is wasted bandwidth a malicious client can use. 32KB is exactly what we need for the maximum legitimate payload.
  2. **Multilingual edge case is acceptable.** Unicode multi-byte characters (CJK, emoji) inflate bytes-per-character — a 10KB notes value of CJK text is up to ~30KB of UTF-8 bytes. If `JSON.stringify({notes: "...", progress: "..."})` exceeds 32KB despite individual fields fitting in their character caps, the validator never runs (413 fires first). **This is acceptable.** Rationale: (1) the validator's character-count caps (10240/20480) are CHARACTER-counting, not BYTE-counting, which is what users care about; (2) heavy-CJK users can split saves into title-only-then-notes-only-then-progress-only (each PATCH ≈ 12KB), or just trim. Phase 2 documents this in the README.
  3. **413 vs 400 disambiguation.** Express's `express.json({ limit })` middleware returns a 413 with the body `PayloadTooLargeError`. Our error envelope at the wire is `{ok: false, code: 'PAYLOAD_TOO_LARGE', error: '...'}` — Phase 2 wraps express's error handler to emit the unified envelope. Documented.

**Why not (b) or (c).** Headroom is YAGNI; 10% buffer is what (b) gives, but the multilingual-edge-case mitigation (split saves or trim) is the same regardless of whether we cap at 32 or 36. Tighter is better.

**Implementation (binding for Phase 2):**

  - PATCH route: `express.json({ limit: '32kb' })` at `items.mutate.ts:149` (replacing `'1kb'`).
  - POST /complete route: `express.json({ limit: '256b' })` at `:306` — UNCHANGED. Complete only carries `{done: boolean}`; 256B is plenty.
  - DELETE route: no body parser; UNCHANGED.
  - Server-level error handler in `server.ts` wraps Express's default `PayloadTooLargeError` to emit `{ok: false, code: 'PAYLOAD_TOO_LARGE', error: 'Request body exceeds 32KB.'}` with status 413.

**Tests required (Phase 2):**

  1. **M-NEW1 (integration):** PATCH with body of exactly 32KB → 200. PATCH with body of 33KB → 413 + unified envelope. Verify the response is JSON, not Express's default HTML error page.
  2. **M-NEW2 (integration):** PATCH with `notes` field of 10KB-1 char → 200. PATCH with notes 10KB+1 char → 400 + `code: 'NOTES_TOO_LONG'`.
  3. **M-NEW3 (integration):** PATCH with `progress` field of 20KB-1 char → 200. PATCH with progress 20KB+1 char → 400 + `code: 'PROGRESS_TOO_LONG'`.

### Decision 5 — Hierarchy filter-intersection edge: orphan renders top-level (Option A)

**Status:** Accepted — when a filter hides a parent goal but its child task/event matches the filter, the child renders at the top level (no parent in the current view).

**Context.** Three candidates:

  - **(a) Child renders top-level if its parent is hidden by filter.** Simplest. UX trade: user sees the task without context. Mitigation: the child's `parentId` field is in the data; clicking the task → detail view shows `parentId: <goal-id>`; user can click through to the parent goal.
  - **(b) Pull in the parent goal as a "ghosted" header (visually distinct, can't expand).** Pro: preserves hierarchy context. Con: UX complexity — user sees a goal that doesn't match their filter, can't tap to expand it (because then they'd see hidden children, or worse, see only some children matching the filter), questions arise about "why is this here?".
  - **(c) Server-side: fetch the parent if any visible item references it.** Adds a join-like step server-side. Server stops being stateless.

**Decision.** (a). Justifications:

  1. **Filter is a user choice; respect it.** When the user filters by "Active", they're saying "show me active items." A done goal showing up because it has an active child violates the contract.
  2. **The user retains discoverability.** The detail view's existing meta block shows `parentId` (verified at `app.js:407-438` — though not currently — Phase 2 adds it as a mini-link). Tapping the parent id navigates to the goal detail.
  3. **Avoids the "ghost rendering" UX surprise.** (b)'s ghosted-but-not-expandable goal is a UI primitive we don't have elsewhere; introducing it is real complexity for marginal benefit.
  4. **Avoids server-side state.** (c) is a clean architectural rule break — D6's "client groups, server stays stateless" principle.

**Implementation (binding for Phase 2):** In the client-side `groupByParent(items)` helper (in `hierarchy.js` per D13), if an item's `parentId` is non-null AND `parentId` is not in the current `items` set, push the item into the top-level array. No special "ghosted parent" rendering.

**Why not (b) or (c).** Above. Both add complexity without proportional UX win.

**Phase 2 UX polish (NOT bound):** the orphan-rendered child MAY have a small visual cue ("↳ part of goal <name>" subtle subline) IF Phase 2 has time. Not bound; not blocked on.

```typescript
// src/organize/storage.ts — new function
/**
 * Restore a soft-deleted item: rename from .trash/<id>.md back to <id>.md.
 *
 * Behavior:
 *   - Strip `deletedAt` from front-matter.
 *   - Stamp `updated: <now>` per D1 (this is a write).
 *   - Re-serialize and atomically write to the live path BEFORE renaming back —
 *     mirrors softDeleteItem's two-stage write pattern. This way the front-matter
 *     update is atomic at the filesystem boundary.
 *
 * Throws:
 *   - ITEM_NOT_FOUND_IN_TRASH if .trash/<id>.md doesn't exist.
 *   - ITEM_ALREADY_LIVE if <id>.md already exists in the live dir (defensive;
 *     should not happen if soft-delete worked correctly, but guards against
 *     filesystem-level corruption).
 *   - FILE_WRITE_FAILED for atomic-write failures (mirrors updateItem error).
 *
 * @returns the restored OrganizeItem (with the new updated:).
 */
export async function restoreItem(
  userId: number,
  dataDir: string,
  itemId: string,
): Promise<OrganizeItem> {
  await ensureUserDir(userId, dataDir);
  await ensureTrashDir(userId, dataDir);
  const liveDir = organizeUserDir(userId, dataDir);
  const trashPath = path.join(liveDir, '.trash', `${itemId}.md`);
  const livePath = path.join(liveDir, `${itemId}.md`);

  if (!existsSync(trashPath)) {
    throw Object.assign(new Error(`Item not in trash: ${itemId}`), { code: 'ITEM_NOT_FOUND_IN_TRASH' });
  }
  if (existsSync(livePath)) {
    throw Object.assign(
      new Error(`Item already live: ${itemId}`),
      { code: 'ITEM_ALREADY_LIVE' },
    );
  }

  // Read trashed file, strip deletedAt, stamp updated
  const raw = await readFile(trashPath, 'utf8');
  const outcome = parseItemFile(raw, itemId);
  if (!outcome.ok) {
    throw Object.assign(
      new Error(`Trashed item is malformed: ${itemId}`),
      { code: 'ITEM_MALFORMED' },
    );
  }

  const { fm: parsedFm, notesBody, progressBody } = outcome.result;
  const fm = stampUpdated({ ...parsedFm, deletedAt: null });
  fm.id = itemId; // R7 normalize

  const content = serializeItem(fm, notesBody, progressBody);
  // Two-stage: write to live path, then unlink trash path. Order matters:
  // if write fails, trash file remains. If unlink fails after write, the user
  // has a duplicate (live + trash) — log at warn; the next restore call will
  // see the live file and refuse with ITEM_ALREADY_LIVE.
  await writeAtomically(livePath, content);
  try {
    await unlink(trashPath);
  } catch (err) {
    log.warn(
      { userId, itemId, err: err instanceof Error ? err.message : String(err) },
      'restoreItem: failed to unlink trash file after live write succeeded',
    );
  }

  log.info({ userId, itemId }, 'organize item restored');
  return { frontMatter: fm, notesBody, progressBody, filePath: livePath };
}
```

**Should restore stamp `updated:`?** YES per D1. The brief asked the question. Justification: it's a write (the file content changes — `deletedAt` is stripped, `updated:` is added). v1.14.4 ETag depends on every write advancing `updated:`. Without it, a restore-then-edit race could produce identical ETags for different content.

### Decision 11 — Audit shape for notes/progress edits

**Status:** Accepted — same `webapp.item_mutate` row; `changedFields: ['notes']` or `['progress']` (or `['notes', 'progress']` if both edited together); NO field VALUES in the audit detail.

**Context.** ADR 010 R12 established the privacy posture: `changedFields: string[]` records WHICH fields changed (closed set), NOT their values. Notes content can be sensitive (personal reflections, private context); the `changedFields` list-of-names gives forensic readers what they need without leaking content.

**Decision.** Confirmed — same shape. The closed-set extension is:

```typescript
// extends WebappItemMutateDetail's changedFields union
changedFields: ('title' | 'due' | 'status' | 'tags' | 'notes' | 'progress')[]
```

The `WebappItemMutateDetail` interface at `items.shared.ts:179-184` carries the new strings. No structural change. Phase 2 verifies: the audit detail's JSON.stringify NEVER contains the user's notes/progress text — Anti-Slop grep-check at the route handler level (mirrors v1.14.2 W5).

**Tests required:**

  - **W5-equivalent (integration):** PATCH with `notes: "secret personal reflection"` → audit row's detail JSON has `changedFields: ['notes']` and `JSON.stringify(detail).includes('secret')` is FALSE.
  - **W5-equivalent (integration):** PATCH with both `notes` and `progress` → `changedFields: ['notes', 'progress']` (sorted; or insertion order — Phase 2 picks; recommend sorted for stable test).

**Why no separate audit category for notes/progress.** Same forensic-shape argument as ADR 010 D4 — `webapp.item_mutate` with `action: 'update'` and `changedFields` covers all edit forensics. Adding `webapp.notes_edit` / `webapp.progress_edit` categories smears the signal across multiple categories for one logical action.

### Decision 13 — Hierarchy grouping logic file location: extract to `hierarchy.js` (Option B)

**Status:** Accepted — pure functions in their own file at `public/webapp/organize/hierarchy.js` (NEW). Vitest can import + test without DOM.

**Context.** Two candidates:

  - **(a) Live in `app.js`.** Client-only, alongside the rest. Pro: one file. Con: harder to unit-test; mixes pure data-shape functions with DOM-rendering side effects.
  - **(b) Extract to `hierarchy.js`.** Pure functions: `groupByParent(items)`, `loadCollapseState`, `saveCollapseState`, `isCollapsed`, `toggleCollapsed`. Pro: vitest imports directly, no DOM mocks. Con: one more file.

**Decision.** (b). Justifications:

  1. **Testability.** The grouping logic is pure: input is a list of items; output is `{goals: [{...goal, children: [...]}], orphans: [...]}`. Vitest can test 10+ permutations (no parents, all parents present, orphan rendering per D5, multi-level — N/A in v1.14.3 since hierarchy is flat 1-level — etc.) without spinning up jsdom.
  2. **app.js is already 962 LOC.** Adding 100 LOC of grouping + collapse logic pushes it past 1000. Anti-Slop's lenient client-side threshold is ~1000 LOC for this codebase (no formal threshold; informal); breaking out hierarchy.js keeps app.js focused on orchestration + DOM rendering.
  3. **Future-proof for v1.16.0+ refactors.** When markdown rendering / drag-drop / multi-level hierarchy ships, the hierarchy file is already the dedicated home for tree logic.
  4. **Tests are the bigger argument.** `tests/unit/webapp.organize.hierarchy.test.ts` (NEW, ~80 LOC) gets full coverage of:
     - Items with no parents → all top-level.
     - Items with parents present → nested correctly.
     - Items with parents NOT in current set → orphan top-level (D5).
     - Items with multiple children per parent → all children attached.
     - Empty input → empty output.
     - Collapse state load/save round-trip.
     - Toggle behavior.

**Module shape (binding for Phase 2):**

```javascript
// public/webapp/organize/hierarchy.js (NEW, ~120 LOC)

/**
 * Group items by parentId. Goals with children become "groups"; standalone
 * items (no parentId, no children) and orphans (parentId present but parent
 * not in this set, per D5) are top-level.
 *
 * @param {Array<{id, parentId, type, ...}>} items  Flat list from the API.
 * @returns {{groups: Array<{goal, children}>, topLevel: Array}}
 */
export function groupByParent(items) {
  const byId = new Map(items.map(it => [it.id, it]));
  const childrenByParentId = new Map();
  const topLevel = [];

  for (const item of items) {
    if (item.parentId && byId.has(item.parentId)) {
      const arr = childrenByParentId.get(item.parentId) ?? [];
      arr.push(item);
      childrenByParentId.set(item.parentId, arr);
    } else {
      topLevel.push(item);
    }
  }

  // Goals with children become groups; goals without children are top-level
  const groups = [];
  const remainingTopLevel = [];
  for (const item of topLevel) {
    const children = childrenByParentId.get(item.id);
    if (item.type === 'goal' && children && children.length > 0) {
      groups.push({ goal: item, children });
    } else {
      remainingTopLevel.push(item);
    }
  }

  return { groups, topLevel: remainingTopLevel };
}

const COLLAPSE_KEY = 'organize-collapse-state-v1';

export function loadCollapseState() { /* per D7 */ }
export function saveCollapseState(state) { /* per D7 */ }
export function isCollapsed(state, goalId) { /* per D7 */ }
export function toggleCollapsed(state, goalId) { /* per D7 */ }

/**
 * Drop collapse-state entries for goalIds that no longer exist in the items
 * list (storage hygiene per D7).
 */
export function pruneCollapseState(state, items) {
  const liveIds = new Set(items.map(it => it.id));
  const next = {};
  for (const [goalId, value] of Object.entries(state)) {
    if (liveIds.has(goalId)) next[goalId] = value;
  }
  return next;
}
```

**ES Modules note.** Existing `app.js` doesn't use ES modules — it's a plain `<script>`. Phase 2 adds `<script type="module" src="./app.js">` and updates the import path from `app.js`. **Or** — simpler — Phase 2 inlines `hierarchy.js`'s exports as global functions exposed via `window.OrganizeHierarchy = { groupByParent, ... }`, and the unit test loads the file via Node's `vm` module or by `eval` of the file contents. Phase 2 picks; recommend the ES Modules approach IF browser compat checks pass for Telegram WebApp embeds. Verify with a quick smoke test in Phase 2; fall back to global-window if needed. Documented as Phase 2 implementation choice.

**Why not (a).** Test friction. The grouping logic is pure; pure logic deserves its own file with its own test suite. Mixing it in `app.js` forces tests to spin up jsdom or to monkey-patch `window`.

## Risks

  1. **Notes/progress overwrite race with `appendProgressEntry`** (D3 / SF-4). Severity: MEDIUM. v1.14.3 mitigation: mtime stale-warning toast (best-effort detection); user is INFORMED. v1.14.4 closes properly. Documented in `KNOWN_ISSUES.md`.
  2. **32KB body cap edge case for heavy CJK.** Severity: LOW. User can split saves into title-only-then-notes-only-then-progress-only. Documented in README v1.14.3 subsection.
  3. **Hierarchy collapse state grows unbounded.** Severity: LOW. Phase 2's `pruneCollapseState` cleanup pass on load handles it.
  4. **`/organize restore` race with delete.** User runs `/organize delete <id>` then immediately `/organize restore <id>` while the soft-delete is still moving the file (~5ms window). Either succeeds (delete done first → restore works) or restore fails with `ITEM_NOT_FOUND_IN_TRASH` (delete still in progress) → user retries → succeeds. No data loss. Documented; no mitigation needed beyond the user-visible 404 + retry message.
  5. **`updated:` field absent on legacy items.** Severity: LOW. Parser tolerates; first edit stamps. v1.14.4 ETag treats missing-`updated:` as "use mtime fallback" (or as 1970-epoch — Phase 2 v1.14.4 designs).
  6. **Hierarchy rendering performance under high item counts.** With 200 items (the active cap), `groupByParent` is O(N) for the byId Map + O(N) for the iteration; total O(N) — fine. With 1000+ items (active + done at the upper end), still O(N); pure JS Map is fast. No virtualization needed for v1.14.3.
  7. **Cross-surface audit visibility.** When the user edits notes in webapp AND the agent appends progress 100ms later, the audit log has TWO rows: `webapp.item_mutate {action: 'update', changedFields: ['notes']}` and the agent's audit row (existing). Both rows show real intent; forensic readers see the sequence. Not a risk; documented.
  8. **Restore command bypasses privacy-filter recheck.** The trashed item's content was filtered at create-time (ADR 003); restore replays the same content. If the privacy filter rules change between create and restore, restored content might NOT match current filter rules. Severity: LOW. Mitigation: documented in CLAUDE.md as accepted behavior; v1.14.x maintenance can add an opt-in re-filter pass if needed.
