# ADR 007 ŌĆö v1.12.0 `/debate` Progress Panel with Expand/Collapse

**Status:** Proposed (Phase 1, v1.12.0)
**Date:** 2026-04-24
**Deciders:** Architect Agent (iteration)
**Extends:** ADR 001 (module boundaries), ADR 006 (`rec:*` callback-namespace + R9 stale-callback pattern). Does NOT supersede any prior ADR ŌĆö this iteration is additive: `/debate`'s multi-message UX becomes a single editable panel, a new reusable `ProgressPanel` primitive lands under `src/gateway/`, and the gateway's callback_query router gains a `debate.*` branch. No schema changes, no new deps, no changes to `runDebate`'s LLM contract.

This ADR records the architectural decisions for the v1.12.0 UX iteration. Three items ship together:

  1. **Generic `ProgressPanel` primitive** ŌĆö a reusable, state-type-parameterized progress panel that handles initial send, debounced edits, collapse/expand state, terminal finalization, and callback dispatch. Consumers supply render callbacks and state; the primitive owns the messaging-adapter mechanics.
  2. **Apply the panel to `/debate`** ŌĆö replace the current N-message-per-debate flow in `src/debate/index.ts` with a single panel that edits in place as the debate progresses. Transcript accumulates in a typed `DebateState` object; the panel renders collapsed summary or expanded transcript based on user taps.
  3. **Gateway callback routing** ŌĆö add a `debate.*` branch to the existing `callback_query:data` router, dispatching to a process-scoped `panelRegistry.handleCallback(data, ctx)`. Ownership checks mirror the `plan.ts` precedent (starter + group admins); per-panel state lives in an LRU + TTL-bounded in-memory `Map`.

Scope is design only ŌĆö no code is in scope here. Developer agents in Phase 2 implement the decisions below without deviating; deviations require an ADR 007 addendum.

  - No Telegram Web App version (deferred separately).
  - No new npm dependencies; `grammY`, `node-cron`, `better-sqlite3`, `zod`, `@anthropic-ai/sdk` already vendored.
  - No change to `runDebate`'s LLM contract (debater system prompt, judge system prompt, final-verdict system prompt, roster picker, round structure).
  - No change to `/debate on|off|[rounds] [exchangesPerRound]` command surface.

## Item 1 ŌĆö Generic `ProgressPanel` primitive

### 1. Callback-data namespace format

**Status:** Accepted.

**Context.** Three precedents live in the codebase today:

  - **`plan.*` (dot separator)** ŌĆö `src/plan/panel.ts:177ŌĆō188` emits callback_data of shape `plan.cancel:<planId>`, `plan.resend:<planId>`, `plan.rerun-sonnet:<planId>`, `plan.approve:<planId>`, `plan.deny:<planId>`. The gateway router branches on `startsWith('plan.')`.
  - **`rec:*` (colon separator)** ŌĆö v1.11.0 `/organize reconcile` (ADR 006 decision 4) uses `rec:<action>:<itemId>`; router branches on `startsWith('rec:')`.
  - **No `sch:*`** ŌĆö ADR 005 decision 16 was misremembered; `/scheduled` does NOT use inline keyboards. Verified against `src/commands/scheduled.ts` (no `InlineKeyboard` emission) and `src/gateway/index.ts` (no `sch:` branch). This was already noted in ADR 006's deviations ┬¦3.

So the precedent is a 1-1 split (plan = dot, rec = colon). Picking for `debate.*` and future ProgressPanel consumers:

**Decision.** Use the `plan.*` shape for ProgressPanel: `<namespace>.<action>:<panelId>`. For `/debate` specifically: `debate.expand:<panelId>`, `debate.collapse:<panelId>`, `debate.cancel:<panelId>`. Length: `debate.collapse:pn_abcdef12` Ōēł 23 chars ŌĆö well under Telegram's 64-byte callback_data budget.

Rationale for matching plan over rec:

  1. `plan.*` is the existing progress-panel precedent in the codebase. ProgressPanel is a GENERALIZATION of the plan-panel pattern, so matching its surface shape preserves the mental model.
  2. The dot-separator on the action (`expand` in `debate.expand:<id>`) reads as "method call" ŌĆö it's the verb on the namespace object. The colon-separator on the id reads as "key/value" ŌĆö it's the target. This split is semantically cleaner than `rec:fix:id` where both separators carry the same meaning.
  3. ProgressPanel's registry needs to parse namespace + action + id; the dot-between-namespace-and-action makes namespace detection a simple `data.startsWith(ns + '.')` check in the gateway router, matching how `plan.*` routes today.

**Unification across existing surfaces.** Not in scope. The `rec:*` callbacks in `/organize reconcile` are active in production and carry stable semantics; changing them would break any in-flight messages a user has from the last 48h (Telegram's edit window). The mismatch is a one-time v1.11.0 inconsistency that's been absorbed. Any new inline-keyboard surface SHOULD use the `plan.*` / ProgressPanel shape. If `/organize reconcile` ever gets a v2.0 UX rework, it can migrate then. Documented in ┬¦Open Questions for future ADRs to honor.

**Consequences.**

  - **Positive.** Consistent with the existing ProgressPanel-like surface (`/plan`). Grep-friendly (`debate.*`).
  - **Positive.** Three-part parse (`namespace.action:id`) is unambiguous. No colon-in-action ambiguity (`rec:fix:2026-04-20-a1b2` has 3 colons but rec's router handles by split-on-colon).
  - **Negative.** `rec:*` and `debate.*` diverge. Accepted as legacy-but-stable.

### 2. Module location ŌĆö `src/gateway/progressPanel.ts`

**Status:** Accepted (gateway layer).

**Context.** Two candidate homes:

  - **`src/gateway/progressPanel.ts`** ŌĆö lives at the gateway layer, alongside the existing `telegramAdapter` factory and the callback_query router that dispatches to it. Imports `MessagingAdapter` + `InlineKeyboard` types (both in `src/messaging/`), the pino logger, and configs. Exports `createPanelRegistry(deps): PanelRegistry`.
  - **`src/messaging/progressPanel.ts`** ŌĆö sits alongside the `MessagingAdapter` interface. Purer (no gateway-specific imports). But the registry state (Map of panels) is process-scoped to the gateway boot, ownership checks read `ctx.from.id` from grammY `Context`, and the callback dispatcher is wired by the gateway's `callback_query:data` handler. None of that belongs in `src/messaging/` which is supposed to be the platform-abstracted layer.

**Decision.** `src/gateway/progressPanel.ts`. Justification:

  1. The primitive depends on grammY `Context` (for callback dispatch) ŌĆö grammY types are gateway-layer, not messaging-layer. `src/messaging/` stays platform-neutral.
  2. The registry is process-singleton owned by `initGateway()`. Registry creation and teardown map 1:1 to gateway lifecycle.
  3. Ownership checks read `ctx.from.id` and `config.groups.adminUserIds` ŌĆö gateway concerns.
  4. The callback router's `debate.*` branch (decision 13) is in `src/gateway/index.ts`; placing ProgressPanel next to it shortens the mental-model distance.

Module's allowed imports: `node:crypto` (for panelId generation), `pino` child via `src/logger/index.js`, `src/messaging/adapter.js` (types only), `src/config/index.js` (AppConfig for admin allowlist). Forbidden: `src/agent/`, `src/tools/`, `src/commands/`, `src/memory/`, `src/organize/`, `src/debate/` (ProgressPanel is domain-neutral; debate-specific rendering lives in `src/debate/`).

**Consequences.**

  - **Positive.** Clear layering ŌĆö `src/messaging/` is the interface; `src/gateway/` composes the interface with platform specifics; `src/debate/` consumes via the primitive.
  - **Positive.** ProgressPanel is reusable: a future `src/gateway/toolCallPreviewPanel.ts` (DM tool-call preview) can call `panelRegistry.create(...)` with its own render callbacks.
  - **Negative.** `src/gateway/` grows by one file (~250 LOC). Accepted ŌĆö still well under the module's per-file LOC norms.

### 3. `ProgressPanel<S>` public surface

**Status:** Accepted.

**Context.** The primitive is generic over state type `S`. Each consumer supplies render callbacks that read `S` and produce HTML-safe text + buttons. The registry owns state storage, debouncing, and callback dispatch.

**Decision.** Exactly these exported types. Names below are binding for Phase 2 developer agents:

```typescript
// src/gateway/progressPanel.ts

import type { Context } from 'grammy';
import type { MessagingAdapter, InlineKeyboard } from '../messaging/adapter.js';
import type { AppConfig } from '../config/index.js';

export type PanelMode = 'collapsed' | 'expanded';

export interface ProgressPanelDeps<S> {
  adapter: MessagingAdapter;
  chatId: number;
  /** userId of the originator; authoritative for owner checks. */
  ownerUserId: number;
  /** Lowercase, dot-free. Becomes the `<namespace>.<action>:<panelId>` prefix. */
  callbackNamespace: string;
  /** pino child `component` label. Mirrors the LOGGING_STANDARD.md one-label-per-module rule. */
  componentTag: string;
  /** Render the collapsed single-paragraph summary. Must return HTML-safe text Ōēż 4000 chars. */
  renderSummary(state: S): string;
  /** Render the expanded detail (full transcript, etc.). Consumer owns any truncation. Must return HTML-safe text Ōēż 4000 chars. */
  renderDetail(state: S): string;
  /** Buttons for the current state+mode. Terminal-state buttons differ from running. */
  renderButtons(state: S, mode: PanelMode, terminal: boolean): InlineKeyboard;
  /**
   * Optional custom callback actions BEYOND the built-in expand/collapse/cancel.
   * Key becomes the `action` in `${namespace}.${action}:${panelId}`.
   * Handler runs AFTER owner check; returns the toast string shown to the tapper.
   * Return `{closePanel: true}` to mark the panel terminal after this action (registry will call finalize on the current state).
   */
  extraActions?: Record<
    string,
    (ctx: Context, state: S, api: ProgressPanelApi<S>) => Promise<{ toast?: string; closePanel?: boolean }>
  >;
  /**
   * Called BY the registry when the panel reaches terminal state (finalize / cancel / post-extraAction closePanel).
   * Consumer uses this to clean its own resources (abort controller, etc.). Registry calls it exactly once.
   */
  onTerminal?(state: S): void;
}

export interface ProgressPanelApi<S> {
  /** Short random id. Appears in callback_data; the user never sees it in UI. */
  readonly panelId: string;
  /** Resolved after create() returns. */
  readonly messageId: number;
  /** Current mode ŌĆö mutated by built-in expand/collapse callbacks, read by consumers that want to log or test. */
  readonly mode: PanelMode;
  /** Debounced re-render based on the current mode. Idempotent ŌĆö consumer may call repeatedly. */
  updateState(state: S): void;
  /**
   * Apply a final state (marks terminal), flush any pending edit synchronously,
   * swap buttons to terminal-state set, stop accepting further updateState calls.
   */
  finalize(finalState: S): Promise<void>;
  /**
   * Drop the panel from the registry's active map. Does NOT edit the message further.
   * State is moved to a "recently-closed" LRU for the stale-callback grace window (decision 7).
   */
  close(): void;
}

export interface PanelRegistry {
  /**
   * Create a new panel. Sends the initial message, resolves panelId + messageId, stores in active map.
   * Returns AFTER the initial send completes (or rejects if adapter.sendMessage throws).
   */
  create<S>(
    deps: ProgressPanelDeps<S>,
    initialState: S,
  ): Promise<ProgressPanelApi<S>>;
  /**
   * Dispatch a `<namespace>.<action>:<panelId>` callback to the appropriate panel.
   * Returns the toast string (gateway's callback_query handler passes to answerCallbackQuery).
   * Handles: unknown panelId (stale/expired), unknown action, owner-check failure, extraActions dispatch.
   */
  handleCallback(data: string, ctx: Context): Promise<{ toast: string }>;
  /**
   * For gateway shutdown: flush all active panels, clear timers. Tests may also use
   * `_stopForTests()` ŌĆö see decision 12.
   */
  stop(): void;
}

export interface PanelRegistryDeps {
  config: AppConfig;        // for groups.adminUserIds ownership rule (decision 8)
  /** LRU cap for active + recently-closed panels combined. Default from config.debate.panelStateCacheMax. */
  cacheMax?: number;
  /** TTL in ms for panel state (ceiling). Default from config.debate.panelStateTtlHours * 3600_000. */
  ttlMs?: number;
}

export function createPanelRegistry(deps: PanelRegistryDeps): PanelRegistry;
```

**Design notes:**

  - State is `S` ŌĆö the registry does not introspect it. It stores a reference, passes it back to `renderSummary`/`renderDetail`/`renderButtons` on every tick.
  - `renderSummary` and `renderDetail` return strings; the primitive does NOT auto-truncate. If a consumer's rendered text exceeds 4000 chars, the primitive slices to 4000 and appends `ŌĆ”` (single char). This is a safety net, NOT the primary truncation mechanism ŌĆö consumers are responsible for meaningful truncation (decision 11 for `/debate`).
  - `renderButtons(state, mode, terminal)` ŌĆö consumer decides every button's label + callback_data based on current mode + terminal state. The primitive ONLY injects the `expand`/`collapse` built-in buttons via the consumer's callback (consumer returns them in its own list). This keeps the button layout fully under consumer control (e.g., `/debate` wants `[Ōīä Expand] [ŌØī Cancel]` in running mode but JUST `[Ōīā Collapse]` in terminal-expanded mode ŌĆö consumer decides).
  - `extraActions` map keys CANNOT include `expand`, `collapse`, or `cancel` ŌĆö those are reserved for the built-in handlers. Registry asserts this at `create()` time; throws if collision.
  - `componentTag` is passed to pino `child({component: tag})` so every log line from a panel's lifecycle is traceable to the originating consumer.

**Consequences.**

  - **Positive.** State-type parameterization means `/debate` gets a typed `DebateState`, future `/plan` refactor gets a `Plan`, future tool-call-preview gets its own type. No stringly-typed union.
  - **Positive.** Generic primitive, domain-specific rendering. Fits Anti-Slop ┬¦5 (Separation of Concerns).
  - **Negative.** Three render callbacks per consumer is a small ceremony. Acceptable ŌĆö the alternative (consumer subclasses a class) is heavier and grammY already leans functional.

### 4. Debounced edit semantics + rate-limit budget

**Status:** Accepted.

**Context.** `src/plan/panel.ts:209ŌĆō256` uses a `MIN_EDIT_INTERVAL_MS = 1500` debounce with coalescing: `update()` schedules at most one pending edit; `flush()` drains it. Pattern is battle-tested. Telegram's group-chat rate limit is 30 edits/min = 2000ms average ŌĆö 1500ms debounce respects it with headroom.

For `/debate`, the edit-trigger frequency is bounded by debater-turn latency (Ollama Cloud turns take 2-10s normally, up to 60s on cold starts). So the primitive rarely rate-limits /debate naturally. But a misbehaving consumer calling `updateState` in a tight loop MUST be bounded by the primitive.

**Decision.** ProgressPanel uses the `plan/panel.ts` debouncer verbatim:

  - `MIN_EDIT_INTERVAL_MS = 1500` (constant, not config-tunable ŌĆö same justification as plan/panel's).
  - `updateState()` coalesces: if a timer is pending, no-op; else schedule one for `Math.max(0, MIN_EDIT_INTERVAL_MS - (Date.now() - lastEditAt))`.
  - `finalize(state)` clears pending timer, awaits in-flight edit, pushes one final edit with terminal-state buttons. No further updates accepted.
  - Edit-text + button layout hash for dedupe: render the text + `JSON.stringify(buttons)` and skip the edit if the result equals `lastSentText`. Prevents "message is not modified" 400s from Telegram.

`finalize` is `async` and awaited by the `/debate` integration before logging the final audit row. This is intentional ŌĆö the consumer needs to know the final edit landed before reporting completion. `plan/panel.ts` has this pattern already (`flush()` returns a Promise<void>).

**Config knob.** None. Debounce interval is not a per-deployment knob; it's a Telegram-API fact. If Telegram's rate limit changes, update the constant with a comment.

**Consequences.**

  - **Positive.** Pattern match with plan/panel.ts ŌĆö Phase-2 dev agent can literally port the `createPanelUpdater` shape with minor changes (the consumer render functions replace the hardcoded `renderPanel(plan)` + `panelButtons(plan)` calls).
  - **Positive.** Dedupe by text+buttons hash is already proven in plan/panel.ts.
  - **Negative.** If a future consumer genuinely needs a faster edit rate (e.g. a tight-loop counter UI), the primitive would be the bottleneck. Fine for the immediate scope; a later iteration can add a `minEditIntervalMs` deps field if a concrete need appears.

### 5. Expand/collapse wiring ŌĆö built-in actions

**Status:** Accepted.

**Context.** Every consumer uses the same expand/collapse logic: user taps `expand` ŌåÆ registry flips `mode` to `expanded` ŌåÆ re-render via `updateState(currentState)` ŌåÆ next debounce tick edits message with the detail view. The primitive owns this; consumers don't re-implement.

**Decision.** Three built-in actions are hardcoded in the registry: `expand`, `collapse`, `cancel`. The consumer chooses whether to emit the buttons (via `renderButtons`). When the button IS emitted and tapped:

  - `expand` ŌĆö owner check ŌåÆ set `mode = 'expanded'` ŌåÆ `updateState(currentState)` to trigger re-render ŌåÆ toast "Expanded.".
  - `collapse` ŌĆö owner check ŌåÆ set `mode = 'collapsed'` ŌåÆ `updateState(currentState)` ŌåÆ toast "Collapsed.".
  - `cancel` ŌĆö owner check (STRICTER per decision 8: starter OR `groups.adminUserIds`) ŌåÆ invoke `extraActions.cancel` if present, else emit "Cancel handler not wired." toast. Consumer implements the actual cancellation semantics (abort controller, final state push) in their `extraActions.cancel` handler. The built-in `cancel` handler ONLY gates the owner check; execution goes through the extraAction.

Implementation detail: `cancel` is NOT a fully built-in action ŌĆö it's a reserved key the primitive dispatches to `extraActions.cancel` after enforcing the stricter owner check. If a consumer doesn't register `cancel` in `extraActions`, taps on a cancel button emit a "Cancel handler not wired." toast (loud in dev, benign in prod). This split lets the primitive own the AUTHORIZATION logic while the consumer owns the EXECUTION logic.

**Consequences.**

  - **Positive.** `/debate` gets expand/collapse for free. Only `cancel` needs a handler in debate/index.ts.
  - **Positive.** Future consumers can skip rendering the expand/collapse buttons if their state is always summary-only ŌĆö they just never emit the button from `renderButtons`.
  - **Negative.** The `cancel` split (auth here, exec in consumer) is slightly non-obvious. Documented in ProgressPanelDeps's JSDoc.

### 6. Initial placeholder message content

**Status:** Accepted.

**Context.** The registry's `create(deps, initialState)` must call `adapter.sendMessage(chatId, initialText, ...)` before it knows the panel's eventual rendered shape. What text goes in the initial send?

Two candidates:

  - **(a)** Call `renderSummary(initialState)` + `renderButtons(initialState, 'collapsed', false)` ŌĆö the panel's real first-frame rendering.
  - **(b)** A hardcoded "StartingŌĆ”" placeholder string.

**Decision.** (a). The consumer's `renderSummary` on the initial state IS the placeholder. For `/debate` this means `ŌÜö’ĖÅ Debate: <topic> (starting...)` ŌĆö the first frame of the collapsed summary (decision 10). No separate placeholder state needed; `DebateState.status` starts as `'starting'` and `renderSummary` switches on it.

Rationale:

  - One rendering path. No hardcoded string in the primitive that could drift from the consumer's intent.
  - The first-edit (triggered when the first debater turn lands) is a REAL change, not a placeholder-to-real transition. Reduces edit count by one per debate.
  - Consumer gets to choose the placeholder tone (`/debate` wants emoji header; a future tool-call-preview might want a neutral "Preparing tool callŌĆ”"). Each consumer's identity is in its renderer.

**Consequences.**

  - **Positive.** Clean. One renderer, no hardcoded placeholders.
  - **Negative.** Consumers MUST handle their "starting" state in `renderSummary` ŌĆö Anti-Slop ┬¦6 (Error Handling) requires `renderSummary` never throws even on a just-initialized state. Documented as a ProgressPanelDeps contract.

### 7. Panel-state storage ŌĆö in-memory LRU + TTL

**Status:** Accepted.

**Context.** Panel state (state object + messageId + deps + mode) has to live somewhere the callback handler can reach it. Two candidates:

  - **In-memory `Map<panelId, PanelEntry>`** ŌĆö registry-scoped, process-lifetime. Simple. Data loss on pm2 restart.
  - **SQLite `progress_panels` table** ŌĆö migration + insert on create + update on state change + delete on finalize. Survives pm2 restart. Adds a write on every `updateState` (amortized by the debouncer, but still a write per 1.5s), plus a query on every callback.

**Decision.** In-memory. Jarvis is single-process today (pm2, one node). Panel state is EPHEMERAL BY DESIGN: a debate in-flight at the moment of a pm2 restart loses its transcript anyway ŌĆö the abort signal fires, `runDebate` throws, the panel was going to land on an error state. Restart-survivability is not a feature that would move the user's UX.

The SQLite alternative costs:

  1. One more migration (+1 migration number, +1 file).
  2. Serialize `DebateState` on every debounced edit (~1.5s cadence). `DebateState.transcript: Turn[]` with 2 rounds ├Ś 2 exchanges ├Ś 3 debaters = 12 turns ├Ś ~400 chars = 4.8 KB per serialize. 20 edits per debate ├Ś 4.8 KB = 96 KB of writes per debate. Small per-debate but O(N) across the day.
  3. A cleanup policy (stale rows from abandoned panels ŌĆö another cron).
  4. Startup-load logic that handles "panel exists in DB but the associated chat's message is gone" ŌĆö messy.

For the benefit (surviving pm2 restart mid-debate), the cost is not justified at today's scale. Documented in ┬¦13 Risk Register as a revisit trigger if multi-process deployment ever happens.

**Storage shape (registry internal):**

```typescript
interface PanelEntry<S = unknown> {
  panelId: string;
  messageId: number;
  chatId: number;
  ownerUserId: number;
  deps: ProgressPanelDeps<S>;
  state: S;
  mode: PanelMode;
  terminal: boolean;
  createdAt: number;      // for TTL
  lastActivityAt: number; // for LRU ordering
}
```

Two maps:

  - `active: Map<panelId, PanelEntry>` ŌĆö currently live panels. Capped at `config.debate.panelStateCacheMax` (default 50) via LRU eviction.
  - `recentlyClosed: Map<panelId, PanelEntry>` ŌĆö panels in the stale-callback grace window (decision 8 below). Same cap. On `finalize()`/`close()`, entry moves from active ŌåÆ recentlyClosed.

**TTL:** 24h hard ceiling (default `config.debate.panelStateTtlHours: 24`). A swept-clean pass runs lazily on every callback dispatch and every `create()` ŌĆö walks both maps, drops entries where `now - createdAt > ttlMs`. No dedicated cron ŌĆö the cost of missing a cleanup for hours is a handful of old Map entries (<1 KB each). The GC-on-access pattern keeps the registry self-maintaining.

**LRU cap:** When `active.size === cacheMax` and a new `create()` lands, evict the least-recently-active entry (min `lastActivityAt`) FIRST to `recentlyClosed`, then (if that's also full) drop it entirely. Consumer's `onTerminal(state)` fires on LRU eviction so abort controllers get a chance to cancel. A warn-level log marks LRU eviction so ops can tune the cap if it actually fires in production (at deployed scale, 50 concurrent panels is far beyond what a single user could generate).

**Consequences.**

  - **Positive.** Zero ceremony. No migration. No per-edit disk write.
  - **Positive.** LRU + TTL bounds the Map so memory never grows unbounded under a pathological consumer.
  - **Negative.** pm2 restart mid-debate loses the panel. Acceptable ŌĆö the abort signal would have killed the debate anyway.
  - **Negative.** Multi-process deployment (future) breaks the primitive. When that happens, migrate panel state to SQLite + worker IPC, per ┬¦13.

### 8. Stale-callback handling (restart / TTL / LRU eviction)

**Status:** Accepted.

**Context.** Three paths to a "stale" callback:

  1. **pm2 restart.** User tapped `[Ōīä Expand]` on a panel created before restart. `active` and `recentlyClosed` are both empty for that panelId.
  2. **TTL elapsed.** Panel state aged out (> 24h). Swept on GC-on-access.
  3. **LRU evicted.** Panel state pushed out of `recentlyClosed` by a burst of new panels.

In all three cases the button exists on a Telegram message (messageId is known to Telegram) but the registry has no matching entry.

**Decision.** `PanelRegistry.handleCallback` handles all three identically:

  1. Parse `data` as `<namespace>.<action>:<panelId>`. If malformed ŌåÆ toast "Bad button." and return (log at warn).
  2. Look up `panelId` in `active` ŌåÆ found, proceed to owner check + dispatch.
  3. Else look up in `recentlyClosed` ŌåÆ found, the panel is terminal; allow read-only actions (`expand`/`collapse`) to proceed, REJECT `cancel` with toast "Already finished." (can't cancel a completed debate).
  4. Else the entry is gone. Respond:
     - `ctx.answerCallbackQuery({text: 'Panel expired ŌĆö please re-run /debate'})` (short; fits Telegram's toast char limit).
     - `ctx.editMessageReplyMarkup(undefined).catch(() => {})` ŌĆö strip the dead buttons so the user can't keep tapping (the message text stays).
     - Return `{toast: 'Panel expired'}` so the gateway's wrapper can log.
     - Log at warn: `{panelId, action, namespace}`.

No error thrown in any branch. All branches return a toast. The `.catch(() => {})` on `editMessageReplyMarkup` swallows "message can't be edited" (Telegram's 48h edit-window expiry), which is expected after long-stale messages and not an error to surface.

**R9-style pre-action verification** (ADR 006 R9 for /organize reconcile): NOT adopted for ProgressPanel. Reasons:

  - `/organize reconcile` has a separate source-of-truth (audit rows) against which the callback can be verified. ProgressPanel's source-of-truth IS the in-memory state ŌĆö if state is gone, there's nothing to verify against.
  - The attack surface here is lower: the callback_data doesn't drive an irrevocable state mutation (unlike reconcile's prune-state-entry). Worst-case malicious replay of `debate.cancel:<panelId>` on a stale panel hits branch 4 above (panel gone ŌåÆ toast expired). No state change.

**Consequences.**

  - **Positive.** All stale paths collapse to one handler. Simple.
  - **Positive.** `recentlyClosed` window gives a grace period for terminal-panel expand/collapse taps ŌĆö the common case of "user sees the final panel, taps Expand to re-read the transcript, tap lands 30 seconds after panel finalized" works.
  - **Negative.** LRU eviction of a long-running panel (>50 concurrent) is silent from the USER's perspective ŌĆö they eventually see "Panel expired." Acceptable because 50 concurrent panels per process is vastly beyond single-user scale and surfaces as a warn log when it ever happens.

### 9. Ownership rules ŌĆö expand/collapse vs. cancel

**Status:** Accepted.

**Context.** `/debate` runs in DMs and groups. When admin A starts a debate in a group, and admin B taps a button:

  - `[Ōīä Expand transcript]` ŌĆö read-only; revealing more of an in-progress debate both admins can already read in the chat.
  - `[Ōīā Collapse]` ŌĆö inverse of expand; still read-only on state.
  - `[ŌØī Cancel]` ŌĆö mutates state (aborts the abort signal, truncates the debate).

**Decision.** Asymmetric owner rules, built into the registry:

  - **Expand / collapse:** any user in the chat who is allowlisted (allowlist middleware already gates the update before it reaches the registry). No per-user check. Rationale: the content is already visible to everyone in the chat; expanding the view doesn't leak anything new.
  - **Cancel:** `ctx.from.id === ownerUserId` OR `config.groups.adminUserIds.includes(ctx.from.id)`. If the check fails, toast `"Only <starter's first_name> or a group admin can cancel this debate."` (look up starter's name from cached entry data; fall back to `"the debate starter"` if name is unavailable). No state mutation.
  - **extraActions (custom):** default to the same rule as `cancel` (starter OR group admin). The deps interface accepts an optional `ownerCheck` function per extraAction if a consumer needs different rules later ŌĆö not in scope for `/debate`.

**DM context:** `config.groups.adminUserIds` is irrelevant (DMs aren't groups); the starter check suffices. In DMs, starter == user == chat, so the check is effectively "the user who started it." Correct.

**Test cases** (required in Phase 3 Sub-Phase A):

  1. Admin A starts debate in group, admin B taps Expand ŌåÆ expand proceeds.
  2. Admin A starts debate in group, admin B taps Cancel ŌåÆ rejected with toast.
  3. Admin A starts debate in group, a group admin C (in `adminUserIds`) taps Cancel ŌåÆ proceeds.
  4. Admin A starts debate in group, admin A taps Cancel ŌåÆ proceeds.
  5. DM: user U starts debate, taps any button ŌåÆ all proceed.
  6. User not in chat taps a forwarded button ŌåÆ allowlist middleware already blocks before registry sees it.

**Consequences.**

  - **Positive.** Permissive on read, conservative on mutation. Matches user expectations.
  - **Positive.** Explicit per-action rule codified in the registry (not left to each consumer) prevents future consumers from accidentally allowing cancel-by-anyone.
  - **Negative.** The "starter's first_name" lookup path: if the starter's Telegram handle is unknown at registry time, the toast falls back to a generic string. Minor UX nit; not correctness.

### 13. `DebateState` interface + state flow

**Status:** Accepted.

**Context.** Today `runDebate` holds `transcript: Turn[]` in a local variable (`src/debate/index.ts:97`). The panel primitive needs state externally readable so `renderSummary`/`renderDetail` can run at any point. Define `DebateState` as the single source of truth.

**Decision.** New interface exported from `src/debate/index.ts`:

```typescript
export type DebateStatus =
  | 'starting'           // panel created; first debater call not yet returned
  | 'running'            // mid-round, a debater is speaking or just spoke
  | 'judging'            // judge call in-flight at round boundary
  | 'synthesizing-verdict' // final-verdict path (no consensus after maxRounds)
  | 'consensus'          // terminal ŌĆö judge returned consensus before maxRounds
  | 'final-verdict'      // terminal ŌĆö Claude arbiter produced a decision
  | 'cancelled';         // terminal ŌĆö abort signal fired

export interface DebateState {
  readonly topic: string;       // user's question (post-HTML-escape, POST-scrub per decision 18)
  readonly rosterLabel: string; // roster.label from pickRoster
  readonly roster: readonly string[]; // roster.models
  readonly maxRounds: number;
  readonly exchangesPerRound: number;
  readonly startedAt: number;   // Date.now() at runDebate entry

  status: DebateStatus;
  currentRound: number;         // 1-indexed; 0 before first round starts
  currentModel: string | null;  // name of debater currently speaking, or null between turns
  transcript: readonly Turn[];  // grows; panel renders from this
  // Consensus/verdict payloads (populated at terminal transitions):
  consensusSummary?: string;    // when status === 'consensus'
  finalVerdict?: {              // when status === 'final-verdict'
    decision: string;
    rationale: string;
    dissent: string;
  };
  roundsCompleted: number;      // how many rounds actually ran (important for cancelled mid-round)
}
```

`Turn` stays as-is (`src/debate/index.ts:77ŌĆō80`).

**State flow inside the revised `runDebate`:**

  - Entry: build `DebateState` with `status: 'starting'`, empty transcript, `currentRound: 0`, `currentModel: null`. Call `panel.updateState(state)` (debounced; panel has already sent the first frame with this state).
  - Start of round N: `state = {...state, status: 'running', currentRound: N, currentModel: speakerOrder[0]}`. `panel.updateState(state)`.
  - Before each debater call: `state = {...state, currentModel: model}`. `panel.updateState(state)`.
  - After each debater call: `state = {...state, transcript: [...state.transcript, turn], currentModel: null}`. `panel.updateState(state)`.
  - Before judge call: `state = {...state, status: 'judging'}`. `panel.updateState(state)`.
  - After judge consensus: `state = {...state, status: 'consensus', consensusSummary: verdict.summary, roundsCompleted: N}`. `panel.finalize(state)`.
  - After maxRounds with no consensus, entering final-verdict: `state = {...state, status: 'synthesizing-verdict'}`. `panel.updateState(state)`.
  - After final-verdict returns: `state = {...state, status: 'final-verdict', finalVerdict: verdict, roundsCompleted: maxRounds}`. `panel.finalize(state)`.
  - Cancel path: abort-signal detection ŌåÆ `state = {...state, status: 'cancelled', roundsCompleted: currentRound - 1}`. `panel.finalize(state)`.

State is treated immutably (`{...state, ...patch}`) ŌĆö the registry stores a reference; mutations in place would make reasoning harder. Spread-update is O(N) in transcript length (small N) and explicit.

**Consequences.**

  - **Positive.** State is introspectable; a test can assert on `DebateState` at any point without running the LLM.
  - **Positive.** Adding a new status (e.g. future `'summarizing'`) is a one-liner on the union.
  - **Negative.** Spread-update per turn is N copies. At 12 turns / debate, negligible.

### 16. Expanded-view truncation strategy

**Status:** Accepted (option a: drop oldest rounds, keep most recent).

**Context.** Rigor ask 4. Three options:

  - **(a)** Oldest rounds dropped first; append `ŌĆ”earlier rounds omitted (N turns) ŌĆö /audit filter debate for full` footer.
  - **(b)** Middle dropped; preserve first + last.
  - **(c)** Split across messages.

Telegram cap is 4096 chars; ProgressPanel's safety net is 4000 (decision 3). Realistic /debate size: 12 turns ├Ś ~400 visible chars = 4800 chars, PLUS header + roster line + per-turn model name markers + final verdict block = easily 5500 chars on a full 2-round ├Ś 2-exchange ├Ś 3-debater run. Truncation WILL be needed.

**Decision.** Option (a). Justifications:

  1. **Recency bias matches reader intent.** A user tapping Expand at the end of a long debate most wants the CLOSING arguments (what won, what got conceded) ŌĆö those are the tail. Early rounds set the terms; late rounds resolve them.
  2. **Option (c) breaks the "single editable panel" contract.** If we split across messages, we lose the "one panel per debate" property. A stream of extra messages re-introduces the v1.11.x spam we're eliminating. Rejected.
  3. **Option (b) preserves the first turn ŌĆö which is valuable in a debate** (debaters open with a stance), but the arithmetic rarely works: first-round opener + last-round closer + middle-omitted marker usually blows the budget anyway for long debates. (a) is simpler and almost always as useful.

**Truncation algorithm** (pseudocode):

```typescript
function renderDebateDetail(state: DebateState): string {
  const header = buildHeader(state); // static ~300 chars
  const footer = buildFooter(state); // verdict + audit hint; up to ~800 chars for final-verdict
  const budget = 4000 - header.length - footer.length - 200; // 200 chars reserve for the truncation marker
  let transcriptLines = state.transcript.map(renderTurn);
  // If total fits, no truncation.
  let totalLen = transcriptLines.reduce((s, l) => s + l.length + 2, 0);
  if (totalLen <= budget) {
    return header + transcriptLines.join('\n\n') + footer;
  }
  // Otherwise drop from the head. Count turns dropped for the marker.
  let droppedTurns = 0;
  while (totalLen > budget && transcriptLines.length > 1) {
    const removed = transcriptLines.shift()!;
    totalLen -= removed.length + 2;
    droppedTurns++;
  }
  const marker = `<i>ŌĆ”earlier rounds omitted (${droppedTurns} turn${droppedTurns === 1 ? '' : 's'}) ŌĆö /audit filter debate for full</i>\n\n`;
  return header + marker + transcriptLines.join('\n\n') + footer;
}
```

Pathological edge case: a single turn longer than the budget. The `while` loop stops when `transcriptLines.length === 1`; the final HTML-safe-slice in the primitive (decision 3's 4000-char safety net) cuts the tail. The turn in question was already tail-sliced to 600 chars by `stripThinkTags` + fallback handling in `src/debate/index.ts:294ŌĆō313`, so this is unreachable in practice for current models.

**Consequences.**

  - **Positive.** Simple, predictable. User always sees "recent plus a marker."
  - **Positive.** The "single editable panel" invariant is preserved.
  - **Negative.** First-round context is lost on very long debates. The audit trail has it; the footer hint surfaces that path. Accepted tradeoff.

  - **Transcript turns** ŌĆö LLM-authored. The Ollama Cloud debaters MAY produce output that quotes file paths the LLM saw in context (though the debate prompts don't pass tool context), hostnames, or credential-shaped strings. Low probability in practice (debaters work off the user's question alone, no tool access), but the surface exists.

Existing scrubber infrastructure:

  - `src/safety/scrubber.ts:scrub(text)` ŌĆö redacts ~20 credential patterns (Anthropic / OpenAI / GitHub / Google / AWS / JWT / Bearer / PEM / Stripe keys, etc.). Base scrubber, always safe to apply.
  - `src/safety/groupScrub.ts:scrubForGroup(text, config)` ŌĆö wraps `scrub` AND additionally redacts Windows paths from `config.filesystem.allowedPaths`, hostname, username. Group-mode only.
  - `src/agent/index.ts:666` ŌĆö the only current caller of `scrubForGroup`, applied to the agent's reply text before sending in group chats.

**Decision.** Two-layer scrubbing for `/debate`:

  1. **Topic (user-authored input) ŌĆö always scrub with `scrub(...)` before storing in `DebateState.topic`.** Rationale: user could have pasted an API key into the question by mistake. `scrub()` is cheap, side-effect-free, and covers the credential patterns. Location: `runDebateTurn` at entry, BEFORE building `initialState`. Done once at panel creation; `state.topic` is the scrubbed form from then on.

  2. **Transcript turns (LLM-authored) ŌĆö scrub per turn, in group chats ALSO path-scrub.** In `src/debate/index.ts`'s `callDebater` return path, BEFORE appending to transcript:
     - Always: `turn.text = scrub(turn.text)`.
     - In group chats: `turn.text = scrubForGroup(turn.text, config)` (which internally calls `scrub` ŌĆö do not double-scrub; use `scrubForGroup` when in group).

     Detection of "group chat" is passed to `runDebate` via a new `DebateParams.chatType: 'private' | 'group' | 'supergroup' | 'channel'` field (sourced from `ctx.chat?.type` at the gateway layer). `runDebate` picks `scrub` vs `scrubForGroup` on this.

  3. **Render-time re-scrub ŌĆö NOT needed.** Scrub at the DATA BOUNDARY (entry into `DebateState`). Render functions work on already-scrubbed text and just HTML-escape. One scrub location, one escape location. Separation of concerns.

  4. **Audit log ŌĆö do NOT store raw LLM transcript.** Per decision 19, the audit detail contains only metadata (topic, rounds, consensus, cancelled, durationMs, roster) ŌĆö never the transcript text. Matches the `/organize` invariant (no raw user text in detail).

**Why topic scrub is always-on (not group-only):** A credential-shaped pattern in the topic is a user mistake regardless of chat type. In DM it's "only" the user seeing it, but the topic is ALSO echoed into debater prompts (`callDebater`'s `userText`). Debaters are Ollama Cloud ŌĆö the credential leaves the local process and flows to an external provider. That's the real leak vector; DMs don't save us. `scrub` on ingest stops it.

**Why path/hostname scrub is group-only:** `scrubForGroup` assumes the user's allowed filesystem paths are "private to this deployment" ŌĆö in a DM the user already owns the deployment and seeing their own path is not a leak. In a group with multiple users, one user's deployment path shouldn't leak to peers. Matches the existing `scrubForGroup` invariant.

**Test coverage:**

  - Topic containing `sk-ant-<real-shape>` ŌåÆ `state.topic` contains `[REDACTED:ANTHROPIC_KEY]` in both summary and detail renders.
  - Debater turn (mocked) producing `C:\Users\Boss\secrets.txt` in group chat ŌåÆ expanded detail shows `<path>` placeholder.
  - DM /debate with the same content ŌåÆ path NOT redacted (group-only scrub).
  - Both scrubs happen before `escapeHtml`, so HTML escaping sees the already-scrubbed text.

**Consequences.**

  - **Positive.** Credentials in user-authored topics never reach the Ollama Cloud debater prompt.
  - **Positive.** Group-mode path/hostname leaks via debater output are scrubbed.
  - **Positive.** One scrub site per input surface (topic at gateway, transcript at callDebater). No double-scrub, no missed paths.
  - **Negative.** `chatType` now flows into `DebateParams`. One more field; trivial.

### 19. Debate audit trail ŌĆö new categories

**Status:** Accepted.

**Context.** Today `/debate` emits no audit rows. For v1.12.0, the panel-based UX condenses user-visible output, which makes an audit trail more valuable (the /audit command becomes the way to see transcripts historically). Rigor ask 13 recommends adding audit events.

**Decision.** Add three new audit categories to `AuditCategory` in `src/memory/auditLog.ts`:

  - `'debate.start'` ŌĆö emitted at `runDebate` entry (after scrub, after panel create). Detail:
    ```typescript
    { topic: string, rounds: number, exchangesPerRound: number, roster: string[] }
    ```
  - `'debate.complete'` ŌĆö emitted at terminal transition for `'consensus'` or `'final-verdict'`. Detail:
    ```typescript
    {
      topic: string,           // scrubbed topic
      rounds: number,          // roundsCompleted
      consensusReached: boolean,
      cancelled: false,
      durationMs: number,      // Date.now() - state.startedAt
      roster: string[],
      outcome: 'consensus' | 'final-verdict',
      summary?: string,        // consensus summary OR final-verdict decision, 200 chars
    }
    ```
  - `'debate.cancel'` ŌĆö emitted at `'cancelled'` terminal transition. Detail:
    ```typescript
    {
      topic: string,
      roundsCompleted: number,
      cancelled: true,
      durationMs: number,
      roster: string[],
    }
    ```

Actor fields: `actor_user_id = ownerUserId`, `actor_chat_id = chatId`, `session_id = memory.sessions.getOrCreate(chatId).id`. The session linkage matters for /audit's session filter.

**No raw transcript in audit.** The TRANSCRIPT is recoverable from the sessions/messages path (already logged by the agent turn mechanism for DM-enabled debates? Verify in Phase 2 ŌĆö debater turns go through Ollama provider directly, NOT `agent.turn`, so they're NOT in `messages` today). If the transcript isn't in `messages`, the audit trail becomes the sole historical record. **Defer to Phase 2 dev agent verification**: if `messages` does not persist debater turns, a further decision is needed on whether to add a `debate.turn` audit category with per-turn detail. Flag as Open Question below.

Why three categories vs. reusing a generic one (per ADR 005 decision 12 precedent of `admin_command` + `detail.tool`):

  - The `/debate` surface is distinct from admin commands and tool calls. Reusing `admin_command` would conflate.
  - `/organize.nudge`, `/organize.reconcile`, `/organize.trash.evict` each got their own category. The `/debate.*` trio matches that pattern (ADR 006 decision 5: "explicit categories > stringly-typed discriminators").
  - Three categories allow `/audit filter debate.start`, `/audit filter debate.complete`, `/audit filter debate.cancel` independently. Grep-friendly.

**Consequences.**

  - **Positive.** `/audit filter debate.*` becomes the historical record.
  - **Positive.** Type-exhaustive `AuditCategory` switch forces consumers to acknowledge new cases (Anti-Slop ┬¦4 compliant).
  - **Negative.** Three new categories. Trivial union addition.
  - **Negative.** Transcript persistence path needs Phase 2 verification; see Open Questions.

### 20. Typing indicator interaction

**Status:** Accepted.

**Context.** Rigor ask 12. v1.11.1 added pulsing typing indicators on long agent turns. The panel's `updateState` calls already produce visible change every ~1.5s max. Is typing redundant, or complementary?

**Decision.** Typing STAYS, scoped to per-turn intervals during debate:

  - In `runDebateTurn` (gateway layer), before each debater call, fire `bot.api.sendChatAction(chatId, 'typing').catch(() => {})` once. Don't pulse every 4s ŌĆö the panel's edit at each turn boundary is the primary liveness signal; typing is a secondary cue that "work is happening between the last edit and the next."
  - Over a long Ollama cold-start (30-60s between panel edits), typing indicator fades after ~5s (Telegram-native behavior). That's fine ŌĆö the user sees the collapsed summary line "<currentModel> speaking..." in the panel and knows the state.
  - No typing indicator during judge call (state already says "Claude judging..."), no typing during final-verdict synth.

Rationale for keeping it (vs. removing):

  - Two liveness signals are complementary: typing is server-pushed ephemeral; panel edit is permanent and content-bearing. Losing typing during the 30s Ollama cold-start would make the UX feel frozen between edits.
  - Cost is zero ŌĆö `sendChatAction` is fire-and-forget and swallowed.

**Consequences.**

  - **Positive.** UX continuity during slow debater turns.
  - **Positive.** No code churn ŌĆö the existing typing call at `src/gateway/index.ts:178` is removed (it was a single pre-debate indicator; v1.12.0 moves it per-turn into runDebate).
  - **Negative.** One more call site inside `runDebate` that touches Telegram. Acceptable ŌĆö scoped and defensive.

### 21. `debate.*` branch in the existing callback_query handler

**Status:** Accepted.

**Context.** The gateway's `callback_query:data` handler at `src/gateway/index.ts:441ŌĆō469` branches on prefix: `plan.*`, `rec:*`, else "Unknown button." v1.12.0 adds `debate.*` to the list, routing through the panel registry.

**Decision.** Add a new branch BEFORE the "Unknown button" fallthrough, after the existing `plan.*` and `rec:*`:

```typescript
} else if (data.startsWith('debate.')) {
  try {
    const result = await panelRegistry.handleCallback(data, ctx);
    await ctx.answerCallbackQuery({ text: result.toast }).catch(() => {});
  } catch (err) {
    log.error(
      { data, err: err instanceof Error ? err.message : String(err) },
      'debate panel callback handler threw',
    );
    await ctx.answerCallbackQuery({ text: 'Panel error.' }).catch(() => {});
  }
}
```

The `panelRegistry.handleCallback` internalizes the namespace parsing (it sees `debate.expand:pn_xxx`, strips `debate.`, dispatches). Any future panel consumer (tool-call preview, plan refactor) adds ITS OWN `else if (data.startsWith('toolpreview.'))` branch similarly, OR ŌĆö cleaner ŌĆö the gateway can iterate over a registry of namespaces. For v1.12.0 with one new consumer, the hardcoded branch is fine.

**Ownership checks.** Mirror `handlePlanButton`'s precedent (`src/commands/plan.ts:410+`):

  - The handler ALWAYS answers the callback (so the button spinner stops) ŌĆö matches the existing pattern.
  - The owner check is IN THE REGISTRY (decision 9), not in the gateway branch ŌĆö the gateway is dumb routing. Registry returns a toast on owner-fail; gateway relays it.

**Consequences.**

  - **Positive.** Five lines added to the router. Fan-out by prefix, same as plan and rec.
  - **Positive.** No duplication of ownership logic in the gateway.

### 22. `panelRegistry` wiring in `initGateway`

**Status:** Accepted.

**Context.** The registry instance is process-scoped, created at gateway init. Shutdown must clear timers so pm2 shutdown is clean.

**Decision.** Wire at the top of `initGateway(deps: GatewayDeps)`, alongside `createTelegramAdapter`:

```typescript
export function initGateway(deps: GatewayDeps): GatewayApi {
  const { config, memory, safety, agent, transcriber, version } = deps;

  const bot = new Bot(config.telegram.botToken);
  const telegramAdapter = createTelegramAdapter(bot.api);
  const panelRegistry = createPanelRegistry({
    config,
    cacheMax: config.debate.panelStateCacheMax,
    ttlMs: config.debate.panelStateTtlHours * 3_600_000,
  });
  // ... existing code continues ...
}
```

Expose on `GatewayApi` for the start/stop lifecycle:

```typescript
export interface GatewayApi {
  // ... existing fields ...
  /** v1.12.0 ŌĆö stop all panel timers. Called by src/index.ts shutdown. */
  stopPanels(): void;
}
```

`initGateway` returns `{ ..., stopPanels: () => panelRegistry.stop() }`. Shutdown order in `src/index.ts`: `... ŌåÆ gateway.stop() ŌåÆ gateway.stopPanels() ŌåÆ scheduler.stop() ŌåÆ ...`. `stopPanels` after `stop()` because callbacks in-flight during `bot.stop()` may still need the registry.

**No separate boot step.** The registry is lazily created inside `initGateway` ŌĆö no top-level boot step like reminders or scheduler. It has no pre-bot-start work to do.

**Consequences.**

  - **Positive.** Minimal boot-order churn.
  - **Positive.** Lifecycle bound to gateway ŌĆö 1:1 with the consumer surface.

## 26. Open questions

  1. **Transcript persistence path.** Verified source: `runDebate` calls `params.ollama.call(...)` directly; debater turns do NOT go through `agent.turn` and are NOT written to the `messages` table. That means historical transcripts are NOT recoverable once the panel's in-memory state is evicted. Three options:
     - (a) Add a `debate.turn` audit category per-turn (matches the brief's preference).
     - (b) Write debater turns to `messages` directly via the existing repo (requires a session binding, which already exists).
     - (c) Accept loss ŌĆö the panel's TTL is 24h and `/audit filter debate.*` captures the start/end metadata, which is probably enough historically.

     **Recommendation for CP1 resolution:** option (a), because it keeps the debate module isolated from the messages schema (no coupling) and the `/audit filter debate.*` surface becomes a single-stop-shop. Open for debate.

  2. **Unifying callback namespaces across subsystems.** This ADR retains `rec:*` as-is and introduces `debate.*` in the plan-style format. A future ADR could migrate `rec:*` to `reconcile.*` for consistency, but it's a breaking change for any in-flight buttons. No action in v1.12.0; flagged for future consideration only.

  3. **Future ProgressPanel consumers and config namespace.** Decision 10 scoped the config knobs under `config.debate.panelStateCacheMax` ŌĆö tightly coupling the primitive's config to the first consumer. When a second consumer (DM tool-call preview, /plan refactor) lands, we should promote to `config.progressPanel.panelStateCacheMax` and have each consumer's module read the shared value. Not in scope for v1.12.0; flagged.
