# ADR 021 — v1.21.0 Avengers MVP (Multi-Bot Identity)

**Status:** Phase 1 deliverable. Architect output. Binding for Phase 2 implementation pending CP1 review.
**Iteration:** v1.21.0
**Owner:** Architect (opus)
**Date:** 2026-04-25
**Predecessors:** ADR 018 (v1.18.0 Coach Jarvis), ADR 019 (v1.19.0 Coach polish + Calendar two-way sync), ADR 020 (v1.20.0 Multi-coach + event triggers)

---

## Context

User asked verbatim, captured 2026-04-25: *"Let's build the avengers. I need clone of Jarvis. I want to create new bots in Telegram all in the same chat room and they can interact with each other and do things."*

The same vision was originally filed at v1.18.0 in TODO.md ("Avengers: multi-bot personas") but vetoed when the user pivoted to Coach Jarvis. Now back as v1.21.0.

v1.20.0 hardened the per-bot abstraction at the **profile** layer (4 coach profiles per user; closed-set markers; profile-agnostic memory). v1.21.0 hardens it at the **process** layer: each bot is a separate pm2 process, with its own Telegram token, persona, data directory, tool allowlist, SQLite DB, audit log, coach memory, and scheduled tasks.

The shape of the abstraction mirrors v1.20.0 `profileTypes.ts` (closed set + type guard + marker convention) and v1.18.0 `intensityTypes.ts` (single-source-of-truth for the value list). Adding a third bot in a future iteration becomes config-only: drop a persona file, add the name to `BOT_NAMES`, update the ecosystem config. No code change required for the persona-only bot path.

This iteration also pre-empts a 5th iteration of the boot-wiring trap class (v1.18.0 `coachTurnCounters`, v1.19.0 calendar breaker, v1.20.0 three monitor callbacks) by extending the v1.20.0 `tests/static/coach-event-wiring.test.ts` pattern to BotIdentity callbacks: storage data path, scheduler bot context, agent persona path, tools allowlist gate.

---

## Pillars (high-level)

- **Pillar 1: Bot identity abstraction.** Pervasive enough that downstream features become config-only. `BOT_NAMES` closed set; per-bot data dir; per-bot persona prompt; per-bot tool allowlist; path-sandbox narrows per process.
- **Pillar 2: Mention-routed inter-bot interaction.** Each bot independently parses for `@<self>` mentions; processes only when called. Loop protection (max 3 bot-to-bot turns per thread). Self-message echo drop. Inter-bot context wrap (`<from-bot>` boundary).
- **Pillar 3: PM2 ops.** `ecosystem.config.cjs` + per-bot `BOT_TOKEN_<NAME>` env vars + `docs/AVENGERS.md` runbook.
- **Pillar 4: Webapp parametrization.** Each bot's webapp on its own port (ai-jarvis 7879, ai-tony 7889). Hub banner shows bot identity.

## Decision 1: BotIdentity placement — standalone `src/config/botIdentity.ts` (~80 LOC)

**Decision:** New file `src/config/botIdentity.ts` (~80 LOC) hosts the type, the closed-set const-array, the type guard, the resolver (from env + config), and small shape helpers (`personaPathFor(name)`, `dataDirFor(name)`, `webappPortFor(name)`).

**Why a new file (not extending profileTypes.ts or intensityTypes.ts):** v1.20.0 D1 already established the precedent — closed-set types live in their own file when they own a domain concept. `intensityTypes.ts` (5 values, coach-scoped) and `profileTypes.ts` (4 values, coach-scoped) are coach-internal closed sets. BotIdentity is a process-scoped closed set with a different consumer graph (config, gateway, agent, tools, safety, organize, coach, calendar, webapp). Cross-pollutating one of the existing coach files would invert that scope. New file = clean static-test isolation per closed set; consistent with v1.20.0 architectural discipline.

```ts
// src/config/botIdentity.ts
export const BOT_NAMES = ['ai-jarvis', 'ai-tony'] as const;
export type BotName = typeof BOT_NAMES[number];
export type BotScope = 'full' | 'specialist';

export interface BotIdentity {
  name: BotName;
  scope: BotScope;
  /** Telegram bot token (resolved from BOT_TOKEN_<NAME> env at boot) */
  telegramToken: string;
  /** Telegram bot username (resolved from getMe at boot; cached for activation gate) */
  botUsername: string;
  /** Absolute path to the persona prompt markdown file */
  personaPath: string;
  /** Webapp port for this bot's tunnel */
  webappPort: number;
}

export function isBotName(v: unknown): v is BotName { ... }
export function resolveBotIdentity(envBotName: string | undefined): { ok: true; identity: BotIdentity } | { ok: false; error: string };
export function personaPathFor(name: BotName): string;
export function dataDirFor(name: BotName): string;
export function webappPortFor(name: BotName): number;
```

Rigor (Anti-Slop §3 closed sets): static test `tests/static/bot-identity-closed-set.test.ts` asserts `BOT_NAMES.length === 2` AND that every entry has a corresponding persona file resolvable via `personaPathFor`.

**User rigor ask quoted:** "PICK: standalone `src/config/botIdentity.ts` (~80 LOC). Mirrors v1.20.0 `profileTypes.ts` + v1.18.0 `intensityTypes.ts` patterns. Closed-set discipline."

## Decision 2: BOT_NAMES closed set — `['ai-jarvis', 'ai-tony']`

**Decision:** v1.21.0 ships exactly TWO bots in the closed set: `ai-jarvis` (the existing Jarvis surface; full tool scope; back-compat) and `ai-tony` (NEW; specialist scope focused on engineering/build/code tasks).

**Adding a 3rd bot is config-only** (e.g., `ai-natasha`, `ai-bruce` in v1.22.0+):
1. Append the name to `BOT_NAMES` array (1-line edit).
2. Drop a persona file at `config/personas/<botName>.md`.
3. Add a `BOT_TOKEN_<NAME>` env entry to `.env.example`.
4. Add a process block to `ecosystem.config.cjs`.
5. Choose a tool scope (`full` or `specialist`) — no new code.

No code change is required for the persona-only bot path because all of {data dir, sandbox path, persona load, tool gate, audit, port assignment, hub banner label} flow through `botIdentity` resolved at boot.

**The contract:** any new bot whose behavior CANNOT be expressed as `(scope, persona)` requires a code change. v1.21.0 explicitly defers a 3rd `scope` value (`'curator'`, `'reader-only'`, etc.) to a future ADR — adding a scope is NOT free; it requires a tool-allowlist closed-set entry (D6) and an audit category review.

Rigor: static test `tests/static/bot-identity-closed-set.test.ts` enforces array length === 2 for v1.21.0. Future iteration that adds a bot updates the test alongside.

## Decision 3: Per-bot data dir migration — legacy `data/` → `data/ai-jarvis/`

**Decision:** Existing flat-layout paths under `data/` migrate to the `data/<botName>/` shape on first v1.21.0 boot for `BOT_NAME=ai-jarvis`. The migration is idempotent and runs ONLY for ai-jarvis (the new ai-tony process starts with an empty `data/ai-tony/` and has no legacy state to migrate).

Affected paths (all owned by ai-jarvis pre-v1.21.0):

| Pre-v1.21.0 location | Post-v1.21.0 location |
|---|---|
| `data/jarvis.db` | `data/ai-jarvis/jarvis.db` |
| `data/organize/<userId>/` | `data/ai-jarvis/organize/<userId>/` |
| `data/coach/<userId>/` | `data/ai-jarvis/coach/<userId>/` |
| `data/calendar/<userId>/` | `data/ai-jarvis/calendar/<userId>/` |
| `data/google-tokens.json` | `data/ai-jarvis/google-tokens.json` |
| `data/workspaces/<chatId>/` | `data/ai-jarvis/workspaces/<chatId>/` |

**Migration boot ordering (BINDING, mirrors v1.20.0 R3.a invariant):** the migration helper `runBotDataMigration(identity)` MUST run BEFORE `initMemory(cfg)` in `src/index.ts`. `initMemory` opens the SQLite DB at the resolved path; if migration runs after, it would either (a) try to move an open file, or (b) initMemory would create a fresh DB at the new path while the legacy DB sits stale at the old path. Static test `tests/static/bot-migration-ordering.test.ts` parses `src/index.ts` AST and asserts `runBotDataMigration` precedes `initMemory`. (Same shape as v1.20.0 `tests/static/coach-migration-ordering.test.ts`; pattern reused verbatim.)

**Idempotency:** the migration first checks if `data/ai-jarvis/jarvis.db` already exists. If yes → migration is a no-op (return `{ migrated: false }`). If no AND `data/jarvis.db` exists → perform fs.rename of each path; audit `bot.migration_completed`. If no AND `data/jarvis.db` ALSO does not exist → fresh install; create `data/ai-jarvis/` empty; no audit row.

**Conflict handling:** if BOTH `data/jarvis.db` AND `data/ai-jarvis/jarvis.db` exist (somehow — pm2 mid-restart with two binaries running, or manual user copy), keep the new path AS-IS, do NOT rename, audit `bot.migration_conflict` with both paths in detail JSON. The legacy `data/jarvis.db` is left in place for manual operator review; the operator decides which to keep.

**For ai-tony:** `runBotDataMigration({name: 'ai-tony', ...})` is a no-op — checks `data/ai-tony/jarvis.db`, which doesn't exist; checks `data/jarvis.db`, but the migration body skips the rename when `identity.name !== 'ai-jarvis'` (legacy path is owned by ai-jarvis only).

Audit categories (D18): `bot.migration_completed`, `bot.migration_conflict`.

## Decision 4: Path-sandbox per bot — `wrapPathForBotIdentity(identity)` helper

**Decision:** The Tier 1 path-sandbox (per `docs/BUILD_SANDBOX.md` factory spec, applied to ai-jarvis since v1.0) narrows from `allowedPaths: ['{build_dir}']` to `allowedPaths: ['{build_dir}/data/<botName>']` per process. ai-tony's process can NOT read `data/ai-jarvis/`; ai-jarvis's process can NOT read `data/ai-tony/`.

**Implementation:** new helper `wrapPathForBotIdentity(identity, allowedPaths)` in `src/safety/paths.ts` (~+30 LOC). Called once at boot from `src/index.ts` after config load + identity resolve, BEFORE `initSafety`. Carry-forward from the v1.18.0 D7 `wrapWriteForCoachRun` pattern (tool-context wrapping for coach turns) — same shape, applied to the safety layer's allowed-paths list.

```ts
// src/safety/paths.ts (additions)
export function wrapPathForBotIdentity(identity: BotIdentity, configAllowedPaths: string[]): string[];
// Returns a NEW array. Each entry that is the build-dir root (or unscoped data/)
// is replaced with the per-bot data path. Entries already scoped (e.g., a project
// path outside the build dir) are passed through unchanged. Idempotent.
```

**For ai-jarvis:** wrapped allowedPaths includes `D:\ai-jarvis\data\ai-jarvis` PLUS any project paths from `config.projects` (which are user-defined work directories outside `data/` — those are NOT bot-scoped; both bots can access them if the user grants them). Workspaces (`config.workspaces.root`) wrap to `D:\ai-jarvis\workspaces\ai-jarvis\` (parallel narrowing).

**For ai-tony:** wrapped allowedPaths = `D:\ai-jarvis\data\ai-tony` + same project paths. ai-tony has the SAME read-deny globs (no .env, no credentials) — those are global, not bot-scoped.

**Why narrow at the safety layer (not just the Tool dispatcher):** a tool that bypasses the dispatcher (e.g., a future MCP server that reads files directly) inherits the safety layer's path check. Tool-level check alone is insufficient defense-in-depth (Anti-Slop §15).

Rigor: `tests/integration/bot-sandbox-isolation.test.ts` boots ai-tony's safety layer with the wrapped paths, calls `isPathAllowed('D:\\ai-jarvis\\data\\ai-jarvis\\organize\\anything')`, asserts `false`.

## Decision 5: Per-bot persona system prompt — `config/personas/<botName>.md`

**Decision:** Each bot loads its system prompt from `config/personas/<botName>.md`. The existing `config/system-prompt.md` migrates to `config/personas/ai-jarvis.md` on first v1.21.0 boot if `config/personas/ai-jarvis.md` does not yet exist.

**New file: `config/personas/ai-tony.md` (~80 LOC).** Persona: dry, technical, build/engineering-focused. Tony Stark voice — short, decisive, doesn't hedge. Specialist scope (D6) means ai-tony's prompt explicitly lists the tools it has and lists the tools it does NOT have (no calendar, no gmail, no organize, no coach). No persona overlap — calling `@ai-jarvis can you build me…` should still go to ai-jarvis (D7 mention routing), and `@ai-tony schedule a meeting…` should respond "that's not my surface — try @ai-jarvis."

**Persona file boot loader:** `src/agent/systemPrompt.ts` is extended (~+20 LOC) to accept a `BotIdentity` and read from `identity.personaPath` instead of the hardcoded `config/system-prompt.md`. Template variables (`{{PROJECTS_CONTEXT}}`, `{{CURRENT_DATETIME}}`, `{{WORKING_DIRECTORY}}`, `{{SYSTEM_INFO}}`) are unchanged. Adds one new variable `{{BOT_NAME}}` so the persona can self-reference if useful.

**Migration helper:** `runPersonaMigration(identity)` in `src/config/botMigration.ts` (~part of D3's migration; +20 LOC inside the same helper). Idempotent: if `config/personas/ai-jarvis.md` exists → no-op. Else if `config/system-prompt.md` exists AND `identity.name === 'ai-jarvis'` → copy (NOT move; the original stays for back-compat in case operator scripts reference it). Audit `bot.migration_completed` with `subject: 'persona'`.

**`config/system-prompt.md` deprecation:** kept on disk for one major version as a fallback. After v1.21.0 ships, future ADR can delete it. Not a build-time concern.

## Decision 6: Per-bot tool allowlist — `scope: 'full' | 'specialist'`

**Decision:** Two scopes in v1.21.0:

- **`scope: 'full'`** (ai-jarvis): all 33+ registered tools, exactly as v1.20.0 ships them. Back-compat preservation invariant. No tool removed.
- **`scope: 'specialist'`** (ai-tony): a subset focused on engineering/build/code work.

**Specialist allowlist (closed set; binding):**

```ts
// src/config/botIdentity.ts
export const SPECIALIST_TOOL_ALLOWLIST: ReadonlySet<string> = new Set([
  'read_file',
  'write_file',
  'list_directory',
  'search_files',
  'system_info',
  'run_command',
  'recall_archive',
  'web_search',
  'browse_url',
  'send_file',
]);
```

**EXCLUDES (specialist):** `organize_create`, `organize_update`, `organize_complete`, `organize_list`, `organize_log_progress`, `organize_delete`, `coach_log_nudge`, `coach_log_research`, `coach_log_idea`, `coach_log_plan`, `coach_read_history`, `coach_log_user_override`, `update_memory`, `forget_memory`, `gmail_search`, `gmail_read`, `gmail_draft`, `calendar_list_events`, `calendar_create_event`, `calendar_update_event`, `calendar_delete_event`, `schedule`. (Total exclusions: ~22 tools.)

**Why allowlist (not deny-list):** the v1.18.0 R6/F1 invariant ("allowlist enforced by code, not prompt") binds. A new tool added in a future iteration would be implicitly granted to ai-tony under a deny-list — surprising. With an allowlist, new tools default-deny for specialist scope unless the architect explicitly adds them. Defense-in-depth (Anti-Slop §15).

**Dispatcher gate:** `src/tools/index.ts dispatch()` extends (~+30 LOC). Before the `ctx.allowedToolNames` check, AFTER the `ctx.coachTurnCounters` check (so coach-disabled-tools-on-ai-jarvis still apply when ai-jarvis fires its coach turn), add:

```ts
if (ctx.botIdentity?.scope === 'specialist' && !SPECIALIST_TOOL_ALLOWLIST.has(name)) {
  log.warn({ toolName: name, botName: ctx.botIdentity.name }, 'Tool rejected: not in specialist allowlist');
  ctx.audit.insert({ category: 'bot.tool_unauthorized', detail: { toolName: name, botName: ctx.botIdentity.name, scope: 'specialist' } });
  return {
    ok: false,
    output: `Tool "${name}" is not available for ${ctx.botIdentity.name} (specialist scope). This bot focuses on engineering/build/code tasks. Try @ai-jarvis for ${name}.`,
    error: { code: 'TOOL_NOT_AVAILABLE_FOR_BOT', message: `Tool "${name}" not in specialist allowlist` },
  };
}
```

Note the order: the gate fires BEFORE `ctx.allowedToolNames` (group-mode filter) and BEFORE the coach allowlist. A specialist bot that's also in a group still hits the specialist gate first; a specialist bot inside a coach turn would hit specialist, but ai-tony has no coach scheduled tasks (D5 — no `coach_*` tools).

**`ToolContext` extension:** add `botIdentity?: BotIdentity` to `ToolContext` in `src/tools/types.ts`. Optional because tests + tools-as-library use cases construct contexts without identity. Boot path always populates it.

**Audit category (D18):** `bot.tool_unauthorized` (NEW).

**`coach.disabledTools` interaction:** ai-tony's specialist allowlist already excludes all `coach_*` tools, so the coach-allowlist gate is structurally unreachable for ai-tony. The v1.18.0 R6/F1 brake stays intact for ai-jarvis (where coach scheduled tasks do fire).

Rigor: static test `tests/static/bot-identity-closed-set.test.ts` asserts `SPECIALIST_TOOL_ALLOWLIST.size === 10` (closed set). Integration test `tests/integration/bot-tool-allowlist.test.ts` boots ai-tony's dispatcher and verifies `organize_create` returns `TOOL_NOT_AVAILABLE_FOR_BOT`.

**User rigor ask quoted:** "Document the closed-set per scope. Dispatcher rejects out-of-allowlist with `TOOL_NOT_AVAILABLE_FOR_BOT`. New audit category `bot.tool_unauthorized`."

## Decision 7: Mention-aware activation gate

**Decision:** In group chats, each bot ONLY processes messages that:

1. Mention `@<botUsername>` (e.g., `@ai_jarvis_bot` or whatever Telegram username is configured for this bot's token), OR
2. Are direct replies to a message THIS bot sent (existing v1.7.13 behavior, kept).

Mentions of OTHER bots in the same group → IGNORE silently (the other bot's process will pick it up). Plain group messages with no bot mention → IGNORE (existing v1.7.13 deterministic gate posture).

DM (private chat) behavior unchanged: process all messages. If the user sends to ai-tony's DM, ai-tony processes; ai-jarvis's DM is independent.

**Implementation:** `src/gateway/groupGate.ts` is extended (~+50 LOC, mostly delegation to a new helper). NEW file `src/gateway/mentionRouter.ts` (~120 LOC) hosts the parsing.

**`isMentionedByUsername(ctx, botUsername)`** parses `ctx.message?.entities` for `mention` and `text_mention` entities (Telegram's structured payload — DON'T regex-match the text, since `@ai-tony-something-else` would match a substring naively). Returns `true` only when an entity exactly equals `@<botUsername>`.

**Boot-time `botUsername` resolution:** the gateway calls `bot.api.getMe()` once at start; caches `bot.username`. The cached value populates `BotIdentity.botUsername` (D1). If `getMe` fails at boot → fail-fast with helpful error mentioning the BOT_TOKEN. (Same posture as v1.18.0 boot fail-fast for missing prompt.)

**v1.7.13 `isJarvisMentioned` retirement:** the existing function regex-matches `/jarvis/i`. v1.21.0 replaces this with `isMentionedByUsername(ctx, identity.botUsername)`. The regex match was Jarvis-specific; the structured-entity match is bot-agnostic. The legacy function is removed (NOT kept as fallback) — keeping it would let users type "hey jarvis" in an ai-tony group and have BOTH bots respond.

**Replies still work cross-bot:** if a user replies to an ai-jarvis message, only ai-jarvis processes (the reply target is ai-jarvis's user_id; isReplyToBot checks against `botUserId`, which differs per process). Same for ai-tony.

**`ctx.from.is_bot` short-circuit:** if the incoming message is FROM another bot (i.e., it's an inter-bot turn — D9), the activation gate fires NORMALLY. Mention check still applies. So if `@ai-jarvis can you check this?` is posted by ai-tony, ai-jarvis sees a mention from a bot user, processes it (subject to D10 loop protection).

**Scope:** v1.21.0 uses ONLY the username-mention path AND reply-to-bot path. The v1.7.13 follow-up heuristic + LLM intent classifier still apply EXCLUSIVELY to ai-jarvis (the existing surface; back-compat). ai-tony has both DISABLED in its config (`groups.intentDetection.enabled = false` in ai-tony's config-side override). Reasoning: specialist bots should have a tighter activation surface — mention or quiet. Document in `docs/AVENGERS.md`.

Rigor: integration test `tests/integration/bot-mention-routing.test.ts` posts a message containing `@ai_tony_bot` to a group with both bots, asserts ai-jarvis ignores AND ai-tony processes.

## Decision 8: Inter-bot mention recognition + self-message echo drop

**Decision:** Bots see each other's messages naturally as group members. To prevent the trivially broken case where a bot processes its OWN message (Telegram echoes outgoing messages back to the bot via the long-poll update stream when other group members include it in update.message), each bot tracks the message_id of every message IT sent in a 20-entry FIFO keyed memory entry `bot.self_messages` with TTL 1h.

**Self-message tracking primitive:**

```ts
// src/gateway/mentionRouter.ts
export interface SelfMessageRecord {
  messageId: number;
  chatId: number;
  sentAt: string; // ISO
}
export async function recordOutgoingMessage(userId: number, dataDir: string, record: SelfMessageRecord): Promise<void>;
export async function isOurEcho(userId: number, dataDir: string, chatId: number, messageId: number): Promise<boolean>;
```

Storage: keyed memory entry `bot.self_messages` (single key; body is a JSON array of the last 20 records — bounded by FIFO eviction at write-time). Within the existing keyed-memory infrastructure (v1.17.0 `userMemoryEntries.ts`). 1h TTL is enforced at READ time: any record with `sentAt < now - 1h` is filtered out before the membership check.

**Why NOT a SQLite table for self-messages:** existing keyed-memory infra is the right shape (single per-bot row, append-with-cap pattern) and reusing it avoids a schema migration. 20 × 100 bytes = 2 KB / bot / hour worst case; trivial.

**Why per-bot, not per-chat:** message_id is unique per (chat, message). A bot might send to multiple chats; we still use one per-bot keyed entry, scoped by `userId` (the bot's owner_user_id). Records carry `chatId` so the membership check is `(chatId, messageId)` joined.

**Echo drop fires BEFORE the mention check:** if `isOurEcho(...)` returns true, return `proceed: false, reason: 'silent', note: 'self-message echo'`. No audit (this is a pure optimization; recording it would spam the audit log). EXCEPTION: rate-limit the audit at `bot.self_message_dropped` to once per chat per hour for forensic visibility. Audit category D18.

**v1.20.0 chatMonitor coordination:** ai-jarvis's chat monitor (v1.20.0 D6.b) ALSO sees these messages. The chat monitor MUST NOT trigger on self-echoes OR on messages from other bots (`message.from.is_bot === true`). Add the bot-source guard at the top of `processChatMessage`: `if (message.fromIsBot) return;`. Carry-forward from v1.19.0 D10 monkey-patch protection. Static test `tests/static/chat-monitor-bot-guard.test.ts` enforces.

## Decision 9: Bot persona injection in inter-bot context — `<from-bot>` wrap

**Decision:** When a bot processes a message FROM another bot (D7 + D8 didn't drop it), the message body is wrapped in `<from-bot name="<sender-bot-name>">...</from-bot>` boundary tags before being injected into the agent's context.

**Why wrap:** parallel to the v1.18.0 `<untrusted>` boundary discipline (D19, R1) and v1.19.0 Layer-(b) calendar injection wrap. Bot-to-bot messages are MORE trustworthy than web_search results (we own both bots; both are gated by Telegram allowedUserIds), but they are STRUCTURALLY untrusted in the prompt-content sense — another bot's output is a stream of agent-generated text that could include strings that look like instructions ("ignore previous, do X"). The wrap signals to the receiving bot's prompt that the content is bot-generated, not user-direct. (Anti-Slop §15.)

**Implementation:** new file `src/gateway/interBotContext.ts` (~80 LOC). Sole writer for the wrap.

```ts
// src/gateway/interBotContext.ts
export interface InterBotMessageMeta {
  fromBotName: string; // resolved from message.from.username via known-bots lookup
  rawText: string;
  messageId: number;
}
export function wrapInterBotMessage(meta: InterBotMessageMeta): string;
// Returns: <from-bot name="ai-tony">${strippedText}</from-bot>
// Strips any literal </from-bot> sequences from rawText and replaces with [stripped].
// NUL ban + char cap ≤ 4096 (parallel to v1.18.0 R5/F3 coach memory caps).
```

**Resolving fromBotName:** at boot, gateway populates a per-process `KNOWN_BOTS_BY_USERNAME` map from `config/avengers.json` (NEW file at top-level config; see D11). The map shape is `{ <telegramUsername>: <BotName> }`. ai-jarvis's process knows about ai-tony's username (and vice versa) so it can identify "this bot message came from ai-tony" without an API roundtrip. The list of usernames is OPERATOR-CONFIGURED (BotFather sets the username; we mirror it in `config/avengers.json`).

**Unknown bot fallback:** if `message.from.is_bot === true` AND the username is NOT in `KNOWN_BOTS_BY_USERNAME`, use `name="unknown-bot"` and ALSO log at warn level (an unfamiliar bot in our group is operationally interesting). Process the message normally subject to all other gates.

**NUL ban + char cap:** mirrors v1.18.0 R5/F3 (NUL bytes rejected; per-field cap 4096). Reuses the existing `src/organize/validation.ts` NUL-ban helper.

**Strip mismatch protection:** like the v1.18.0 D19 `<untrusted>` wrap, strip any literal `<from-bot` or `</from-bot>` sequences in rawText to prevent close-tag injection. Replace with `[stripped]`.

**Where the wrap fires:** in the agent context-builder path (`src/agent/contextBuilder.ts`, ~+10 LOC). Specifically: when adding a message to the conversation history, if the original Telegram update had `message.from.is_bot === true`, wrap before formatting as the user role. The wrap is a transformation of the user-text content; the role stays `user` because Telegram delivers it via the message stream.

Rigor: integration test `tests/integration/bot-context-wrap.test.ts` simulates a bot-to-bot message and asserts the formatted user message body contains `<from-bot name="ai-tony">`.

## Decision 10: Inter-bot loop protection — max 3 bot-to-bot turns per thread

**Decision:** Hard cap of 3 bot-to-bot turns per Telegram message thread. The 4th would-be turn is dropped silently with audit `bot.loop_protection.engaged`.

**Thread keying:** Telegram has a structured `message_thread_id` (forum topics). When present, use it. When absent (regular groups, supergroup main thread), use the first bot-to-bot message's `message_id` as the thread key (the originating turn).

**Counter storage:** keyed memory `bot.thread.<threadKey>.botToBotCount` (counter; integer). TTL 1h (the entry is auto-cleaned by the v1.20.0 trash-evictor pattern; new TTL value in the keyed-memory entry).

**Increment ordering (BINDING):** counter increments on the BOT's outgoing message, not the incoming one. So the flow is:

1. User: `@ai-jarvis hey, ask @ai-tony to check the build`. Counter starts at 0.
2. ai-jarvis turn: counter check (0 < 3, OK). Replies with `@ai-tony check the build`. Counter increments to 1.
3. ai-tony sees the message (mention; D7 OK). Counter check (1 < 3, OK). Replies. Counter increments to 2.
4. ai-jarvis sees ai-tony's reply (D9 wrap; D7 if mentioned or reply-to). Counter check (2 < 3, OK). Replies. Counter increments to 3.
5. ai-tony sees this. Counter check (3 < 3, FAIL). Drops + audits `bot.loop_protection.engaged` + does NOT reply.
6. The thread effectively ends.

**Why 3 (not 2 or 5):** 2 is too tight (a clarification round-trip is reasonable). 5 is too loose (loops have already cost compute by then). 3 lets a meaningful exchange happen but caps the worst case at 6 messages of bot-bot chatter. User-tunable in `config.botInteraction.maxBotToBotTurnsPerThread` if v1.22.0 needs it; v1.21.0 hardcodes for simplicity.

**User-initiated reset:** if the user posts a new message in the thread (NOT from is_bot), reset the counter to 0. The user breaking in is a fresh intent. Implementation: chat-message receive path also calls `resetThreadCounter(threadKey)` for non-bot senders.

**Implementation:** new file `src/gateway/loopProtection.ts` (~80 LOC). Sole writer for the counter.

```ts
// src/gateway/loopProtection.ts
export const MAX_BOT_TO_BOT_TURNS_PER_THREAD = 3;
export async function checkAndIncrementBotTurn(userId: number, dataDir: string, threadKey: string): Promise<{ ok: true; count: number } | { ok: false; reason: 'LOOP_PROTECTION'; count: number }>;
export async function resetThreadCounter(userId: number, dataDir: string, threadKey: string): Promise<void>;
export function deriveThreadKey(message: { message_thread_id?: number; message_id: number }): string;
```

**Where it fires:** in the gateway, AFTER the activation gate (D7) AND AFTER self-echo drop (D8) BUT BEFORE the agent.turn() is enqueued. Specifically: in `src/gateway/index.ts` chat-receive path (~+30 LOC), check `if (ctx.message.from.is_bot)` → call `checkAndIncrementBotTurn`; if NOT ok → audit + return.

Audit category (D18): `bot.loop_protection.engaged`.

Rigor: integration test `tests/integration/bot-loop-protection.test.ts` simulates 4 round-trips and asserts the 4th is dropped.

## Decision 17: Per-bot data dir centralization helper — `resolveBotDataPath`

**Decision:** All `data/` path construction goes through ONE helper `resolveBotDataPath(identity, ...subpath)`. NEW file `src/config/botPaths.ts` (~60 LOC).

**Why a helper:** the codebase currently has 30+ files that construct `data/...` paths via ad-hoc `path.join(dataDir, ...)`. Each is a candidate drift site under the per-bot regime. Consolidating to one helper:

1. Lets the v1.21.0 implementation update one file rather than 30.
2. Lets a future ADR (e.g., shard-by-userId) update one helper.
3. Provides a single location to enforce the path-sandbox invariant (helper internally calls `wrapPathForBotIdentity` to build paths that are structurally allowed).

```ts
// src/config/botPaths.ts
import path from 'node:path';
import type { BotIdentity } from './botIdentity.js';

/**
 * Build an absolute path under the bot's isolated data directory.
 * Sole writer for data-dir paths. All callers MUST use this.
 *
 * Static test bot-data-path-centralization.test.ts asserts that `src/**`
 * contains zero hardcoded `data/organize/`, `data/coach/`, `data/jarvis.db`,
 * `data/calendar/`, `data/google-tokens.json`, `data/workspaces/` strings
 * outside of this file + the migration helper.
 */
export function resolveBotDataPath(identity: BotIdentity, ...subpath: string[]): string {
  return path.resolve(process.cwd(), 'data', identity.name, ...subpath);
}

export function resolveBotDbPath(identity: BotIdentity): string {
  return resolveBotDataPath(identity, 'jarvis.db');
}

export function resolveBotOrganizePath(identity: BotIdentity, userId?: number): string {
  return userId !== undefined
    ? resolveBotDataPath(identity, 'organize', String(userId))
    : resolveBotDataPath(identity, 'organize');
}

export function resolveBotCoachPath(identity: BotIdentity, userId?: number): string {
  return userId !== undefined
    ? resolveBotDataPath(identity, 'coach', String(userId))
    : resolveBotDataPath(identity, 'coach');
}

export function resolveBotCalendarPath(identity: BotIdentity, userId?: number): string {
  return userId !== undefined
    ? resolveBotDataPath(identity, 'calendar', String(userId))
    : resolveBotDataPath(identity, 'calendar');
}

export function resolveGoogleTokensPath(identity: BotIdentity): string {
  return resolveBotDataPath(identity, 'google-tokens.json');
}

export function resolveBotWorkspacesPath(identity: BotIdentity, chatId?: number): string {
  return chatId !== undefined
    ? path.resolve(process.cwd(), 'workspaces', identity.name, String(chatId))
    : path.resolve(process.cwd(), 'workspaces', identity.name);
}
```

**Static test `tests/static/bot-data-path-centralization.test.ts` (~50 LOC):** walks `src/**/*.ts`, scans for the substrings `'data/organize/'`, `'data/coach/'`, `'data/jarvis.db'`, `'data/calendar/'`, `'data/google-tokens.json'`, `'data/workspaces/'`. Allowed exception files: `src/config/botPaths.ts`, `src/config/botMigration.ts`, `src/config/dataDir.ts` (legacy fallback if retained), and any test fixtures. Any other match → fail.

**Migration impact:** Phase 2 commits 1-2 (Dev-A) replace ad-hoc constructions in `src/organize/storage.ts`, `src/coach/coachMemory.ts`, `src/calendar/syncCursor.ts`, `src/calendar/breakerState.ts`, etc. with helper calls. Mechanical refactor; LOC delta neutral or slightly negative.

**Existing `src/config/dataDir.ts`** (16 LOC, returns `path.resolve(path.dirname(memory.dbPath))`) becomes a back-compat thin wrapper: `resolveDataDir(config)` returns `path.dirname(resolveBotDbPath(identity))` — same shape, but identity-routed. The 3 known callers stay on `resolveDataDir` (no churn there); they'll get the per-bot path automatically because `cfg.memory.dbPath` ends up pointing at `data/<botName>/jarvis.db` after D3 migration.

## Open questions for CP1 (DA + Anti-Slop reviewers)

1. **DA adversarial: bot-to-bot loop edge cases.** D10 caps at 3 turns per thread. What if user posts 2 user messages between bot turns? Counter resets on user message, so each user post buys 3 more bot-to-bot turns. Adversarial: a malicious user could spam "go again" between bot turns to escape the cap. Mitigation candidate: ALSO rate-limit total bot-to-bot turns per chat per hour (e.g., 10). Defer to DA; if accepted, add to D10 as a second-layer cap.

2. **Anti-Slop §15 defense-in-depth: persona-prompt-stated tool list vs `SPECIALIST_TOOL_ALLOWLIST` SSOT.** D5 says ai-tony's persona prompt explicitly lists tools it has and doesn't have. D6 says enforcement is at the dispatcher. The persona prompt is duplicated knowledge — if v1.22.0 grants ai-tony `update_memory`, the prompt must also be updated, or the bot will refuse internally ("I don't have that tool") even though the dispatcher accepts. Resolution candidate: persona files use a `{{TOOL_LIST}}` template variable that `systemPrompt.ts` populates from `SPECIALIST_TOOL_ALLOWLIST`. Then SSOT is in `botIdentity.ts`. Defer to Anti-Slop.

3. **Scalability: shared SQLite DB on the same disk.** v1.21.0 separates DB FILES (`data/ai-jarvis/jarvis.db` vs `data/ai-tony/jarvis.db`) but both are on the same disk and same I/O scheduler. At single-user scale, irrelevant. Multi-user (deferred): same disk could become I/O-bound with N bots × M users. Filed for v1.23.0+ ("multi-instance scaling").

4. **Anti-Slop §6 single-source-of-truth: `KNOWN_BOTS_BY_USERNAME` in `config/avengers.json` vs `BOT_NAMES` in `botIdentity.ts`.** Two sources of truth: the closed-set names are in `botIdentity.ts`; the per-name Telegram username is in `config/avengers.json`. Drift risk: BotFather rename of a bot doesn't update `BOT_NAMES`. Resolution: keep `BOT_NAMES` as the canonical-name list; `avengers.json` is operational-only (Telegram-username mapping; can change). Document the asymmetry. Defer to Anti-Slop.

5. **DA adversarial: `getMe()` API roundtrip at boot fails for one bot in a multi-bot deployment.** Per D20, missing or invalid `BOT_TOKEN_<NAME>` exits the process. But what if ai-jarvis boots fine and ai-tony's getMe times out? In pm2, ai-tony restarts in a loop; ai-jarvis stays up. Acceptable behavior (process isolation is the point). Document in `docs/AVENGERS.md` operator runbook.

6. **Anti-Slop §13 module boundaries: `gateway/mentionRouter.ts` reading `bot.self_messages` from keyed memory.** mentionRouter reads from memory; that's an upward edge (gateway → memory). Already established as OK in the existing architecture (gateway/groupGate.ts reads groupSettings via injection). Keep self-messages access via the same DI shape (`UserMemoryEntriesRepo` injected). Defer to Anti-Slop.

7. **Scalability: `bot.self_messages` keyed entry growth.** 20 records × 100 bytes = 2 KB / bot / hour. With 1h TTL, the entry is pruned at read-time. Growth is bounded. OK.

8. **DA adversarial: a 3rd party adds a bot to the same Telegram group with a Telegram username that ALMOST matches one of ours.** E.g., `@ai_jarvis_bot_evil`. The mention `@ai_jarvis_bot_evil` is a structured Telegram entity that exactly equals `@ai_jarvis_bot_evil`, NOT `@ai_jarvis_bot`. So our entity-equality check (D7) returns false for our bots. Their bot might process it; ours don't. Acceptable. Document in `docs/AVENGERS.md` ("entity equality is exact, not prefix").
