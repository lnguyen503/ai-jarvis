# ADR 020 — v1.20.0 Multi-Coach Profiles + Event-Driven Proactive Coach

**Status:** Phase 1 deliverable. Architect output. Binding for Phase 2 implementation pending CP1 review.
**Iteration:** v1.20.0
**Owner:** Architect (opus)
**Date:** 2026-04-25
**Predecessors:** ADR 018 (v1.18.0 Coach Jarvis), ADR 019 (v1.19.0 Coach polish + Calendar two-way sync)

---

## Context

v1.18.0 + v1.19.0 shipped Coach Jarvis as a daily-cron assistant. It worked, but felt rigid.

User feedback (verbatim from session captured 2026-04-23): *"Coach Jarvis is a singular cron job? Why can't I set more than 1 time? Why can't it tell me things on its own?"*

Two distinct gaps:

1. **One scheduled time per user.** v1.18.0 D9 fixed `description='__coach__'` as the single sentinel — no room for multiple coach schedules. The user wants morning + midday + evening + weekly.
2. **No spontaneous behavior.** Coach only fires on its cron tick. It never reacts to "this item is due in 24 hours and the user hasn't touched it" or "the user just typed 'I'm blocked on the design review' in chat."

v1.20.0 closes both gaps without breaking the v1.18.0 R6/F1 brake (allowlist enforced by code, not prompt) or the v1.19.0 R3/W1 brake (NL parser is PURE, sole-writer chain).

It also pre-empts a 4th iteration of the trap class that bit v1.18.0 (`coachTurnCounters` plumbing) and v1.19.0 (calendar breaker stubs, audit emission shims, calendar boot wiring): **interface declared, boot-time stubbed**. Pillar 3 ships a static `tests/static/coach-event-wiring.test.ts` lint that asserts callbacks at the boot site are non-stub functions.

---

## Pillars (high-level)

- **Pillar 1: Multi-coach profiles.** Up to 4 distinct coach schedules per user (`morning`, `midday`, `evening`, `weekly`). Each has its own cron + fires independently. Shared coach memory (profiles affect WHEN, not WHAT remembered).
- **Pillar 2: Event-driven proactive coach.** Coach now fires on events too — item-state changes, conversation patterns, Calendar reverse-sync events. All rate-limited (1/4h per item; 3/day global) + kill switch (`/coach quiet 4h`).
- **Pillar 3: Boot-wiring lint.** Static test catches stubs in src/index.ts callback registrations BEFORE QA.

## Decision 1: Profile closed set + types placement

**Decision:** Closed set of profile names: `morning`, `midday`, `evening`, `weekly`. New file `src/coach/profileTypes.ts` (~30 LOC) hosts the type, the const-array, the type guard, and small parser helpers (`parseHHMM`, `parseWeeklyDay`).

**Why a new file (not extending `intensityTypes.ts`):** `intensityTypes.ts` is 20 LOC and has earned a tight scope (the v1.18.0 closed-set static test asserts its array length === 5). Rigor (Anti-Slop §3 closed sets): static test `tests/static/coach-profile-closed-set.test.ts` asserts `COACH_PROFILES.length === 4`.

## Decision 3: Profile data model

**Decision:** Profile schedule lives in the existing `scheduled_tasks.cron_expression` column. The profile name is encoded in `description`. **No new DB columns. No migration of the `scheduled_tasks` table itself** (only the per-row `description` rewrite of D2's migration).

Rationale: a profile IS a (cron_expression, marker) pair. Both already have homes. Adding a `coach_profile` column would duplicate the marker as data — a §6 violation of single-source-of-truth.

Look-up patterns:
- Per-user list: `scheduledTasks.listByOwner(userId).filter(t => isCoachMarker(t.description))`
- Per-profile lookup: `scheduledTasks.listByOwner(userId).find(t => t.description === COACH_MARKER_BY_PROFILE[profile])`
- Aggregate "any profile active": same filter; `length > 0`.

`findCoachTask(memory, userId)` (v1.18.0) is RETAINED for back-compat reads but soft-deprecated; new code uses `findCoachTaskByProfile(memory, userId, profile)`.

## Decision 4: Chat command grammar

**Decision:** Extend `/coach` namespace; preserve back-compat aliases.

| Command | Behavior |
|---|---|
| `/coach on` | Alias → `/coach setup morning 08:00` (v1.19.0 back-compat) |
| `/coach off` | Alias → `/coach off all` (pauses every profile; preserves memory) |
| `/coach setup [profile] HH:MM` | profile defaults to `morning` if omitted; HH:MM required |
| `/coach setup weekly <day> HH:MM` | `<day>` ∈ `mon\|tue\|wed\|thu\|fri\|sat\|sun` (case-insensitive); cron `<m> <h> * * <0-6>` |
| `/coach off [profile\|all]` | profile defaults to `all` if omitted |
| `/coach status` | Multi-profile status (see format below) |
| `/coach quiet <duration>` | `2h` / `3d` / `until tomorrow` / `until monday` (case-insensitive) |
| `/coach quiet status` | Remaining quiet time |
| `/coach quiet off` | Clear quiet mode |

`/coach status` output format:

```
Morning: 8:00am ✅
Midday: not set
Evening: 7:00pm ✅
Weekly: Mon 9:00am ✅

Event triggers: ON
Quiet mode: not active
```

(If quiet active: `Quiet mode: active until <ISO> (3h 42m remaining)`)

**Parser strategy (binding):** regex + keyword tokens; **NO LLM** (per v1.19.0 R3 / W1 invariant — chat-command parsing is deterministic). Invalid HH:MM rejected with helpful error: `Couldn't parse "{input}" — use HH:MM (e.g. 08:00, 14:30) or '/coach status' to see all profiles.` Invalid day rejected similarly.

**Parser locality:** stays in `src/commands/coachSubcommands.ts`. The HH:MM / weekly-day parsing helpers live in `src/coach/profileTypes.ts` (D1). The duration parser for `/coach quiet` lives in a NEW pure helper `src/coach/rateLimits.ts.parseQuietDuration` (D8).

## Decision 5: Coach memory namespace unchanged across profiles

**Decision:** Profiles affect WHEN coach fires, not WHAT it remembers. Coach memory keys are profile-agnostic:

```
coach.<itemId>.lastNudge.<timestamp>   — shared across all profile fires
coach.<itemId>.research.<timestamp>    — shared
coach.<itemId>.idea.<timestamp>        — shared
coach.<itemId>.plan.<timestamp>        — shared
coach.<itemId>.fatigue                 — shared (3 ignored gentle nudges in 7d, regardless of which profile fired)
coach.<itemId>.userOverride.<...>      — shared (v1.19.0 D10)
```

Rationale: the LLM's mental model of "I have nudged X three times and they ignored every one" should not reset because we crossed midnight or a profile boundary. Fatigue counter is a property of the item, not of the cron schedule.

**Asymmetry to document in coachPrompt.md:** profile = WHEN; memory = WHAT.

The fatigue rule from v1.19.0 D13 (3 ignored gentle nudges → mark fatigued for 7d) is unchanged; it counts nudges across all profile fires.

## Decision 6: Event trigger architecture

**Decision:** Three trigger source modules in `src/coach/`. Each is pure-detect (returns a TriggerRecord OR null) + dispatches to a SHARED firing helper (D7).

### 6.a — `src/coach/itemStateMonitor.ts` (~150 LOC)

Hooked into `storage.updateItem()` via callback registry pattern (precedent: v1.19.0 D4 calendar-sync hook in `src/organize/storage.ts`). Registered at boot from `src/index.ts` via NEW `registerItemStateMonitorCallback(cb)`. Inspects post-write item state.

Trigger conditions (mutually exclusive — first match wins per item):

| Condition | Detection |
|---|---|
| `due-in-24h-no-progress` | `due` is set, `due - now ∈ [0, 24h)`, `progress.updatedAt < now - 48h` (or absent) |
| `goal-stale-14d` | `type === 'goal'`, `updated < now - 14d`, `status !== 'done'` |
| `persistent-zero-engagement-7d` | `coachIntensity === 'persistent'`, no `lastNudge` reply marked `engaged` in last 7d |
| `new-vague-goal` | `type === 'goal'`, age `< 24h`, title shorter than 8 tokens AND no `notes` |

Pure-detect: returns `TriggerRecord | null`. Caller dispatches.

### 6.b — `src/coach/chatMonitor.ts` (~200 LOC)

Hooked into `agent.turn()` post-turn callback. Registered at boot from `src/index.ts` via NEW `registerChatMessageCallback(cb)`. Receives the user's just-sent message + recent items (read via the same `listItems` injection used by the agent).

Regex parses for four pattern families (same trip-class as v1.19.0 NL override parser, different intent set):

| Family | Trigger condition |
|---|---|
| commitment | `\b(?:i'?ll\|i\s+will\|going\s+to\|gonna\|let\s+me)\s+\w` near a fuzzy-matched item title (≥0.7 score) |
| blocker | `\b(?:blocked\s+on\|stuck\s+on\|can'?t\s+make\s+progress\s+(?:on\|with))\b` near fuzzy match |
| procrastination | `\b(?:i\s+keep\s+(?:putting\s+off\|delaying)\|haven'?t\s+gotten\s+to\|been\s+avoiding)\b` near fuzzy match |
| completion-claim | `\b(?:done\s+with\|finished\|wrapped\s+up\|completed)\b` near fuzzy match (already in v1.19.0 OverrideIntentKind 'done_signal'; chat trigger surfaces as `done-signal-confirmation` to suggest user marks the item done) |

Fuzzy-match algorithm reused via NEW `src/coach/textPatternMatcher.ts` (~120 LOC) extracted from `userOverrideParser.ts` per Anti-Slop §6 single-source-of-truth.

### 6.c — `src/coach/calendarMonitor.ts` (~120 LOC)

Hooked into `src/calendar/sync.ts` reverse-sync entry path. Registered at boot from `src/index.ts` via NEW `registerCalendarEventCallback(cb)`. Fires AFTER reverse-sync creates an organize item from a Google Calendar event.

Trigger conditions:

| Condition | Detection |
|---|---|
| recurring-meeting-detected | reverse-sync event has `recurringEventId` set in calendar API response |
| standalone-meaningful-event | event title length ≥ 12 tokens OR description present, single-occurrence (no recurringEventId), `start - now > 1h` |

These are documented as low-priority triggers (see D11 priority order); recurring meetings often need prep notes; standalone meaningful events benefit from coach offering to track action items. **Both surfaces are passive** — coach offers to track, never auto-creates extra items.

### Trigger record shape (binding)

```ts
// src/coach/triggerFiring.ts
export interface TriggerRecord {
  source: 'item-state' | 'chat' | 'calendar';
  triggerType:
    | 'due-in-24h-no-progress' | 'goal-stale-14d' | 'persistent-zero-engagement-7d' | 'new-vague-goal'
    | 'commitment' | 'blocker' | 'procrastination' | 'done-signal-confirmation'
    | 'recurring-meeting-detected' | 'standalone-meaningful-event';
  itemId: string;
  reason: string;            // short structural string for audit (no content)
  fromMessageHash?: string;  // chat trigger only — sha256 of the user message; NEVER the message itself
  detectedAt: string;        // ISO
}
```

## Decision 7: Trigger-firing flow

**Decision:** NEW gateway path `gateway.fireSpontaneousCoachTurn(trigger: TriggerRecord)` that takes a coach turn directly without going through `scheduled_tasks`. Lighter than dynamic-create-then-delete (which would also create cron-table churn).

**No new sentinel marker for spontaneous fires.** The previous draft of D2 considered `'__coach_event_<trigger>__'`; that's REMOVED. Spontaneous fires are an in-process flow, not a row.

Flow:

1. **Rate-limit + quiet checks (D8 + D9 + D10).** If any blocks, audit `coach.event_trigger.suppressed` + return.
2. **Build coach prompt** with `${trigger_context}` template variable populated (trigger type + reason + focus item) — see D15.
3. **Call `agent.turn()`** with `coachTurnCounters` initialized to `{nudges: 0, totalWrites: 0}` AND a NEW `isSpontaneousTrigger: true` flag on `TurnParams`. The flag gates a few coach behaviors (focus on the single triggered item; don't re-pick from full active list).
4. **Coach proceeds normally:** research / plan / nudge / log nudge memory. Same allowlist as scheduled fires (v1.18.0 R6/F1 brake holds — `disabledTools` enforced at dispatcher).
5. **On success:** increment per-item `lastSpontaneousAt` + global daily counter (D8).
6. **Audit `coach.event_trigger.fired`** with structural-only metadata (trigger type, itemId, condition matched, reason). NO content (titles, notes, message body). v1.17.0 H gate carry-forward.

Implementation site: `src/gateway/index.ts` adds ~50 LOC alongside existing `enqueueSchedulerTurn`. Re-uses the same per-chat queue infrastructure; the queue already handles ordering + back-pressure.

## Decision 8: Rate limiting primitive

**Decision:** Three keyed-memory entries form the rate-limit primitive. All in `src/coach/rateLimits.ts` (~80 LOC).

| Key | Body | Read check |
|---|---|---|
| `coach.<itemId>.lastSpontaneousAt` | `{ "at": "<ISO>" }` | `now - at < 4h` blocks; else allow |
| `coach.global.spontaneousCount.<YYYYMMDD>` | `{ "count": <int> }` | `count >= 3` blocks; else allow |
| `coach.global.quietUntil` | `{ "at": "<ISO>" }` | `now < at` blocks; else allow |

Storage location: existing keyed-memory infrastructure in `src/memory/userMemoryEntries.ts`. The sentinel parser handles dotted keys per v1.18.0 R3/F3 (regex `[a-zA-Z0-9._-]{1,128}`). All three key shapes fit the regex.

API:

```ts
// src/coach/rateLimits.ts
export async function checkPerItemRate(userId: number, dataDir: string, itemId: string, nowMs: number): Promise<{ ok: true } | { ok: false; reason: 'PER_ITEM_BACKOFF'; retryAfterMs: number }>;
export async function checkGlobalDailyRate(userId: number, dataDir: string, todayYYYYMMDD: string): Promise<{ ok: true } | { ok: false; reason: 'GLOBAL_DAILY_CAP'; current: number; cap: number }>;
export async function checkQuietMode(userId: number, dataDir: string, nowMs: number): Promise<{ ok: true } | { ok: false; reason: 'QUIET_ACTIVE'; quietUntilIso: string }>;
export async function recordPerItemFire(userId: number, dataDir: string, itemId: string, nowIso: string): Promise<void>;
export async function recordGlobalDailyFire(userId: number, dataDir: string, todayYYYYMMDD: string): Promise<void>;
export async function setQuietUntil(userId: number, dataDir: string, untilIso: string): Promise<void>;
export async function clearQuietMode(userId: number, dataDir: string): Promise<void>;
export function parseQuietDuration(input: string, nowIso: string): { ok: true; untilIso: string } | { ok: false; error: string };
```

Constants (binding):

```ts
export const PER_ITEM_RATE_WINDOW_MS = 4 * 60 * 60 * 1000;  // 4h
export const GLOBAL_DAILY_CAP = 3;
```

`coach.global.spontaneousCount.<YYYYMMDD>` keys accumulate one per day with no cleanup. **Garbage collection:** the trash evictor (`src/organize/trashEvictor.ts` precedent) registers a daily sweep that deletes `coach.global.spontaneousCount.<date>` entries older than 30 days. Worst case: ~30 stale rows per user = trivial. Adding a cleanup-on-write would race with the read; once-daily eviction is simpler.

## Decision 9: Quiet mode semantics

**Decision:** Global flag, applies to ALL event triggers. **Does NOT mute scheduled cron coach.**

- User can still receive a morning DM during quiet hours — they explicitly scheduled morning.
- To mute scheduled coach: `/coach off morning` (or specific profile) instead.

Rationale: scheduled fires are USER intent expressed at schedule time. Spontaneous fires are SYSTEM intent. Quiet mode silences the system without overriding the user's expressed schedule.

This asymmetry is surprising at first; the `/coach quiet <duration>` reply must explain it:

```
Quiet mode active until 2026-04-26 23:59 UTC.
This silences event-driven coach nudges (item state changes, chat patterns, calendar events).
Your scheduled times still fire — use `/coach off [morning|midday|evening|weekly|all]` to mute those.
```

Documented in coachPrompt.md asymmetry note (D5) AND in user-facing /coach help text.

## Decision 10: Conversation trigger frequency + false-positive guards

**Decision:** Every user DM is parsed by `chatMonitor.ts` (regex match; ~1ms cost; on the post-turn callback path so it never blocks the turn).

**Threshold:** pattern match + fuzzy item match score ≥ 0.7 (reuse v1.19.0 `userOverrideParser.ts` algorithm via shared `src/coach/textPatternMatcher.ts` ~120 LOC, extracted per Anti-Slop §6 single-source-of-truth — KEEP one fuzzy-match implementation across the codebase).

**Cooldown:** triggers are SUPPRESSED if `now - coach.global.lastCoachDmAt < 30min`. This prevents feedback loops where coach DMs the user, the user replies, that reply re-triggers chat monitor, etc. New keyed entry `coach.global.lastCoachDmAt` (updated by every coach DM send — both scheduled + spontaneous).

**Multi-message buffering:** triggers run on each individual user message; no concatenation across messages. If two consecutive messages BOTH match, the second is suppressed by D12 debounce (60s after last user message).

**Audit posture:** chat-trigger reasons store `fromMessageHash` (sha256 of the message), NEVER the message text. v1.17.0 H gate.

## Decision 11: Trigger priority (Devil's Advocate adversarial)

**Decision:** Mutually-exclusive per fire — first match wins. Order:

1. **`back_off` user override** (highest priority — user explicitly said no; suppresses ALL triggers for the item until override expires)
2. **`push` user override** (user explicitly said push; the next trigger that fires for this item is treated as `persistent` for that one fire)
3. **`coach.<itemId>.fatigue`** (3 ignored gentle nudges; suppresses non-`persistent` triggers for 7d)
4. **`done_signal` user override** (suppresses standard triggers; permits `done-signal-confirmation` chat trigger)
5. **Standard triggers** (D6 conditions; first-match per item)

If MULTIPLE triggers match the same item simultaneously (e.g., item is both `due-in-24h-no-progress` AND just had a `commitment` chat match): the chat trigger wins (more recent user signal; more contextual). Item-state triggers are background; chat triggers are foreground.

Within item-state triggers: detection order is the order in D6 (due-in-24h > goal-stale > persistent-zero-engagement > new-vague-goal). First match wins.

Calendar triggers always fire if not blocked by overrides/fatigue — they don't compete with state/chat triggers because they're tied to a specific calendar-created itemId.

## Decision 12: Event trigger debounce

**Decision:** Triggers wait 60 seconds after the last user DM before firing. Avoids mid-message interruptions.

**Implementation:** keyed entry `coach.global.lastUserMessageAt` updated on every incoming user DM (in `src/agent/index.ts` post-turn). Trigger checks `now - lastUserMessageAt > 60s` before calling `gateway.fireSpontaneousCoachTurn`.

If the check fails, the trigger is **dropped, not deferred** — re-detection is cheap (next item write or chat message will re-fire detection). Deferring would create state churn (timer, cancellation, race with the second-message arrival).

Exception: the chat trigger itself is the SOURCE of `lastUserMessageAt` updates, so its 60s-window will always block its own immediate fire. The detect-then-defer-by-dropping behavior is correct: the next detection naturally happens 60s+ later.

## Decision 13: Calendar trigger interaction with reverse sync

**Decision:** Calendar reverse-sync ALREADY creates organize items from new events (v1.19.0 D5). Calendar trigger fires AFTER item creation completes — coach offers prep notes / action items.

**Order:**

1. Reverse-sync detects new Google Calendar event (`pollCalendarChanges`).
2. `syncCalendarEventToItem()` creates the organize item (existing v1.19.0 path).
3. The new item write fires `_calendarSyncCallback` → no-op for forward-sync of brand-new item (already in calendar).
4. The new item write ALSO fires `_itemStateMonitorCallback` → typically no trigger fires because the item is fresh.
5. **NEW:** the calendar reverse-sync entry path also fires `_calendarEventCallback(userId, item, source: 'reverse-sync')` → calendarMonitor inspects the event metadata (recurring? long description? meaningful start time?) → may produce a TriggerRecord.
6. If trigger produced AND rate limits permit → `gateway.fireSpontaneousCoachTurn` → coach DMs offering "Want me to track this with prep notes?"

The trigger and the item creation are sequenced (item exists when coach reads it), not parallel.

## Decision 15: Coach prompt template extension

**Decision:** Add a Step 0.5 to `src/coach/coachPrompt.md` BETWEEN existing Step 0 (read recent activity) and the existing item-selection logic.

```markdown
## Step 0.5 — Spontaneous trigger context (only when fired by event trigger)

When fired by an event trigger (NOT a scheduled cron tick), the trigger context will populate the
`${trigger_context}` template variable. Format:

```
Trigger source: <item-state|chat|calendar>
Trigger type: <one of D6 trigger types>
Focus item: <itemId>
Reason: <short structural string>
```

When the trigger context is populated:
1. **Focus on the single triggered item.** Do NOT re-pick from the full active list. The user's
   most recent signal is about this one item; respect that.
2. **Use the trigger reason to shape your nudge.** A `due-in-24h-no-progress` trigger calls for
   urgency + offering help breaking the task down. A `commitment` chat trigger calls for
   acknowledgement + a single follow-up question. A `recurring-meeting-detected` calls for
   offering prep notes.
3. **Per-turn write caps still apply** (≤5 nudges, ≤10 total writes per turn — the spontaneous
   path uses the same `coachTurnCounters` brake as scheduled fires).

When the trigger context is EMPTY (template variable not populated): you were fired by a
scheduled cron tick. Proceed with the existing Step 0 → Step 1 → multi-item picker flow.
```

**Implementation:** `src/coach/index.ts.expandCoachPromptToken` is extended to accept an optional second argument `triggerContext?: string`. When provided, it ALSO replaces a NEW placeholder `${trigger_context}` in the prompt with the populated string. When not provided, the placeholder is replaced with empty string.

```ts
export function expandCoachPromptToken(command: string, triggerContext?: string): string;
```

The new placeholder constant `COACH_TRIGGER_CONTEXT_PLACEHOLDER = '${trigger_context}'` is exported alongside `COACH_PROMPT_PLACEHOLDER`.

`coachPrompt.md` adds the literal `${trigger_context}` line near the top (above Step 0) so the trigger context is visible BEFORE the LLM enters the existing Step 0 monitoring loop. For scheduled fires, this expands to empty string and the existing Step 0 fires normally.

## R2 — Institutional memory (6 KI entries + 6 CLAUDE.md invariants)

**KNOWN_ISSUES.md additions** (under NEW header `## v1.20.0 Issues`):

1. **`COACH_PROFILES` closed set + marker convention extension.** Profiles ∈ `{morning, midday, evening, weekly}`. Each maps to `__coach_<profile>__` marker. v1.18.0's `__coach__` is migration-rewritten to `__coach_morning__` on first v1.20.0 boot. Adding a 5th profile = closed-set update + new marker constant + webapp UI button + migration not needed (additive). Static test asserts the array length (4).

2. **Profile sharing memory (WHEN vs WHAT distinction).** `coach.<itemId>.*` keyed memory is shared across ALL profile fires for a user. Fatigue counter accumulates across profiles. Profile = WHEN coach fires; memory = WHAT coach remembers. Adding a per-profile memory namespace (e.g., `coach.<profile>.<itemId>.fatigue`) was REJECTED in D5 — fatigue is a property of the item, not the schedule.

3. **Event trigger boot-wiring discipline (the v1.18.0/v1.19.0 trap-class fix).** Three new callbacks at boot: `registerItemStateMonitorCallback`, `registerChatMessageCallback`, `registerCalendarEventCallback`. Each wires from `src/index.ts` to its respective monitor module. Static test `tests/static/coach-event-wiring.test.ts` rejects identity stubs (`async () => undefined` and friends). Land the static test SCAFFOLD as commit 0a, BEFORE the wiring commits — the test fails until 12 lands real handlers; that's the point.

4. **Rate-limit primitives — three keyed-memory entries.** `coach.<itemId>.lastSpontaneousAt` (per-item 4h window), `coach.global.spontaneousCount.<YYYYMMDD>` (global daily cap of 3), `coach.global.quietUntil` (kill switch). All three are read on EVERY trigger fire (D8). The trash evictor sweeps stale `spontaneousCount.<date>` rows older than 30 days.

5. **Trigger priority order (override > fatigue > standard).** Mutually-exclusive per item per fire. `back_off > push > fatigue > done_signal > standard`. If multiple triggers match the same item simultaneously, chat triggers > item-state triggers (more recent user signal). Calendar triggers don't compete with state/chat (distinct itemId paths).

6. **Conversation trigger 30-min cooldown after coach DM (feedback-loop prevention).** `coach.global.lastCoachDmAt` updated on every coach DM (scheduled + spontaneous). chatMonitor suppresses trigger if `now - lastCoachDmAt < 30min`. Without this, a coach DM that elicits a "thanks, will do" reply re-triggers chatMonitor.commitment, which DMs again, ad infinitum.

**CLAUDE.md invariants additions** (under NEW header `## v1.20.0 Invariants (Multi-coach + event triggers)`):

1. **`COACH_PROFILES` is a closed set of 4 (RA1 invariant 1).** Defined in `src/coach/profileTypes.ts`. Static test `tests/static/coach-profile-closed-set.test.ts` asserts length 4. Adding a profile = update array + add marker constant + add webapp button + update migration if back-compat needed. Don't proliferate profiles ad-hoc; each one ships UI + cron + audit fan-out.

2. **Coach memory is profile-agnostic; profile = WHEN, memory = WHAT (RA1 invariant 2).** `coach.<itemId>.*` keys NEVER include the profile name. Adding profile-scoped memory (`coach.morning.<itemId>.lastNudge`) breaks the v1.18.0 fatigue rule (3 ignored gentle nudges across all profile fires). Reference: ADR 020 D5.

3. **Event trigger callbacks at boot must be non-stub (RA1 invariant 3 — trap-class fix, 4th iteration).** `tests/static/coach-event-wiring.test.ts` asserts `registerItemStateMonitorCallback` / `registerChatMessageCallback` / `registerCalendarEventCallback` are NOT registered with identity stubs (`async () => undefined`, `() => {}`, etc.). v1.18.0 had this trap with `coachTurnCounters` plumbing (3 fix iterations); v1.19.0 had it 3 ways with calendar (breaker, audit shims, boot wiring). v1.20.0 introduces 3 new callbacks; the lint pre-empts the trap. Reference: ADR 020 D17.

4. **Rate-limit reads are mandatory before every spontaneous fire (RA1 invariant 4).** `gateway.fireSpontaneousCoachTurn` MUST call `checkPerItemRate` + `checkGlobalDailyRate` + `checkQuietMode` BEFORE building the prompt. Failing to call any of the three opens the floodgate. Audit emission for suppressed triggers is part of the contract — silent suppression is forbidden. Reference: ADR 020 D7 + D8.

5. **Trigger priority (RA1 invariant 5).** `back_off override > push override > fatigue > done_signal override > standard triggers`. Within standard: chat > item-state > calendar (most-recent-user-signal-wins). The priority is enforced in the monitor modules' `detect()` return logic, NOT in the gateway. Reference: ADR 020 D11.

6. **`textPatternMatcher.ts` is the SOLE owner of fuzzy item-title matching + tokenization (RA1 invariant 6 — Anti-Slop §6).** `userOverrideParser.ts` (v1.19.0) imports `tokenize` + `jaccardScore` + `STOP_WORDS` + `FUZZY_MATCH_THRESHOLD` from `textPatternMatcher.ts`. `chatMonitor.ts` imports the same. NEVER reimplement tokenization or jaccard locally. Static test `tests/static/coach-textpattern-shared.test.ts` enforces. Reference: ADR 020 D10 + D16.

## Open questions for CP1 (DA + Anti-Slop reviewers)

1. **Marker migration idempotency.** D2's migration rewrites `__coach__` → `__coach_morning__` on first v1.20.0 boot. What if pm2 restarts mid-migration? Helper is single-`UPDATE` per row; SQLite WAL guarantees atomicity per row. But if user creates `__coach_morning__` AND `__coach__` exists (somehow — pre-migration binary running)? Migration helper should be conditional: skip if a `__coach_morning__` already exists for the user. **Architect commit:** make migration helper UPDATE only where `description = '__coach__'` AND no row exists with `description = '__coach_morning__'` for the same `owner_user_id`. Single SQL. Otherwise drop the legacy row + audit `coach.migration_skipped`.

2. **DA adversarial: feedback-loop window.** D10 sets 30-min cooldown after coach DM. Is 30 min long enough? An engaged user might respond within 5 min, and we WANT to ack. But chatMonitor doesn't ack — it triggers a coach turn. The 30-min cooldown means we won't re-trigger, but normal `agent.turn()` still happens (user gets a normal chat reply, not a coach DM). Trade-off: shorter cooldown = more chat-trigger fires but more risk of feedback loops. Defer to DA for adversarial test cases.

3. **Anti-Slop §15 defense-in-depth: chat trigger and prompt injection.** `chatMonitor.ts` runs regex on user messages — is that already covered by the existing v1.18.0 `<untrusted>` boundary? The user's own DM is NOT wrapped in `<untrusted>` (it's the user's intent, not external content). But we treat it as input to a regex match, not an LLM prompt. Trigger context that flows to the coach prompt embeds `triggerType` + `itemId` + structural reason — NEVER the message body. Confirm with Anti-Slop reviewer that the trigger context expansion is safe.

4. **Scalability: keyed memory growth from `coach.<itemId>.lastSpontaneousAt`.** One entry per item EVER triggered. At 100 active items + steady state, ~100 entries. Single-user; trivial. Multi-user (deferred): bounded same way per-user. OK.

5. **DA adversarial: what if user has 4 profiles + all 3 trigger sources active + global cap of 3/day?** Worst case = 3 spontaneous DMs/day + 4 scheduled DMs/day = 7 DMs. Compared to v1.19.0's 1 DM/day, that's 7x. Is that acceptable UX? The kill switch (`/coach quiet 7d`) is the escape hatch; the heavy-hammer hub banner button auto-engages it. Document the math in the v1.20.0 release notes so user is aware.

6. **Anti-Slop §6 (DRY): `parseHHMM` vs the v1.18.0 `hhmm_to_cron` helper.** The v1.18.0 helper is in `coachSubcommands.ts` as a private function. D1 puts `parseHHMM` in `profileTypes.ts`. Is this duplication? Resolution: `parseHHMM` returns `{hour, minute}`; `hhmm_to_cron` builds the cron string. Refactor `coachSubcommands.ts` to use `parseHHMM` internally — single tokenizer, two consumers. Filed as a sub-bullet of commit 0d.

7. **Anti-Slop §13 module boundaries: `calendar/syncTypes.ts` import from `calendarMonitor.ts`.** D6.c says `calendarMonitor.ts → calendar/syncTypes`. That's a forward edge (coach reads calendar types). v1.19.0 ADR 019 D14 forbids `calendar/** → coach/**` but allows `coach/** → calendar/**` (types only). Confirm this stays types-only.
