## D1 — `parentId` PATCH semantics (allowed values + validation rules)

**Decision.** `parentId` is added to `ALLOWED_PATCH_FIELDS` in `src/organize/validation.ts`. Accepted values:

  - A string matching `/^\d{4}-\d{2}-\d{2}-[a-z0-9]{4}$/` (the canonical item-id shape; same regex as `src/commands/organize.ts:122`).
  - `null` (explicit clear — item becomes top-level / orphan / standalone).
  - Field absent — leaves the existing `parentId` untouched.

Validation rules (in order; first failure short-circuits):

  1. **Format check.** Regex above OR `null`. Otherwise `PARENT_ID_INVALID_FORMAT`.
  2. **Self-reference forbidden.** If the supplied `parentId` equals the path-param `:id`, reject with `PARENT_ID_SELF_REFERENCE`.
  3. **Cross-type forbidden (mirrors create-time R13).** If the item being PATCHed is itself a goal AND the patch sets `parentId` to non-null, reject with `GOAL_CANNOT_HAVE_PARENT`. Reuses the code already shipped by v1.14.3 `organize_create`. Determined by reading the current item's type from disk (existing `readItem` call at the handler level — no new IO).
  4. **Cross-user forbidden.** By construction: validation runs against the requesting user's data dir only. No additional check needed.
  5. **Existence check.** When non-null and format-valid, the target id must exist as a live (not trashed) goal in the user's dir AND its status must NOT be `'abandoned'`. Otherwise `PARENT_NOT_FOUND` (when missing or trashed) or `PARENT_NOT_GOAL` (when present but `type !== 'goal'`).

**Why reject 'abandoned' goals as parents.** The v1.14.3 hierarchy renderer treats `status='abandoned'` items as still-rendered (just badge-styled differently). An abandoned goal CAN have children today; the system tolerates dangling parentId references (D3 below), but accepting a NEW reparent to an abandoned goal is semantically an anti-pattern: the user is moving an active task under a goal they've explicitly given up on. The right UX is "abandoned goals are dead-ends — re-parent to an active goal, or clear the parent." If the user genuinely wants to revive the goal-task pairing, the workflow is: first reactivate the goal (PATCH `status: active`), then re-parent the task. The picker (D4) reflects this by showing only active goals.

**Validator codes (added to `ValidatorErrorCode` union).**

  - `PARENT_ID_INVALID_FORMAT`
  - `PARENT_ID_SELF_REFERENCE`
  - `PARENT_NOT_FOUND`
  - `PARENT_NOT_GOAL`
  - `PARENT_NOT_ACTIVE` (new: distinguishes abandoned-goal rejection from missing-goal rejection — forensic clarity)
  - `GOAL_CANNOT_HAVE_PARENT` (NOT new — the same string code is already used by `organize_create` R13; the wire shape is "code: string" so reuse is automatic; tests assert the literal value once across the codebase)

**`AllowedPatch` interface gains `parentId?: string | null`.**

The `null` semantic is "clear the parent." Absence is "leave untouched." Mirrors `due` field's existing tri-state at `validation.ts:217-230`.

**Options weighed.**

  - Allow abandoned goals as parents: rejected (UX argument above; no concrete user demand).
  - Allow re-parenting under done goals: ACCEPTED. Done goals are completed achievements; sub-items continuing under them is meaningful (e.g., sub-tasks completed alongside the goal). The 'active OR done' set is the picker's source.
  - Allow free-text id input: rejected (D4 picker covers it).

## D2 — Where the existence check runs (validator vs storage vs route helper)

**Decision: Option C — shared async helper.** A new exported function `parentExistsAndIsActiveGoal(userId, dataDir, parentId): Promise<{ ok: true } | { ok: false; reason: 'NOT_FOUND' | 'NOT_GOAL' | 'NOT_ACTIVE' }>` lives in `src/organize/storage.ts`. The PATCH route handler in `items.mutate.ts` calls it AFTER `validatePatchBody` returns ok and BEFORE invoking `updateItem`. The validator stays purely synchronous and IO-free.

**Options weighed.**

  - **Option A — validator-time existence check.** Couples the validator to file IO. Validator becomes async. Breaks the "validators are pure functions" invariant established by v1.14.2 W3. Below the bar.
  - **Option B — storage-time check inside `updateItem`.** Mixes the existence-of-parent semantic into a primitive that already handles the (read, patch, write) atom. Adds a second disk read (the parent's own file) inside an operation that today reads only the patched item. Storage primitives stay focused; this is route-level coordination. Below the bar.
  - **Option C — shared async helper called by route handler.** Validator stays pure (format + self-ref only). Storage stays focused. Route handler is the natural orchestrator. Helper is exported from storage.ts because that's where the IO neighbors live (mirrors `readItem`, `readItemFrontMatter` precedent). PICK.

**Helper signature (binding for D13 location):**

```typescript
// src/organize/storage.ts — new exported function

export type ParentRefRejection = 'NOT_FOUND' | 'NOT_GOAL' | 'NOT_ACTIVE';

export interface ParentRefResult {
  ok: boolean;
  reason?: ParentRefRejection;
}

/**
 * Verify that `parentId` references an existing, non-trashed goal whose status
 * is 'active' or 'done' (NOT 'abandoned' — D1 rationale).
 *
 * Reads only the parent item's file via readItemFrontMatter. Does NOT call
 * readItem (notes/progress not needed). Returns NOT_FOUND for a missing or
 * trashed file, NOT_GOAL for a non-goal item, NOT_ACTIVE for an abandoned goal.
 */
export async function parentExistsAndIsActiveGoal(
  userId: number,
  dataDir: string,
  parentId: string,
): Promise<ParentRefResult>;
```

**Call site (`src/webapp/items.mutate.ts`, immediately after `validatePatchBody`):**

```typescript
if (validated.parentId !== undefined && validated.parentId !== null) {
  // Self-reference is caught by the validator (D1 rule 2) before this point.
  // Cross-type (the patched item is itself a goal): caught by reading
  // currentItem before the helper call (one readItem call already needed
  // for the goal-cannot-have-parent guard — see D1 rule 3).
  const ref = await parentExistsAndIsActiveGoal(userId, dataDir, validated.parentId);
  if (!ref.ok) {
    const code = ref.reason === 'NOT_FOUND' ? 'PARENT_NOT_FOUND'
               : ref.reason === 'NOT_GOAL'  ? 'PARENT_NOT_GOAL'
               : 'PARENT_NOT_ACTIVE';
    res.status(400).json({ ok: false, code, error: <appropriate message> });
    return;
  }
}
```

The cross-type guard (D1 rule 3 — `GOAL_CANNOT_HAVE_PARENT`) requires reading the patched item itself to check its `type`. This is one extra `readItem` call PER PATCH that includes `parentId`. The PATCH handler today does NOT read the item before calling `updateItem` — `updateItem` reads it internally and stamps. We could either: (a) add a second read in the handler (one for type-check, then `updateItem` reads again); (b) extend `updateItem` to surface the pre-patch type via the error/return shape (couples concerns); (c) add a thin `readItemFrontMatter` call (cheap — front-matter only, ~80 bytes). Pick (c). Adds ~5 LOC to the handler; cost is one front-matter parse per parentId-bearing PATCH (rare relative to total PATCH volume).

## D3 — TOCTOU between parentId existence check and PATCH commit

**Decision: ACCEPT.** A user-or-agent could delete the parent goal between `parentExistsAndIsActiveGoal` returning `{ok:true}` and `updateItem` writing the new parentId. The result: a stored child references a goal that was just trashed. Acceptable because:

  1. v1.14.3 hierarchy renderer (`hierarchy.js` `groupByParent`) already handles dangling parentId references: the orphaned child renders top-level. No UI surprise.
  2. The race window is microseconds (one front-matter read + one rename in `softDeleteItem`); single-user posture today; in practice the user can't out-race themselves.
  3. The parent-delete via webapp DELETE today does NOT cascade-delete or re-parent children. Today's DELETE leaves children dangling already; this is consistent with status-quo.
  4. Locking would add complexity (per-user mutex; FS-level locking; coordination with chat-tool `softDeleteItem` calls). Re-checking on write would double the IO. Both below the bar for v1.14.5.

**Options weighed.**

  - **Accept the dangling-reference outcome.** PICK. Documented; renders as orphan; user can re-parent if needed.
  - **Per-user mutex around (existence check, write).** Adds a Map-based async lock. Coordinates with chat-side `softDeleteItem` only if both share the lock. Multi-process posture (if pm2 cluster ever ships) breaks the lock anyway. Skip.
  - **Storage-layer re-check.** `updateItem` reads the parent's file as a second IO. Cost: 2x storage IO per parentId-bearing PATCH. Skip.

**Documentation binding.** The PATCH handler comment block notes this trade-off explicitly: "TOCTOU race: parent could be trashed between existence check and write. Hierarchy renderer treats orphan children as top-level (`hierarchy.js` v1.14.3); the dangling reference is observable but not user-fatal."

## D4 — `parentId` editing UI: picker shape

**Decision: native `<select>`.** A `<select id="edit-parent-id">` is added to the edit form between the Status fieldset and the Tags input (D5). The `<option>` list is `[<option value="">— None (top level) —</option>, ...active+done goals]`. `value=""` represents the "clear parent / orphan" choice; mapped to `null` in the PATCH body. Active goal items are listed by `goal.title` text (textContent only; same security posture as the tag rendering at app.js:382-388).

**Source of goals.** A separate `GET /api/webapp/items?type=goal&status=active` call is fired ONCE on edit-mode entry per detail session — cached in a module-level variable until the user navigates back to the list (where the cache is invalidated). The fetch returns the LIST projection (`OrganizeListItem`), which already includes `id`, `title`, and `type`. Status is filtered server-side by the existing `?status=active` query.

**Why this picker, not the others.**

  - **Custom searchable dropdown.** Over-engineered for v1.14.5 single-user posture; goal counts are typically <20; sort + first-letter typeahead in a native `<select>` is sufficient. Skip.
  - **Free-text id input.** UX hostile (users don't memorize `2026-04-25-ab1c`). Skip.
  - **Goals from `renderedItems` only.** Insufficient: if user filtered to status='done' or type='task', the active goals aren't in the list. The picker would be empty or sparse. Skip.

**Active-only fetch rationale (matches D1 picker semantics).** The picker offers only `status='active'` goals because re-parenting under a done goal is meaningful but uncommon, and adding it dilutes the picker UX. POST-v1.14.5 follow-up: if users complain, expand to `status=active OR done`.

**Edge case: 0 goals.** Picker shows only "— None (top level) —"; the user cannot set a parent. Currently-set parent (if any) is preserved as a stale option with title "(current — id)" so the form doesn't silently un-parent. Documented in D5.

**Edge case: current parent is abandoned.** The form's currently-set value is shown as `<option value="<id>" selected>(current: title) — abandoned</option>`. The user can KEEP it (unchanged PATCH) or CHANGE it. Server-side validator only runs the active-goal check when the value DIFFERS from the current `parentId` in the front-matter — i.e., we permit "leave a stale ref alone" but reject "set a NEW ref to abandoned." The handler determines this by comparing `validated.parentId` to `currentFm.parentId` (already read for the goal-cannot-have-parent check, D2 call site). One extra `if` in the handler.

## D6 — `/organize trash list` shape

**Decision.** A new chat subcommand. Dispatch added to `src/commands/organize.ts:96` block (alongside `restore`). Optional first arg is an integer offset (default 0).

**Per-entry data shown.**

  - Title (privacy posture: titles ARE shown — same as `findClosestTrashedIds` already shows them at `:494`).
  - Type icon prefix (🗑 for visual marker, then type's icon).
  - Type label.
  - Deleted-relative ("3d ago").
  - Item id (monospaced for copy-paste into `/organize restore <id>`).

**Sort.** `deletedAt` descending (most recent first). Falls back to mtime when `deletedAt` is missing (legacy pre-v1.11.0 trash). If both missing, sort by filename id (stable, low signal but predictable).

**Pagination.** Cap at 50 per reply. If more, append: `Showing 1-50 of <N>. Use /organize trash list 50 for the next page.` Offset advances by 50 each page.

**Reply format (Markdown → existing `markdownToTelegramHtml` pipeline):**

```
**Trash (12 items):**

🗑 📌 Clean garage — deleted 3d ago
   id: `2026-04-22-x7j2`

🗑 ⚑ Q1 wiki rewrite — deleted 5d ago
   id: `2026-04-20-a4q1`

[...]
```

**Empty case.** Reply: `Trash is empty.`

**Edge cases.**

  - Mixed live and trash dirs (FS inconsistency from `restoreItem` collisions): listing only reads `.trash/`; live-and-trash collisions produce a duplicate-id warning surfaced via log, not chat reply.
  - Malformed front-matter file in trash: tolerated. `parseItemFile` returns `{ok:false}`; the helper falls back to filename id + "(unreadable)" title + `mtime` for date. Same posture as `findClosestTrashedIds` at `:572-576`.
  - Files matching the rename-collision suffix (`<id>--<unix>-<hex>.md` per storage.ts:842-845): the helper extracts the base id (everything before `--`) for the reply, falls back to full filename if no `--` separator. Documented.

**Options weighed.**

  - Show titles: PICK. Privacy is per-user (single-user posture); chat is DM-only; v1.14.3 `restoreItem` 404 path already shows titles via `findClosestTrashedIds`. Consistent.
  - Hide titles (id-only): considered; rejected. The user's claim "I want to know what's in trash" is the use case; ids alone are unreadable.
  - Inline keyboard with restore buttons: considered; rejected. v1.14.3 `restoreItem` is a typed command and works fine; adding callback-button surface introduces gateway-callback wiring (`gateway.ts` updates), which is out of scope. v1.14.6+ may add this.

## D7 — `listTrashedItems` storage primitive

**Decision.** A new exported helper in `src/organize/storage.ts`:

```typescript
export interface TrashedItemSummary {
  id: string;                       // base id (suffix-collision suffix stripped)
  fileBasename: string;             // full filename (for diagnostic; may include --<unix>-<hex>.md)
  title: string;                    // from front-matter; "(unreadable)" on parse failure
  type: OrganizeType;               // task | event | goal
  deletedAt: string;                // ISO from front-matter; falls back to mtime ISO; "(unknown)" if neither
  originalStatus: OrganizeStatus;   // status pre-delete (from front-matter); 'active' fallback
}

/**
 * List trashed items for a user.
 *
 * Reads `data/organize/<userId>/.trash/`. Each .md file is parsed via the
 * existing tolerant parser. Parse failures are surfaced as `{title:"(unreadable)"}`
 * entries (not omitted — the user needs to know they exist).
 *
 * Sorted: deletedAt desc; ties broken by fileBasename ascending.
 *
 * @param options.limit  Max entries to return (default 50; cap 200).
 * @param options.offset Number of entries to skip (default 0).
 */
export async function listTrashedItems(
  userId: number,
  dataDir: string,
  options?: { limit?: number; offset?: number },
): Promise<{ items: TrashedItemSummary[]; total: number }>;
```

**Implementation outline.**

  1. `ensureUserDir` (creates if missing); skip if no `.trash/`.
  2. `readdir` `.trash/`; filter `.md`; for each filename, derive `baseId` = everything before `--` if present, else `filename.slice(0,-3)`.
  3. For each: `readItemFrontMatter(filePath, baseId)` — tolerant of malformed.
  4. If parse failed: synthesize `{id: baseId, title: '(unreadable)', type: 'task' (default), deletedAt: <mtime ISO>, originalStatus: 'active'}`.
  5. Sort by `deletedAt` desc.
  6. Total = full sorted list length (used by D6 pagination "Showing 1-50 of N").
  7. Slice by offset/limit.
  8. Return.

**LOC estimate.** ~70 LOC.

**Test coverage.** Happy (3 items, descending sort), empty (no `.trash/`), legacy (no `deletedAt` → mtime fallback), malformed file (1 unreadable + 2 ok, all returned), collision-suffix file (`<id>--<unix>-<hex>.md` parsed correctly), pagination (offset=2 limit=2 over 5 items returns items 3-4).

## D8 — BroadcastChannel cross-tab sync (channel name + payload + listener policy)

**Decision: single channel `organize-mutations`; banner-on-detail; silent-refetch-on-list.**

**Channel name.** `organize-mutations`. Single-user app today; no per-user scoping. If multi-user ever ships, scope to `organize-mutations:{userId}` in one location (one constant + one fetch site). Documented as a pre-emptive scaling note.

**Message payload (JSON-stringified per `BroadcastChannel.postMessage` defaults):**

```typescript
interface OrganizeMutationMessage {
  kind: 'patch' | 'delete' | 'complete';
  itemId: string;            // the mutated item's id
  newEtag: string | null;    // server's response ETag (delete → null since no item)
  ts: number;                // Date.now() of the post — consumer uses to dedupe stale replays
  parentId?: string | null;  // if PATCH changed parentId — receivers can avoid full list-refetch in some paths
  changedFields?: string[];  // PATCH only — for receiver-side optimization (banner vs silent)
}
```

**Sender (sender side per `app.js`):** PATCH/DELETE/POST-/complete success handlers post ONE message after the response is parsed and local state is updated. Sender does NOT post on failure (412, network error, validation reject). Consumers see only successful mutations.

**Listener policy.**

  - **Detail view (currentDetailItem matches itemId):** show non-blocking banner: "This item was just updated in another tab. [Reload] [Dismiss]". Reload triggers `fetchAndShowDetail(itemId)`. Dismiss hides the banner. The banner is a separate `<div id="bc-banner">` element styled like the existing conflict panel.
  - **Detail view (currentDetailItem mismatches OR null):** ignore. Other tabs' work doesn't concern this tab's view.
  - **List view (no detail open):** silent refetch via the existing `loadItems()` function. No banner; the list re-renders with fresh data; user sees no input loss because no input is open.

**Why banner-on-detail (vs auto-refetch / silent-overwrite / conditional).**

  - **Banner always (detail).** PICK. Predictability over freshness; user-initiated refresh; no surprise input loss while editing.
  - **Auto-refetch always.** REJECTED. User mid-edit could see their textarea blow away. Bad UX.
  - **Invalidate ETag.** REJECTED. `currentDetailEtag = null` causes the next save to send no `If-Match` → server-side silent overwrite path. Worse than the conflict-panel UX v1.14.4 just shipped. Bad.
  - **Conditional (banner if editing, refetch if read-only).** Considered. Adds branching on `editFormEl.hidden`. Marginal UX gain (read-only viewers see fresh data faster) for added complexity. The list-view silent-refetch already covers most "freshness" cases. Skip.

**Sender-side guard.** A tab that posts a message ALSO receives it (the BroadcastChannel API echoes locally by default? No — it does NOT echo to the same tab. From MDN: "Messages are not sent to the source that posted them." Verified with Phase 2 unit test). No filtering needed. If the spec ever changes, the `ts` field disambiguates.

**Edge case: detail view shows itemA; banner already up; user opens itemB; channel fires for itemA again.** The banner targets `currentDetailItem.id`. When `itemB` is opened, the banner is dismissed (state cleared in `fetchAndShowDetail`). New itemA event arrives, doesn't match itemB, ignored. Correct.

**Edge case: channel fires DURING a 412 conflict panel display.** The conflict panel and the BC banner share screen real estate. Decision: BC banner is suppressed when conflict panel is visible (`!conflictPanelEl.hidden`). The conflict panel's own Reload action fetches fresh data; redundant BC banner is noise.

## D10 — BroadcastChannel availability fallback

**Decision: feature-detect; fall back to no-op.**

```javascript
let bcChannel = null;
if (typeof BroadcastChannel !== 'undefined') {
  try {
    bcChannel = new BroadcastChannel('organize-mutations');
    bcChannel.addEventListener('message', handleBroadcastMessage);
  } catch (err) {
    bcChannel = null; // suppressed; logging via console.warn (vetted with CSP — same-origin script)
  }
}
```

When `bcChannel === null`:

  - Sender-side `broadcastMutation()` is a no-op.
  - Receiver-side: no listener; tab does not auto-refresh on other-tab activity.
  - Status quo before v1.14.5; no regression.

**Affected environments (per MDN compat table, current as of 2026-04):**

  - All current desktop browsers: supported.
  - Safari: 15.4+ (released 2022-03). Older iOS may lack it.
  - Telegram WebApp on iOS uses WKWebView; current iOS 17+ inherits modern Safari support.
  - Telegram WebApp on Android uses WebView (Chrome-based); supported.

**Audit posture.** The fallback is silent — no audit row, no toast. The user does not know the feature is degraded. Phase 2 may add a one-time `console.info` for diagnostic visibility.

## D11 — Audit row impact

**Decision: NO new audit category. NO schema migration.** `parentId` is treated as just another field in the existing `webapp.item_mutate` audit detail's `changedFields` array. When a PATCH changes `parentId`, the audit row's `changedFields` includes `'parentId'` alongside `'title'`, `'tags'`, etc. The detail JSON shape from ADR 012 D10 is unchanged.

**`/organize trash list`.** Read-only; no audit row written. Same posture as `/organize all` and `/organize tasks` today. Documented.

**BroadcastChannel.** Client-side messaging; no server audit surface. Sender posts AFTER server PATCH success, and the PATCH success path already wrote a `webapp.item_mutate` row — that row is the canonical audit trail for the mutation. The cross-tab notification is UI-only.

## D12 — `items.mutate.ts` split decision (Anti-Slop W2 watchlist)

**Decision: defer.** Current LOC: 525. v1.14.5 adds:

  - ~5 LOC: front-matter type-check helper invocation (D1 rule 3).
  - ~6 LOC: `parentExistsAndIsActiveGoal` invocation + error mapping (D2).
  - ~3 LOC: `parentId` field added to `storagePatch` explicit-copy block (mirrors lines 207-208).
  - ~3 LOC: BroadcastChannel `broadcastMutation()` post-success hook (3 sites — PATCH, DELETE, POST /complete).

Total delta: ~17 LOC. Projected total: ~542 LOC. Anti-Slop §13 soft threshold is 500; the v1.14.4 ADR-acknowledged margin remains.

**Why defer the split.**

  - The PATCH/DELETE/POST-/complete handlers are cohesive (all mutate the same resource; share the same If-Match/X-Force-Override/conflict-tracker plumbing). Splitting them would duplicate ~30 LOC of header parsing across two files.
  - The natural split driver would be a NEW resource (e.g., `/api/webapp/goals` separate from items), not field-count growth. That driver does not exist in v1.14.5.
  - W2 watchlist explicitly tolerates the margin until ~600 LOC OR until a new resource lands.

**Re-evaluation trigger (added to `KNOWN_ISSUES.md` follow-up table per D15).** If `items.mutate.ts` crosses 600 LOC OR a v1.14.6+ iteration adds a non-items webapp mutation surface, split into `items.mutate.patch.ts` + `items.mutate.delete.ts` + `items.mutate.complete.ts` with shared helpers in `items.shared.ts`.
