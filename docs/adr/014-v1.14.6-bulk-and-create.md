# ADR 014 — v1.14.6 picker tier completion (trash.ts extraction + multi-select bulk actions + create-new-item form)

**Status:** Proposed (CP1 pending). Phase 2 is gated on Devil's Advocate + Anti-Slop Phase 1 review.
**Parent line:** ADR 010 (v1.14.2 mutations) → ADR 011 (v1.14.3 notes/progress/hierarchy) → ADR 012 (v1.14.4 ETag) → ADR 013 (v1.14.5 picker tier) → **ADR 014 (v1.14.6, this doc)**.
**Closes:** R5 (deferred from ADR 013 — `src/organize/trash.ts` extraction; trigger fired at v1.14.5 close — storage.ts at 1391 LOC > 1300 threshold). Closes the bulk-actions and create-form gaps in the webapp surface.

---

## 0. Context

v1.14.5 shipped the picker tier (parentId PATCH + `/organize trash list` chat command + BroadcastChannel cross-tab sync) and brought the webapp to feature parity with the chat-side organize commands except for two structural holes:

  1. **No bulk actions in the list view.** Completing or deleting 10 items is 10 individual taps. Re-parenting 10 tasks under a goal is 10 individual edit-mode visits. This is the most-asked-for affordance for users with > 5 active items.
  2. **No create-new-item form.** Adding an item requires dropping into the Telegram chat ("add a task: …"). The webapp's empty-state and footer both currently point users back to chat. The asymmetry is jarring once a user has internalized the webapp as their organize surface.

v1.14.5 also closed with `src/organize/storage.ts` at 1391 LOC. ADR 013 R5 specified an explicit re-evaluation trigger: extract `src/organize/trash.ts` when storage.ts crosses 1300 LOC OR v1.14.6 starts. Both triggers fired (1391 > 1300; v1.14.6 starts now). ADR 014 lands the trash.ts extraction as the FIRST commit of Phase 2, cleanly before the new bulk + create work goes in. ZERO logic change; mechanical line moves; tests update import paths only.

The three deliverables compose cleanly:

  - **trash.ts extraction (server, mechanical).** Phase 2 commit 1. Lifts ~280 LOC of trash-domain code out of storage.ts (1391 → ~1110) and out of commands/organize.ts (758 → ~680) into a new `src/organize/trash.ts` (~330 LOC including header/imports/JSDoc). No behavior change. Closes ADR 013 R5.
  - **Bulk actions (client-side).** Phase 2 commit 2-3. Multi-select state + bulk dispatch UI in `public/webapp/organize/app.js`. Server is unchanged — bulk operations are N parallel single-item PATCH/DELETE/POST `/complete` calls, throttled to 10 in-flight via a client-side promise pool. No new endpoints. Audit forensics granularity preserved (one row per item; per-item ETag/forced/bypassAfter412).
  - **Create form (client + server).** Phase 2 commit 4-5. NEW `src/webapp/items.create.ts` (POST /api/webapp/items handler, ~150 LOC) symmetric with items.complete.ts. NEW `validateCreateBody` in validation.ts. Inline create form at top of list view, toggled by "+ New" button. Reuses parent picker logic from v1.14.5. Reuses existing storage primitive `createItem` (storage.ts:414).

This ADR makes 18 decisions (D1–D18). The chronological ordering of impact for Phase 2:

  1. **D1, D2, D6, D7, D8, D17 (server-side):** trash.ts extraction; create-endpoint placement + audit category + body schema. Land first.
  2. **D9–D16, D18 (client-side):** multi-select UX + create form UX + dedup + double-submit guards. Land in parallel with server work after D1.
  3. **D3–D5 (server posture toward bulk):** N parallel calls; throttle is client-side; partial-failure UX is client-side. Server unchanged.

## D1 — trash.ts extraction scope

**Question.** Which trash-domain symbols move from `storage.ts` (and `commands/organize.ts`) into the new `src/organize/trash.ts`?

**Options weighed.**

  - **A (recommended).** Move four functions: `listTrashedItems` (storage.ts:1301), `evictExpiredTrash` (storage.ts:955), `restoreItem` (storage.ts:1115), and `findClosestTrashedIds` (relocated from `src/commands/organize.ts:563`). Move `ensureTrashDir` (storage.ts:76) since it's only called by trash-touching ops (verified: 4 call sites — softDeleteItem, restoreItem, evictExpiredTrash plus an indirect existence check, all trash-domain). KEEP `softDeleteItem` (storage.ts:765) in storage.ts: it is the CORE CRUD primary alongside createItem/readItem/updateItem; it WRITES to trash but the operation is "soft delete a live item," not "operate on trash as source of truth." Storage still owns the live-dir → trash transition; trash.ts owns the trash-as-source-of-truth lifecycle (read/restore/evict).
  - **B.** Move `softDeleteItem` too. Cohesive ("all trash-touching ops together") but breaks the live-CRUD primitives apart and the storage.ts `createItem`/`readItem`/`updateItem`/`softDeleteItem` symmetry that 8 call sites in commands/, agent tools, and tests depend on (refactor cost is large for marginal cohesion gain).
  - **C.** Move only the trash-as-source-of-truth ops (listTrashedItems, evictExpiredTrash, restoreItem); leave `findClosestTrashedIds` in `commands/organize.ts`. Keeps the chat-command file self-contained but leaves a piece of trash-domain code stranded outside trash.ts and forces commands/organize.ts to keep its `readdir` import.

**Decision — A.** Four functions move (listTrashedItems, evictExpiredTrash, restoreItem, findClosestTrashedIds). `softDeleteItem` stays in storage.ts.

**Helper-symbol disposition:**

  - `ensureTrashDir` (storage.ts:76) → moves to trash.ts as a private helper. softDeleteItem (which stays in storage.ts) imports it from trash.ts. This creates a controlled storage.ts → trash.ts dependency edge; storage.ts already imports from `./etag.js` so the directional import is normal.
  - `ITEM_FILE_RE` (the `^\d{4}-\d{2}-\d{2}-[a-z0-9]{4}\.md$` regex) — verified absent from current storage.ts source (the closest is `PARENT_ID_RE` in validation.ts and `ITEM_ID_RE` in items.shared.ts; neither covers the `.md`-suffix filename form). `listTrashedItems` and `findClosestTrashedIds` use plain `.endsWith('.md')` filtering today (storage.ts:1316 + commands/organize.ts:578); we keep that pattern in trash.ts. NO new constant introduced; NO new shared constants module.
  - `readItemFrontMatter` (storage.ts:914) — stays in storage.ts (used by parentExistsAndIsActiveGoal and many other paths). trash.ts imports it.
  - `readItemFrontMatterFromPath` (storage.ts:1195) — stays in storage.ts (used outside trash by items.read.ts via no path; verified by grep — only commands/organize.ts:587 in `findClosestTrashedIds` calls it). Even so it stays in storage.ts because the public-API surface area is published via storage's existing exports. trash.ts imports it.
  - `parseItemFile` (storage.ts:192, private) — stays in storage.ts. evictExpiredTrash uses it inline at storage.ts:1012; we replace that inline use with a call to a helper that trash.ts can use. **Decision:** Add a new TINY exported helper `parseItemFileFromRaw(raw, filenameId)` in storage.ts that wraps the existing private `parseItemFile`. trash.ts imports it. This avoids moving the parser and keeps the existing parseItemFile symmetric across all callers (live + trash).
  - `OrganizeFrontMatter`, `OrganizeStatus`, `OrganizeType` types — already exported from `./types.js`; trash.ts imports from there.

**Justify A.** R5 of ADR 013 explicitly preferred Option A reasoning ("listTrashedItems + evictExpiredTrash + findClosestTrashedIds + restoreItem form a trash-domain group"); ADR 013 R5 deferred only because v1.14.5 already had three deliverables. The deferral hit its trigger; A is what was always intended. softDeleteItem-stays preserves the storage CRUD-primitives quartet which is a separate cohesion axis from trash-domain.

**LOC accounting (D1).**

  - storage.ts: 1391 LOC HEAD → 1391 − 70 (listTrashedItems body+JSDoc) − 130 (evictExpiredTrash body+JSDoc) − 70 (restoreItem body+JSDoc) − 16 (ensureTrashDir + JSDoc) + 8 (parseItemFileFromRaw export wrapper) + 4 (import line for ensureTrashDir from trash.ts) = **~1117 LOC** (well below the 1300 threshold and ~285 LOC under HEAD).
  - commands/organize.ts: 758 LOC HEAD → 758 − 35 (findClosestTrashedIds + JSDoc) − 2 (readdir import line if no other use) + 1 (import findClosestTrashedIds from trash.ts) = **~722 LOC**. (existsSync stays — used elsewhere; verified.)
  - trash.ts (NEW): 70 + 130 + 70 + 35 + 16 + ~30 (header + imports + module-comment) = **~351 LOC**.
  - **Net source LOC delta: roughly 0** (file boundaries shift; total LOC changes by +20 to cover the parseItemFileFromRaw wrapper + import lines). Mechanical extraction.

## D2 — Bulk pattern: client-fired N parallel requests vs server-side bulk endpoint

**Question.** Bulk complete / bulk delete / bulk re-parent — does the client send N independent HTTP requests, or do we add a `POST /api/webapp/items/bulk` endpoint that takes an array?

**Options weighed.**

  - **A (recommended).** N parallel client-fired requests. Each item is a separate REST resource with its own ETag, audit row, conflict-tracker entry, and 412 response path. The client does the orchestration.
  - **B.** Server-side bulk endpoint. Body shape `{op: 'complete'|'delete'|'reparent', ids: string[], parentId?: string|null}`. Server iterates internally. One request, one audit row (or one big composite audit row), one response with per-item status array.
  - **C.** Hybrid — a server-side bulk endpoint that internally fans out to the existing handlers (composes them) and returns per-item status. Cleaner audit (per-item rows still emitted internally) but adds a new endpoint surface to maintain.

**Decision — A.** N parallel client-fired requests, throttled by D3, surfaced by D4.

**Justify A over B and C.**

  1. **Symmetry with existing API surface.** Every item is a REST resource with its own ETag; PATCH /:id, DELETE /:id, POST /:id/complete each round-trip exactly one item. Bulk-as-N-requests preserves that contract. A new bulk endpoint introduces a second mutation pattern that future iterations have to either keep symmetric or document the asymmetry. Maintenance debt.
  2. **Audit forensics granularity preserved.** The webapp.item_mutate audit row carries `changedFields`, `etag`, `forced`, and `bypassAfter412` per item (per ADR 010 decision 9 + ADR 012 R2). A bulk endpoint either emits N rows internally (in which case why have the endpoint?) or emits a composite row that loses per-item audit fields (privacy regression). N rows fall out naturally from N requests.
  3. **Partial failures naturally surface as per-request 412/4xx.** No bulk-endpoint composite-error envelope to design + version. The client orchestrator collects per-request results and does the UX (D4).
  4. **No new endpoint to maintain.** v1.14.6 ships zero new server routes for bulk (just the create endpoint, which is structurally orthogonal). Surface area stays tight.
  5. **Server stays stateless toward batch intent.** Different tabs / clients may bulk-act in overlapping windows; the server doesn't have to coordinate — every request is independently authenticated, rate-limited, validated, and audited. Good HTTP/REST citizenship.

**Cost of A (acknowledged scope-tradeoff).**

  - 1000-item bulk delete = 1000 HTTP requests, with all the per-request auth/validation/audit overhead. At ~5ms per round-trip in a loopback test environment, that's ~5 seconds total (with 10 in-flight per D3 throttle); at ~50ms over real-world Telegram WebApp transport, that's ~50 seconds. This is acceptable because realistic bulk operations are 5-50 items, not 1000. D18 carries forward a v1.15.0+ TODO for server-side bulk if/when usage justifies it.
  - Rate-limit interaction: webapp routes are bound by the existing `webappAuthRateLimit` config (current default 60/min per user per ADR 008-ish; verified inactive at the route layer for now). With D3's 10-in-flight client throttle, sustained bulk above ~10/sec would still trip a tight rate limit; we don't have one tight enough to matter today (60/min is much more permissive than 10/sec * 60 = 600/min). When the rate limit tightens for production, D18 carries forward "revisit bulk pattern."

**Document the scope tradeoff in ADR 014.** This is a conscious choice: simplicity + symmetry over throughput.

## D3 — Bulk client-side throttling

**Question.** Devil's Advocate would observe: 1000-item bulk delete + tight rate limit = N-1 of those fail with 429. How does the client throttle?

**Options weighed.**

  - **A (recommended).** Client-side concurrency limiter — max 10 in-flight at any time; queue the rest. Simple Promise-pool pattern.
  - **B.** No throttle; let rate limit do the work; show 429s in the results UI (D4).
  - **C.** Server-side per-user concurrency control (e.g., a Redis-backed semaphore or in-memory map). Defense-in-depth alternative.

**Decision — A.** Client-side concurrency limiter, max 10 in-flight, queue-based.

**Specification.**

```javascript
// public/webapp/organize/app.js — bulk dispatcher

const MAX_BULK_INFLIGHT = 10;  // top-of-file constant; tunable per UX feedback

async function dispatchBulk(items, perItemAction) {
  const queue = [...items];
  const results = [];  // {item, ok, error?} — preserves order via index
  results.length = items.length;
  let nextIdx = 0;

  async function worker() {
    while (true) {
      const idx = nextIdx++;
      if (idx >= items.length) return;
      try {
        const r = await perItemAction(items[idx]);
        results[idx] = { item: items[idx], ok: true, response: r };
      } catch (err) {
        results[idx] = { item: items[idx], ok: false, error: err };
      }
    }
  }

  const workerCount = Math.min(MAX_BULK_INFLIGHT, items.length);
  await Promise.all(Array.from({length: workerCount}, () => worker()));
  return results;
}
```

**Justify A over B / C.**

  - 10 in-flight settles 50 items in ~5 round-trips × ~50ms = ~250ms (perceptual instant). 1000 items in ~100 round-trips × ~50ms = ~5 seconds (acceptable progress UX with a "Working… 47/1000" toast).
  - B's "let rate limit do the work" gives users a 940-failure pile they have to retry-or-give-up — terrible UX.
  - C is server-side defense in depth which is the right answer at scale, but it requires a stateful per-user concurrency map (memory cost; cleanup logic; cross-process coordination if we ever multi-process). Defer to v1.15.0+ via D18.
  - 10 is the same magic number Chrome / Firefox use for HTTP concurrency-per-origin; familiar shape; trivially adjustable via the constant.

**The 1000-item edge case.** Realistic bulk size in v1.14.6 is 5-50 items (the active-cap for organize is 200, of which a typical bulk is a tag/parent group of ≤50). 1000-item bulk is theoretical worst-case; D3 + D4 + D18 cover it without optimizing for it.

## D4 — Bulk partial-failure UX

**Question.** 50 items, 30 succeed, 20 fail (mostly 412 from concurrent edits in another tab + a couple of 400s from invalid state). User can't tell which are which.

**Options weighed.**

  - **A (recommended).** Results toast: "30 done, 20 failed — see selection". Stay in select mode. SUCCEEDED items become deselected. FAILED items remain selected (visually distinguishing them). User can retry the failed batch with the same bulk action OR open them individually.
  - **B.** Full-screen results modal listing each item + status. More information density but more friction.
  - **C.** Force a list refresh (loses selection state but shows truth).

**Decision — A.**

**Toast format (specification).**

  - When all succeed: "Completed 5 items." (TOAST_DEFAULT_MS = 3000ms).
  - When all fail: "Failed: 5 items. Tap any item to see details." (TOAST_LONG_MS = 5000ms).
  - When mixed (the most common case for partial failures): "30 done, 20 failed (kept selected). Tap to retry." (TOAST_LONG_MS = 5000ms).
  - 412-specific toast (ALL failures are 412s — likely cross-tab race): "Some items changed in another tab. Reload to see latest, then retry." (TOAST_LONG_MS = 5000ms).
  - Toast uses `textContent` only — no HTML interpolation (consistent with v1.14.4 D6 invariant).

**Selection state contract (specification).**

  1. After bulk dispatch resolves, iterate `results`.
  2. For each `ok: true` entry, REMOVE the item from `selectedIds`.
  3. For each `ok: false` entry, KEEP the item in `selectedIds`.
  4. Re-render the list — successfully-completed items now show their new state; failed items stay selected (highlighted via the selection-checkbox card style from D11).
  5. Bulk-action toolbar count updates: "Selected: 20" (was 50 before the action).
  6. If `selectedIds.size === 0`, exit select mode (D9 transition).

**Justify A.** Mid-friction. The user gets per-item visibility (failed items are visually selected; the SUCCEEDED items are gone from the selection so they don't pollute the next bulk action) without a modal interruption. Retry is one more tap of the same bulk action button. B is overkill for the 95% case (5-50 items). C is the worst-of-all (loses information AND selection state).

## D5 — Bulk-as-PATCH for re-parent

**Question.** Bulk re-parent uses N parallel `PATCH {parentId}` requests. Each one re-validates `parentExistsAndIsActiveGoal` (per ADR 013 D2 + R1). If the chosen parent goal is soft-deleted MID-batch (from another tab or chat), later items in the batch get 400 PARENT_NOT_FOUND.

**Decision.** This is acceptable per D2 (per-item independence) + D4 (partial-failure UX surfaces it). Document the race; test for it.

**Race semantics (specification).**

  1. Client picks parent goal G via the same picker UI used in single-item edit (with the "self-id filter" R2 invariant from ADR 013).
  2. Client dispatches N parallel `PATCH /:id {parentId: G}` requests, each with its own `If-Match: <etag>` (the per-item ETag captured at list-fetch time, retained in `renderedItems[]`).
  3. The server's parentExistsAndIsActiveGoal check (ADR 013 R1) runs per-request. If G is trashed mid-batch:
     - Items whose request fired BEFORE G's softDeleteItem completed: 200 + parentId set successfully (acceptable; user reparented to a target that became invalid microseconds later — same as any other write that races a delete).
     - Items whose request fired AFTER G's softDeleteItem: 400 PARENT_NOT_FOUND.
  4. D4 surfaces the failures: "30 done, 20 failed". Failed items stay selected. User can pick a different parent and retry.
  5. The 412 race (G was edited but not deleted) is also handled by ADR 012 + ADR 013 invariants — the per-item If-Match check catches it; the 412 envelope carries `currentItem`; the client surfaces 412 as a failure with kind "changed in another tab" instead of "parent not found."

**Failure ordering note.** The per-item failures may not be in submission order (parallel + non-deterministic completion). The results array preserves submission order (D3 spec); the UX presents the count, not the order. Acceptable.

**Test required.** Phase 2 integration test: mock G's softDeleteItem to fire mid-batch; verify some items succeed and some fail with PARENT_NOT_FOUND; verify D4 selection-state contract holds.

## D6 — Create endpoint route placement

**Question.** Add the create endpoint to `items.mutate.ts` or to a new `items.create.ts`?

**Options weighed.**

  - **A (recommended).** NEW `src/webapp/items.create.ts` (~150 LOC). Symmetric with items.read.ts + items.mutate.ts + items.complete.ts + items.shared.ts. Five files; each ≤500 LOC; each cohesive on its HTTP verb / semantic axis.
  - **B.** Add POST /api/webapp/items to items.mutate.ts. Reuses validatePatchBody → validateCreateBody integration site, the auth chain, the audit emit pattern, ETag header set. Risk: cycles past 500 LOC again — exactly the W2 v1.14.4 lesson + R3 ADR 013 split that's still less than three months in.

**Decision — A.**

**Justify A with the W2 cycle history.**

  - v1.14.2: items.mutate.ts at 516 LOC.
  - v1.14.3: 525 LOC.
  - v1.14.4: 529 LOC at HEAD.
  - v1.14.5 projection (ADR 013 D12 deferral): would have been 546 LOC.
  - v1.14.5 R3 (Anti-Slop W2 + DA P15 convergence): items.mutate.ts split into items.mutate.ts (PATCH+DELETE, ~370 LOC) + items.complete.ts (POST /complete, ~339 LOC). Both back below the 500-LOC soft threshold.
  - v1.14.6 if we put POST `/items` in items.mutate.ts: add ~150 LOC of create handler → 520 LOC. Right back over the soft threshold, four iterations after splitting away the W2 problem.
  - The lesson the v1.14.5 R3 split was meant to teach: don't accumulate handlers in items.mutate.ts. The right split axis is per-HTTP-verb / per-semantic-group. Create has neither If-Match nor 412 path nor conflict tracker — fits naturally as its own file (like items.complete.ts).

**Mounting (specification).**

  - Wired in `src/webapp/itemsRoute.ts` after `mountItemsCompleteRoutes(app, deps)`. Order: read → mutate → complete → create. Read mounts the no-store Cache-Control middleware on the `/api/webapp/items` prefix; create benefits from it.
  - Test harness inherits the wiring; no test changes for mount integrity beyond a +1 mount-integrity check (T-create-1).

## D7 — Audit category for create

**Question.** Use `webapp.item_mutate` (existing) or NEW `webapp.item_create`?

**Decision.** NEW `webapp.item_create` category.

**Justify.**

  1. **Creates have no `changedFields`.** All fields are new. The `changedFields: string[]` field of `webapp.item_mutate` is structured as "which fields the user updated"; for a create, "all of them" is meaningless / forensically empty. Separate category avoids polluting the field semantics.
  2. **Creates have no ETag/forced/bypassAfter412 axis.** No existing item to conflict with; no If-Match envelope; no 412 path. Optional fields would always be absent — they'd just be noise in the row.
  3. **Operator-facing forensics distinguish "added 50 tasks today" from "edited 50 tasks today" without parsing metadata.** A SELECT on `category = 'webapp.item_mutate' AND ts >= today` should NOT include creates. Adding a category is one ALTER-table-free SQLite migration step (the auditLog category column is a free-form string per existing schema; no migration; no schema bump).
  4. **Forward-compatible.** Future iterations may want create-specific fields (e.g., the user-provided `type`, whether `parentId` was set at create time) without disturbing the mutate row shape. Separate category is the clean home.

**Audit row metadata for create (specification).**

```typescript
interface WebappItemCreateDetail {
  itemId: string;        // newly assigned id (for the audit forensic trail)
  type: 'task' | 'event' | 'goal';
  hasParent: boolean;    // true if parentId was set; false otherwise. NOT the parentId itself (privacy).
  ip?: string;           // redacted IP (3-octet, same as v1.14.5 redactIp)
}
```

**No field VALUES recorded.** Title, due, tags, notes, progress are NOT recorded. Same privacy posture as v1.14.3 W5 (no-change-parentId no-recorded-value pattern) and ADR 010 decision 4 (changedFields-not-values for mutate audits). Operators auditing user behavior see "user N created K items today, mostly tasks, J of them with parents" — actionable for usage telemetry, opaque to user content.

**Audit emitter (specification).** New helper `auditItemCreate(memory, userId, itemId, type, hasParent, ip)` in `items.shared.ts` (sibling to `auditItemMutate`). Failures are logged at warn but do NOT propagate (same pattern as auditItemMutate at items.shared.ts:217-246).

## D8 — Create body schema + validator

**Question.** What's the request body shape? What error codes are emitted? Where does validation live?

**Decision.** New exported `validateCreateBody(body)` in `src/organize/validation.ts`. Sync (no IO). Returns `{ok: true, input: CreateItemInput} | {ok: false, code, error}`. Async parentId existence check happens in the route handler (mirroring v1.14.5 R1 same-read invariant where the existence check is route-side, not validator-side).

**Required fields.**

  - `type: 'task' | 'event' | 'goal'` — code `CREATE_TYPE_REQUIRED` if missing/invalid.
  - `title: string` (1-500 chars after trim, NUL-byte ban) — REUSE codes `TITLE_REQUIRED` / `TITLE_TOO_LONG` / `TITLE_NOT_STRING`. ADD code `TITLE_INVALID_CHARS` (NUL-byte ban) for symmetry with notes/progress (v1.14.3 D2 fix 4 pattern). NOTE: validatePatchBody currently does NOT NUL-byte-check title (verified at validation.ts:211-224). v1.14.6 adds the NUL-byte check to BOTH the new validateCreateBody AND retrofits it to validatePatchBody for consistency. **D8.a.**

**Optional fields.**

  - `due: string | null` — REUSE `DUE_INVALID_FORMAT`.
  - `tags: string[]` (max 10, each ≤40 chars matching TAG_RE) — REUSE `TAGS_NOT_ARRAY`, `TAG_TOO_LONG`, `TAG_INVALID_CHARS`, `TAGS_TOO_MANY`.
  - `notes: string` (max 10240, NUL-banned) — REUSE `NOTES_NOT_STRING`, `NOTES_TOO_LONG`, `NOTES_INVALID_CHARS`.
  - `progress: string` (max 20480, NUL-banned) — REUSE `PROGRESS_NOT_STRING`, `PROGRESS_TOO_LONG`, `PROGRESS_INVALID_CHARS`. **(Note: storage's createItem signature only accepts `notes`; `progress` would need to be added as a separate appendProgressEntry call after createItem returns. D17 handles the wiring.)**
  - `parentId: string` (item-id format, no `null` here — absent means top-level) — REUSE `PARENT_ID_INVALID_FORMAT`. ADD route-handler-level code `GOAL_CANNOT_HAVE_PARENT` (reused from v1.14.5 D1 rule 3) when `type === 'goal' && parentId !== undefined`. Same v1.14.5 parentId existence chain (NOT_FOUND, NOT_GOAL, NOT_ACTIVE, deletedAt filter from R1).

**Status: NOT settable on create.** Default to `'active'` (storage's createItem already does this — storage.ts:445). v1.14.6 does NOT expose status as a creation field. D18 carries forward "settable status on create" as a v1.14.7+ TODO (the legitimate use case is "log a completed task" — power-user feature).

**Body-shape strictness.** Same posture as validatePatchBody (RA1 / RA2 from ADR 010): unknown fields trigger explicit reject (`CREATE_UNKNOWN_FIELDS`). `type` and `title` REQUIRED, the rest optional. NEW code `CREATE_TYPE_REQUIRED` for missing/invalid type. Reuse `PATCH_NO_VALID_FIELDS` posture as `CREATE_NO_FIELDS` — but actually a create with only `type+title` is the minimum-valid case, so the empty-body check is "no `type` AND no `title`" rather than "no recognized fields"; emit `CREATE_TYPE_REQUIRED` first.

**`validateCreateBody` signature (specification).**

```typescript

  | 'TITLE_REQUIRED' | 'TITLE_TOO_LONG' | 'TITLE_NOT_STRING' | 'TITLE_INVALID_CHARS'
  | 'DUE_INVALID_FORMAT'
  | 'TAGS_NOT_ARRAY' | 'TAGS_TOO_MANY' | 'TAG_TOO_LONG' | 'TAG_INVALID_CHARS'
  | 'NOTES_NOT_STRING' | 'NOTES_TOO_LONG' | 'NOTES_INVALID_CHARS'
  | 'PROGRESS_NOT_STRING' | 'PROGRESS_TOO_LONG' | 'PROGRESS_INVALID_CHARS'
```

## D9 — Multi-select state model

**Question.** Where lives the `selectedIds` set? sessionStorage, localStorage, or in-memory only? Persist across navigation?

**Options weighed.**

  - **A (recommended).** Client-side `Set<string>` of selectedIds + boolean `multiSelectMode`. In-memory only — NEITHER sessionStorage NOR localStorage. Exiting select mode resets `selectedIds`. Navigating to detail and back resets too (ergo: don't navigate to detail while in select mode; tap-on-card means "toggle selection" not "navigate").
  - **B.** Persist `selectedIds` in sessionStorage so user can navigate to detail and back without losing selection.
  - **C.** Persist across reloads (localStorage). Surprising UX.

**Decision — A.**

**Justify A.**

  1. **Bulk operations are short-lived sessions.** User enters select mode, picks items, fires a bulk action, exits. Persistence creates "wait what's selected" surprise on the next page load (especially if the underlying items changed between sessions — a previously-selected item may now be in a different filter view, or deleted, or completed).
  2. **Predictable state machine.** `multiSelectMode` is binary; `selectedIds` is empty when mode is off; mode toggles cleanly via the "Select" button (D10) and via the ESC / cancel affordance.
  3. **No cross-page-load consistency to verify.** No "is the selected item still in the list?" check. No "the parentId I had selected was just deleted by another tab" check.
  4. **Tap-on-card behavior depends on mode.** In list mode, tap navigates; in select mode, tap toggles selection. Symmetric; documented in D11.

**State variables (specification).**

```javascript
// public/webapp/organize/app.js

const selectedIds = new Set();        // ids selected for bulk action; cleared on mode exit

// Convenience getters used by render fns + bulk action handlers:
function inSelectMode() { return multiSelectMode; }
function isSelected(id) { return selectedIds.has(id); }
function getSelectedCount() { return selectedIds.size; }
```

**Mode-exit transitions (specification).** Any of these resets state: explicit "Cancel" button tap; ESC key; navigating to detail panel; navigating to a different filter (changing the type or status filter exits select mode and clears selectedIds — different filter = different working set). Bulk action completes with all results successful.

## D11 — Multi-select rendering: card transformation

**Question.** When in select mode, how does the existing card UI change?

**Decision.** Card morphs as follows when `multiSelectMode === true`:

  1. **The complete checkbox (the v1.14.2 ⭕/✅ button at the start of the card)** — REPLACED with a real selection checkbox. Same tap target hit area; visual distinction:
     - Square outline + check mark when selected; square empty when unselected (CSS class `select-checkbox`). Round (⭕) is reserved for the done-status semantic.
     - For abandoned items (where the complete checkbox is currently hidden — v1.14.2 R14), the selection checkbox is STILL rendered (you can still bulk-delete or bulk-re-parent abandoned items).   2. **Card click semantics** — toggles selection (NOT navigation to detail). `event.stopPropagation()` on the inner select-checkbox so the click event doesn't double-fire.
  3. **Goal-group chevron (collapse/expand)** — KEEPS WORKING in select mode. Tap on chevron expands/collapses; tap on goal header body toggles selection. ## D12 — Bulk action confirmation flow

**Question.** Does bulk delete need a confirmation? Bulk complete? Bulk re-parent?

**Decision.**

  - **Bulk Delete: SINGLE 6-second two-tap confirm covers the whole batch.** Matches the single-item v1.14.2 R5 pattern. Toast: "Delete 5 items? (tap again within 6s to confirm)". User taps "Delete" again → fires N parallel DELETE requests via D3's dispatcher. Timer auto-cancels at 6s.
  - **Bulk Complete: NO confirm.** Same as single-item ⭕ tap (which has no confirm). Complete is reversible by tapping again or by editing the item.
  - **Bulk Re-parent: NO confirm.** Same as single-item parentId edit (which has no confirm — submitEdit goes straight to PATCH). Reversible by re-editing parent.

**Justify.** Symmetric with single-item flows. No new confirmation pattern to learn. Delete is the only destructive bulk; one confirm covers it. Bulk-as-N-actions amplifies impact, but the confirm is the same shape — users already trained on the v1.14.2 R5 pattern.

## D16 — BroadcastChannel dedup window

**Question.** A 50-item bulk × N tabs = 50N broadcast messages. The listener does 50 silent refetches in a row in each non-active tab. CPU spike + redundant network.

**Options weighed.**

  - **A (recommended).** 1-second dedup window. Listener queues incoming BC messages; if another arrives within 1 second, replace the queued action; only execute after 1 second of silence.
  - **B.** 500ms (more aggressive — collapses bulk faster but tighter window for legitimate single-item bursts).
  - **C.** No dedup; accept the noise.

**Decision — A.**

**Implementation specification.**

```javascript
// public/webapp/organize/app.js — handleBroadcastMessage extension

const BC_DEDUP_WINDOW_MS = 1000;
let _bcDedupTimer = null;
let _bcQueuedAction = null;   // {kind: 'refetch-list' | 'refetch-detail', itemId?: string}

function handleBroadcastMessage(ev) {
  if (!ev.data || !ev.data.itemId) return;
  const { itemId, kind } = ev.data;

  // Suppress when conflict panel is already visible (existing v1.14.5 invariant)
  if (_conflictPanelEl && !_conflictPanelEl.hidden) return;

  // Determine the action this message would trigger
  let action = null;
  if (currentDetailItem && currentDetailItem.id === itemId) {
    if (editFormEl && !editFormEl.hidden) {
      // Edit form open — banner action (queue with the dedup window too — banner spam is also noise)
      action = { kind: 'show-banner', bannerKind: kind };
    } else {
      action = { kind: 'refetch-detail', itemId };
    }
  } else if (!currentDetailItem) {
    action = { kind: 'refetch-list' };
  } else {
    return; // different item in detail — ignore (existing invariant)
  }

  // Dedup: queue the latest action; reset the timer
  _bcQueuedAction = action;
  if (_bcDedupTimer) clearTimeout(_bcDedupTimer);
  _bcDedupTimer = setTimeout(() => {
    const a = _bcQueuedAction;
    _bcQueuedAction = null;
    _bcDedupTimer = null;
    if (!a) return;
    if (a.kind === 'refetch-list') {
      fetchItems();
    } else if (a.kind === 'refetch-detail') {
      fetchAndShowDetail(a.itemId);
    } else if (a.kind === 'show-banner') {
      showBcBanner(a.bannerKind);
    }
  }, BC_DEDUP_WINDOW_MS);
}
```

**Why 1s and not 500ms.** Bulk-of-50 sends 50 broadcasts in ~5 seconds (from the dispatching tab) at 10/sec via the D3 throttle. Within a single batch, broadcasts arrive every ~100ms; 1s window collapses the entire batch into ONE refetch. 500ms would split a 50-item batch into two refetches — still better than 50 but not optimal. 1s is the sweet spot for batches of any plausible size at the throttle rate.

**Edge case: broadcast during an in-flight refetch.** If a fresh broadcast arrives while a `fetchItems()` is in flight from a previous timer fire, the listener queues the new action; the timer fires 1s later; if `fetchItems()` is STILL in flight (slow network), the dedup just rolls forward — at worst one extra refetch fires after the network resolves. Acceptable.

**Edge case: cross-tab interleaving.** Tab A and Tab B both bulk-act simultaneously. Tab A sees its own broadcasts AND Tab B's. The BC listener runs on both tabs (BroadcastChannel echoes to all subscribers EXCEPT the sender — verified per spec). Each tab dedup-queues independently; both end up with one extra refetch each at the end of the window. Acceptable.

```typescript
        status: item.frontMatter.status,
        title: item.frontMatter.title,
        created: item.frontMatter.created,
        due: item.frontMatter.due,
        parentId: item.frontMatter.parentId,
        calendarEventId: item.frontMatter.calendarEventId,
        tags: item.frontMatter.tags,
        notes: item.notesBody,
        progress: item.progressBody,
        fileBasename,
        mtimeMs,
        updated: item.frontMatter.updated ?? null,
      };

      const etag = computeETag(item.frontMatter, mtimeMs);
      res.setHeader(ETAG_HEADER, etag);

      // 7. Audit (D7)
      auditItemCreate(deps.memory, userId, item.frontMatter.id, item.frontMatter.type, item.frontMatter.parentId !== null, ip);

      res.status(201).json({ ok: true, item: detail });
    },
  );
}
```

**Status code: 201 Created** (REST convention) — distinguishes from PATCH/POST-complete which use 200. Tests assert 201.

## CP1 surface — what DA + Anti-Slop are likely to push back on

**P1 (HIGH likelihood) — Bulk DoS / accidental mass deletion.** A user mis-taps "Select All" + "Delete" + the 6s confirm → 200 items gone, all in trash. The 6s confirm covers it but it's a single confirm for an N-amplified action. DA may push: add an N-aware confirm threshold ("Delete 50 items? Tap again. … Delete 200 items? Type DELETE to confirm.").
  - **Architect's likely response:** Acknowledge; counter that the 6s window is forgiving (any not-completed-tap dismisses), and items go to trash with 30-day TTL (recoverable via /organize restore). NOT lost. Keep the single-confirm pattern symmetric with single-item delete. May concede a "tap N times, escalate to a typed confirm at >50 items" amendment.

**P2 (HIGH likelihood) — Dual-tab BC storm + bulk thrash.** Tab A bulk-deletes 50 items; Tab B receives 50 BC messages; D16 dedup collapses to one refetch — but Tab B's user is mid-edit on item X (which was NOT in the bulk delete). Banner shows "deleted in another tab" misleadingly. Or Tab B's edit form is open on item Y that WAS in the bulk delete; user finishes editing; sends PATCH; gets 404 (item is in trash now); confused.
  - **Architect's likely response:** D16 dedup queues by action-kind, not item. The banner shows a generic message; the user reloads to see truth. The 404 path on a trashed-during-edit item is an existing v1.14.4 surface (the conflict-panel shows a relevant error). v1.14.6 doesn't make this worse. May add a regression test for "edit item Y; another tab bulk-deletes Y; user submits PATCH; gets 404; sees conflict panel."

**P3 (MEDIUM likelihood) — Create double-submit on slow network.** D15's `_createSubmitInFlight` flag closes the JS-thread case but a user who taps Create, the request is queued in the iOS Safari networking stack but not yet dispatched, and they background the app + foreground it → some weird race. Likely DA/Anti-Slop says "what about the wakeup race?"
  - **Architect's likely response:** btn.disabled + the in-flight flag should hold across foreground/background — neither resets. The fetch promise is in-flight; either it resolves (finally re-enables) or it rejects (finally re-enables). Test for it (T-79 covers single-thread; add T-79a for visibility-change interleaving).

**P4 (MEDIUM likelihood) — Bulk patch with stale ETags.** Each item in `renderedItems[]` was captured at list-fetch time. By the time a 50-item bulk fires, some of those ETags may be stale (other tab edits, reminders, agent updates). Each request returns 412; D4 surfaces "all 50 failed in another tab" — false alarm; the user's intent was valid; the problem is the client used stale ETags.
  - **Architect's likely response:** Bulk operations send ETags from the moment of select-mode entry, not from list-fetch. Or, more aggressively: bulk operations send `If-Match: *` (skip-the-check semantics — equivalent to single-item Save Anyway). ADR 014 takes the LATTER position: bulk operations send NO If-Match (omit the header); the audit row records `etag: null, forced: false`; the conflict tracker still notes any 412s. This is a deliberate scope choice: bulk is "I want this state for these items, regardless of others' concurrent edits." Document the choice. **(Folded back into D5.)**

**P5 (MEDIUM likelihood) — trash.ts extract is mechanical but storage.ts → trash.ts → storage.ts can produce a circular import if the wrapper helper isn't placed cleanly.** parseItemFileFromRaw export from storage.ts that trash.ts imports is the only directional dependency. If trash.ts is imported by storage.ts (it is — softDeleteItem needs ensureTrashDir), the cycle exists.
  - **Architect's likely response:** Move ensureTrashDir back to storage.ts; trash.ts imports it. OR: have trash.ts re-export ensureTrashDir for storage.ts (one-way). The cleanest break: ensureTrashDir is a path-helper, fits in storage.ts's path-helper section (storage.ts:36-91). KEEP IT IN STORAGE.TS. Update D1 to KEEP ensureTrashDir in storage.ts. **(Concession in CP1 revisions.)**

**P6 (LOW likelihood) — D7 audit category split (webapp.item_create) breaks an existing audit-query path.** Operators may have SQL like `SELECT COUNT(*) FROM audit_log WHERE category = 'webapp.item_mutate'` to count "user activity"; adding a new category means they have to update queries.
  - **Architect's likely response:** Audit operators are a small, internal population. Document the new category in `docs/PRIVACY.md` audit-row enumeration. Acceptable.

**Anti-Slop W likelihoods:**
  - **W-A:** D8.a (NUL-byte retrofit on validatePatchBody title) is added to a different ADR's territory (v1.14.2 invariant). Anti-Slop will want a doc note in ADR 014 that the retrofit is intentional + a regression test. Both are present.
  - **W-B:** D9's "in-memory only, no persistence" needs a documented rationale matching the v1.14.5 pattern. Present at D9.
  - **W-C:** D11's dual-mode contract (chevron vs select) might be mis-implemented if the dev agent doesn't realize the existing v1.14.3 chevron has its own stopPropagation; explicit binding required.
  - **W-D:** D14's char counters are a copy-paste from edit form. Must reuse the shared helper, not duplicate.
