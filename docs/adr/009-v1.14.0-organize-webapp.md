## Decisions

### 1. Per-feature directory pattern — `public/webapp/organize/` (subdirectory under hub)

**Status:** Accepted (subdirectory).

**Context (rigor ask 1).** Two candidate paths:

  - **(a) `public/webapp/organize/`** — subdirectory under the hub. URL is `${publicUrl}/webapp/organize/`. Static middleware in `src/webapp/server.ts` is already `app.use('/webapp', express.static(staticDir))`; nothing changes server-side — `staticDir` (= `public/webapp/`) gains a child directory and `express.static` resolves nested paths automatically.
  - **(b) `public/webapp-organize/`** — sibling of `public/webapp/`. URL is `${publicUrl}/webapp-organize/`. Requires a new `app.use('/webapp-organize', express.static(...))` registration in `src/webapp/server.ts`. Each new feature page in v1.16.0+ adds another sibling AND another `app.use` line.

**Decision.** (a). Justifications:

  1. **Zero server-side change.** The existing `app.use('/webapp', express.static(staticDir))` already serves nested files: `staticDir/organize/index.html` → `/webapp/organize/index.html`. The CSP-and-cache-control `setHeaders` callback runs on `index.html` regardless of depth (it only checks `filePath.endsWith('.html')`). v1.14.0 adds zero LOC to `server.ts` for static serving — the directory is just files on disk.
  2. **Hub-and-spoke mental model.** `/webapp/` is the root entry point users see in Telegram; everything reachable is "under" the hub. A sibling `/webapp-organize/` URL feels disconnected — users don't navigate to it from the hub, they receive it as a separate button. Subdirectory matches the navigation reality.
  3. **CSP locality.** All Web App pages share the same CSP (set by `setHeaders` on any HTML file under `staticDir`). With sibling directories each new mount needs its own CSP wiring; with subdirectories the existing wiring covers them all by construction.
  4. **Forward compatibility.** v1.16.0 adds `public/webapp/debate/`, v1.17.0 adds `public/webapp/schedule/`. Each is a one-directory addition with no `server.ts` edit. Sibling-directory pattern requires a `server.ts` edit per feature, which is per-iteration churn for no architectural benefit.

**Module shape:**

```
public/webapp/
├── index.html        ← hub (was the v1.13.0 ping skeleton; converted in Item 4)
├── app.js            ← hub script (rewritten — see decision 4)
├── styles.css        ← hub styles (NEW; Item 4 splits inline styles out for reuse)
└── organize/
    ├── index.html    ← organize list + detail view
    ├── app.js        ← organize page script
    └── styles.css    ← organize page styles
```

The hub's `styles.css` may be `@import`ed by `organize/styles.css` for shared theme variables (`--tg-theme-bg-color`, etc.) — but this is a Phase 2 implementation choice; the design doesn't mandate it. Each page's CSS may also stand alone; total per-page payload is small enough that duplication is fine (~30 lines × 2 pages = 60 lines).

**Consequences.**

  - **Positive.** Zero `server.ts` change.
  - **Positive.** Future feature pages drop in as new subdirectories.
  - **Positive.** CSP uniform across all Web App pages by construction.
  - **Negative.** `public/webapp/` accumulates depth over time. Acceptable — the alternative is N flat siblings with the same eventual file count.

### 2. Pagination on `/api/webapp/items` — YAGNI now, response shape carries `total` only

**Status:** Accepted — no `nextCursor`; just `{items, total, serverTime}`.

**Context (rigor ask 2).** Today's storage layer has no pagination primitive. `listItems` reads the full user directory, parses each file's front-matter + body, applies in-memory filter, and returns the array. At the per-user 200-active-item cap × ~150 bytes of metadata per `ListItem` ≈ 30 KB of JSON max for the active set — well under any realistic mobile-network concern. Three response-shape candidates:

  - **(a)** `{items, total, serverTime}` — simple. v1.15.0+ pagination will need a breaking change OR a separate route.
  - **(b)** `{items, total, serverTime, nextCursor: null}` — pre-allocates the field; v1.15.0 starts populating it without changing the response schema.
  - **(c)** `{items, page: {data: [...], total, nextCursor: null}, serverTime}` — full envelope; future-proof but verbose for v1.14.0's needs.

**Decision.** (a). YAGNI-pure. v1.14.0 ships without `nextCursor`; v1.15.0 (which is the FIRST iteration where pagination becomes plausible — kanban + calendar grid views need to render hundreds of items) introduces it as an additive field. Adding an optional field to a JSON response is non-breaking by client-coding convention (the v1.14.0 client doesn't read it; the v1.15.0 client does). Pre-allocating a `nextCursor: null` field today communicates an API surface we don't yet have, which is worse than its absence.

**Why not (b).** A `nextCursor: null` field invites a v1.14.x developer to "improve" the spec by populating it before the actual pagination semantics are designed. That's premature commitment — pagination on this surface needs a decision about cursor stability across creates/deletes, which is a v1.15.0 design task and not something to half-ship.

**Why not (c).** Same reason, more so. Adds verbosity without value.

**Response schema (v1.14.0):**

```typescript
// GET /api/webapp/items?type=task&status=active&tag=urgent
// 200 OK, application/json:
interface ItemsListResponse {
  items: ListItem[];        // metadata-only projection (decision 3)
  total: number;            // items.length AFTER filtering
  serverTime: string;       // ISO-8601, e.g. "2026-04-24T15:30:00.000Z"
}
```

`serverTime` is included because the client renders relative dates ("due in 2 days") against the server's clock — without it, a client whose phone clock is wrong renders "due yesterday" for items due today. It's also useful diagnostic data when the user complains "my list looks stale." Cheap; ~30 bytes; ship it.

`total === items.length` is an invariant in v1.14.0 (no pagination). When v1.15.0 introduces pagination, `items.length <= total` will hold (returned-page-size vs filtered-total). Documenting the invariant now in the JSDoc lets v1.15.0 evolve without confusion.

**Consequences.**

  - **Positive.** Smallest possible response envelope.
  - **Positive.** v1.15.0 pagination is an additive change; v1.14.0 clients keep working.
  - **Negative.** A v1.14.x client that wants to know "will the server paginate me?" can't tell from the response. Acceptable — there's exactly one client today and it's controlled by the same release cycle.

### 3. Item-detail endpoint shape — pre-parsed sections only, no raw markdown

**Status:** Accepted — `{item: {...frontMatter, notes: string, progress: string, fileBasename: string}}`.

**Context (rigor ask 3).** `OrganizeItem` from storage already separates `notesBody` and `progressBody` (the parser owns the section split — see `extractBodySections` at `storage.ts:296`). Three shape candidates for the API response:

  - **(a) Pre-parsed only:** `{frontMatter, notes, progress, fileBasename}` — exactly what storage gives us, minus the absolute filePath.
  - **(b) Raw + pre-parsed:** include both `raw: string` (full file content with `---` fences and `## Notes` / `## Progress` headers) AND the pre-parsed sections. Future-proofs for clients that want to render with their own markdown engine.
  - **(c) Raw only:** ship the file content, let the client parse. Zero server-side semantics.

**Decision.** (a). Pre-parsed only.

Justifications:

  1. **Storage already separates.** The `parseItemFile` → `extractBodySections` pipeline is the canonical split. The API simply mirrors it. Shipping `raw` would mean the client needs the same parser logic OR we're encouraging the client to reinvent it (see Anti-Slop §5 — Separation of Concerns).
  2. **Brief's "no markdown rendering" boundary.** v1.14.0's detail view renders notes + progress as preserved-whitespace text. There is no engine on the client that would benefit from raw markdown. v1.14.1+ MAY add markdown rendering — but if/when it does, the renderer takes the pre-parsed `notes` string as input. `raw` doesn't help.
  3. **Front-matter shape stability.** Returning `frontMatter` as a typed object (id, type, status, title, due, etc.) means the client never has to parse YAML — it consumes a JSON object. Future spec changes (a new front-matter field) are an API-version concern, not a per-client parsing concern.
  4. **`fileBasename: string` instead of absolute path.** ADR 008 § "Module isolation" + ADR 005's multi-user posture: never expose absolute paths over an external API. `fileBasename` is the `.md` filename (e.g. `2026-04-23-a1b2.md`) — useful for diagnostic display ("file: 2026-04-23-a1b2.md") without leaking host directory structure. The route handler computes it via `path.basename(item.filePath)`.

**Response schema:**

```typescript
// GET /api/webapp/items/:id
// 200 OK, application/json:
interface ItemDetailResponse {
  item: {
    // frontMatter fields, flattened:
    id: string;
    type: 'task' | 'event' | 'goal';
    status: 'active' | 'done' | 'abandoned';
    title: string;
    created: string;          // ISO-8601
    due: string | null;
    parentId: string | null;
    calendarEventId: string | null;
    tags: string[];
    // body sections:
    notes: string;            // notesBody verbatim (CRLF normalized to LF by parser)
    progress: string;         // progressBody verbatim
    // metadata:
    fileBasename: string;     // e.g. "2026-04-23-a1b2.md" — never the absolute path
  };
}

// 404 Not Found, application/json:
interface NotFoundResponse {
  error: 'NOT_FOUND';
}
```

The `frontMatter` shape is FLATTENED (not nested under a `frontMatter` key) for client ergonomics: `item.title` is more natural than `item.frontMatter.title`. The flatten is one line of mapping at the route handler. v1.14.1+ can add a `meta` sub-object for fields like `lastEdited` without restructuring; flatten doesn't fight extensibility.

`deletedAt` is intentionally omitted from the response. Read-only v1.14.0 doesn't expose trashed items (`storage.listItems` skips `.trash/`); this endpoint will never see a deletedAt-stamped item from the live directory. If it somehow did (legacy data), the field is dropped from the response — clients have no use for it. Documented.

**Consequences.**

  - **Positive.** Smallest payload that's complete; no client-side parsing.
  - **Positive.** Future markdown rendering is a renderer-on-`notes` decision, orthogonal to this shape.
  - **Negative.** A future client wanting the raw file (e.g. for "edit in raw mode" — speculative) can't get it from this endpoint. Acceptable; we'd add a `?raw=true` query at that point and ship the raw view as an explicit choice.

### 4. Scale at 5000 items — punt; rely on the active-cap + acknowledge done/abandoned growth

**Status:** Accepted — option (a) in the brief, with explicit risk-register entry per SF-1.

**Context (rigor ask 4 + SF-1).** `listItems(userId, dataDir, filter)` reads + parses every `.md` file in the user's directory (excluding `.trash/`). The 200-active-item cap holds active count bounded; done + abandoned items accumulate without bound. Three candidates:

  - **(a) Punt.** Rely on the active-item cap. Add a §13 risk-register row for done/abandoned growth.
  - **(b) Fast metadata-only path.** Add `listItemsMetadata(userId, dataDir, filter)` to `src/organize/storage.ts` that uses a hypothetical front-matter-only parse. But: the existing parser at `parseItemFile` is already mostly front-matter parsing — the body extraction is ~30 lines of in-memory string-split, not a meaningful cost. The "fast path" would save the body-section extraction call but not the readFile call (which dominates I/O).
  - **(c) Compromise — measure.** Ship v1.14.0 with the existing path; add a benchmark to the test suite; if the median user shows >100ms list latency at deployed scale, ship v1.14.1 with the optimization.

**Decision.** (a). Punt with explicit acknowledgment. Justifications:

  1. **The body extraction is cheap.** `extractBodySections` is a single-pass line splitter (~30 LOC of pure JS). On a 5KB item it adds well under 1ms; for a 200-item active set the cumulative body-extraction cost is sub-millisecond. The storage cost is dominated by `readFile`, which a "metadata-only" path doesn't avoid.
  2. **Done/abandoned growth IS real, but not urgent.** A user creating 5 done/abandoned items per day takes ~2 years to reach 5000. By then v1.14.x has had multiple read-path optimizations OR the user has used `/organize` reconcile to clean up (v1.11.0 ADR 006 — already shipped). Acting today on a hypothetical 2-years-out scale concern is YAGNI.
  3. **The endpoint serves a single user per request.** The cost is `O(N items for THIS user)`. There's no cross-user fan-out. A power user with 5000 items pays the cost themselves; other users on the same deployment are unaffected.
  4. **A future migration is plumb.** When growth becomes real, the migration path is: (i) materialize a `data/organize/<userId>/.index.json` file on every write that holds the metadata-only projection; (ii) `listItems` for the API path reads `.index.json` directly when present; (iii) on cache miss, rebuild from disk and write the index. This is a v1.14.x or v1.15.x decision when measurements warrant it. Documented.

**Risk-register entry (added to §13):** "Done/abandoned items accumulate without bound; list endpoint cost grows O(N total items per user). MITIGATION: relies on per-user reconcile workflow (ADR 006). v1.14.x revisits if measured P95 list latency exceeds 200ms."

**Why not (b) shipping the optimization now.** It's a net-zero refactor — saves microseconds, adds code paths to maintain, and the optimization that actually matters (an index file) is a different pattern entirely. Ship the obvious thing now; optimize when measurement says so.

**Why not (c) instrumenting in v1.14.0.** The test fixtures don't have realistic per-user item counts to measure against. A synthetic benchmark with 5000 generated items proves nothing about real users' workflows. v1.14.x acceptance-monitoring (real production latency from cloudflared logs) is the right place to gather signal.

**Consequences.**

  - **Positive.** Zero refactor for v1.14.0.
  - **Positive.** Future optimization has a clean trigger (measured latency exceeds threshold).
  - **Negative.** A power user with thousands of done/abandoned items pays a real cost. Acceptable for v1.14.0; flagged for revisit.

### 5. API auth replay window — keep 24h default; document the consequence

**Status:** Accepted — keep `initDataMaxAgeSeconds: 86400` for `/api/webapp/items*`; do NOT introduce a per-route override.

**Context (rigor ask 5).** v1.13.0's default `initDataMaxAgeSeconds: 86400` (24h) was set for the echo endpoint, which leaks `userId` only. The items endpoints expose user-authored task titles — more sensitive (titles can include "Quarterly review meeting with Acme Corp"). Three candidates:

  - **(a) Keep 24h** — single replay window for all `/api/webapp/*` routes. Document the consequence.
  - **(b) Tighten to 1h** — for the items endpoints specifically. Requires per-route override in the verifier call.
  - **(c) Tighten to 1h globally** — change the default in `config.webapp.initDataMaxAgeSeconds`. Affects echo too.

**Decision.** (a). Keep 24h. Justifications:

  1. **Replay surface is small.** A 24h-replay against `/api/webapp/items` returns the user's items at the moment of replay (not at the moment of original auth) — the data is "what's in their organize folder right now," which the legitimate user accesses freely. The leak is "an attacker who captured initData via screen-recording or referer-leak gets a 24h window to peek at the user's task list." Real-world impact: the same screen recording almost certainly already shows the task list itself. The marginal harm of the replay is small.
  2. **Cost of multiple routes with different windows.** A per-route window pollutes the verifier API, the verify-call sites, and the test surface. Three routes today (echo, items, items/:id) × two windows (24h vs 1h) is 6 test cases just for the window. Unjustified ceremony for the actual risk.
  3. **The right defense is initData hygiene, not window shrinking.** The page does not log initData, does not store it client-side (it's stripped on each fetch), does not put it in URLs (it's in the Authorization header). The CSP `connect-src 'self'` prevents a leaked `initData` from being POSTed to an attacker's domain by a future page bug. v1.13.0's defenses HOLD for v1.14.0; the window is not the load-bearing layer.
  4. **Operators can tighten globally.** `config.webapp.initDataMaxAgeSeconds` is configurable; an operator who wants 1h global (or 5 min) can set it. The DEFAULT stays at 24h because it tolerates a user opening a saved Web App link in a slow-network situation.

**Documentation requirement.** README's v1.14.0 subsection (Item 7) MUST include: "initData is valid for 24 hours by default. To shorten, set `webapp.initDataMaxAgeSeconds` in config.json. The window applies to all `/api/webapp/*` routes uniformly."

**Risk-register entry (added to §13):** "24h initData window leaks task titles to a successful replay. MITIGATION: `connect-src 'self'` CSP, no URL-based initData, header-only auth (R5). Operator may tighten via `webapp.initDataMaxAgeSeconds` config. v1.14.x revisits if a real leak channel is identified."

**Why not (b).** Per-route overrides multiply test surface for hypothetical risk. A future v1.14.x decision can introduce route-specific windows when an actual attack pattern justifies it.

**Why not (c).** Tightening the global default re-litigates v1.13.0's decision without new information. v1.13.0's choice was "24h tolerates legitimate UX (open a saved link)"; that rationale still holds.

**Consequences.**

  - **Positive.** No verifier API change; no test bloat.
  - **Positive.** Operators with stricter compliance needs can tighten via config.
  - **Negative.** A successful initData replay within 24h leaks task titles. Documented; risk register row.

### 6. HTML escape contract for client-side rendering — `textContent` only for user-authored data

**Status:** Accepted — strict contract documented in `public/webapp/organize/app.js` top comment.

**Context (rigor ask 6).** The API returns raw strings: `title`, `tags[]`, `notes`, `progress`. The client renders these into the DOM. XSS is prevented by NEVER setting these strings via `innerHTML`. Three candidates:

  - **(a)** Use `textContent` for all user-authored text. CSS sets `white-space: pre-wrap` for multi-line content. No escape function needed (the DOM API auto-escapes).
  - **(b)** Build an `escapeHtml(s)` function and use `innerHTML` carefully where layout requires.
  - **(c)** Adopt a tiny templating helper (lit-html-style tagged template) for safe DOM construction.

**Decision.** (a). Strictest of the three; matches v1.13.0's `app.js` precedent (which uses `textContent` exclusively — verified line-by-line in `public/webapp/app.js:29-71`).

**Contract** (binding for Phase 2 developer agents):

  1. **Never assign user-authored strings to `innerHTML`.** Specifically: `title`, `tags[i]`, `notes`, `progress`, `fileBasename`. Always set via `element.textContent = value` or `element.append(document.createTextNode(value))`.
  2. **`innerHTML` is permitted ONLY for static, hardcoded HTML strings authored in `app.js`.** Example: `el.innerHTML = '<span class="empty-state">No items match.</span>'` — the string is a literal in the source, never mixed with user data.
  3. **Multi-line content (notes, progress) renders into an element with `white-space: pre-wrap` CSS.** Set via `el.textContent = item.notes`. The browser renders newlines as line breaks; `textContent` auto-escapes any `<`, `>`, `&`, `"` in the text. No regex-based escaper needed.
  4. **Tags render as separate elements.** `tags.forEach(t => { const span = document.createElement('span'); span.className = 'tag'; span.textContent = t; container.append(span); })`. Each tag is its own DOM node — never concatenated into a parent's `innerHTML`.
  5. **Filter chip clicks set `data-*` attributes via `dataset`, not `innerHTML`.** `chip.dataset.type = 'task'` is safe; `chip.innerHTML = '<button data-type="' + raw + '">...'` is forbidden.
  6. **API URLs are constructed via `URL` and `URLSearchParams`, not string concatenation.** `const url = new URL('/api/webapp/items', location.origin); url.searchParams.set('type', filter.type);`. The browser's URL API encodes special characters; concatenation does not.

**Detail-view-specific rule.** The detail panel's notes/progress containers are styled `white-space: pre-wrap; word-break: break-word; font-family: monospace`. Content is set via `notesEl.textContent = item.notes; progressEl.textContent = item.progress`. No markdown rendering (boundary). No code-block detection. No link auto-detection (a v1.14.x decision; auto-link of URLs requires a regex pass that has its own injection surface — explicitly out of scope for v1.14.0).

**Empty-string handling.** When `notes === ''` or `progress === ''`, render the placeholder `'(empty)'` in a separate `<span class="empty-text">` element with hint-color CSS. Don't try to render an empty string as content; the placeholder is a hardcoded literal so `innerHTML` is fine for it.

**Validation in tests.** Item 6's integration test asserts that a known-malicious item title (e.g. `'<script>alert(1)</script>'`) renders into the page as escaped text, NOT as a script element. The test loads the page via Node's `fs.readFileSync` + a happy-path fetch mock that returns the malicious title; asserts `document.querySelector('script')` does NOT find a new script after rendering.

**Consequences.**

  - **Positive.** XSS surface is zero by construction. The contract is short enough for Anti-Slop to grep for violations: any `\.innerHTML\s*=` next to a non-literal string is a fail.
  - **Positive.** No escape-helper to maintain (and accidentally break in a refactor).
  - **Negative.** Slightly more DOM-construction code than templating would produce. Acceptable — vanilla-JS posture is part of the "no build tooling" boundary.

### 7. Module isolation — direct import from `src/organize/storage.ts`; thin adapter optional

**Status:** Accepted — direct import. NO new wrapper file.

**Context (rigor ask 7).** Two candidate patterns:

  - **(a) Direct import.** `src/webapp/server.ts` imports `listItems`, `readItem` from `src/organize/storage.ts`. The route handlers call them directly with `userId` (from verifier) and `dataDir` (computed from config).
  - **(b) Wrapper file.** Add `src/webapp/organizeApi.ts` that imports `src/organize/storage.ts` and re-exports a thin façade (e.g. `listForWebapp(userId, dataDir, filter)` returning `ListItem[]` directly — pre-projected). `server.ts` imports `organizeApi.ts`.

**Decision.** (a). Direct import.

Justifications:

  1. **`listItems` and `readItem` are stable and well-tested.** The interface has been frozen since v1.8.6 (verified — `listItems` signature unchanged across v1.8.6 → v1.13.1). Wrapping a stable interface adds indirection without buying anything.
  2. **The projection is small and route-specific.** The list-endpoint projection (`OrganizeItem[]` → `ListItem[]`) is ~10 LOC of map operations. Putting it in a wrapper file means one file with one function, used by one caller. A wrapper is overweight.
  3. **`server.ts` is allowed to grow.** ADR 008's R2 explicitly placed the webapp Express factory inside `src/webapp/` and acknowledged future routes would land there. The factory currently has the echo route inline; the items routes are the same pattern. v1.16.0 (debate webapp) lives in its own subtree (`src/webapp/debate/` if needed) with its own modular split. The "decompose `server.ts` when it's >500 LOC" rule applies if/when it actually hits that size; v1.14.0 brings server.ts to ~300 LOC, well below.
  4. **Anti-Slop §13 (Module Isolation) does NOT forbid cross-module imports.** It forbids reaching into another module's INTERNALS. `listItems` is the public API of `src/organize/storage.ts` (exported; documented; tested). Importing it is the correct discipline.

**However:** Phase 2 developer agents MAY split the items-route logic into a sibling file if `server.ts` exceeds 500 LOC after the v1.14.0 changes — a refactoring decision that lives in Phase 2, not in this ADR. The split, if needed, would be `src/webapp/itemsRoute.ts` exporting a `(app: Express, deps) => void` registration function — same pattern as Express's per-feature router idiom. Decision is delegated to Phase 2 with the 500-LOC trigger.

**`dataDir` resolution.** `src/commands/organize.ts:62` already has the canonical `dataDirFromConfig` helper:
```typescript
function dataDirFromConfig(config: AppConfig): string {
  return path.resolve(config.memory?.dbPath ? path.dirname(config.memory.dbPath) : 'data');
}
```
Phase 2 dev agents MUST extract this to a shared helper (location: `src/organize/storage.ts` makes sense since it's a storage-layer concern, OR `src/config/dataDir.ts` if a clean home is preferred — Phase 2 picks). The webapp route handler imports it. Duplicating the inline expression is a fail; both call sites must reach the same `dataDir`.

**Consequences.**

  - **Positive.** Zero new file. Import direction is gateway-tier-equivalent → domain-tier (allowed by ADR 001's module boundaries).
  - **Positive.** Refactor trigger (500 LOC) is explicit and measurable — Phase 2 can act or skip as the actual code size dictates.
  - **Negative.** `server.ts` accumulates routes over v1.14.x → v1.17.x. Documented; refactoring trigger covers it.

### 8. In-memory cache for the list endpoint — no cache; fetch every time

**Status:** Accepted — option (a) in the brief.

**Context (rigor ask 8).** Three candidates:

  - **(a) No cache.** Fetch from disk every time. Cleanest; slightly slower on filter-toggle navigation.
  - **(b) Cache with explicit invalidation.** Hook into `organize_create`, `organize_update`, `organize_delete` tool dispatches to invalidate the cache for that user. Requires cross-module wiring.
  - **(c) Short TTL cache.** 5-second TTL per user. Auto-self-heals. May serve stale data for up to 5s after a chat-side mutation.

**Decision.** (a). Justifications:

  1. **Single-user scale.** At 200 active items per user × ~5KB per file = 1MB read. On a local SSD that's <50ms. Filter-toggle latency is dominated by network round-trip (cloudflared + Telegram SDK), not disk read. A cache saves disk I/O time that's already lost in the noise.
  2. **Cross-module invalidation cost.** Option (b) requires the webapp module to know about `organize_create` / `organize_update` / `organize_delete` tool calls. These currently live in `src/tools/organize/*.ts` and are dispatched by the agent loop in `src/agent/index.ts`. Adding a "publish cache-invalidation event" hook there is real coupling — the webapp module suddenly cares about tool-side mutations. Bad layering for the size of the perf win.
  3. **Stale-by-5s UX is bad for THIS feature.** A user creates a task via Telegram chat ("add a task: pick up dry cleaning"), opens the Web App 3s later, taps "Refresh" or filter chip — and the new task isn't there. Confusion is real even though the perf win is negligible. The TTL approach (c) creates the exact bug the user is most likely to spot.
  4. **A cache layer is reversible.** If v1.14.x measurement shows the filter-toggle latency is genuinely user-perceptible, we add the cache then with measured numbers backing it. Adding a cache is an additive change; removing one is hard (clients build expectations around timing). Default-no-cache is the conservative choice.

**Future migration.** When (if) caching is added, the right pattern is:
  - In-memory map keyed by `${userId}` → `{items: ListItem[], expires: number, etag: string}`.
  - Cache invalidation: TTL-only, 60s. Cross-module event-bus is overkill.
  - HTTP-level: respond with `ETag` + handle `If-None-Match` for 304 responses. The browser cache becomes the primary cache; the server cache is a fallback.

That's a v1.14.x or v1.15.x decision when measurement warrants. v1.14.0 ships without it.

**Consequences.**

  - **Positive.** No staleness window. Chat-side creates are immediately visible.
  - **Positive.** No cross-module coupling.
  - **Negative.** A user toggling filters rapidly pays a fresh 50-100ms disk read each time. Acceptable; humans don't toggle filters faster than ~1Hz.

### 9. Audit emission for `/api/webapp/items*` — failures only; existing debouncer handles it

**Status:** Accepted — no new audit category, no per-success row.

**Context (rigor ask 9).** v1.13.0 audit-logs ONLY auth FAILURES. The new items endpoints inherit the same auth gate (the `Authorization: tma <initData>` header path). Three candidates:

  - **(a) No new audit.** Authentication failures continue to fire `webapp.auth_failure` per the existing debounced path. Successful list/detail calls do not audit.
  - **(b) Audit successful list/detail.** Add `webapp.list_request` / `webapp.detail_request` for forensic visibility on what users access.
  - **(c) Audit on rate-limit only.** Add a `webapp.rate_limit` audit row when the per-IP limit fires.

**Decision.** (a). The existing `webapp.auth_failure` audit category handles all auth failures uniformly — including failures on the new routes (the `pathHit` field captures which route was hit). The route-handler's auth gate runs the same `verifyTelegramInitData` and the same `emitAuditIfDue` call site. ZERO new audit logic for v1.14.0.

Justifications:

  1. **Successful audits would balloon under normal usage.** A user who opens the Web App and taps 3 filter chips fires 4 list requests. Each would be an audit row. Across days, the table grows linearly with usage for no operational signal.
  2. **The pino info log is the success record.** `serverLog.info({userId, count, filter}, 'organize list served')` is greppable, ages out per the log rotation policy (LOGGING_STANDARD.md), and isn't permanently stored. That's the right tier for "this user opened their list."
  3. **Rate-limit auditing is incomplete without the wider signal.** A single rate-limit event could be a flaky network OR an attack. The existing rate-limit warn log is the right tier; lifting it to audit doesn't add information.
  4. **`pathHit` already encodes the route.** Existing audit row's `detail.pathHit` field captures `/api/webapp/items` vs `/api/webapp/echo`. Forensic queries can already distinguish "auth failure on items endpoint" from "auth failure on echo endpoint" without a new category.

**Pino logging tiers** (matches ADR 008 §9 conventions):

  - **info** — successful list (`{userId, total, filter, durationMs}`), successful detail (`{userId, itemId, durationMs}`). Component tag `webapp.server` for both (reuse existing logger; do NOT introduce a new component tag).
  - **warn** — itemId regex mismatch (`{userId, attemptedItemId, ip}` — interesting, possibly a probe). 404 on detail (`{userId, itemId}` — informational; not necessarily an attack but useful for debugging).
  - **debug** — per-item front-matter parse warnings already emitted by `storage.ts:498`. No new debug logs in this iteration.

**No new audit category.** No new pino component tag. Both reuse `webapp.server`.

**Consequences.**

  - **Positive.** Zero new audit-table growth for the normal case.
  - **Positive.** Auth failures on the new routes are already covered by the existing debounced path.
  - **Positive.** Operator forensics (search for `webapp.auth_failure` rows with `pathHit` containing `/items`) works without code change.
  - **Negative.** No "who accessed which item" trail. Acceptable for v1.14.0 (single-user); revisit when multi-user becomes real.

### 10. Detail-view back-navigation — Telegram BackButton + Android hardware back, single-page hide/show

**Status:** Accepted — single-page state machine with two views, BackButton wired explicitly.

**Context (rigor ask 10).** The brief ruled out a real router. Two views (list + detail) within one HTML page is the agreed shape. Wiring the back behavior is the open question. Three candidates:

  - **(a) Hide/show only, no BackButton.** User taps the in-page "← Back" button to return. Hardware back closes the webview entirely.
  - **(b) Hide/show + Telegram BackButton API.** When detail is shown, call `Telegram.WebApp.BackButton.show()` and register a click handler that returns to list. Android hardware back also pops to list (via the BackButton handler — Telegram routes hardware back to BackButton when shown). Second back closes the webview.
  - **(c) Hide/show + history.pushState.** Use the browser's history API. `popstate` event handles back. Works in any browser; Telegram's BackButton may not show.

**Decision.** (b). Mirrors Telegram's official Mini Apps pattern.

**Wiring** (binding for Phase 2):

  1. Initial state: list view visible, detail view hidden. `Telegram.WebApp.BackButton` is hidden (it isn't shown on the hub or the list — Telegram's default).
  2. User taps an item card → `enterDetail(itemId)`:
     - Fetch `/api/webapp/items/:id`.
     - On success: hide list container, show detail container, populate fields.
     - Call `Telegram.WebApp.BackButton.show()`.
     - Register the BackButton handler: `Telegram.WebApp.BackButton.onClick(() => exitDetail())`.
  3. User taps Telegram BackButton OR Android hardware back → fires the BackButton handler → `exitDetail()`:
     - Show list, hide detail.
     - Call `Telegram.WebApp.BackButton.offClick(handler)` AND `Telegram.WebApp.BackButton.hide()`.
     - Detail view's container is fully cleared (so a stale item never flashes on the next entry).
  4. Second back from list state → Telegram closes the webview natively (because BackButton is hidden). NO custom handler.

**Edge cases:**

  - User taps two items rapidly (race): the second `enterDetail` cancels the first's pending fetch via an `AbortController`. The detail view shows the second item's fields when the second fetch resolves. Documented; tested in Item 6.
  - Detail fetch fails (404 or network): the detail view shows an error message; BackButton is still shown so user can return. The error message renders via `textContent` (decision 6).
  - User opens a deep link with `?item=<id>` query (future, NOT v1.14.0): not supported. The page loads to the list view always. Documented.

**Why not (c) `history.pushState`.** Two reasons: (i) Telegram's BackButton is the user's primary expectation in a Mini App — users tap the platform's back, not the browser's. (ii) `history.pushState` may interact poorly with Telegram's webview lifecycle on some clients. (b) is the documented Telegram pattern.

**Why not (a).** Hardware back closing the webview when the user is in detail view is a footgun — a single accidental back lose the user's context. Telegram BackButton is the right intercept.

**Consequences.**

  - **Positive.** Matches platform expectations; one back from detail returns to list, one back from list closes the webview.
  - **Positive.** Pure DOM-level state; no router complexity.
  - **Negative.** A user's deep link to a specific item (`?item=2026-04-23-a1b2`) is not supported. Acceptable for v1.14.0; flagged in TODO.md for v1.14.1+.

### 12. CSP for the new page — inherited from existing `setHeaders` middleware; no per-file override

**Status:** Accepted — same CSP as v1.13.1's `index.html`, applied automatically.

**Context (rigor ask 12).** v1.13.1 fixed the CSP `frame-ancestors` directive; the policy lives in `src/webapp/server.ts`'s `express.static` `setHeaders` callback (lines 271-309). The callback fires for every served file — checks `filePath.endsWith('.html')` and applies the CSP only to HTML responses. Two candidates for v1.14.0:

  - **(a) Inherit.** New `public/webapp/organize/index.html` is served by the existing `app.use('/webapp', ...)` middleware. The `setHeaders` callback applies the existing CSP automatically. ZERO server-side change.
  - **(b) Per-file CSP.** `<meta http-equiv="Content-Security-Policy" content="...">` inside the `index.html`. The HTTP header still fires (`setHeaders` doesn't go away); the meta tag is a duplicate. Useless.

**Decision.** (a). Inherit.

**Verification of inheritance** (from `server.ts:271-309`):

```typescript
setHeaders(res: Response, filePath: string) {
  const isHtml = filePath.endsWith('.html');
  if (isHtml) {
    res.setHeader('Cache-Control', 'no-cache');
    res.setHeader(
      'Content-Security-Policy',
      "default-src 'self'; " +
      "script-src 'self' https://telegram.org; " +
      "style-src 'self' 'unsafe-inline'; " +
      "connect-src 'self'; " +
      "img-src 'self' data:; " +
      "frame-ancestors https://web.telegram.org https://*.telegram.org; " +
      "base-uri 'self'; " +
      "form-action 'self'",
    );
  } else {
    res.setHeader('Cache-Control', 'public, max-age=300');
  }
},
```

Both `public/webapp/organize/index.html` and `public/webapp/organize/app.js` are served by this middleware. The HTML file gets the CSP + `Cache-Control: no-cache`; the JS file gets `Cache-Control: public, max-age=300`. Phase 2 confirms by integration test (Item 6) — assert `CSP` header is on the organize page response, AND that `frame-ancestors` includes `https://web.telegram.org`.

**Consequences.**

  - **Positive.** Zero `server.ts` change.
  - **Positive.** Future Web App pages inherit the CSP automatically — security posture is by-construction.
  - **Negative.** A future page that needs a different CSP (e.g. embeds an external chart library) would have to break this pattern. Accepted; that's a v1.16.x+ problem and would need its own ADR.

### 13. Theme-param fallback — match v1.13.0's `--bg-color, #ffffff` pattern

**Status:** Accepted — explicit fallbacks per CSS variable.

**Context (rigor ask 13).** Telegram's theme params arrive via `Telegram.WebApp.themeParams.bg_color` etc. v1.13.0's `app.js` sets `--bg-color`, `--text-color`, `--hint-color`, `--button-bg`, `--button-text` as inline styles on `:root`. CSS uses `var(--bg-color, #ffffff)` with a fallback hex. When the SDK isn't loaded (browser direct access) or the theme is incomplete, the fallback hex applies.

**Decision.** Match v1.13.0's pattern verbatim. The organize page's `styles.css` uses these variables:

```css
:root {
  --bg-color: #ffffff;
  --text-color: #000000;
  --hint-color: #999999;
  --button-bg: #2481cc;
  --button-text: #ffffff;
  --link-color: #2481cc;
  --secondary-bg: #f4f4f5;
  --destructive: #ff3b30;
}

body {
  background: var(--bg-color);
  color: var(--text-color);
}
```

Naming convention: variables match the v1.13.0 naming (`--bg-color`, `--text-color` — NOT the longer `--tg-theme-bg-color` form). The Telegram SDK injects its own `--tg-theme-*` variables on `<html>` automatically; the page MAY use either, but using the short form (set explicitly by `app.js`) is one less SDK dependency. v1.13.0 chose the short form; v1.14.0 keeps consistency.

**Defaults justification.** The fallback hex matches Telegram's default light theme:
  - `#ffffff` background, `#000000` text, `#999999` hint, `#2481cc` button — these are Telegram's own light-theme defaults observable from any Telegram client without a custom theme.

**Dark-mode fallback.** When `themeParams.bg_color` arrives as a dark hex (e.g. `#212121`), the SDK's value overrides the `:root` default — the page renders dark automatically. NO `prefers-color-scheme` media query — Telegram's theme is the authoritative source; the OS theme is not. Documented.

**Theme-update events.** `Telegram.WebApp.onEvent('themeChanged', updateTheme)` listener re-applies the variables on the fly when the user changes their Telegram theme mid-session. v1.13.0's `app.js` does NOT subscribe to this event; v1.14.0's organize page DOES (small enhancement, no behavioral risk). The hub page (Item 4) ALSO subscribes for parity.

**Consequences.**

  - **Positive.** Theme-aware in light + dark + custom Telegram themes.
  - **Positive.** Browser-direct access (no SDK) gets a clean light-mode fallback.
  - **Negative.** None material.

### 14. Filter chip default — All types + Active; reset on each load (no sessionStorage)

**Status:** Accepted — defaults reset every load.

**Context (rigor ask 14).** Two candidates:

  - **(a) Reset every load.** Initial state is "All types + Active." User-tapped chips DO NOT persist across page loads.
  - **(b) Persist via sessionStorage.** User's last filter is remembered. On reload, restore.

**Decision.** (a). Justifications:

  1. **Predictability.** Telegram opens the Web App in a fresh webview each time the user taps the button — unlike a browser tab, there's no "back to where I was" expectation. Users open the app to see the current state; "active items" is the obvious default.
  2. **Simpler implementation.** No storage layer; no schema for the saved filter; no migration when the schema changes.
  3. **No surprise.** A user who set a filter to "abandoned + tasks" three days ago doesn't expect to see it next time — they expect to see what's actionable now.
  4. **The chip UI is fast.** Re-applying a filter is one tap. Users who want a non-default filter set it on each visit; the cost is single-tap.

**Default state:**

```
Type: [All*] [Tasks] [Events] [Goals]
Status: [Active*] [Done] [Abandoned] [All]
Tag: <none — chip absent in v1.14.0>
```

Where `*` denotes the selected chip. `Tag` filter is NOT included in v1.14.0's chip UI — the API supports `?tag=...` but no chip is rendered. Tag filtering is a v1.14.1+ UX decision (auto-completion, multi-select, etc. need design work that's out of scope here). The API field is documented in case advanced users hit it via curl; the UI does not surface it.

**Chip-state mechanics.** Two filter dimensions (type + status) are mutually independent. Selecting "Tasks" doesn't change the status filter. Selecting "Done" doesn't change the type filter. Each chip is a radio (one selected per dimension); the "All" chip is the dimension's reset.

**Consequences.**

  - **Positive.** Zero state to persist; zero migration risk.
  - **Positive.** Predictable default for first-time and returning users alike.
  - **Negative.** A user with a strong "I always want to see tasks only" preference re-selects on each visit. Acceptable; persistence is a v1.14.x decision.

**Why call `/echo` from the hub at all.** Two reasons: (i) belt-and-braces — confirm initData is valid before letting the user navigate to a feature page that will fetch authenticated data. (ii) early failure surfacing — if initData is corrupt, show a clear error on the hub instead of having the failure happen on the first feature page click.

**Why NOT a "this hub is read-only, skip auth" optimization.** Skipping the auth call would mean the user could navigate to `/webapp/organize/` with bad initData and only see the failure there. The hub's role as the entry point is the right place for the early-fail UX.

**Consequences.**

  - **Positive.** Minimal-diff edit to existing files; reviewers can see the conversion as a single PR-style change.
  - **Positive.** Forward-compatible structure (`<nav class="feature-grid">` accepts more `<a>` entries).
  - **Negative.** The v1.13.0 ping button feature is gone. The corresponding `bot.on('message:web_app_data', ...)` handler in `src/gateway/index.ts` is now orphaned. Phase 2 dev MUST remove the handler — keeping a no-op handler is dead code. Test `tests/unit/gateway.webAppData.test.ts` (per ADR 008 R12.1) is also removed. Documented as a Phase 2 cleanup task.

### 16. Rollback posture — feature-flagged by file existence; zero-coordination revert

**Status:** Accepted — each item is reversible by removing files OR reverting commits.

**Context (Item 16 in brief).** Standard practice for the factory is "ring-fenced changes — each iteration's revert is a single bisect-friendly window." Specific to v1.14.0:

**Decision.** The revert plan, item by item:

| Item | Revert action |
|---|---|
| 1. `GET /api/webapp/items` | Remove the route registration in `src/webapp/server.ts`. The route handler function is local; deleting its `app.get(...)` call is sufficient. |
| 2. `GET /api/webapp/items/:id` | Same — remove the second `app.get(...)` call. |
| 3. `public/webapp/organize/` | `rm -rf public/webapp/organize/`. Express returns 404 from the catchall (`server.ts:353`). |
| 4. Hub conversion | `git revert` the commit that touched `public/webapp/index.html` + `app.js`. The v1.13.0 ping skeleton is restored verbatim. |
| 5. `/webapp` slash command | NO change to revert (no code change in v1.14.0). |
| 6. `tests/integration/webapp.organize.test.ts` | `rm` the file. |
| 7. README subsection | `git revert` the README commit. |

**Bot handler cleanup (decision 15 consequence).** Removal of `bot.on('message:web_app_data', ...)` and `tests/unit/gateway.webAppData.test.ts` is part of Item 4's revert window — the `git revert` of the hub conversion commit restores both. Phase 2 commits MUST group these changes into the hub conversion commit (NOT split across two commits) so the revert atom is correct. Documented as a Phase 2 commit-discipline rule.

**No data migration to undo.** v1.14.0 makes no changes to `data/` directory contents, no SQLite schema migration, no audit-log row format change. The audit table continues to gain `webapp.auth_failure` rows; their format is unchanged from v1.13.1. Reverting v1.14.0 leaves the database in a consistent state.

**Pm2 restart required after revert.** The Express server's route table is built at boot; revert + restart shows the new (reverted) state. `pm2 restart jarvis` is the standard operator action.

**Risk-register entry (added to §13).** "v1.14.0 changes the public surface of `/webapp/`'s root URL (was ping page, becomes hub). User who has the v1.13.0 page cached MAY see a stale ping page until they hard-refresh. MITIGATION: `Cache-Control: no-cache` on HTML responses (server.ts:274) — cache is bypassed on every load. No real exposure."

**Consequences.**

  - **Positive.** Each item ring-fenced; revert is straightforward.
  - **Positive.** No data layer churn.
  - **Negative.** None material.

## §17 Detailed API specs (for Phase 2 — binding)

### `GET /api/webapp/items`

**Auth.** `Authorization: tma <initData>` header REQUIRED. Same path as `/api/webapp/echo`. On failure: 401 + existing `webapp.auth_failure` audit (no change).

**Query parameters** (all optional):
  - `type` — one of `'task' | 'event' | 'goal'`. Invalid value → 400 + `{error: 'INVALID_TYPE'}`.
  - `status` — one of `'active' | 'done' | 'abandoned' | 'all'`. Default: `'active'`. Invalid → 400 + `{error: 'INVALID_STATUS'}`. `'all'` is mapped to `undefined` before passing to `listItems`.
  - `tag` — string, 1-40 chars. Invalid (empty after trim, >40 chars) → 400 + `{error: 'INVALID_TAG'}`.

**Response 200:**
```typescript
interface ListItem {
  id: string;
  type: 'task' | 'event' | 'goal';
  status: 'active' | 'done' | 'abandoned';
  title: string;
  due: string | null;
  tags: string[];
  created: string;
  hasNotes: boolean;        // notesBody.trim().length > 0
  hasProgress: boolean;     // progressBody.trim().length > 0
  calendarEventId: string | null;
}

interface ItemsListResponse {
  items: ListItem[];
  total: number;
  serverTime: string;       // ISO-8601
}
```

**Projection function** (Phase 2 implements verbatim):
```typescript
function toListItem(item: OrganizeItem): ListItem {
  return {
    id: item.frontMatter.id,
    type: item.frontMatter.type,
    status: item.frontMatter.status,
    title: item.frontMatter.title,
    due: item.frontMatter.due,
    tags: item.frontMatter.tags,
    created: item.frontMatter.created,
    hasNotes: item.notesBody.trim().length > 0,
    hasProgress: item.progressBody.trim().length > 0,
    calendarEventId: item.frontMatter.calendarEventId,
  };
}
```

**Sort order:** by `created` DESC (newest first). The route handler sorts after projection — `items.sort((a, b) => b.created.localeCompare(a.created))`. Documented in JSDoc.

**Logging:** `serverLog.info({userId, total, filter, durationMs}, 'organize list served')`. Component `webapp.server` (existing).

### `GET /api/webapp/items/:id`

**Auth.** Same as above.

**Path parameter validation.** `:id` must match `^\d{4}-\d{2}-\d{2}-[a-z0-9]{4}$`. Validation runs BEFORE any filesystem touch:
```typescript
const ITEM_ID_RE = /^\d{4}-\d{2}-\d{2}-[a-z0-9]{4}$/;
if (!ITEM_ID_RE.test(req.params.id)) {
  serverLog.warn({userId, attemptedId: req.params.id, ip: req.ip}, 'invalid itemId format');
  res.status(400).json({error: 'INVALID_ITEM_ID'});
  return;
}
```

**Response 200:** per decision 3.

**Response 404 (item not found for this user):**
```typescript
{error: 'NOT_FOUND'}
```

`readItem` returns `null` when the file doesn't exist; the route handler converts to 404. A malformed item file (`ITEM_MALFORMED` thrown from `readItem`) → 500 + `{error: 'INTERNAL'}` with serverLog.error. Operator inspects the underlying file via SSH; user sees a generic error.

**Logging:** `serverLog.info({userId, itemId, durationMs}, 'organize detail served')` on success; `warn` on 404; `error` on parse-malformed.
